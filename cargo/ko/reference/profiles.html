The Cargo Book

프로파일

프로파일은 컴파일러 설정을 변경하는 방법을 제공하여 최적화 및 디버깅 심볼과 같은 항목에 영향을 줍니다.

Cargo에는 dev, release, test, bench 라는 4개의 내장 프로파일이 있습니다. 명령줄에서 프로파일을 지정하지 않으면 실행 중인 명령에 따라 프로파일이 자동으로 선택됩니다. 내장 프로파일 외에도 사용자 정의 프로파일을 지정할 수 있습니다.

프로파일 설정은 Cargo.toml 의 [profile] 테이블에서 변경할 수 있습니다. 명명된 각 프로파일 내에서 개별 설정은 다음과 같이 키/값 쌍으로 변경할 수 있습니다.

[profile.dev]
opt-level = 1               # Use slightly better optimizations.
overflow-checks = false     # Disable integer overflow checks.

Cargo는 워크스페이스 루트의 Cargo.toml 매니페스트에 있는 프로파일 설정만 확인합니다. 의존성에 정의된 프로파일 설정은 무시됩니다.

또한 프로파일은 config 정의에서 재정의할 수 있습니다. 설정 파일이나 환경 변수에 프로파일을 지정하면 Cargo.toml 의 설정보다 우선 적용됩니다.

프로파일 설정

다음은 프로파일에서 제어할 수 있는 설정 목록입니다.

opt-level

opt-level 설정은 최적화 수준을 제어하는 -C opt-level 플래그 를 제어합니다. 최적화 수준이 높을수록 컴파일 시간은 길어지지만 더 빠른 런타임 코드를 생성할 수 있습니다. 높은 수준의 최적화는 컴파일된 코드를 변경하고 재배치하여 디버거와 함께 사용하기 어렵게 만들 수도 있습니다.

사용 가능한 옵션은 다음과 같습니다.

  • 0: 최적화 없음
  • 1: 기본 최적화
  • 2: 일부 최적화
  • 3: 모든 최적화
  • "s": 바이너리 크기 최적화
  • "z": 바이너리 크기 최적화 및 루프 벡터화 비활성화

프로젝트에 적합한 균형을 찾기 위해 다양한 수준을 실험해보는 것이 좋습니다. 레벨 3 이 2 보다 느리거나, "s" 와 "z" 레벨이 반드시 더 작지 않은 것과 같은 예상치 못한 결과가 나올 수 있습니다. 최신 버전의 rustc 가 최적화 동작을 변경함에 따라 시간이 지남에 따라 설정을 재평가하고 싶을 수도 있습니다.

더 발전된 최적화 기술에 대해서는 프로파일 기반 최적화(Profile Guided Optimization) 를 참조하세요.

debug

debug 설정은 컴파일된 바이너리에 포함되는 디버그 정보의 양을 제어하는 -C debuginfo 플래그 를 제어합니다.

사용 가능한 옵션은 다음과 같습니다.

  • 0, false, 또는 "none": 디버그 정보 없음, release 의 기본값
  • "line-directives-only": 라인 정보 지시문만 포함. nvptx* 타겟의 경우 프로파일링 을 활성화합니다. 다른 유스케이스의 경우 line-tables-only 가 더 나은 호환성을 제공하는 선택입니다.
  • "line-tables-only": 라인 테이블만 포함. 파일 이름/라인 번호 정보를 포함한 백트레이스를 위한 최소한의 디버그 정보만 생성하며, 변수나 함수 매개변수 정보 등은 포함하지 않습니다.
  • 1 또는 "limited": 타입이나 변수 수준의 정보가 없는 디버그 정보. line-tables-only 보다 더 자세한 모듈 수준 정보를 생성합니다.
  • 2, true, 또는 "full": 모든 디버그 정보 포함, dev 의 기본값

각 옵션이 수행하는 작업에 대한 자세한 내용은 rustc 의 debuginfo 문서를 참조하세요.

필요에 따라 split-debuginfo 옵션도 구성할 수 있습니다.

MSRV: none, limited, full, line-directives-only, line-tables-only 에는 1.71 버전이 필요합니다.

split-debuginfo

split-debuginfo 설정은 생성된 디버그 정보를 실행 파일 자체에 배치할지 아니면 인접하게 배치할지를 제어하는 -C split-debuginfo 플래그 를 제어합니다.

이 옵션은 문자열이며 허용되는 값은 컴파일러가 허용하는 값 과 동일합니다. 이 옵션의 기본값은 디버그 정보가 활성화된 프로파일의 경우 macOS에서 unpacked 입니다. 그렇지 않은 경우 이 옵션의 기본값은 rustc 문서 에 명시되어 있으며 플랫폼에 따라 다릅니다. 일부 옵션은 나이틀리 채널 에서만 사용할 수 있습니다. 더 많은 테스트가 수행되고 DWARF 지원이 안정화되면 향후 Cargo 기본값이 변경될 수 있습니다.

Cargo와 rustc는 이 옵션에 대해 서로 다른 기본값을 가질 수 있음에 유의하세요. 이 옵션은 Cargo가 다양한 플래그 조합을 실험하여 더 나은 디버깅 및 개발자 경험을 제공할 수 있도록 하기 위해 존재합니다.

strip

strip 옵션은 바이너리에서 심볼이나 디버그 정보를 제거하도록 rustc에 지시하는 -C strip 플래그 를 제어합니다. 다음과 같이 활성화할 수 있습니다.

[package]
# ...

[profile.release]
strip = "debuginfo"

strip 에 가능한 문자열 값은 "none", "debuginfo", "symbols" 입니다. 기본값은 "none" 입니다.

이 옵션은 불리언 값인 true 또는 false 로도 구성할 수 있습니다. strip = true 는 strip = "symbols" 와 동일합니다. strip = false 는 strip = "none" 과 동일하며 strip 을 완전히 비활성화합니다.

debug-assertions

debug-assertions 설정은 cfg(debug_assertions) 조건부 컴파일 을 켜거나 끄는 -C debug-assertions 플래그 를 제어합니다. 디버그 어설션은 디버그/개발 빌드에서만 사용할 수 있는 런타임 유효성 검사를 포함하기 위한 것입니다. 이는 릴리스 빌드에서는 너무 비용이 많이 들거나 바람직하지 않은 작업일 수 있습니다. 디버그 어설션은 표준 라이브러리의 debug_assert! 매크로 를 활성화합니다.

사용 가능한 옵션은 다음과 같습니다.

  • true: 활성화됨
  • false: 비활성화됨

overflow-checks

overflow-checks 설정은 런타임 정수 오버플로 동작을 제어하는 -C overflow-checks 플래그 를 제어합니다. 오버플로 검사가 활성화되면 오버플로 발생 시 패닉이 발생합니다.

사용 가능한 옵션은 다음과 같습니다.

  • true: 활성화됨
  • false: 비활성화됨

lto

lto 설정은 LLVM의 링크 타임 최적화(Link Time Optimization) 를 제어하는 rustc 의 -C lto, -C linker-plugin-lto, -C embed-bitcode 옵션을 제어합니다. LTO는 전체 프로그램 분석을 사용하여 더 최적화된 코드를 생성할 수 있지만, 링크 시간이 더 길어집니다.

사용 가능한 옵션은 다음과 같습니다.

  • true 또는 "fat": 의존성 그래프 내의 모든 크레이트에 대해 최적화를 시도하는 “fat” LTO를 수행합니다.
  • "thin": “thin” LTO 를 수행합니다. 이는 “fat“과 유사하지만, 실행 시간이 상당히 단축되면서도 “fat“과 유사한 성능 이득을 얻을 수 있습니다.
  • false: 로컬 크레이트의 codegen units 전체에 대해서만 “thin” LTO를 수행하는 “thin local LTO“를 수행합니다. codegen units가 1이거나 opt-level 이 0인 경우 LTO는 수행되지 않습니다.
  • "off": LTO를 비활성화합니다.

언어 간 LTO에 관심이 있다면 linker-plugin-lto 장 을 참조하세요. 이는 아직 Cargo에서 기본적으로 지원되지는 않지만, RUSTFLAGS 를 통해 수행할 수 있습니다.

panic

panic 설정은 어떤 패닉 전략을 사용할지 제어하는 -C panic 플래그 를 제어합니다.

사용 가능한 옵션은 다음과 같습니다.

  • "unwind": 패닉 발생 시 스택을 언와인드(unwind)합니다.
  • "abort": 패닉 발생 시 프로세스를 즉시 종료(abort)합니다.

"unwind" 로 설정된 경우, 실제 값은 대상 플랫폼의 기본값에 따라 달라집니다. 예를 들어, NVPTX 플랫폼은 언와인딩을 지원하지 않으므로 항상 "abort" 를 사용합니다.

테스트, 벤치마크, 빌드 스크립트 및 절차적 매크로는 panic 설정을 무시합니다. rustc 테스트 하네스는 현재 unwind 동작이 필요합니다. abort 동작을 활성화하는 panic-abort-tests 불안정 플래그를 참조하세요.

또한, abort 전략을 사용하면서 테스트를 빌드할 때는 모든 의존성도 강제로 unwind 전략으로 빌드됩니다.

incremental

incremental 설정은 증분 컴파일(incremental compilation) 활성화 여부를 제어하는 -C incremental 플래그 를 제어합니다. 증분 컴파일을 사용하면 rustc 가 디스크에 추가 정보를 저장하고 크레이트를 다시 컴파일할 때 이를 재사용하여 재컴파일 시간을 단축합니다. 이 추가 정보는 target 디렉토리에 저장됩니다.

사용 가능한 옵션은 다음과 같습니다.

  • true: 활성화됨
  • false: 비활성화됨

증분 컴파일은 워크스페이스 구성원 및 “경로(path)” 의존성에 대해서만 사용됩니다.

incremental 값은 CARGO_INCREMENTAL 환경 변수 나 build.incremental 설정 변수를 통해 전역적으로 재정의될 수 있습니다.

codegen-units

codegen-units 설정은 크레이트가 몇 개의 “코드 생성 단위(code generation units)“로 나뉠지를 제어하는 -C codegen-units 플래그 를 제어합니다. 코드 생성 단위가 많을수록 크레이트의 더 많은 부분을 병렬로 처리할 수 있어 컴파일 시간을 줄일 수 있지만, 생성된 코드의 속도가 느려질 수 있습니다.

이 옵션은 0보다 큰 정수 값을 가집니다.

기본값은 증분(#incremental) 빌드의 경우 256이고, 비증분 빌드의 경우 16입니다.

rpath

rpath 설정은 rpath 활성화 여부를 제어하는 -C rpath 플래그 를 제어합니다.

기본 프로파일

dev

dev 프로파일은 일반적인 개발 및 디버깅에 사용됩니다. cargo build 와 같은 빌드 명령어의 기본값이며, cargo install --debug 에 사용됩니다.

dev 프로파일의 기본 설정은 다음과 같습니다.

[profile.dev]
opt-level = 0
debug = true
split-debuginfo = '...'  # 플랫폼별 상이
strip = "none"
debug-assertions = true
overflow-checks = true
lto = false
panic = 'unwind'
incremental = true
codegen-units = 256
rpath = false

release

release 프로파일은 릴리스 및 프로덕션에서 사용되는 최적화된 결과물을 위한 것입니다. --release 플래그를 사용할 때 사용되며, cargo install 의 기본값입니다.

release 프로파일의 기본 설정은 다음과 같습니다.

[profile.release]
opt-level = 3
debug = false
split-debuginfo = '...'  # 플랫폼별 상이
strip = "none"
debug-assertions = false
overflow-checks = false
lto = false
panic = 'unwind'
incremental = false
codegen-units = 16
rpath = false

test

test 프로파일은 cargo test 에서 사용되는 기본 프로파일입니다. test 프로파일은 dev 프로파일의 설정을 상속합니다.

bench

bench 프로파일은 cargo bench 에서 사용되는 기본 프로파일입니다. bench 프로파일은 release 프로파일의 설정을 상속합니다.

빌드 의존성

빠른 컴파일을 위해 모든 프로파일은 기본적으로 빌드 의존성(빌드 스크립트, 절차적 매크로 및 그 의존성)을 최적화하지 않으며, 빌드 의존성이 런타임 의존성으로 사용되지 않는 한 디버그 정보를 계산하지 않습니다. 빌드 재정의에 대한 기본 설정은 다음과 같습니다.

[profile.dev.build-override]
opt-level = 0
codegen-units = 256
debug = false # when possible

[profile.release.build-override]
opt-level = 0
codegen-units = 256

하지만 빌드 의존성을 실행하는 동안 오류가 발생하는 경우, 전체 디버그 정보를 활성화하면 백트레이스 정보가 개선되어 디버깅에 도움이 됩니다.

debug = true

그 외의 경우 빌드 의존성은 프로파일 선택 에 설명된 대로 사용 중인 활성 프로파일의 설정을 상속합니다.

사용자 정의 프로파일

내장 프로파일 외에도 추가적인 사용자 정의 프로파일을 정의할 수 있습니다. 이는 여러 워크플로나 빌드 모드를 설정할 때 유용할 수 있습니다. 사용자 정의 프로파일을 정의할 때는 설정이 지정되지 않았을 때 어떤 프로파일로부터 설정을 상속받을지 inherits 키를 사용하여 명시해야 합니다.

예를 들어, 일반적인 릴리스 빌드와 LTO 최적화가 적용된 릴리스 빌드를 비교하고 싶다면 Cargo.toml 에 다음과 같이 지정할 수 있습니다.

[profile.release-lto]
inherits = "release"
lto = true

그런 다음 --profile 플래그를 사용하여 이 사용자 정의 프로파일을 선택할 수 있습니다.

cargo build --profile release-lto

각 프로파일에 대한 출력은 target 디렉토리 내의 프로파일 이름과 동일한 디렉토리에 배치됩니다. 위의 예시에서 출력은 target/release-lto 디렉토리에 저장됩니다.

프로파일 선택

사용되는 프로파일은 명령어, --release 또는 --profile 과 같은 명령줄 플래그, 그리고 (의존성 재정의(#overrides) 의 경우) 패키지에 따라 달라집니다. 아무것도 지정되지 않았을 때의 기본 프로파일은 다음과 같습니다.

명령어기본 프로파일
cargo run, cargo build,
cargo check, cargo rustc		dev 프로파일
cargo testtest 프로파일
cargo benchbench 프로파일
cargo installrelease 프로파일

--profile=NAME 옵션을 사용하여 다른 프로파일로 전환할 수 있으며, 이 경우 지정된 프로파일이 사용됩니다. --release 플래그는 --profile=release 와 동일합니다.

선택된 프로파일은 라이브러리, 바이너리, 예제, 테스트, 벤치마크 를 포함한 모든 Cargo 타겟에 적용됩니다.

특정 패키지에 대한 프로파일은 아래에 설명된 재정의(overrides) 를 통해 지정할 수 있습니다.

재정의(Overrides)

특정 패키지 및 빌드 시 크레이트에 대해 프로파일 설정을 재정의할 수 있습니다. 특정 패키지의 설정을 재정의하려면 package 테이블을 사용하여 해당 패키지의 설정을 변경하세요.

# The `foo` package will use the -Copt-level=3 flag.
[profile.dev.package.foo]
opt-level = 3

패키지 이름은 실제로는 패키지 ID 사양(Package ID Spec) 이므로, [profile.dev.package."foo:2.1.0"] 과 같은 구문을 사용하여 패키지의 특정 버전을 대상으로 지정할 수 있습니다.

모든 의존성(워크스페이스 구성원 제외)에 대한 설정을 재정의하려면 "*" 패키지 이름을 사용하세요.

# 의존성에 대한 기본값 설정
[profile.dev.package."*"]
opt-level = 2

빌드 스크립트, 절차적 매크로 및 그 의존성에 대한 설정을 재정의하려면 build-override 테이블을 사용하세요.

# Set the settings for build scripts and proc-macros.
[profile.dev.build-override]
opt-level = 3

참고: 의존성이 일반 의존성과 빌드 의존성 모두에 해당하는 경우, --target 이 지정되지 않았다면 Cargo는 이를 한 번만 빌드하려고 시도합니다. build-override 를 사용하면 의존성을 두 번 빌드해야 할 수도 있습니다 (한 번은 일반 의존성으로, 또 한 번은 재정의된 빌드 설정으로). 이로 인해 초기 빌드 시간이 늘어날 수 있습니다.

어떤 값이 사용될지에 대한 우선순위는 다음 순서에 따릅니다 (가장 먼저 일치하는 항목 적용).

  1. [profile.dev.package.name] — 지정된 이름의 패키지
  2. [profile.dev.package."*"] — 워크스페이스 구성원이 아닌 모든 패키지
  3. [profile.dev.build-override] — 빌드 스크립트, 절차적 매크로 및 그 의존성에만 해당
  4. [profile.dev] — Cargo.toml 의 설정
  5. Cargo에 내장된 기본값

재정의 설정에서는 panic, lto, rpath 설정을 지정할 수 없습니다.

재정의와 제네릭

제네릭 코드가 인스턴스화되는 위치가 해당 제네릭 코드에 사용되는 최적화 설정에 영향을 미칩니다. 이로 인해 프로파일 재정의를 사용하여 특정 크레이트의 최적화 레벨을 변경할 때 미묘한 상호 작용이 발생할 수 있습니다. 제네릭 함수를 정의하는 의존성의 최적화 레벨을 높이려고 시도하더라도, 해당 제네릭 함수가 로컬 크레이트에서 사용될 때는 최적화되지 않을 수 있습니다. 이는 코드가 인스턴스화되는 크레이트에서 생성될 수 있으며, 따라서 해당 크레이트의 최적화 설정을 사용할 수 있기 때문입니다.

예를 들어, nalgebra 는 제네릭 매개변수를 많이 사용하는 벡터와 행렬을 정의하는 라이브러리입니다. 로컬 코드에서 Vector4<f64> 와 같은 구체적인 nalgebra 타입을 정의하고 그 메서드들을 사용하는 경우, 해당 nalgebra 코드는 사용자의 크레이트 내에서 인스턴스화되고 빌드됩니다. 따라서 프로파일 재정의를 통해 nalgebra 의 최적화 레벨을 높이려고 시도하더라도, 실제 성능 향상으로 이어지지 않을 수 있습니다.

문제를 더 복잡하게 만드는 것은, rustc 가 크레이트 간에 단일화된(monomorphized) 제네릭을 공유하려는 최적화 기법을 가지고 있다는 점입니다. opt-level이 2 또는 3인 경우, 크레이트는 다른 크레이트의 단일화된 제네릭을 사용하지 않으며, 로컬에서 정의된 단일화된 아이템을 다른 크레이트와 공유하기 위해 내보내지도 않습니다. 개발 시 의존성 최적화를 실험할 때는 opt-level 1을 시도해 보는 것을 고려해 보세요. 이는 어느 정도의 최적화를 적용하면서도 단일화된 아이템의 공유를 허용합니다.
