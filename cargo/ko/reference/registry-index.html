The Cargo Book

인덱스 형식

다음은 인덱스의 형식을 정의합니다. 새로운 기능이 가끔 추가되는데, 이는 해당 기능을 도입한 Cargo 버전부터 이해할 수 있습니다. 오래된 버전의 Cargo는 새로운 기능을 사용하는 패키지를 사용하지 못할 수도 있습니다. 하지만 오래된 패키지에 대한 형식은 변경되지 않아야 하며, 따라서 오래된 버전의 Cargo도 이를 사용할 수 있어야 합니다.

인덱스 설정

인덱스의 루트에는 Cargo가 레지스트리에 접근할 때 사용하는 JSON 정보가 포함된 config.json 파일이 있습니다. 다음은 crates.io 설정 파일의 예시입니다.

{
    "dl": "https://crates.io/api/v1/crates",
    "api": "https://crates.io"
}

키는 다음과 같습니다.

  • dl: 인덱스에 나열된 크레이트를 다운로드하기 위한 URL입니다. 이 값은 대응하는 값으로 대체될 다음과 같은 마커들을 가질 수 있습니다.

    • {crate}: 크레이트의 이름입니다.
    • {version}: 크레이트 버전입니다.
    • {prefix}: 크레이트 이름으로부터 계산된 디렉토리 접두사입니다. 예를 들어, 이름이 cargo 인 크레이트는 ca/rg 라는 접두사를 가집니다. 자세한 내용은 아래를 참조하세요.
    • {lowerprefix}: {prefix} 의 소문자 형태입니다.
    • {sha256-checksum}: 크레이트의 sha256 체크섬입니다. 마커가 하나도 없는 경우, 값 끝에 /{crate}/{version}/download 가 추가됩니다.

  • api: 웹 API의 기본 URL입니다. 이 키는 선택 사항이지만, 지정하지 않으면 cargo publish와 같은 명령어가 작동하지 않습니다. 웹 API는 아래에 설명되어 있습니다.

  • auth-required: API 요청, 크레이트 다운로드, sparse 인덱스 업데이트를 포함한 모든 작업에 인증이 필요한 프라이빗 레지스트리인지 여부를 나타냅니다.

다운로드 엔드포인트

다운로드 엔드포인트는 요청된 패키지의 .crate 파일을 전송해야 합니다. Cargo는 https, http, file URL 및 HTTP 리다이렉트, HTTP1, HTTP2를 지원합니다. TLS 지원에 대한 구체적인 사항은 Cargo가 실행 중인 플랫폼, Cargo 버전 및 컴파일 방식에 따라 달라집니다.

config.json 에 auth-required: true 가 설정된 경우, http(s) 다운로드 요청에 Authorization 헤더가 포함됩니다.

인덱스 파일

인덱스 저장소의 나머지 부분에는 각 패키지당 하나의 파일이 포함되어 있으며, 파일 이름은 소문자로 된 패키지 이름입니다. 패키지의 각 버전은 파일 내에서 별도의 줄을 차지합니다. 파일들은 계층 구조의 디렉토리에 정리됩니다.

  • 이름이 1글자인 패키지들은 1 이라는 이름의 디렉토리에 위치합니다.
  • 이름이 2글자인 패키지들은 2 라는 이름의 디렉토리에 위치합니다.
  • 이름이 3글자인 패키지들은 3/{첫-글자} 디렉토리에 위치하며, 여기서 {첫-글자} 는 패키지 이름의 첫 번째 글자입니다.
  • 그 외 모든 패키지는 {앞-두-글자}/{다음-두-글자} 라는 이름의 디렉토리에 저장됩니다. 상위 디렉토리는 패키지 이름의 처음 두 글자이고, 다음 하위 디렉토리는 패키지 이름의 세 번째와 네 번째 글자입니다. 예를 들어, cargo 는 ca/rg/cargo 라는 파일에 저장됩니다.

참고: 인덱스 파일 이름은 소문자이지만, Cargo.toml 및 인덱스 JSON 데이터의 패키지 이름 필드는 대소문자를 구분하며 대문자와 소문자를 모두 포함할 수 있습니다.

위의 디렉토리 이름은 소문자로 변환된 패키지 이름을 기반으로 계산되며, 마커 {lowerprefix} 로 표현됩니다. 대소문자 변환 없이 원래 패키지 이름을 사용하는 경우, 결과 디렉토리 이름은 마커 {prefix} 로 표현됩니다. 예를 들어, MyCrate 패키지는 My/Cr 이라는 {prefix} 와 my/cr 이라는 {lowerprefix} 를 가집니다. 일반적으로 {lowerprefix} 보다 {prefix} 를 사용하는 것이 권장되지만, 각 선택에는 장단점이 있습니다. 대소문자를 구분하지 않는 파일 시스템에서 {prefix} 를 사용하면 (해롭지는 않지만 우아하지 않은) 디렉토리 별칭(aliasing)이 발생합니다. 예를 들어, crate 와 CrateTwo 는 각각 cr/at 및 Cr/at 라는 {prefix} 값을 가집니다. 이들은 유닉스 머신에서는 서로 다르지만 윈도우에서는 동일한 디렉토리로 별칭 처리됩니다. 정규화된 대소문자를 사용하는 디렉토리를 쓰면 별칭 문제를 피할 수 있지만, 대소문자를 구분하는 파일 시스템에서는 {prefix}/{lowerprefix} 기능이 없는 이전 버전의 Cargo를 지원하기가 더 어렵습니다. 예를 들어, nginx 리라이트 규칙은 {prefix} 를 쉽게 생성할 수 있지만 소문자 변환을 수행하여 {lowerprefix} 를 생성할 수는 없습니다.

이름 제한

레지스트리는 인덱스에 추가되는 패키지 이름에 대한 제한을 두는 것을 고려해야 합니다. Cargo 자체는 모든 영문자 및 숫자, -, _ 문자를 포함한 이름을 허용합니다. crates.io 는 다음과 같은 자체 제한 사항을 두고 있습니다.

  • ASCII 문자만 허용합니다.
  • 영문자 및 숫자, -, _ 문자만 허용합니다.
  • 첫 번째 문자는 반드시 알파벳이어야 합니다.
  • 대소문자를 구분하지 않는 충돌 감지를 수행합니다.
  • - 와 _ 의 차이로 인한 혼동을 방지합니다.
  • 특정 길이(최대 64자) 미만이어야 합니다.
  • “nul“과 같은 윈도우 특수 파일 이름 등의 예약된 이름은 거부합니다.

레지스트리는 이와 유사한 제한 사항을 적용하는 것을 고려해야 하며, IDN 호모그래프 공격(IDN homograph attacks) 및 UTR36과 UTS39 에서 다루는 다른 보안 문제들도 고려해야 합니다.

버전 유일성

인덱스는 각 패키지에 대해 각 버전이 단 한 번만 나타나도록 보장 해야 합니다. 여기에는 SemVer 빌드 메타데이터를 무시하는 것도 포함됩니다. 예를 들어, 인덱스는 버전 1.0.7 과 1.0.7+extra 를 가진 두 개의 항목을 포함해서는 안 됩니다.

JSON 스키마

패키지 파일의 각 줄에는 배포된 패키지 버전을 설명하는 JSON 객체가 포함되어 있습니다. 다음은 항목의 형식을 설명하는 주석이 포함된 예시입니다.

{
    // 패키지의 이름입니다.
    // 영문자 및 숫자, `-`, `_` 문자만 포함해야 합니다.
    "name": "foo",
    // 이 행이 설명하는 패키지의 버전입니다.
    // https://semver.org/의 유의적 버전(Semantic Versioning) 2.0.0 사양에 따른 유효한 버전 번호여야 합니다.
    "vers": "0.1.0",
    // 패키지의 직접적인 의존성 배열입니다.
    "deps": [
        {
            // 의존성의 이름입니다.
            // 만약 의존성이 원래 패키지 이름에서 변경되었다면, 이것이 새 이름이 됩니다.
            // 원래 패키지 이름은 `package` 필드에 저장됩니다.
            "name": "rand",
            // 이 의존성에 대한 SemVer 요구 사항입니다.
            // 이는 https://doc.rust-lang.org/cargo/reference/specifying-dependencies.html 에 정의된
            // 유효한 버전 요구 사항이어야 합니다.
            "req": "^0.6",
            // Array of features (as strings) enabled for this dependency.
            // Since Cargo 1.84, defaults to `[]` if not specified.
            "features": ["i128_support"],
            // 이것이 선택적 의존성인지 여부를 나타내는 불리언 값입니다.
            // Cargo 1.84부터는 지정되지 않은 경우 기본값이 `false` 입니다.
            "optional": false,
            // 기본 기능(default features)이 활성화되었는지 여부를 나타내는 불리언 값입니다.
            // Cargo 1.84부터는 지정되지 않은 경우 기본값이 `true` 입니다.
            "default_features": true,
            // 의존성에 대한 타겟 플랫폼입니다.
            // 지정되지 않았거나 `null` 인 경우, 타겟 의존성이 아닙니다.
            // 그 외의 경우, "cfg(windows)"와 같은 문자열입니다.
            "target": null,
            // 의존성 종류입니다.
            // "dev", "build", 혹은 "normal"입니다.
            // 지정되지 않았거나 `null` 인 경우 기본값은 "normal"입니다.
            "kind": "normal",
            // 이 의존성의 출처인 레지스트리의 인덱스 URL 문자열입니다.
            // 지정되지 않았거나 `null` 인 경우, 의존성이 현재 레지스트리에 있는 것으로 간주됩니다.
            "registry": null,
            // If the dependency is renamed, this is a string of the actual
            // package name. If not specified or `null`, this dependency is not
            // renamed.
            "package": null,
        }
    ],
    // `.crate` 파일의 SHA256 체크섬입니다.
    "cksum": "d867001db0e2b6e0496f9fac96930e2d42233ecd3ca0413e0753d4c7695d289c",
    // Set of features defined for the package.
    // Each feature maps to an array of features or dependencies it enables.
    // Since Cargo 1.84, defaults to `{}` if not specified.
    "features": {
        "extras": ["rand/simd_support"]
    },
    // 이 버전이 철회(yank)되었는지 여부를 나타내는 불리언 값입니다.
    "yanked": false,
    // 패키지 매니페스트의 `links` 문자열 값이거나, 지정되지 않은 경우 null입니다.
    // 이 필드는 선택 사항이며 기본값은 null입니다.
    "links": null,
    // 이 항목의 스키마 버전을 나타내는 부호 없는 32비트 정수 값입니다.
//
// 지정되지 않은 경우 기본값인 1로 해석해야 합니다.
//
// Cargo(버전 1.51부터)는 인식하지 못하는 버전을 무시합니다. 이는 인덱스 항목에 변경 사항을 안전하게 도입하고,
// 이전 버전의 cargo가 이해하지 못하는 새로운 항목을 무시할 수 있도록 하는 방법을 제공합니다. 1.51보다 오래된
// 버전은 이 필드를 무시하므로 인덱스 항목의 의미를 잘못 해석할 수 있습니다.
//
// 현재 값은 다음과 같습니다:
//
// * 1: 여기에 문서화된 스키마이며, 최신 추가 사항은 포함하지 않습니다.
//      이 값은 Rust 버전 1.51 이상에서 존중됩니다.
// * 2: `features2` 필드가 추가되었습니다.
//      이 값은 Rust 버전 1.60 이상에서 존중됩니다.
"v": 2,
    // 이 선택적 필드는 새로운 확장 구문을 사용하는 기능(feature)들을 포함합니다.
    // 구체적으로는 네임스페이스 기능(`dep:`) 및 약한 의존성(`pkg?/feat`)입니다.
    //
    // 이 필드가 `features` 와 분리된 이유는 1.19보다 오래된 버전의 경우 `Cargo.lock` 파일이 있더라도
    // 새 구문을 파싱하지 못해 로드에 실패하기 때문입니다.
    //
    // Cargo는 여기에 나열된 모든 값을 "features" 필드와 병합합니다.
    //
    // 이 필드가 포함된 경우, "v" 필드는 최소 2로 설정되어야 합니다.
    //
    // 레지스트리가 확장 기능 구문을 위해 반드시 이 필드를 사용해야 하는 것은 아니며, "features" 필드에
    // 이를 포함하는 것도 허용됩니다. 이 필드를 사용하는 것은 오직 1.19 이전 버전의 cargo를 지원하고자 하는
    // 경우에만 필요하며, 실제로는 crates.io만이 해당됩니다 (오래된 버전들은 다른 레지스트리를 지원하지 않기 때문입니다).
    "features2": {
        "serde": ["dep:serde", "chrono?/serde"]
    }
    // 최소 지원 Rust 버전 (선택 사항)
    // 연산자(예: `=`)가 없는 유효한 버전 요구 사항이어야 합니다.
    "rust_version": "1.60",
    // The publish time of this package version (optional).
    //
    // The format is a subset of ISO8601:
    // - `yyyy-mm-ddThh:mm:ssZ`
    // - no fractional seconds
    // - always `Z` for UTC timezone, no timezone offsets supported
    // - fields are 0-padded
    //
    // Example: 2025-11-12T19:30:12Z
    //
    // This should be the original publish time and not changed on any status changes,
    // like `yanked`.
    "pubtime": "2025-11-12T19:30:12Z"
}

JSON 객체는 추가된 이후에 수정되어서는 안 됩니다. 다만 yanked 필드는 언제든지 값이 변경될 수 있습니다.

참고: 인덱스 JSON 형식은 배포 API(Publish API) 및 cargo metadata 의 JSON 형식과 미세한 차이가 있습니다. 만약 이들 중 하나를 소스로 사용하여 인덱스 항목을 생성하려는 경우, 문서상에 기술된 차이점을 주의 깊게 확인하는 것이 좋습니다.

배포 API 와의 차이점은 다음과 같습니다.

  • deps
    • name — 의존성이 Cargo.toml 에서 이름 변경(renamed)된 경우, 배포 API는 원래 패키지 이름을 name 필드에, 별칭 이름을 explicit_name_in_toml 필드에 넣습니다. 인덱스는 별칭 이름을 name 필드에, 원래 패키지 이름을 package 필드에 넣습니다.
    • req — 배포 API 필드 이름은 version_req 입니다.
  • cksum — 배포 API는 체크섬을 지정하지 않으며, 인덱스에 추가하기 전에 레지스트리에서 직접 계산해야 합니다.
  • features — 일부 기능은 features2 필드에 위치할 수 있습니다. 참고: 이는 crates.io를 위한 레거시 요구 사항일 뿐이며, 다른 레지스트리는 기능 맵을 수정할 필요가 없습니다. v 필드는 features2 필드의 존재 여부를 나타냅니다.
  • 배포 API는 description 및 readme 와 같이 인덱스에는 나타나지 않는 여러 다른 필드를 포함합니다. 이는 레지스트리가 .crate 파일을 압축 해제하고 파싱할 필요 없이 웹사이트에 표시할 크레이트 메타데이터를 쉽게 얻을 수 있도록 하기 위함입니다. 이 추가 정보는 일반적으로 레지스트리 서버의 데이터베이스에 추가됩니다.
  • rust_version 이 여기에 포함되어 있더라도, crates.io 는 이 필드를 무시하고 대신 .crate 파일에 포함된 Cargo.toml 에서 이를 읽습니다.

cargo metadata 와의 차이점은 다음과 같습니다.

  • vers — cargo metadata 필드 이름은 version 입니다.
  • deps
    • name — 의존성이 Cargo.toml 에서 이름 변경(renamed)된 경우, cargo metadata 는 원래 패키지 이름을 name 필드에, 별칭 이름을 rename 필드에 넣습니다. 인덱스는 별칭 이름을 name 필드에, 원래 패키지 이름을 package 필드에 넣습니다.
    • default_features — cargo metadata 필드 이름은 uses_default_features 입니다.
    • registry — cargo metadata 는 의존성이 crates.io 에서 왔음을 나타내기 위해 null 값을 사용합니다. 인덱스는 의존성이 인덱스와 동일한 레지스트리에서 왔음을 나타내기 위해 null 값을 사용합니다. 인덱스 항목을 생성할 때 crates.io 가 아닌 레지스트리는 null 값을 https://github.com/rust-lang/crates.io-index 로 변환해야 하며, 현재 인덱스와 일치하는 URL은 null 로 변환해야 합니다.
    • cargo metadata 는 source 및 path 와 같은 몇 가지 추가 필드를 포함합니다.
  • 인덱스는 yanked, cksum, v 와 같은 추가 필드를 포함합니다.

인덱스 프로토콜

Cargo는 git 과 sparse 두 가지 원격 레지스트리 프로토콜을 지원합니다. git 프로토콜은 인덱스 파일들을 git 저장소에 저장하며, sparse 프로토콜은 HTTP를 통해 개별 파일들을 가져옵니다.

Git 프로토콜

git 프로토콜은 인덱스 URL에 별도의 프로토콜 접두사가 없습니다. 예를 들어 crates.io 의 git 인덱스 URL은 https://github.com/rust-lang/crates.io-index 입니다.

Cargo는 업데이트를 효율적으로 증분해서 가져올 수 있도록 디스크에 git 저장소를 캐시합니다.

Sparse 프로토콜

sparse 프로토콜은 레지스트리 URL에 sparse+ 프로토콜 접두사를 사용합니다. 예를 들어 crates.io의 sparse 인덱스 URL은 sparse+https://index.crates.io/ 입니다.

sparse 프로토콜은 각 인덱스 파일을 개별 HTTP 요청으로 다운로드합니다. 이로 인해 수많은 작은 HTTP 요청이 발생하므로, 파이프라이닝과 HTTP/2를 지원하는 서버를 사용하면 성능이 크게 향상됩니다.

Sparse 인증

Cargo는 다른 파일을 가져오기 전에 config.json 파일을 먼저 가져오려고 시도합니다. 서버가 HTTP 401로 응답하면, Cargo는 레지스트리가 인증을 필요로 한다고 가정하고 인증 토큰을 포함하여 config.json 요청을 다시 시도합니다.

인증 실패(또는 인증 토큰 누락) 시, 서버는 사용자가 토큰을 받을 수 있는 위치를 나타내기 위해 Cargo login_url="<URL>" 챌린지가 포함된 www-authenticate 헤더를 포함할 수 있습니다.

인증이 필요한 레지스트리는 config.json 에 auth-required: true 를 설정해야 합니다.

캐싱

Cargo는 크레이트 메타데이터 파일을 캐시하며, 각 항목에 대해 서버로부터 ETag 혹은 Last-Modified HTTP 헤더를 캡처합니다. 크레이트 메타데이터를 갱신할 때, Cargo는 If-None-Match 혹은 If-Modified-Since 헤더를 전송하여 로컬 캐시가 유효한 경우 서버가 HTTP 304 “Not Modified“로 응답할 수 있도록 하여 시간과 대역폭을 절약합니다. ETag 와 Last-Modified 헤더가 모두 존재하는 경우, Cargo는 ETag 만 사용합니다.

캐시 무효화

레지스트리가 인덱스 파일 접근을 캐시하는 CDN이나 프록시 종류를 사용하고 있다면, 파일이 업데이트될 때 어떤 형태로든 캐시 무효화를 구현하는 것이 권장됩니다. 이러한 캐시가 업데이트되지 않으면 사용자는 캐시가 만료될 때까지 새로운 크레이트에 접근하지 못할 수 있습니다.

존재하지 않는 크레이트

존재하지 않는 크레이트에 대해, 레지스트리는 404 “Not Found”, 410 “Gone”, 혹은 451 “Unavailable For Legal Reasons” 코드로 응답해야 합니다.

Sparse 제한 사항

레지스트리 URL이 락파일(lockfile)에 저장되기 때문에, 한 레지스트리에 대해 두 프로토콜을 모두 제공하는 것은 권장되지 않습니다. 전환 계획에 대한 논의는 이슈 #10964 에서 진행 중입니다. crates.io 레지스트리는 예외인데, Cargo가 sparse 프로토콜 사용 시 내부적으로 그에 상응하는 git URL로 대체하기 때문입니다.

만약 레지스트리가 두 프로토콜을 모두 제공한다면, 현재로서는 하나를 정식 프로토콜로 선택하고 다른 하나에 대해서는 소스 대체(source replacement)를 사용하도록 하는 것이 권장됩니다.
