The Cargo Book

기능

Cargo “기능(features)“은 조건부 컴파일(conditional compilation) 및 선택적 의존성(optional dependencies)을 표현하기 위한 메커니즘을 제공합니다. 패키지는 Cargo.toml의 [features] 테이블에 명명된 기능 세트를 정의하며, 각 기능은 활성화하거나 비활성화할 수 있습니다. 빌드되는 패키지의 기능은 --features와 같은 플래그를 사용하여 명령줄에서 활성화할 수 있습니다. 의존성의 기능은 Cargo.toml의 의존성 선언부에서 활성화할 수 있습니다.

참고: 이제 crates.io에 게시되는 새로운 크레이트나 버전은 최대 300개의 기능으로 제한됩니다. 예외는 사안별로 허용됩니다. 자세한 내용은 이 블로그 포스트를 참조하세요. 해결 방안 논의에 대한 참여는 crates.io Zulip 스트림을 통해 권장됩니다.

기능이 어떻게 사용될 수 있는지에 대한 몇 가지 예시는 기능 예제(Features Examples) 장을 참조하세요.

[features] 섹션

기능은 Cargo.toml의 [features] 테이블에 정의됩니다. 각 기능은 해당 기능이 활성화할 다른 기능들이나 선택적 의존성들의 배열을 지정합니다. 다음 예시들은 서로 다른 이미지 형식 지원을 선택적으로 포함할 수 있는 2D 이미지 처리 라이브러리에서 기능이 어떻게 사용될 수 있는지 보여줍니다.

[features]
# Defines a feature named `webp` that does not enable any other features.
webp = []

이 기능이 정의되면, 컴파일 타임에 요청된 기능을 지원하는 코드를 조건부로 포함하기 위해 cfg 식을 사용할 수 있습니다. 예를 들어, 패키지의 lib.rs 내부에 다음을 포함할 수 있습니다.

#![allow(unused)]
fn main() {
// WEBP 지원을 구현하는 모듈을 조건부로 포함합니다.
#[cfg(feature = "webp")]
pub mod webp;
}

Cargo는 rustc --cfg 플래그를 사용하여 패키지에 기능을 설정하며, 코드는 cfg 속성 또는 cfg 매크로를 사용하여 기능의 존재 여부를 테스트할 수 있습니다.

기능은 활성화할 다른 기능들을 나열할 수 있습니다. 예를 들어, ICO 이미지 형식은 BMP 및 PNG 이미지를 포함할 수 있으므로, ICO 기능이 활성화될 때 해당 다른 기능들도 활성화되도록 해야 합니다.

[features]
bmp = []
png = []
ico = ["bmp", "png"]
webp = []

기능 이름은 (대부분의 문자를 포함하는) Unicode XID 표준의 문자들을 포함할 수 있으며, 추가적으로 _나 0에서 9까지의 숫자로 시작할 수 있고, 첫 번째 문자 이후에는 -, + 또는 .을 포함할 수도 있습니다.

참고: crates.io는 기능 이름 구문에 대해 ASCII 영문자 및 숫자 문자 또는 _, -, +만 사용해야 한다는 추가 제약을 둡니다.

default 기능

기본적으로 모든 기능은 명시적으로 활성화되지 않는 한 비활성화됩니다. 이는 default 기능을 지정하여 변경할 수 있습니다.

[features]
default = ["ico", "webp"]
bmp = []
png = []
ico = ["bmp", "png"]
webp = []

패키지가 빌드될 때 default 기능이 활성화되며, 이는 나열된 기능들을 차례로 활성화합니다. 이 동작은 다음과 같이 변경할 수 있습니다.

  • --no-default-features 명령줄 플래그는 패키지의 기본 기능을 비활성화합니다.
  • default-features = false 옵션을 의존성 선언부에 지정할 수 있습니다.

참고: 기본 기능 세트를 선택할 때 주의하세요. 기본 기능은 사용자가 일반적인 용도를 위해 어떤 기능을 활성화할지 신중하게 선택하도록 강요하지 않고 패키지를 더 쉽게 사용할 수 있게 해주는 편의성이지만, 몇 가지 단점이 있습니다. 의존성은 default-features = false가 지정되지 않는 한 자동으로 기본 기능을 활성화합니다. 이는 특히 의존성 그래프에 여러 번 나타나는 의존성에 대해 기본 기능이 활성화되지 않도록 보장하는 것을 어렵게 만들 수 있습니다. 모든 패키지는 기본 기능이 활성화되는 것을 피하기 위해 default-features = false가 지정되었는지 확인해야 합니다.

또 다른 문제는 기본 세트에서 기능을 제거하는 것이 SemVer 비호환 변경이 될 수 있다는 것이므로, 해당 기능들을 계속 유지할 것인지 확신이 있어야 합니다.

선택적 의존성

의존성은 “선택적(optional)“으로 표시될 수 있으며, 이는 기본적으로 컴파일되지 않음을 의미합니다. 예를 들어, 우리의 2D 이미지 처리 라이브러리가 GIF 이미지를 처리하기 위해 외부 패키지를 사용한다고 가정해 봅시다. 이는 다음과 같이 표현될 수 있습니다.

[dependencies]
gif = { version = "0.11.1", optional = true }

기본적으로, 이 선택적 의존성은 다음과 같은 기능을 암시적으로 정의합니다.

[features]
gif = ["dep:gif"]

이는 gif 기능이 활성화된 경우에만 이 의존성이 포함됨을 의미합니다. 코드 내에서 동일한 cfg(feature = "gif") 구문을 사용할 수 있으며, 해당 의존성은 --features gif와 같은 여타 기능들과 마찬가지로 활성화될 수 있습니다 (아래의 명령줄 기능 옵션 참조).

경우에 따라 선택적 의존성과 동일한 이름을 가진 기능을 노출하고 싶지 않을 수도 있습니다. 예를 들어, 선택적 의존성이 내부 상세 구현이거나, 여러 선택적 의존성을 하나로 묶고 싶거나, 혹은 더 나은 이름을 사용하고 싶은 경우가 해당됩니다. [features] 테이블의 어느 곳에서든 dep: 접두사를 사용하여 선택적 의존성을 지정하면, 해당 암시적 기능이 비활성화됩니다.

참고: dep: 구문은 Rust 1.60부터 사용할 수 있습니다. 이전 버전에서는 암시적 기능 이름만 사용할 수 있습니다.

예를 들어, AVIF 이미지 형식을 지원하기 위해 라이브러리에서 다른 두 의존성을 활성화해야 한다고 가정해 봅시다.

[dependencies]
ravif = { version = "0.6.3", optional = true }
rgb = { version = "0.8.25", optional = true }

[features]
avif = ["dep:ravif", "dep:rgb"]

이 예시에서 avif 기능은 나열된 두 의존성을 활성화합니다. 또한 ravif 및 rgb는 우리 크레이트의 내부 상세 구현이므로 사용자가 개별적으로 활성화하는 것을 원치 않기 때문에, 암시적 기능이 생성되는 것을 방지합니다.

참고: 의존성을 선택적으로 포함하는 또 다른 방법은 플랫폼별 의존성을 사용하는 것입니다. 이는 기능을 사용하는 대신 대상 플랫폼에 따라 조건부로 적용됩니다.

의존성 기능

의존성의 기능은 의존성 선언 내에서 활성화할 수 있습니다. features 키는 어떤 기능을 활성화할지 나타냅니다.

[dependencies]
# serde의 `derive` 기능을 활성화합니다.
serde = { version = "1.0.118", features = ["derive"] }

default 기능은 default-features = false를 사용하여 비활성화할 수 있습니다.

[dependencies]
flate2 = { version = "1.0.3", default-features = false, features = ["zlib-rs"] }

참고: 이것만으로는 기본 기능이 비활성화되는 것을 보장하지 못할 수도 있습니다. 다른 의존성이 default-features = false를 지정하지 않고 flate2를 포함한다면, 기본 기능이 활성화됩니다. 자세한 내용은 아래의 기능 통합(feature unification)을 참조하세요.

의존성의 기능은 [features] 테이블에서도 활성화할 수 있습니다. 구문은 "패키지-이름/기능-이름"입니다. 예를 들어 다음과 같습니다.

[dependencies]
jpeg-decoder = { version = "0.1.20", default-features = false }

[features]
# jpeg-decoder의 "rayon" 기능을 활성화하여 병렬 처리 지원을 활성화합니다.
parallel = ["jpeg-decoder/rayon"]

"패키지-이름/기능-이름" 구문은 패키지-이름이 선택적 의존성인 경우 해당 패키지까지 활성화합니다. 대개의 경우 이는 의도한 바가 아닐 것입니다. "패키지-이름?/기능-이름"과 같이 ?를 추가하면, 다른 누군가가 해당 선택적 의존성을 활성화했을 때만 지정된 기능을 활성화합니다.

참고: ? 구문은 Rust 1.60부터 사용할 수 있습니다.

예를 들어, 라이브러리에 직렬화(serialization) 지원을 추가했고, 일부 선택적 의존성에서도 그에 대응하는 기능을 활성화해야 한다고 가정해 봅시다. 다음과 같이 할 수 있습니다.

[dependencies]
serde = { version = "1.0.133", optional = true }
rgb = { version = "0.8.25", optional = true }

[features]
serde = ["dep:serde", "rgb?/serde"]

이 예시에서 serde 기능을 활성화하면 serde 의존성이 활성화됩니다. 또한 rgb 의존성에 대해서도 serde 기능을 활성화하지만, 이는 다른 누군가가 rgb 의존성을 활성화했을 때만 해당됩니다.

명령줄 기능 옵션

다음 명령줄 플래그를 사용하여 활성화할 기능을 제어할 수 있습니다.

  • --features FEATURES: 나열된 기능을 활성화합니다. 여러 기능은 쉼표나 공백으로 구분할 수 있습니다. 공백을 사용하는 경우, 셸에서 Cargo를 실행할 때 모든 기능 주위에 따옴표를 사용해야 합니다(예: --features "foo bar"). 워크스페이스에서 여러 패키지를 빌드하는 경우, 패키지-이름/기능-이름 구문을 사용하여 특정 워크스페이스 구성원의 기능을 지정할 수 있습니다.
  • --all-features: Activates all features of all packages selected on the command line.
  • --no-default-features: 선택된 패키지의 default 기능을 활성화하지 않습니다.

NOTE: check the individual subcommand documentation for details. Not all flags are available for all subcommands.

기능 통합

기능은 해당 기능을 정의하는 패키지에 고유합니다. 한 패키지에서 특정 기능을 활성화한다고 해서 다른 패키지에서 동일한 이름의 기능이 활성화되지는 않습니다.

여러 패키지에서 하나의 의존성을 사용하는 경우, Cargo는 해당 의존성을 빌드할 때 활성화된 모든 기능의 합집합을 사용합니다. 이는 의존성의 사본이 단 하나만 사용되도록 보장하는 데 도움이 됩니다. 자세한 내용은 리졸버 문서의 기능 섹션을 참조하세요.

예를 들어, 수많은 기능을 사용하는 winapi 패키지를 살펴봅시다. 사용자의 패키지가 winapi의 “fileapi” 및 “handleapi” 기능을 활성화하는 foo 패키지에 의존하고, 또 다른 의존성인 bar가 winapi의 “std” 및 “winnt” 기능을 활성화한다면, winapi는 이 네 가지 기능이 모두 활성화된 상태로 빌드됩니다.

이러한 동작의 결과로 기능은 _가산적(additive)_이어야 합니다. 즉, 기능을 활성화한다고 해서 기존 기능이 비활성화되어서는 안 되며, 어떤 기능의 조합이든 활성화해도 안전해야 합니다. 기능은 SemVer 비호환 변경을 도입해서는 안 됩니다.

예를 들어, no_std 환경을 선택적으로 지원하고 싶다면 no_std 기능을 사용하지 마세요. 대신 std를 활성화하는 std 기능을 사용하세요. 예시:

#![allow(unused)]
#![no_std]

fn main() {
#[cfg(feature = "std")]
extern crate std;

#[cfg(feature = "std")]
pub fn function_that_requires_std() {
    // ...
}
}

상호 배타적 기능

드문 경우지만 기능들이 서로 호환되지 않을 수 있습니다. 이는 의존성 그래프 내에서 해당 패키지를 사용하는 모든 곳이 서로 협력하여 기능들을 동시에 활성화하지 않도록 조정해야 하므로, 가능하면 피해야 합니다. 만약 피할 수 없다면, 이러한 시나리오를 감지하기 위해 컴파일 오류를 추가하는 것을 고려해 보세요. 예시:

#[cfg(all(feature = "foo", feature = "bar"))]
compile_error!("feature \"foo\" and feature \"bar\" cannot be enabled at the same time");

상호 배타적인 기능을 사용하는 대신 다음과 같은 다른 옵션들을 고려해 보세요.

  • 기능을 별도의 패키지로 분리합니다.
  • 충돌이 발생할 경우, 하나의 기능을 다른 기능보다 우선시합니다. cfg-if 패키지를 사용하면 더 복잡한 cfg 식을 작성하는 데 도움이 될 수 있습니다.
  • 기능들이 동시에 활성화될 수 있도록 코드를 설계하고, 어떤 기능을 사용할지 제어하기 위해 런타임 옵션을 사용합니다. 예를 들어, 설정 파일, 명령줄 인수 또는 환경 변수를 사용하여 활성화할 동작을 선택합니다.

해결된 기능 검사하기

복잡한 의존성 그래프에서는 여러 패키지에서 서로 다른 기능들이 어떻게 활성화되는지 파악하기 어려울 때가 있습니다. cargo tree 명령어는 어떤 기능들이 활성화되었는지 검사하고 시각화하는 데 도움이 되는 몇 가지 옵션을 제공합니다. 다음 옵션들을 시도해 보세요.

  • cargo tree -e features: 의존성 그래프에 기능을 표시합니다. 각 기능은 어떤 패키지가 이를 활성화했는지와 함께 나타납니다.
  • cargo tree -f "{p} {f}": 각 패키지에서 활성화된 기능들을 쉼표로 구분된 목록으로 보여주는 더 간결한 뷰입니다.
  • cargo tree -e features -i foo: 트리를 반전시켜 지정된 패키지 “foo“로 기능들이 어떻게 유입되는지 보여줍니다. 전체 그래프를 보는 것이 너무 방대하고 복잡할 때 유용할 수 있습니다. 특정 패키지에 어떤 기능들이 왜 활성화되었는지 파악하려고 할 때 이 옵션을 사용하세요. 이를 읽는 방법에 대해서는 cargo tree 페이지 하단의 예시를 참조하세요.

기능 리졸버 버전 2

다음과 같이 Cargo.toml의 resolver 필드를 사용하여 다른 기능 리졸버를 지정할 수 있습니다.

[package]
name = "my-package"
version = "1.0.0"
resolver = "2"

리졸버 버전을 지정하는 방법에 대한 자세한 내용은 리졸버 버전 섹션을 참조하세요.

버전 "2" 리졸버는 통합이 원치 않는 몇 가지 상황에서 기능 통합을 피합니다. 구체적인 상황은 리졸버 장에 설명되어 있지만, 요약하자면 다음과 같은 상황에서 통합을 피합니다.

  • 현재 빌드 중이 아닌 대상 아키텍처에 대한 타겟별 의존성 기능은 무시됩니다.
  • 빌드 의존성 및 절차적 매크로(proc-macro)는 일반 의존성과 기능을 공유하지 않습니다.
  • 개발 의존성은 해당 기능이 필요한 Cargo 타겟 (테스트나 예제 등)을 빌드할 때만 기능을 활성화합니다.

일부 상황에서는 통합을 피하는 것이 필요합니다. 예를 들어, 빌드 의존성이 std 기능을 활성화하는데 동일한 의존성이 no_std 환경을 위한 일반 의존성으로 사용되는 경우, std를 활성화하면 빌드가 깨지게 됩니다.

하지만 단점은 의존성이 여러 번 빌드되기 때문에(각각 다른 기능으로) 빌드 시간이 늘어날 수 있다는 것입니다. 버전 "2" 리졸버를 사용할 때는 전체 빌드 시간을 단축하기 위해 여러 번 빌드되는 의존성이 있는지 확인하는 것이 좋습니다. 만약 해당 중복 패키지들을 별도의 기능으로 빌드할 필요가 없다면, 중복된 패키지들이 동일한 기능을 갖도록(따라서 Cargo가 한 번만 빌드하도록) 의존성 선언부의 features 목록에 기능을 추가하는 것을 고려해 보세요. 이러한 중복 의존성은 cargo tree --duplicates 명령어로 감지할 수 있습니다. 이 명령어는 여러 번 빌드된 패키지들을 보여주며, 동일한 버전으로 나열된 항목들을 찾으면 됩니다. 해결된 기능에 대한 정보를 가져오는 방법은 해결된 기능 검사하기를 참조하세요. 빌드 의존성의 경우, --target 플래그를 사용하여 교차 컴파일 중이라면 빌드 의존성은 항상 일반 의존성과 별도로 빌드되므로 이 작업이 필요하지 않습니다.

리졸버 버전 2 명령줄 플래그

resolver = "2" 설정은 --features 및 --no-default-features 명령줄 옵션의 동작도 변경합니다.

버전 "1"에서는 현재 작업 디렉토리에 있는 패키지의 기능만 활성화할 수 있습니다. 예를 들어, foo와 bar 패키지가 있는 워크스페이스에서 foo 패키지 디렉토리에 있는 상태로 cargo build -p bar --features bar-feat 명령을 실행하면 실패합니다. --features 플래그가 foo에 대한 기능 활성화만 허용했기 때문입니다.

버전 "2" 리졸버를 사용하면, 기능 플래그를 통해 -p 및 --workspace 플래그로 선택된 모든 패키지에 대해 기능을 활성화할 수 있습니다. 예를 들어 다음과 같습니다.

# 이 명령은 현재 어느 디렉토리에 있든 관계없이 resolver = "2"에서 허용됩니다.
cargo build -p foo -p bar --features foo-feat,bar-feat

# 이 명시적인 동등한 명령은 모든 리졸버 버전에서 작동합니다.
cargo build -p foo -p bar --features foo/foo-feat,bar/bar-feat

또한, resolver = "1"에서는 --no-default-features 플래그가 현재 디렉토리의 패키지에 대한 기본 기능만 비활성화합니다. 버전 “2“에서는 모든 워크스페이스 구성원에 대해 기본 기능을 비활성화합니다.

빌드 스크립트

빌드 스크립트는 CARGO_FEATURE_<name> 환경 변수를 검사하여 패키지에 어떤 기능이 활성화되었는지 감지할 수 있습니다. 여기서 <name>은 대문자로 변환되고 -는 _로 변환된 기능 이름입니다.

필수 기능

required-features 필드를 사용하면 특정 기능이 활성화되지 않은 경우 특정 Cargo 타겟을 비활성화할 수 있습니다. 자세한 내용은 링크된 문서를 참조하세요.

SemVer 호환성

기능을 활성화해도 SemVer와 호환되지 않는 변경이 도입되어서는 안 됩니다. 예를 들어, 기능이 기존 사용을 깨뜨릴 수 있는 방식으로 기존 API를 변경해서는 안 됩니다. 어떤 변경이 호환되는지에 대한 자세한 내용은 SemVer 호환성 장에서 확인할 수 있습니다.

기능 정의 및 선택적 의존성을 추가하거나 제거할 때는 하위 호환성을 깨뜨리는 변경이 될 수 있으므로 주의해야 합니다. 자세한 내용은 SemVer 호환성 장의 Cargo 섹션에서 확인할 수 있습니다. 요약하자면 다음 규칙을 따르세요.

  • 부 버전(minor release)에서 다음 작업은 일반적으로 안전합니다.
    • 새로운 기능 또는 선택적 의존성 추가
    • 의존성에 사용된 기능 변경
  • 부 버전에서 다음 작업은 일반적으로 수행해서는 안 됩니다.
    • 기능 제거 또는 선택적 의존성 제거
    • 기존 공개 코드를 기능 뒤로 이동(기능 게이트 적용)
    • 기능 목록에서 기능 제거

주의 사항 및 예시는 링크를 참조하세요.

기능 문서화 및 탐색

패키지에서 어떤 기능들을 사용할 수 있는지 문서화할 것을 권장합니다. 이는 lib.rs 상단에 독 커멘트(doc comments)를 추가하여 수행할 수 있습니다. 예시로 regex 크레이트 소스를 참조하세요. 렌더링된 결과는 docs.rs에서 확인할 수 있습니다. 사용자 가이드와 같은 다른 문서가 있다면 해당 문서에도 추가하는 것을 고려해 보세요 (예를 들어 serde.rs 참조). 바이너리 프로젝트인 경우 README나 프로젝트의 다른 문서에 기능을 문서화하는 것을 고려해 보세요 (예를 들어 sccache 참조).

기능을 명확하게 문서화하면 “불안정“하거나 다른 이유로 사용해서는 안 되는 기능들에 대한 기대치를 설정할 수 있습니다. 예를 들어, 선택적 의존성이 있지만 사용자가 해당 선택적 의존성을 기능으로 명시적으로 나열하는 것을 원치 않는다면, 문서화된 목록에서 제외하세요.

docs.rs에 게시되는 문서는 Cargo.toml의 메타데이터를 사용하여 문서 빌드 시 어떤 기능을 활성화할지 제어할 수 있습니다. 자세한 내용은 docs.rs 메타데이터 문서를 참조하세요.

Note: Rustdoc has experimental support for annotating the documentation to indicate which features are required to use certain APIs. See the doc_cfg documentation for more details. An example is the syn documentation, where you can see colored boxes which note which features are required to use it.

기능 탐색하기

라이브러리 API에 기능이 문서화되어 있으면 사용자가 어떤 기능을 사용할 수 있고 그 기능이 무엇을 하는지 더 쉽게 파악할 수 있습니다. 패키지의 기능 문서가 즉시 제공되지 않는 경우 Cargo.toml 파일을 살펴볼 수 있지만, 때로는 이를 찾아내기 어려울 수 있습니다. crates.io의 크레이트 페이지에는 사용 가능한 경우 소스 저장소에 대한 링크가 있습니다. cargo vendor 또는 cargo-clone-crate와 같은 도구를 사용하여 소스를 다운로드하고 검사할 수 있습니다.

기능 조합

기능은 조건부 컴파일의 한 형태이기 때문에, 100% 테스트하려면 기하급수적인 수의 구성과 테스트 케이스가 필요합니다. 기본적으로 테스트, 문서 및 Clippy와 같은 다른 도구들은 기본 기능 세트만 사용하여 실행됩니다.

다양한 기능 조합에 관한 전략과 도구 사용을 고려해 보시기 바랍니다. 프로젝트마다 시간, 리소스, 특정 시나리오를 다루는 데 따른 비용 대비 효과 등 서로 다른 요구 사항이 있을 것입니다. 일반적인 구성으로는 기본 기능을 포함하거나 포함하지 않는 경우, 특정 기능들의 조합, 또는 모든 기능의 조합 등이 있을 수 있습니다.
