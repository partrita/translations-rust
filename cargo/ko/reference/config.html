The Cargo Book

설정

이 문서는 Cargo의 설정(configuration) 시스템 작동 방식과 사용 가능한 키 또는 설정에 대해 설명합니다. 매니페스트를 통한 패키지 설정에 대해서는 매니페스트 형식 을 참조하세요.

계층적 구조

Cargo는 특정 패키지에 대한 로컬 설정뿐만 아니라 전역 설정도 허용합니다. 현재 디렉토리와 모든 상위 디렉토리에서 설정 파일을 찾습니다. 예를 들어, Cargo가 /projects/foo/bar/baz 에서 호출되었다면 다음과 같은 설정 파일들을 찾아 이 순서대로 통합합니다.

  • /projects/foo/bar/baz/.cargo/config.toml
  • /projects/foo/bar/.cargo/config.toml
  • /projects/foo/.cargo/config.toml
  • /projects/.cargo/config.toml
  • /.cargo/config.toml
  • 기본값이 다음과 같은 $CARGO_HOME/config.toml:
    • Windows: %USERPROFILE%\.cargo\config.toml
    • Unix: $HOME/.cargo/config.toml

이러한 구조를 통해 패키지별로 설정을 지정할 수 있으며, 가능한 경우 버전 관리 시스템에 체크인할 수도 있습니다. 또한 홈 디렉토리에 있는 설정 파일을 사용하여 개인적인 기본값을 지정할 수도 있습니다.

여러 설정 파일에 동일한 키가 지정된 경우, 값들이 하나로 통합됩니다. 숫자, 문자열 및 불리언 값은 더 깊은 디렉토리에 있는 설정이 상위 디렉토리보다 우선하며, 홈 디렉토리의 우선순위가 가장 낮습니다. 배열은 서로 합쳐지며, 우선순위가 높은 항목이 통합된 배열의 뒤쪽에 배치됩니다.

현재로서, 워크스페이스에서 호출될 때 Cargo는 워크스페이스 내 크레이트들의 설정 파일을 읽지 않습니다. 즉, /projects/foo/bar/baz/mylib 와 /projects/foo/bar/baz/mybin 이라는 두 크레이트가 있고 각각에 설정 파일이 있더라도, 워크스페이스 루트(/projects/foo/bar/baz/)에서 호출되면 해당 설정 파일들을 읽지 않습니다.

참고: Cargo는 .cargo/config 와 같이 .toml 확장자가 없는 설정 파일도 읽습니다. .toml 확장자 지원은 1.39 버전에서 추가되었으며 권장되는 형식입니다. 두 파일이 모두 존재하는 경우 Cargo는 확장자가 없는 파일을 사용합니다.

환경 변수

Cargo는 TOML 설정 파일 외에도 환경 변수를 통해 설정할 수 있습니다. foo.bar 형식의 각 설정 키에 대해 CARGO_FOO_BAR 환경 변수를 사용하여 값을 정의할 수 있습니다. 키는 대문자로 변환되며, 마침표와 대시는 밑줄로 변환됩니다. 예를 들어 target.x86_64-unknown-linux-gnu.runner 키는 CARGO_TARGET_X86_64_UNKNOWN_LINUX_GNU_RUNNER 환경 변수로 정의될 수 있습니다.

환경 변수는 TOML 설정 파일보다 우선합니다. 현재로서는 정수, 불리언, 문자열 및 일부 배열 값만 환경 변수로 정의할 수 있습니다. 아래의 설명 에서 어떤 키가 환경 변수를 지원하는지 확인할 수 있으며, 그렇지 않은 키는 기술적인 문제 로 인해 지원되지 않습니다.

위의 시스템 외에도 Cargo는 몇 가지 다른 특정 환경 변수 를 인식합니다.

명령줄 재정의

Cargo는 --config 명령줄 옵션을 통해 임의의 설정 재정의를 허용합니다. 인수는 KEY=VALUE 와 같은 TOML 구문이거나 추가 설정 파일의 경로여야 합니다.

# With `KEY=VALUE` in TOML syntax
cargo --config net.git-fetch-with-cli=true fetch

# With a path to a configuration file
cargo --config ./path/to/my/extra-config.toml fetch

--config 옵션은 여러 번 지정될 수 있으며, 이 경우 여러 설정 파일이 적용될 때와 동일한 병합 로직을 사용하여 왼쪽에서 오른쪽 순서로 병합됩니다. 이 방식으로 지정된 설정 값은 환경 변수보다 우선하며, 환경 변수는 설정 파일보다 우선합니다.

--config 옵션에 추가 설정 파일이 제공된 경우, 이 방식으로 로드된 설정 파일은 --config 로 직접 지정된 다른 옵션과 동일한 우선순위 규칙을 따릅니다.

Bourne 셸 구문을 사용한 몇 가지 예시는 다음과 같습니다.

# Most shells will require escaping.
cargo --config http.proxy=\"http://example.com\" …

# Spaces may be used.
cargo --config "net.git-fetch-with-cli = true" …

# TOML array example. Single quotes make it easier to read and write.
cargo --config 'build.rustdocflags = ["--html-in-header", "header.html"]' …

# Example of a complex TOML key.
cargo --config "target.'cfg(all(target_arch = \"arm\", target_os = \"none\"))'.runner = 'my-runner'" …

# Example of overriding a profile setting.
cargo --config profile.dev.package.image.opt-level=3 …

Including extra configuration files

Configuration can include other configuration files using the top-level include key. This allows sharing configuration across multiple projects or splitting complex configurations into multiple files.

include

  • Type: array of strings or tables
  • 기본값: 없음
  • 환경 변수: 지원되지 않음

Loads additional configuration files. Paths are relative to the configuration file that includes them. Only paths ending with .toml are accepted.

Supports the following formats:

# array of paths
include = [
    "frodo.toml",
    "samwise.toml",
]

# inline tables for more control
include = [
    { path = "required.toml" },
    { path = "optional.toml", optional = true },
]

Note: For better readability and to avoid confusion, it is recommended to:

  • Place include at the top of the configuration file
  • Put one include per line for clearer version control diffs
  • Use inline table syntax when optional includes are needed

When using table syntax, the following fields are supported:

  • path (string, required): Path to the config file to include.
  • optional (boolean, default: false): If true, missing files are silently skipped instead of causing an error.

The merge behavior of include is different from other config values:

  1. Config values are first loaded from the include paths.
    • Included files are loaded left to right, with values from later files taking precedence over earlier ones.
    • This step recurses if included config files also contain include keys.
  2. Then, the config file’s own values are merged on top of the included config, taking highest precedence.

설정 파일 기준 상대 경로

설정 파일의 경로는 절대 경로, 상대 경로, 또는 경로 구분 기호가 없는 단순 이름일 수 있습니다. 경로 구분 기호가 없는 실행 파일 경로의 경우 PATH 환경 변수를 사용하여 실행 파일을 검색합니다. 실행 파일이 아닌 경우의 경로는 설정 값이 정의된 위치를 기준으로 하는 상대 경로입니다.

구체적인 규칙은 다음과 같습니다.

  • 환경 변수의 경우, 경로는 현재 작업 디렉토리를 기준으로 합니다.
  • --config KEY=VALUE 옵션에서 직접 로드된 설정 값의 경우, 경로는 현재 작업 디렉토리를 기준으로 합니다.
  • 설정 파일의 경우, 해당 파일이 계층적 탐색 을 통한 것이든 --config <path> 옵션을 통한 것이든 관계없이, 경로는 설정 파일이 정의된 디렉토리의 부모 디렉토리를 기준으로 합니다.

참고: 기존의 .cargo/config.toml 탐색 동작과 일관성을 유지하기 위해, --config <path> 를 통해 전달된 설정 파일의 경로도 설정 파일 자체에서 두 단계 상위 디렉토리를 기준으로 하도록 설계되었습니다.

예기치 않은 결과를 방지하기 위한 기본 원칙은 추가 설정 파일을 프로젝트에서 발견된 .cargo/config.toml 과 동일한 수준에 두는 것입니다. 예를 들어, 프로젝트가 /my/project 인 경우 설정 파일을 /my/project/.cargo 아래에 두거나 동일한 수준의 새 디렉토리(예: /my/project/.config)에 두는 것이 좋습니다.

# 예시: `/my/project/.cargo/config.toml` 이 있다면 결과는 `/my/project/vendor` 가 됩니다.
directory = "vendor"

인수가 포함된 실행 파일 경로

일부 Cargo 명령어는 외부 프로그램을 호출하며, 이는 경로와 여러 인수로 구성할 수 있습니다.

값은 ['/path/to/program', 'somearg'] 와 같은 문자열 배열이거나 '/path/to/program somearg' 와 같이 공백으로 구분된 문자열일 수 있습니다. 실행 파일 경로에 공백이 포함된 경우 반드시 리스트 형식을 사용해야 합니다.

Cargo가 열기나 실행을 위한 경로와 같이 다른 인수를 프로그램에 전달하는 경우, 이러한 형식의 옵션 값에 지정된 마지막 인수 뒤에 전달됩니다. 지정된 프로그램에 경로 구분 기호가 없는 경우, Cargo는 PATH 에서 실행 파일을 검색합니다.

자격 증명

민감한 정보가 포함된 설정 값은 $CARGO_HOME/credentials.toml 파일에 저장됩니다. 이 파일은 cargo:token 자격 증명 제공자를 사용할 때 cargo login 및 cargo logout 에 의해 자동으로 생성되고 업데이트됩니다.

토큰은 원격 레지스트리 인증을 위해 cargo publish 와 같은 일부 Cargo 명령어에서 사용됩니다. 토큰을 보호하고 비밀로 유지하도록 주의해야 합니다.

Cargo 설정 파일과 동일한 형식을 따릅니다.

[registry]
token = "…"   # crates.io용 액세스 토큰

[registries.<이름>]
token = "…"   # 명명된 레지스트리용 액세스 토큰

다른 대부분의 설정 값과 마찬가지로, 토큰도 환경 변수로 지정할 수 있습니다. crates.io 용 토큰은 CARGO_REGISTRY_TOKEN 환경 변수로 지정할 수 있습니다. 다른 레지스트리용 토큰은 CARGO_REGISTRIES_<이름>_TOKEN 형식의 환경 변수로 지정할 수 있으며, 여기서 <이름> 은 레지스트리 이름을 모두 대문자로 쓴 것입니다.

참고: Cargo는 .cargo/credentials 와 같이 .toml 확장자가 없는 자격 증명 파일도 읽고 씁니다. .toml 확장자 지원은 1.39 버전에서 추가되었습니다. 1.68 버전부터 Cargo는 기본적으로 확장자가 있는 파일에 씁니다. 하지만 하위 호환성을 위해 두 파일이 모두 존재하는 경우 Cargo는 확장자가 없는 파일을 읽고 씁니다.

설정 키

이 섹션은 모든 설정 키를 문서화합니다. 가변적인 부분이 있는 키에 대한 설명은 target.<triple> 과 같이 꺽쇠 괄호로 표시되며, 여기서 <triple> 부분은 target.x86_64-pc-windows-msvc 와 같은 모든 [타겟 트리플](../appendix/glossary.md#target “타겟 트리플” (용어집))이 될 수 있습니다.

paths

  • 타입: 문자열 배열 (경로)
  • 기본값: 없음
  • 환경 변수: 지원되지 않음

의존성 재정의를 위해 사용될 로컬 패키지들의 경로 배열입니다. 자세한 내용은 의존성 재정의 가이드 를 참조하세요.

[alias]

  • 타입: 문자열 또는 문자열 배열
  • 기본값: 아래 참조
  • 환경 변수: CARGO_ALIAS_<name>

Cargo가 보통 직접 관리하는 플래그를 전달하는 것은 권장되지 않습니다. 예를 들어, 프로파일(profiles) 에 의해 결정되는 플래그들은 적절한 프로파일 설정을 통해 처리하는 것이 가장 좋습니다.

주의: 컴파일러에 직접 플래그를 전달하는 저수준 작업의 특성상, 향후 버전의 Cargo가 자체적으로 동일하거나 유사한 플래그를 사용하여 사용자가 지정한 플래그와 충돌을 일으킬 수 있습니다. 이 영역은 Cargo가 항상 하위 호환성을 보장하지 않을 수 있는 부분입니다.

[cache]

The [cache] table defines settings for cargo’s caches.

Global caches

When running cargo commands, Cargo will automatically track which files you are using within the global cache. Periodically, Cargo will delete files that have not been used for some period of time. It will delete files that have to be downloaded from the network if they have not been used in 3 months. Files that can be generated without network access will be deleted if they have not been used in 1 month.

The automatic deletion of files only occurs when running commands that are already doing a significant amount of work, such as all of the build commands (cargo build, cargo test, cargo check, etc.), and cargo fetch.

Automatic deletion is disabled if cargo is offline such as with --offline or --frozen to avoid deleting artifacts that may need to be used if you are offline for a long period of time.

Note: This tracking is currently only implemented for the global cache in Cargo’s home directory. This includes registry indexes and source files downloaded from registries and git dependencies. Support for tracking build artifacts is not yet implemented, and tracked in cargo#13136.

Additionally, there is an unstable feature to support manually triggering cache cleaning, and to further customize the configuration options. See the Unstable chapter for more information.

cache.auto-clean-frequency

  • 타입: 문자열
  • Default: "1 day"
  • Environment: CARGO_CACHE_AUTO_CLEAN_FREQUENCY

This option defines how often Cargo will automatically delete unused files in the global cache. This does not define how old the files must be, those thresholds are described above.

It supports the following settings:

  • "never" — Never deletes old files.
  • "always" — Checks to delete old files every time Cargo runs.
  • An integer followed by “seconds”, “minutes”, “hours”, “days”, “weeks”, or “months” — Checks to delete old files at most the given time frame.

  • 환경 변수: CARGO_HTTP_PROXY 또는 HTTPS_PROXY 또는 https_proxy 또는 http_proxy

사용할 HTTP 및 HTTPS 프록시를 설정합니다. 형식은 [protocol://]host[:port] 와 같은 libcurl 형식 입니다. 설정되지 않은 경우 Cargo는 전역 git 구성의 http.proxy 설정도 확인합니다. 둘 다 설정되지 않은 경우, HTTPS_PROXY 또는 https_proxy 환경 변수가 HTTPS 요청용 프록시를 설정하고, http_proxy 가 HTTP 요청용 프록시를 설정합니다.

http.timeout

  • 타입: 정수
  • 기본값: 30
  • 환경 변수: CARGO_HTTP_TIMEOUT 또는 HTTP_TIMEOUT

각 HTTP 요청에 대한 타임아웃을 초 단위로 설정합니다.

http.cainfo

  • 타입: 문자열 (경로)
  • 기본값: 없음
  • 환경 변수: CARGO_HTTP_CAINFO

TLS 인증서를 확인하는 데 사용되는 인증 기관(CA) 번들 파일의 경로입니다. 지정하지 않으면 Cargo는 시스템 인증서 사용을 시도합니다.

http.proxy-cainfo

  • 타입: 문자열 (경로)
  • Default: falls back to http.cainfo if not set
  • Environment: CARGO_HTTP_PROXY_CAINFO

Path to a Certificate Authority (CA) bundle file, used to verify proxy TLS certificates.

http.check-revoke

  • 타입: 불리언
  • 기본값: true (Windows), false (기타 모든 운영체제)
  • 환경 변수: CARGO_HTTP_CHECK_REVOKE

TLS 인증서 폐기(revocation) 확인을 수행할지 여부를 결정합니다. 이 기능은 윈도우에서만 작동합니다.

http.ssl-version

  • 타입: 문자열 또는 min/max 테이블
  • 기본값: 없음
  • 환경 변수: CARGO_HTTP_SSL_VERSION

사용할 최소 TLS 버전을 설정합니다. 문자열 값을 가지며, 가능한 값은 "default", "tlsv1", "tlsv1.0", "tlsv1.1", "tlsv1.2", "tlsv1.3" 중 하나입니다.

대신 min 과 max 라는 두 개의 키가 있는 테이블을 가질 수도 있으며, 각 키는 사용할 TLS 버전의 최소 및 최대 범위를 지정하는 동일한 종류의 문자열 값을 가집니다.

기본값은 최소 버전 "tlsv1.0" 이며, 최대 버전은 사용자의 플랫폼에서 지원하는 최신 버전(일반적으로 "tlsv1.3")입니다.

http.low-speed-limit

  • 타입: 정수
  • 기본값: 10
  • 환경 변수: CARGO_HTTP_LOW_SPEED_LIMIT

이 설정은 느린 연결에 대한 타임아웃 동작을 제어합니다. 초당 평균 전송 속도(바이트)가 http.timeout 초(기본값 30초) 동안 지정된 값보다 낮으면 연결이 너무 느린 것으로 간주되어 Cargo가 중단하고 다시 시도합니다.

http.multiplexing

  • 타입: 불리언
  • 기본값: true
  • 환경 변수: CARGO_HTTP_MULTIPLEXING

true 인 경우, Cargo는 멀티플렉싱이 포함된 HTTP2 프로토콜 사용을 시도합니다. 이를 통해 여러 요청이 동일한 연결을 사용할 수 있으며, 보통 여러 파일을 가져올 때 성능이 향상됩니다. false 인 경우, Cargo는 파이프라이닝이 없는 HTTP 1.1을 사용합니다.

http.user-agent

  • 타입: 문자열
  • 기본값: Cargo 버전
  • 환경 변수: CARGO_HTTP_USER_AGENT

사용할 사용자 정의 user-agent 헤더를 지정합니다. 지정하지 않을 경우 기본값은 Cargo 버전을 포함하는 문자열입니다.

net.ssh

[net.ssh] 테이블은 SSH 연결에 대한 설정을 포함합니다.

net.ssh.known-hosts

  • 타입: 문자열 배열
  • 기본값: 설명 참조
  • 환경 변수: 지원되지 않음

known-hosts 배열은 SSH 서버에 연결할 때(예: SSH git 의존성) 유효한 것으로 수락되어야 하는 SSH 호스트 키 목록을 포함합니다. 각 항목은 OpenSSH known_hosts 파일과 유사한 형식의 문자열이어야 합니다. 각 문자열은 쉼표로 구분된 하나 이상의 호스트 이름, 공백, 키 타입 이름, 공백, 그리고 base64로 인코딩된 키로 시작해야 합니다. 예를 들어 다음과 같습니다.

[net.ssh]
known-hosts = [
    "example.com ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIFO4Q5T0UV0SQevair9PFwoxY9dl4pQl3u5phoqJH3cF"

Cargo는 OpenSSH에서 지원하는 일반적인 위치에서 알려진 호스트 키를 로드하려고 시도하며, 이를 Cargo 설정 파일에 나열된 키와 병합합니다. 일치하는 항목 중 올바른 키가 있으면 연결이 허용됩니다.

Cargo에는 github.com 의 호스트 키가 내장되어 있습니다. 이 키가 변경되는 경우 설정 파일이나 known_hosts 파일에 새 키를 추가할 수 있습니다.

자세한 내용은 Git 인증 을 참조하세요.

[patch]

Cargo.toml 의 [patch]를 사용하여 의존성을 재정의할 수 있는 것처럼, Cargo 설정 파일에서도 의존성을 재정의하여 영향을 받는 모든 빌드에 해당 패치를 적용할 수 있습니다. 형식은 Cargo.toml 에서 사용하는 것과 동일합니다.

.cargo/config.toml 파일은 보통 소스 제어 시스템에 체크인되지 않으므로, 다른 개발자들이 자신의 환경에서 크레이트를 컴파일할 수 있도록 가급적 Cargo.toml 을 사용하여 패치하는 것이 좋습니다. Cargo 설정 파일을 통한 패치는 일반적으로 패치 섹션이 외부 빌드 도구에 의해 자동으로 생성되는 경우에만 적합합니다.

특정 의존성이 Cargo 설정 파일과 Cargo.toml 파일 모두에서 패치된 경우, 설정 파일의 패치가 사용됩니다. 여러 설정 파일이 동일한 의존성을 패치하는 경우 표준 Cargo 설정 병합 규칙이 사용되며, 현재 디렉토리에 가장 가까운 곳에 정의된 값이 우선되고 $HOME/.cargo/config.toml 이 가장 낮은 우선순위를 갖습니다.

이러한 [patch] 섹션의 상대 path 의존성은 해당 설정 파일이 위치한 디렉토리를 기준으로 해결됩니다.

[resolver]

[resolver] 테이블은 로컬 개발을 위한 의존성 해결 동작 을 재정의합니다 (예: cargo install 은 제외됨).

resolver.incompatible-rust-versions

  • 타입: 문자열
  • 기본값: resolver 문서 참조
  • 환경 변수: CARGO_RESOLVER_INCOMPATIBLE_RUST_VERSIONS

어떤 버전의 의존성을 사용할지 해결할 때, package.rust-version 이 호환되지 않는 버전을 어떻게 처리할지 선택합니다. 가능한 값은 다음과 같습니다.

  • allow: rust-version 이 호환되지 않는 버전을 다른 버전과 동일하게 취급합니다.
  • fallback: 일치하는 다른 버전이 없는 경우에만 rust-version 이 호환되지 않는 버전을 고려합니다.

다음과 같은 방법으로 재정의할 수 있습니다.

  • --ignore-rust-version CLI 옵션 사용
  • 의존성의 버전 요구 사항을 호환되는 rust-version 을 가진 어떤 버전보다 높게 설정하기
  • cargo update 시 --precise 로 특정 버전을 명시하기

자세한 내용은 리졸버(resolver) 장을 참조하세요.

MSRV:

  • allow 는 모든 버전에서 지원됩니다.
  • fallback 은 1.84 버전부터 적용됩니다.

registries.crates-io.protocol

  • 타입: 문자열
  • 기본값: "sparse"
  • 환경 변수: CARGO_REGISTRIES_CRATES_IO_PROTOCOL

crates.io에 접속할 때 사용할 프로토콜을 지정합니다. 허용되는 값은 git 또는 sparse 입니다.

git 을 사용하면 Cargo가 https://github.com/rust-lang/crates.io-index/에서 crates.io 에 게시된 모든 패키지의 전체 인덱스를 복제합니다. 이는 인덱스의 크기 때문에 성능에 영향을 미칠 수 있습니다. sparse 는 https://index.crates.io/에서 필요한 것만 다운로드하기 위해 HTTPS를 사용하는 최신 프로토콜입니다. 이는 대부분의 상황에서 새로운 의존성을 해결할 때 상당한 성능 향상을 가져올 수 있습니다.

레지스트리 프로토콜에 대한 자세한 내용은 레지스트리(Registries) 장 에서 확인할 수 있습니다.

이 값은 더 이상 허용되지 않으며 사용해서는 안 됩니다.

  • 기본값: "crates-io"
  • 환경 변수: CARGO_REGISTRY_DEFAULT

--registry 명령줄 옵션으로 재정의할 수 있습니다.

registry.credential-provider

  • 타입: 문자열 또는 경로와 인수의 배열
  • 기본값: 없음
  • 환경 변수: CARGO_REGISTRY_CREDENTIAL_PROVIDER

crates.io 에 대한 자격 증명 제공자를 지정합니다. 설정하지 않으면 registry.global-credential-providers 에 있는 제공자들이 사용됩니다.

문자열로 지정된 경우 경로와 인수는 공백을 기준으로 나뉩니다. 공백이 포함된 경로 또는 인수의 경우 배열을 사용하세요.

값이 [credential-alias] 테이블에 존재하면 해당 별칭이 사용됩니다.

자세한 내용은 레지스트리 인증 을 참조하세요.

registry.token

  • 타입: 문자열
  • 기본값: 없음
  • 환경 변수: CARGO_REGISTRY_TOKEN

crates.io 에 대한 인증 토큰을 지정합니다. 이 값은 자격 증명(#credentials) 파일에만 나타나야 합니다. 이는 인증이 필요한 cargo publish 와 같은 레지스트리 명령어에 사용됩니다.

--token 명령줄 옵션으로 재정의할 수 있습니다.

registry.global-credential-providers

  • 타입: 배열
  • 기본값: ["cargo:token"]
  • 환경 변수: CARGO_REGISTRY_GLOBAL_CREDENTIAL_PROVIDERS

전역 자격 증명 제공자 목록을 지정합니다. registries.<name>.credential-provider 를 사용하여 특정 레지스트리에 대해 자격 증명 제공자가 설정되지 않은 경우, Cargo는 이 목록에 있는 자격 증명 제공자를 사용합니다. 목록의 뒤쪽에 있는 제공자가 우선순위를 갖습니다.

경로와 인수는 공백을 기준으로 나뉩니다. 경로 또는 인수에 공백이 포함된 경우, 자격 증명 제공자를 [credential-alias] 테이블에 정의하고 여기에서 그 별칭으로 참조해야 합니다.

자세한 내용은 레지스트리 인증 을 참조하세요.

[target]

[target] 테이블은 특정 플랫폼 타겟에 대한 설정을 지정하는 데 사용됩니다. 이는 [플랫폼 트리플](../appendix/glossary.md#target “타겟” (용어집)) 또는 cfg() 식 인 하위 테이블로 구성됩니다. 대상 플랫폼이 <triple> 값이나 <cfg> 식과 일치하면 지정된 값이 사용됩니다.

[target.thumbv7m-none-eabi]
linker = "arm-none-eabi-gcc"
runner = "my-emulator"
rustflags = ["…", "…"]

[target.'cfg(all(target_arch = "arm", target_os = "none"))']
runner = "my-arm-wrapper"
rustflags = ["…", "…"]

cfg 값은 컴파일러에 내장된 값(rustc --print=cfg 명령으로 확인 가능)과 rustc 에 전달된 추가적인 --cfg 플래그(RUSTFLAGS 에 정의된 것 등)에서 가져옵니다. debug_assertions, test, Cargo 기능(예: feature="foo"), 또는 빌드 스크립트 에 의해 설정된 값과 일치시키려 하지 마세요.

타겟 사양 JSON 파일을 사용하는 경우, [<triple>](../appendix/glossary.md#target “타겟” (용어집)) 값은 파일 이름의 몸체(stem)가 됩니다. 예를 들어 --target foo/bar.json 은 [target.bar] 와 일치합니다.

target.<triple>.ar

이 옵션은 사용이 중단되었으며 사용되지 않습니다.

target.<triple>.linker

  • 타입: 문자열 (프로그램 경로)
  • 기본값: 없음
  • 환경 변수: CARGO_TARGET_<triple>_LINKER

해당 [<triple>](../appendix/glossary.md#target “타겟” (용어집))용으로 컴파일할 때 rustc 에 (-C linker 를 통해) 전달될 링커를 지정합니다. 기본적으로 링커는 재정의되지 않습니다.

target.<cfg>.linker

이것은 타겟 링커 와 유사하지만, cfg() 식 을 사용합니다. [<triple>](../appendix/glossary.md#target “타겟” (용어집))과 <cfg> 러너가 모두 일치하는 경우 <triple> 이 우선합니다. 현재 타겟과 일치하는 <cfg> 러너가 두 개 이상이면 오류가 발생합니다.

target.<triple>.runner

  • 타입: 문자열 또는 문자열 배열 (인수가 포함된 프로그램 경로)
  • 기본값: 없음
  • 환경 변수: CARGO_TARGET_<triple>_RUNNER

러너가 제공된 경우, 해당 타겟 [<triple>](../appendix/glossary.md#target “타겟” (용어집))에 대한 실행 파일은 실제 실행 파일을 인수로 전달하여 지정된 러너를 호출함으로써 실행됩니다. 이는 cargo run, cargo test, cargo bench 명령어에 적용됩니다. 기본적으로 컴파일된 실행 파일은 직접 실행됩니다.

target.<cfg>.runner

이것은 타겟 러너 와 유사하지만, cfg() 식 을 사용합니다. [<triple>](../appendix/glossary.md#target “타겟” (용어집))과 <cfg> 러너가 모두 일치하는 경우 <triple> 이 우선합니다. 현재 타겟과 일치하는 <cfg> 러너가 두 개 이상이면 오류가 발생합니다.

target.<triple>.rustflags

  • 타입: 문자열 또는 문자열 배열
  • 기본값: 없음
  • Environment: CARGO_TARGET_<triple>_RUSTFLAGS

이 [<triple>](../appendix/glossary.md#target “타겟” (용어집))에 대해 컴파일러에 전달할 사용자 정의 플래그 세트입니다. 값은 문자열 배열이거나 공백으로 구분된 문자열일 수 있습니다.

추가 플래그를 지정하는 다양한 방법에 대한 자세한 내용은 build.rustflags 를 참조하세요.

target.<cfg>.rustflags

이것은 타겟 rustflags 와 유사하지만, cfg() 식 을 사용합니다. 여러 <cfg> 및 [<triple>](../appendix/glossary.md#target “타겟” (용어집)) 항목이 현재 타겟과 일치하면 플래그들이 하나로 결합됩니다.

target.<triple>.rustdocflags

  • 타입: 문자열 또는 문자열 배열
  • 기본값: 없음
  • 환경 변수: CARGO_TARGET_<triple>_RUSTDOCFLAGS

이 [<triple>](../appendix/glossary.md#target “타겟” (용어집))에 대해 컴파일러에 전달할 사용자 정의 플래그 세트입니다. 값은 문자열 배열이거나 공백으로 구분된 문자열일 수 있습니다.

추가 플래그를 지정하는 다양한 방법에 대한 자세한 내용은 build.rustdocflags 를 참조하세요.

target.<triple>.<links>

links 하위 테이블은 빌드 스크립트를 재정의 하는 방법을 제공합니다. 이를 지정하면 해당 links 라이브러리에 대한 빌드 스크립트가 실행되지 않으며, 대신 지정된 값들이 사용됩니다.

[target.x86_64-unknown-linux-gnu.foo]
rustc-link-lib = ["foo"]
rustc-link-search = ["/path/to/foo"]
rustc-flags = "-L /some/path"
rustc-cfg = ['key="value"']
rustc-env = {key = "value"}
rustc-cdylib-link-arg = ["…"]
metadata_key1 = "value"
metadata_key2 = "value"

[term]

[term] 테이블은 터미널 출력 및 상호 작용을 제어합니다.

term.quiet

  • 타입: 불리언
  • 기본값: false
  • 환경 변수: CARGO_TERM_QUIET

Cargo가 로그 메시지를 표시할지 여부를 제어합니다.

--quiet 플래그를 지정하면 설정을 재정의하여 강제로 출력을 하지 않게 됩니다. --verbose 플래그를 지정하면 설정을 재정의하여 출력을 하지 않는 기능을 비활성화합니다.

term.verbose

  • 타입: 불리언
  • 기본값: false
  • 환경 변수: CARGO_TERM_VERBOSE

Cargo가 추가적인 상세 메시지를 표시할지 여부를 제어합니다.

--quiet 플래그를 지정하면 설정을 재정의하여 상세 출력을 비활성화합니다. --verbose 플래그를 지정하면 설정을 재정의하여 상세 출력을 강제합니다.

term.color

  • 타입: 문자열
  • 기본값: "auto"
  • 환경 변수: CARGO_TERM_COLOR

터미널에서 색상 출력을 사용할지 여부를 제어합니다. 가능한 값은 다음과 같습니다.

  • auto (기본값): 터미널의 색상 지원 여부를 자동으로 감지합니다.
  • always: 항상 색상을 표시합니다.
  • never: 색상을 절대 표시하지 않습니다.

--color 명령줄 옵션으로 재정의할 수 있습니다.
