The Cargo Book

Cargo 타겟

Cargo 패키지는 크레이트로 컴파일될 수 있는 소스 파일에 대응하는 타겟(targets) 들로 구성됩니다. 패키지는 라이브러리, 바이너리, 예제, 테스트, 벤치마크 타겟을 가질 수 있습니다. 타겟 목록은 Cargo.toml 매니페스트에서 구성할 수 있으며, 종종 소스 파일의 디렉토리 레이아웃 에 따라 자동으로 유추 됩니다.

타겟 설정을 구성하는 방법에 대한 자세한 내용은 아래의 타겟 구성하기 를 참조하세요.

라이브러리

라이브러리 타겟은 다른 라이브러리 및 실행 파일에서 사용하고 링크할 수 있는 “라이브러리“를 정의합니다. 파일 이름의 기본값은 src/lib.rs 이며, 라이브러리 이름의 기본값은 패키지 이름에서 대시(-)를 밑줄(_)로 바꾼 이름입니다. 패키지는 단 하나의 라이브러리만 가질 수 있습니다. 라이브러리 설정은 Cargo.toml 의 [lib] 테이블에서 사용자 정의 할 수 있습니다.

# Cargo.toml에서 라이브러리를 사용자 정의하는 예시
[lib]
crate-type = ["cdylib"]
bench = false

바이너리

바이너리 타겟은 컴파일 후 실행할 수 있는 실행 프로그램입니다. 바이너리의 소스는 src/main.rs 일 수 있으며, 또는 src/bin/ 디렉토리 에 저장될 수도 있습니다. src/main.rs 의 경우 기본 바이너리 이름은 패키지 이름입니다. 각 바이너리의 설정은 Cargo.toml 의 [[bin]] 테이블에서 사용자 정의 할 수 있습니다.

바이너리는 패키지 라이브러리의 공개 API를 사용할 수 있습니다. 또한 Cargo.toml 에 정의된 [dependencies]와 링크됩니다.

개별 바이너리는 cargo run 명령어에 --bin <bin-이름> 옵션을 주어 실행할 수 있습니다. cargo install 을 사용하면 실행 파일을 공통 위치로 복사할 수 있습니다.

# Cargo.toml에서 바이너리를 사용자 정의하는 예시
[[bin]]
name = "cool-tool"
test = false
bench = false

[[bin]]
name = "frobnicator"
required-features = ["frobnicate"]

예제

examples 디렉토리 에 있는 파일들은 라이브러리가 제공하는 기능의 사용 예제입니다. 컴파일되면 target/debug/examples 디렉토리 에 배치됩니다.

예제는 패키지 라이브러리의 공개 API를 사용할 수 있습니다. 또한 Cargo.toml 에 정의된 [dependencies] 및 [dev-dependencies]와 링크됩니다.

기본적으로 예제는 실행 가능한 바이너리(main() 함수 포함)입니다. crate-type 필드 를 지정하여 예제를 라이브러리로 컴파일할 수 있습니다.

[[example]]
name = "foo"
crate-type = ["staticlib"]

개별 실행 예제는 cargo run 명령어에 --example <예제-이름> 옵션을 주어 실행할 수 있습니다. 라이브러리 예제는 cargo build 에 --example <예제-이름> 옵션을 주어 빌드할 수 있습니다. cargo install 에 --example <예제-이름> 옵션을 사용하면 실행 바이너리를 공통 위치로 복사할 수 있습니다. 예제는 소스 코드가 노후화(bit-rotting)되는 것을 방지하기 위해 기본적으로 cargo test 에 의해 컴파일됩니다. 예제 안에 cargo test 로 실행하고 싶은 #[test] 함수가 있다면 the test 필드 를 true 로 설정하세요.

테스트

Cargo 프로젝트 내에는 두 가지 스타일의 테스트가 있습니다.

  • 단위 테스트(Unit tests) 는 라이브러리나 바이너리(또는 test 필드 가 활성화된 모든 타겟) 내에 위치하며 #[test] 속성이 표시된 함수입니다. 이 테스트들은 정의된 타겟 내부의 비공개 API에 접근할 수 있습니다.
  • 통합 테스트(Integration tests) 는 별도의 실행 바이너리로, 역시 #[test] 함수를 포함하며 프로젝트의 라이브러리와 링크되어 공개 API에 접근할 수 있습니다.

테스트는 cargo test 명령어로 실행됩니다. 기본적으로 Cargo와 rustc 는 #[test] 속성이 붙은 함수들을 수집하고 병렬로 실행하며 각 테스트의 성공 및 실패를 보고하는 libtest 하네스 를 사용합니다. 다른 하네스나 테스트 전략을 사용하고 싶다면 harness 필드 를 참조하세요.

참고: Cargo에는 또 다른 특별한 스타일의 테스트인 문서 테스트 가 있습니다. 이는 rustdoc 에 의해 처리되며 실행 모델이 약간 다릅니다. 자세한 내용은 cargo test 를 참조하세요.

통합 테스트

tests 디렉토리 에 있는 파일들은 통합 테스트입니다. cargo test 를 실행하면 Cargo는 이 파일들을 각각 별도의 크레이트로 컴파일하고 실행합니다.

통합 테스트는 패키지 라이브러리의 공개 API를 사용할 수 있습니다. 또한 Cargo.toml 에 정의된 [dependencies] 및 [dev-dependencies]와 링크됩니다.

여러 통합 테스트 간에 코드를 공유하고 싶다면 tests/common/mod.rs 와 같은 별도의 모듈에 코드를 배치한 다음, 각 테스트에서 mod common; 을 사용하여 가져올 수 있습니다.

각 통합 테스트는 별도의 실행 바이너리로 생성되며, cargo test 는 이를 순차적으로 실행합니다. 어떤 경우에는 컴파일 시간이 더 길어지고 테스트 실행 시 여러 CPU를 충분히 활용하지 못해 비효율적일 수 있습니다. 통합 테스트가 많은 경우, 단일 통합 테스트 파일을 만들고 테스트를 여러 모듈로 나누는 것을 고려해 보세요. libtest 하네스는 #[test] 속성이 붙은 모든 함수를 자동으로 찾아 병렬로 실행합니다. cargo test 에 모듈 이름을 전달하여 해당 모듈 내의 테스트만 실행할 수도 있습니다.

Binary targets are automatically built if there is an integration test. This allows an integration test to execute the binary to exercise and test its behavior. The CARGO_BIN_EXE_<name> environment variable is set when the integration test is built and run so that it can use the env macro or var function to locate the executable.

벤치마크

벤치마크는 cargo bench 명령어를 사용하여 코드의 성능을 테스트하는 방법을 제공합니다. 벤치마크는 테스트 와 동일한 구조를 따르며, 각 벤치마크 함수에는 #[bench] 속성이 표시됩니다. 테스트와 마찬가지로 다음과 같습니다.

  • 벤치마크는 benches 디렉토리 에 위치합니다.
  • 라이브러리 및 바이너리에 정의된 벤치마크 함수는 정의된 타겟 내부의 비공개 API에 접근할 수 있습니다. benches 디렉토리에 있는 벤치마크는 공개 API를 사용할 수 있습니다.
  • bench 필드 를 사용하여 어떤 타겟을 기본적으로 벤치마크할지 정의할 수 있습니다.
  • harness 필드 를 사용하여 내장 하네스를 비활성화할 수 있습니다.

참고: #[bench] 속성은 현재 불안정하며 나이틀리 채널(nightly channel) 에서만 사용할 수 있습니다. Criterion 과 같이 안정 채널에서 벤치마크를 실행하는 데 도움이 될 수 있는 일부 패키지들이 crates.io 에 있습니다.

타겟 구성하기

Cargo.toml 의 모든 [lib], [[bin]], [[example]], [[test]], [[bench]] 섹션은 타겟 빌드 방식을 지정하기 위한 유사한 구성을 지원합니다. [[bin]] 과 같이 이중 대괄호를 사용하는 섹션은 TOML의 테이블 배열(array-of-table) 로, 크레이트 내에 여러 개의 실행 파일을 만들기 위해 하나 이상의 [[bin]] 섹션을 작성할 수 있음을 의미합니다. 라이브러리는 하나만 지정할 수 있으므로 [lib] 은 일반적인 TOML 테이블입니다.

다음은 각 타겟에 대한 TOML 설정의 개요이며, 각 필드에 대한 자세한 설명은 아래에 나와 있습니다.

[lib]
name = "foo"           # The name of the target.
path = "src/lib.rs"    # The source file of the target.
test = true            # Is tested by default.
doctest = true         # Documentation examples are tested by default.
bench = true           # Is benchmarked by default.
doc = true             # Is documented by default.
proc-macro = false     # Set to `true` for a proc-macro library.
harness = true         # Use libtest harness.
crate-type = ["lib"]   # The crate types to generate.
required-features = [] # Features required to build this target (N/A for lib).

name 필드

name 필드는 생성될 결과물의 파일 이름에 대응하는 타겟의 이름을 지정합니다. 라이브러리의 경우, 의존성 패키지들이 해당 라이브러리를 참조할 때 사용하는 크레이트 이름이 됩니다.

라이브러리 타겟의 경우, 기본값은 패키지 이름에서 모든 대시(-)를 밑줄(_)로 바꾼 것입니다. 기본 바이너리(src/main.rs)의 경우에도 기본값은 패키지 이름이지만 대시가 유지됩니다. [자동 검색(#target-auto-discovery)된 타겟의 경우, 디렉토리 또는 파일 이름이 기본값이 됩니다.

이 설정은 [lib] 를 제외한 모든 타겟에 필수입니다.

path 필드

path 필드는 Cargo.toml 파일의 위치를 기준으로 크레이트의 소스가 어디에 있는지 지정합니다.

지정하지 않으면 타겟 이름을 기반으로 유추된 경로 가 사용됩니다.

test 필드

test 필드는 해당 타겟이 cargo test 에 의해 기본적으로 테스트되는지 여부를 나타냅니다. 라이브러리, 바이너리, 테스트 타겟의 기본값은 true 입니다.

참고: 예제는 컴파일이 계속 가능한지 확인하기 위해 기본적으로 cargo test 에 의해 빌드되지만, 기본적으로 테스트 되지는 않습니다. 예제에 대해 test = true 를 설정하면 테스트로도 빌드되며, 예제 내에 정의된 모든 #[test] 함수를 실행합니다.

doctest 필드

doctest 필드는 문서 예제 가 cargo test 에 의해 기본적으로 테스트되는지 여부를 나타냅니다. 이는 라이브러리에만 해당되며 다른 섹션에는 영향을 주지 않습니다. 라이브러리의 기본값은 true 입니다.

bench 필드

bench 필드는 해당 타겟이 cargo bench 에 의해 기본적으로 벤치마크되는지 여부를 나타냅니다. 라이브러리, 바이너리, 벤치마크 타겟의 기본값은 true 입니다.

doc 필드

doc 필드는 해당 타겟이 cargo doc 에 의해 생성되는 문서에 기본적으로 포함되는지 여부를 나타냅니다. 라이브러리와 바이너리의 기본값은 true 입니다.

참고: 바이너리 이름이 라이브러리 타겟 이름과 같으면 바이너리는 건너뜁니다.

plugin 필드

이 옵션은 사용이 중단되었으며 사용되지 않습니다.

proc-macro 필드

proc-macro 필드는 라이브러리가 절차적 매크로(procedural macro) (참조)임을 나타냅니다. 이는 [lib] 타겟에만 유효합니다.

harness 필드

harness 필드는 rustc 에 --test 플래그 를 전달할지 여부를 나타냅니다. 이 플래그를 전달하면 #[test] 속성이 붙은 테스트나 #[bench] 속성이 붙은 벤치마크를 수집하고 실행하는 드라이버인 libtest 라이브러리를 자동으로 포함합니다. 모든 타겟의 기본값은 true 입니다.

false 로 설정된 경우, 테스트와 벤치마크를 실행하기 위한 main() 함수를 직접 정의해야 할 책임이 사용자에게 있습니다.

하네스 활성화 여부와 관계없이, 테스트 시에는 cfg(test) 조건식 이 활성화됩니다.

crate-type 필드

crate-type 필드는 타겟에 의해 생성될 크레이트 타입 을 정의합니다. 문자열 배열 형식이므로 하나의 타겟에 대해 여러 크레이트 타입을 지정할 수 있습니다. 이 설정은 라이브러리와 예제 타겟에만 지정할 수 있습니다. 바이너리, 테스트, 벤치마크는 항상 “bin” 크레이트 타입입니다. 기본값은 다음과 같습니다.

타겟크레이트 타입
일반 라이브러리"lib"
절차적 매크로(Proc-macro) 라이브러리"proc-macro"
예제"bin"

사용 가능한 옵션은 bin, lib, rlib, dylib, cdylib, staticlib, proc-macro 입니다. 다양한 크레이트 타입에 대한 자세한 내용은 Rust 레퍼런스 매뉴얼 을 참조하세요.

required-features 필드

required-features 필드는 타겟을 빌드하는 데 필요한 기능(features) 을 지정합니다. 필요한 기능 중 하나라도 활성화되지 않으면 해당 타겟은 건너뜁니다. 이 설정은 [[bin]], [[bench]], [[test]], [[example]] 섹션에만 해당되며 [lib] 에는 영향을 주지 않습니다.

[features]
# ...
postgres = []
sqlite = []
tools = []

[[bin]]
name = "my-pg-tool"
required-features = ["postgres", "tools"]

edition 필드

The edition field defines the Rust edition the target will use. If not specified, it defaults to the edition field for the [package].

Note: This field is deprecated and will be removed in a future Edition

타겟 자동 검색

기본적으로 Cargo는 파일 시스템의 파일 레이아웃 을 기반으로 빌드할 타겟을 자동으로 결정합니다. 표준 디렉토리 레이아웃을 따르지 않는 추가 타겟을 더하려면 [lib], [[bin]], [[test]], [[bench]], [[example]] 과 같은 타겟 구성 테이블을 사용할 수 있습니다.

수동으로 구성된 타겟만 빌드되도록 타겟 자동 검색을 비활성화할 수 있습니다. [package] 섹션에서 autolib, autobins, autoexamples, autotests, autobenches 키를 false 로 설정하면 해당 타겟 유형의 자동 검색이 비활성화됩니다.

[package]
# ...
autolib = false
autobins = false
autoexamples = false
autotests = false
autobenches = false

자동 검색 비활성화는 특수한 상황에서만 필요합니다. 예를 들어 라이브러리에 bin 이라는 이름의 모듈 을 두고 싶은 경우, Cargo는 보통 bin 디렉토리의 모든 파일을 실행 파일로 컴파일하려고 하므로 문제가 발생합니다. 다음은 이러한 시나리오의 레이아웃 예시입니다.

├── Cargo.toml
└── src
    ├── lib.rs
    └── bin
        └── mod.rs

Cargo가 src/bin/mod.rs 를 실행 파일로 유추하지 못하도록 하려면, Cargo.toml 에서 autobins = false 를 설정하여 자동 검색을 비활성화하세요.

[package]
# …
autobins = false

참고: 2015 에디션 패키지의 경우, Cargo.toml 에 수동으로 정의된 타겟이 하나라도 있으면 자동 검색의 기본값은 false 입니다. 2018 에디션부터는 기본값이 항상 true 입니다.

MSRV: autobins, autoexamples, autotests, autobenches 는 1.27 버전부터 적용됨

MSRV: autolib 는 1.83 버전부터 적용됨
