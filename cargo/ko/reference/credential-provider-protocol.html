자격 증명 제공자 프로토콜

이 문서는 Cargo 자격 증명 제공자(credential provider)를 빌드하기 위한 정보를 설명합니다. 자격 증명 제공자를 설정하거나 사용하는 방법에 대한 정보는 레지스트리 인증(Registry Authentication)을 참조하세요.

외부 자격 증명 제공자를 사용할 때, Cargo는 한 줄의 JSON으로 전달되는 stdin/stdout 메시지를 사용하여 자격 증명 제공자와 통신합니다.

Cargo는 항상 자격 증명 제공자를 --cargo-plugin 인자와 함께 실행합니다. 이를 통해 자격 증명 제공자 실행 파일이 Cargo가 필요로 하는 것 이상의 추가 기능을 가질 수 있습니다. 추가 인자들은 JSON의 args 필드를 통해 포함됩니다.

JSON 메시지

이 문서의 JSON 메시지에는 가독성을 위해 줄바꿈이 추가되었습니다. 실제 메시지에는 줄바꿈이 포함되어서는 안 됩니다.

자격 증명 헬로(Credential hello)

  • 보낸 사람: 자격 증명 제공자
  • 용도: 프로세스 시작 시 지원되는 프로토콜을 식별하는 데 사용됨
{
    "v":[1]
}

Cargo가 보내는 요청에는 여기에 나열된 버전 중 하나로 설정된 v 필드가 포함됩니다. 만약 Cargo가 자격 증명 제공자가 제안한 버전 중 어느 것도 지원하지 않는다면, 에러를 발생시키고 자격 증명 프로세스를 종료합니다.

레지스트리 정보

  • 보낸 사람: Cargo. 그 자체로 하나의 메시지는 아닙니다. Cargo가 보내는 모든 메시지에 registry 필드로 포함됩니다.
{
    // 레지스트리의 인덱스 URL
    "index-url":"https://github.com/rust-lang/crates.io-index",
    // 설정에서의 레지스트리 이름 (선택 사항)
    "name": "crates-io",
    // 인증이 필요한 레지스트리에 접근을 시도하며 받은 HTTP 헤더 (선택 사항)
    "headers": ["WWW-Authenticate: cargo"]
}

로그인 요청

  • 보낸 사람: Cargo
  • 용도: 자격 증명 수집 및 저장
{
    // 프로토콜 버전
    "v":1,
    // 수행할 작업: 로그인
    "kind":"login",
    // 레지스트리 정보 (레지스트리 정보 참조)
    "registry":{"index-url":"sparse+https://registry-url/index/", "name": "my-registry"},
    // 표준 입력 혹은 커맨드 라인에서 사용자가 지정한 토큰 (선택 사항)
    "token": "<토큰 값>",
    // 사용자가 토큰을 받기 위해 방문할 수 있는 URL (선택 사항)
    "login-url": "http://registry-url/login",
    // 추가적인 커맨드 라인 인자 (선택 사항)
    "args":[]
}

token 필드가 설정되어 있다면, 자격 증명 제공자는 제공된 토큰을 사용해야 합니다. token 이 설정되어 있지 않다면, 자격 증명 제공자는 사용자에게 토큰 입력을 요청해야 합니다.

설정에 따라 자격 증명 제공자에게 전달될 수 있는 인자 외에도, cargo login 은 cargo login -- <추가 인자> 를 통해 추가적인 커맨드 라인 인자를 전달하는 것을 지원합니다. 이러한 추가 인자들은 Cargo 설정에 있는 인자들 뒤에 JSON의 args 필드에 포함됩니다.

읽기 요청

  • 보낸 사람: Cargo
  • 용도: 크레이트 정보를 읽기 위한 자격 증명 가져오기
{
    // 프로토콜 버전
    "v":1,
    // 요청 종류: 자격 증명 가져오기
    "kind":"get",
    // 수행할 작업: 크레이트 정보 읽기
    "operation":"read",
    // 레지스트리 정보 (레지스트리 정보 참조)
    "registry":{"index-url":"sparse+https://registry-url/index/", "name": "my-registry"},
    // 추가적인 커맨드 라인 인자 (선택 사항)
    "args":[]
}

배포 요청

  • 보낸 사람: Cargo
  • 용도: 크레이트 배포를 위한 자격 증명 가져오기
{
    // 프로토콜 버전
    "v":1,
    // 요청 종류: 자격 증명 가져오기
    "kind":"get",
    // 수행할 작업: 크레이트 배포
    "operation":"publish",
    // 크레이트 이름
    "name":"sample",
    // 크레이트 버전
    "vers":"0.1.0",
    // 크레이트 체크섬
    "cksum":"...",
    // 레지스트리 정보 (레지스트리 정보 참조)
    "registry":{"index-url":"sparse+https://registry-url/index/", "name": "my-registry"},
    // 추가적인 커맨드 라인 인자 (선택 사항)
    "args":[]
}

Get 성공 응답

  • 보낸 사람: 자격 증명 제공자
  • 용도: Cargo에 자격 증명 제공
{"Ok":{
    // 응답 종류: get 요청에 대한 응답임
    "kind":"get",
    // 레지스트리로 보낼 토큰
    "token":"...",
    // 캐시 제어. 다음 중 하나일 수 있습니다:
    // * "never": 캐시하지 않음
    // * "session": 현재 cargo 세션 동안 캐시
    // * "expires": 만료될 때까지 현재 cargo 세션 동안 캐시
    "cache":"expires",
    // 유닉스 타임스탬프 ("cache": "expires"인 경우에만 해당)
    "expiration":1693942857,
    // 토큰이 작업에 독립적인가요?
    "operation_independent":true
}}

token 은 Authorization HTTP 헤더의 값으로 레지스트리에 전송됩니다.

operation_independent 는 토큰이 서로 다른 작업(예: 배포 혹은 가져오기) 간에 캐시될 수 있는지 여부를 나타냅니다. 일반적으로 제공자가 특정 작업으로 범위가 한정된 토큰을 생성하려는 것이 아니라면 이는 true 여야 합니다.

로그인 성공 응답

  • 보낸 사람: 자격 증명 제공자
  • 용도: 로그인이 성공했음을 나타냄
{"Ok":{
    // 응답 종류: 로그인 요청에 대한 응답임
    "kind":"login"
}}

로그아웃 성공 응답

  • 보낸 사람: 자격 증명 제공자
  • 용도: 로그아웃이 성공했음을 나타냄
{"Ok":{
    // 응답 종류: 로그아웃 요청에 대한 응답임
    "kind":"logout"
}}

실패 응답 (URL 미지원)

  • 보낸 사람: 자격 증명 제공자
  • 용도: Cargo에 에러 정보 제공
{"Err":{
    "kind":"url-not-supported"
}}

자격 증명 제공자가 특정 레지스트리 URL만 처리하도록 설계되었으나 주어진 URL을 지원하지 않는 경우 전송됩니다. Cargo는 가능한 경우 다른 제공자를 시도합니다.

실패 응답 (찾을 수 없음)

  • 보낸 사람: 자격 증명 제공자
  • 용도: Cargo에 에러 정보 제공
{"Err":{
    // 에러: 제공자에서 자격 증명을 찾을 수 없습니다.
    "kind":"not-found"
}}

자격 증명을 찾을 수 없을 때 전송됩니다. 자격 증명을 사용할 수 없는 get 요청이나, 삭제할 항목을 찾을 수 없는 logout 요청 시에 발생할 수 있는 정상적인 응답입니다.

실패 응답 (지원되지 않는 작업)

  • 보낸 사람: 자격 증명 제공자
  • 용도: Cargo에 에러 정보 제공
{"Err":{
    // 에러: 제공자에서 자격 증명을 찾을 수 없습니다.
    "kind":"operation-not-supported"
}}

자격 증명 제공자가 요청된 작업을 지원하지 않을 때 전송됩니다. 예를 들어 제공자가 get 만 지원하는데 login 이 요청된 경우, 제공자는 이 에러로 응답해야 합니다.

실패 응답 (기타)

  • 보낸 사람: 자격 증명 제공자
  • 용도: Cargo에 에러 정보 제공
{"Err":{
    // 에러: 다른 이유로 실패했습니다.
    "kind":"other",
    // 표시될 에러 메시지 문자열
    "message": "자유 형식의 에러 메시지 문자열",
    // 에러에 대한 상세 원인 체인 (선택 사항)
    "caused-by": ["원인 1", "원인 2"]
}}

읽기 작업을 위한 토큰 요청 시의 통신 예시:

  1. Cargo가 자격 증명 프로세스를 생성하고 표준 입력 및 출력을 캡처합니다.

  2. 자격 증명 프로세스가 Cargo에 Hello 메시지를 보냅니다.

    { "v": [1] }

  3. Cargo가 자격 증명 프로세스에 CredentialRequest 메시지를 보냅니다 (가독성을 위해 줄바꿈이 추가됨).

    {
    "v": 1,
    "kind": "get",
    "operation": "read",
    "registry":{"index-url":"sparse+https://registry-url/index/"}
}

  4. 자격 증명 프로세스가 Cargo에 CredentialResponse를 보냅니다 (가독성을 위해 줄바꿈이 추가됨).

    {
    "token": "...",
    "cache": "session",
    "operation_independent": true
}

  5. Cargo가 자격 증명 제공자와의 stdin 파이프를 닫고 프로세스가 종료됩니다.

  6. Cargo는 이 레지스트리와 상호작용할 때 세션이 끝날 때까지(Cargo가 종료될 때까지) 이 토큰을 사용합니다.
