워크스페이스

_워크스페이스(workspace)_는 함께 관리되는 하나 이상의 패키지(구성원 패키지라고 함)의 모음입니다.

워크스페이스의 핵심 요점은 다음과 같습니다.

  • cargo check --workspace와 같이 공통 명령어를 모든 워크스페이스 구성원에 대해 실행할 수 있습니다.
  • 모든 패키지는 _워크스페이스 루트_에 있는 공통 Cargo.lock 파일을 공유합니다.
  • 모든 패키지는 공통 출력 디렉토리를 공유하며, 기본값은 _워크스페이스 루트_의 target 디렉토리입니다.
  • workspace.package를 사용하는 것처럼 패키지 메타데이터를 공유합니다.
  • Cargo.toml의 [patch], [replace], [profile.*] 섹션은 루트 매니페스트에서만 인식되며 구성원 크레이트의 매니페스트에서는 무시됩니다.

워크스페이스의 루트 Cargo.toml은 다음 섹션을 지원합니다.

  • [workspace] — 워크스페이스를 정의합니다.
    • resolver — 사용할 의존성 해결사를 설정합니다.
    • members — 워크스페이스에 포함할 패키지입니다.
    • exclude — 워크스페이스에서 제외할 패키지입니다.
    • default-members — 특정 패키지가 선택되지 않았을 때 작동할 패키지입니다.
    • package — 패키지에서 상속할 키입니다.
    • dependencies — 패키지 의존성에서 상속할 키입니다.
    • lints — 패키지 린트에서 상속할 키입니다.
    • metadata — 외부 도구를 위한 추가 설정입니다.
  • [patch] — 의존성을 재정의합니다.
  • [replace] — 의존성을 재정의합니다 (사용 중단됨).
  • [profile] — 컴파일러 설정 및 최적화입니다.

[workspace] 섹션

워크스페이스를 만들려면 Cargo.toml에 [workspace] 테이블을 추가합니다.

[workspace]
# ...

최소한 워크스페이스는 루트 패키지나 가상 매니페스트 중 하나를 통해 구성원을 가져야 합니다.

루트 패키지

[workspace] 섹션이 이미 [package]를 정의한 Cargo.toml에 추가된 경우, 해당 패키지는 워크스페이스의 _루트 패키지_입니다. _워크스페이스 루트_는 워크스페이스의 Cargo.toml이 위치한 디렉토리입니다.

[workspace]

[package]
name = "hello_world" # 패키지 이름
version = "0.1.0"    # 현재 버전 (semver 준수)

가상 워크스페이스

또는 [workspace] 섹션은 있지만 [package] 섹션이 없는 Cargo.toml 파일을 만들 수 있습니다. 이를 _가상 매니페스트(virtual manifest)_라고 합니다. 이는 보통 “주요” 패키지가 없거나 모든 패키지를 별도의 디렉토리로 구성하고 싶을 때 유용합니다.

# [프로젝트_디렉토리]/Cargo.toml
[workspace]
members = ["hello_world"]
resolver = "3"

# [프로젝트_디렉토리]/hello_world/Cargo.toml
[package]
name = "hello_world" # 패키지 이름
version = "0.1.0"    # 현재 버전 (semver 준수)
edition = "2024"     # 에디션 (워크스페이스에서 사용되는 리졸버에는 영향을 주지 않음)

루트 패키지 없이 워크스페이스를 구성하면 다음과 같습니다.

  • 가상 워크스페이스는 리졸버 버전을 유추할 수 있는 package.edition이 없으므로 resolver를 명시적으로 설정해야 합니다.
  • 워크스페이스 루트에서 실행되는 명령어는 기본적으로 모든 워크스페이스 구성원에 대해 실행됩니다. default-members를 참조하세요.

members 및 exclude 필드

members 및 exclude 필드는 워크스페이스의 구성원인 패키지를 정의합니다.

[workspace]
members = ["member1", "path/to/member2", "crates/*"]
exclude = ["crates/foo", "path/to/other"]

워크스페이스 디렉토리에 있는 모든 경로(path) 의존성은 자동으로 구성원이 됩니다. 추가 구성원은 members 키를 사용하여 나열할 수 있으며, 이는 Cargo.toml 파일이 있는 디렉토리를 포함하는 문자열 배열이어야 합니다.

members 목록은 * 및 ?와 같은 일반적인 파일 이름 글로브 패턴을 사용하여 여러 경로를 일치시키는 글로브(globs)를 지원합니다.

exclude 키는 경로가 워크스페이스에 포함되는 것을 방지하는 데 사용할 수 있습니다. 이는 일부 경로 의존성이 워크스페이스에 포함되는 것을 전혀 원하지 않거나, 글로브 패턴을 사용 중인데 특정 디렉토리를 제거하고 싶을 때 유용할 수 있습니다.

워크스페이스 내의 하위 디렉토리에 있을 때, Cargo는 사용할 워크스페이스를 결정하기 위해 상위 디렉토리에서 [workspace] 정의가 있는 Cargo.toml 파일을 자동으로 검색합니다. 구성원 크레이트의 package.workspace 매니페스트 키를 사용하여 워크스페이스 루트를 가리킴으로써 이 자동 검색을 무시할 수 있습니다. 수동 설정은 구성원이 워크스페이스 루트의 하위 디렉토리에 있지 않은 경우 유용할 수 있습니다.

패키지 선택

워크스페이스에서 cargo build와 같은 패키지 관련 Cargo 명령어는 -p / --package 또는 --workspace 명령줄 플래그를 사용하여 작동할 패키지를 결정할 수 있습니다. 해당 플래그가 지정되지 않은 경우, Cargo는 현재 작업 디렉토리의 패키지를 사용합니다. 그러나 현재 디렉토리가 워크스페이스 루트인 경우 default-members가 사용됩니다.

default-members 필드

default-members 필드는 워크스페이스 루트에 있고 패키지 선택 플래그를 사용하지 않을 때 작동할 구성원의 경로를 지정합니다.

[workspace]
members = ["path/to/member1", "path/to/member2", "path/to/member3/*"]
default-members = ["path/to/member2", "path/to/member3/foo"]

참고: 루트 패키지가 있을 때는 --package 및 --workspace 플래그를 사용해야만 해당 패키지에 대해 작업할 수 있습니다.

지정되지 않은 경우 루트 패키지가 사용됩니다. 가상 워크스페이스의 경우 모든 구성원이 사용됩니다 (명령줄에서 --workspace가 지정된 것과 같음).

package 테이블

workspace.package 테이블은 워크스페이스의 구성원이 상속할 수 있는 키를 정의하는 곳입니다. 이러한 키는 구성원 패키지에서 {key}.workspace = true로 정의하여 상속할 수 있습니다.

지원되는 키는 다음과 같습니다.
authorscategories
descriptiondocumentation
editionexclude
homepageinclude
keywordslicense
license-filepublish
readmerepository
rust-versionversion
  • license-file 및 readme는 워크스페이스 루트를 기준으로 합니다.
  • include 및 exclude는 사용자의 패키지 루트를 기준으로 합니다.

예시:

# [프로젝트_디렉토리]/Cargo.toml
[workspace]
members = ["bar"]

[workspace.package]
version = "1.2.3"
authors = ["Nice Folks"]
description = "내 패키지에 대한 짧은 설명"
documentation = "https://example.com/bar"

# [프로젝트_디렉토리]/bar/Cargo.toml
[package]
name = "bar"
version.workspace = true
authors.workspace = true
description.workspace = true
documentation.workspace = true

MSRV: 1.64 이상 필요

dependencies 테이블

workspace.dependencies 테이블은 워크스페이스의 구성원이 상속할 의존성을 정의하는 곳입니다.

워크스페이스 의존성을 지정하는 것은 다음을 제외하고 패키지 의존성과 유사합니다.

  • 이 테이블의 의존성은 optional로 선언될 수 없습니다.
  • 이 테이블에 선언된 기능(features)은 [dependencies]의 features에 추가됩니다.

그런 다음 워크스페이스 의존성을 패키지 의존성으로 상속할 수 있습니다.

예시:

# [프로젝트_디렉토리]/Cargo.toml
[workspace]
members = ["bar"]

[workspace.dependencies]
cc = "1.0.73"
rand = "0.8.5"
regex = { version = "1.6.0", default-features = false, features = ["std"] }

# [프로젝트_디렉토리]/bar/Cargo.toml
[package]
name = "bar"
version = "0.2.0"

[dependencies]
regex = { workspace = true, features = ["unicode"] }

[build-dependencies]
cc.workspace = true

[dev-dependencies]
rand.workspace = true

MSRV: 1.64 이상 필요

lints 테이블

workspace.lints 테이블은 워크스페이스의 구성원이 상속할 린트 구성을 정의하는 곳입니다.

워크스페이스 린트 구성을 지정하는 것은 패키지 린트와 유사합니다.

예시:

# [프로젝트_디렉토리]/Cargo.toml
[workspace]
members = ["crates/*"]

[workspace.lints.rust]
unsafe_code = "forbid"

# [프로젝트_디렉토리]/crates/bar/Cargo.toml
[package]
name = "bar"
version = "0.1.0"

[lints]
workspace = true

MSRV: 1.74 버전부터 적용됨

metadata 테이블

workspace.metadata 테이블은 Cargo에 의해 무시되며 경고가 발생하지 않습니다. 이 섹션은 Cargo.toml에 워크스페이스 구성을 저장하려는 도구에 사용할 수 있습니다. 예를 들어 다음과 같습니다.

[workspace]
members = ["member1", "member2"]

[workspace.metadata.webcontents]
root = "웹프로젝트/경로"
tool = ["npm", "run", "build"]
# ...

패키지 수준에도 package.metadata라는 유사한 테이블 세트가 있습니다. Cargo는 이러한 테이블의 내용에 대한 형식을 지정하지 않지만, 외부 도구에서 관련 도구에 적합하다면 package.metadata에 데이터가 없는 경우 workspace.metadata의 데이터를 참조하는 등 일관된 방식으로 사용하도록 제안됩니다.
