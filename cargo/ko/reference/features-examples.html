The Cargo Book

기능 예제

다음은 실제 사례에서 기능이 어떻게 작동하는지 보여주는 몇 가지 예시입니다.

빌드 시간 및 파일 크기 최소화

일부 패키지는 기능을 사용하여, 해당 기능이 활성화되지 않았을 때 크레이트의 크기를 줄이고 컴파일 시간을 단축합니다. 몇 가지 예시는 다음과 같습니다.

  • syn은 Rust 코드를 파싱하는 데 널리 사용되는 크레이트입니다. 매우 널리 사용되기 때문에, 많은 프로젝트에 영향을 미치므로 컴파일 시간을 줄이는 것이 도움이 됩니다. 포함되는 코드의 양을 최소화하는 데 사용할 수 있는 명확하게 문서화된 기능 목록을 제공합니다.
  • regex는 문서화가 잘 된 여러 기능을 가지고 있습니다. 유니코드 지원을 제외하면 일부 거대한 테이블을 제거할 수 있어 결과 파일 크기를 줄일 수 있습니다.
  • winapi는 지원하는 Windows API 바인딩을 제한하는 수많은 기능을 가지고 있습니다.
  • web-sys는 winapi와 유사한 또 다른 예시로, 기능을 사용하여 제한할 수 있는 방대한 범위의 API 바인딩을 제공합니다.

동작 확장

serde_json 패키지는 키가 삽입된 순서를 유지하도록 JSON 맵의 동작을 변경하는 preserve_order 기능을 가지고 있습니다. 이 기능은 새로운 동작을 구현하기 위해 선택적 의존성인 indexmap을 활성화한다는 점에 주목하세요.

이와 같이 동작을 변경할 때는 해당 변경 사항이 SemVer 호환(SemVer compatible)되는지 주의 깊게 확인해야 합니다. 즉, 기능을 활성화했을 때, 해당 기능이 꺼진 상태에서 정상적으로 빌드되던 코드가 깨져서는 안 됩니다.

no_std 지원

일부 패키지는 no_std와 std 환경을 모두 지원하고 싶어 합니다. 이는 임베디드 및 리소스가 제한된 플랫폼을 지원하면서도, 전체 표준 라이브러리를 지원하는 플랫폼에 대해서는 확장된 기능을 허용하는 데 유용합니다.

wasm-bindgen 패키지는 기본적으로 활성화되는 std 기능을 정의합니다. 라이브러리 상단에서 무조건적으로 no_std 속성을 활성화합니다. 이를 통해 std와 std 프렐류드(std prelude)가 자동으로 스코프에 들어오지 않도록 보장합니다. 그런 다음 코드의 여러 곳에서 #[cfg(feature = "std")] 속성을 사용하여 std를 필요로 하는 추가 기능을 조건부로 활성화합니다.

의존성 기능 재내보내기

의존성의 기능을 재내보내는 것이 편리할 때가 있습니다. 이를 통해 해당 크레이트에 의존하는 사용자가 의존성을 직접 지정하지 않고도 해당 기능들을 제어할 수 있게 됩니다. 예를 들어, regex는 regex_syntax 패키지의 기능들을 재내보냅니다. regex 사용자는 regex_syntax 패키지에 대해 알 필요가 없지만, 여전히 그 안에 포함된 기능들을 사용할 수 있습니다.

C 라이브러리 벤더링

일부 패키지는 일반적인 C 라이브러리에 대한 바인딩을 제공합니다(가끔 “sys” 크레이트라고 불림). 이러한 패키지들은 시스템에 설치된 C 라이브러리를 사용할지, 아니면 소스에서 직접 빌드할지 선택권을 주기도 합니다. 예를 들어, openssl 패키지는 openssl-sys의 대응하는 vendored 기능을 활성화하는 vendored 기능을 가지고 있습니다. openssl-sys 빌드 스크립트에는 시스템의 라이브러리를 사용하는 대신 로컬에 복사된 OpenSSL 소스 코드에서 빌드하도록 하는 조건부 로직이 포함되어 있습니다.

curl-sys 패키지는 static-curl 기능을 통해 libcurl을 소스에서 빌드하는 또 다른 예입니다. 또한 force-system-lib-on-osx 기능을 통해 static-curl 설정을 무시하고 시스템 libcurl을 사용하도록 강제할 수도 있습니다.

기능 우선순위

일부 패키지는 상호 배타적인 기능을 가질 수 있습니다. 이를 처리하는 한 가지 방법은 한 기능을 다른 기능보다 우선시하는 것입니다. log 패키지가 그 예입니다. 이 패키지는 여기에 설명된 대로 컴파일 타임에 최대 로깅 레벨을 선택하기 위한 여러 기능을 제공합니다. 이 패키지는 cfg-if를 사용하여 우선순위를 선택합니다. 여러 기능이 활성화된 경우, 더 높은 “최대(max)” 레벨이 낮은 레벨보다 우선 적용됩니다.

절차적 매크로 동반 패키지

일부 패키지는 그와 밀접하게 연관된 절차적 매크로(proc-macro)를 가지고 있습니다. 하지만 모든 사용자가 해당 매크로를 사용할 필요는 없을 것입니다. 절차적 매크로를 선택적 의존성(optional-dependency)으로 만들면, 포함 여부를 편리하게 선택할 수 있게 됩니다. 이는 때때로 절차적 매크로의 버전이 부모 패키지와 동기화되어야 할 때 특히 유용한데, 사용자가 두 의존성을 모두 직접 지정하고 동기화 상태를 유지하도록 강요하고 싶지 않기 때문입니다.

한 예로 serde가 있는데, 이 패키지는 serde_derive 절차적 매크로를 활성화하는 derive 기능을 가지고 있습니다. serde_derive 크레이트는 serde와 매우 긴밀하게 연결되어 있으므로, 서로의 동기화를 보장하기 위해 일치 버전 요구 사항(equals version requirement)을 사용합니다.

나이틀리 전용 기능

일부 패키지는 Rust 나이틀리 채널에서만 사용 가능한 API나 언어 기능을 실험해보고 싶어 할 수 있습니다. 하지만 사용자들에게도 나이틀리 채널 사용을 강제하고 싶지는 않을 것입니다. 한 예로 wasm-bindgen이 있는데, 이 패키지는 이 글을 쓰는 시점에 나이틀리 채널에서만 사용 가능한 Unsize 마커 트레이트를 사용하는 확장 API를 활성화하는 nightly 기능을 가지고 있습니다.

크레이트 루트에서 나이틀리 기능을 활성화하기 위해 cfg_attr을 사용함에 주목하세요. feature 속성은 Cargo 기능과 무관하며, 실험적인 언어 기능을 사용하기 위해 명시적으로 선택하는 데 사용된다는 점을 기억하세요.

rand 패키지의 simd_support 기능은 나이틀리 채널에서만 빌드되는 의존성에 의존하는 또 다른 예입니다.

실험적인 기능

일부 패키지는 해당 API의 안정성을 보장하지 않으면서 새로운 기능을 실험해보고 싶어 할 수 있습니다. 이러한 기능들은 보통 실험적이라고 문서화되며, 따라서 미래에(심지어 마이너 릴리스 중에도) 변경되거나 깨질 수 있습니다. 한 예로 async-std 패키지가 있는데, 이 패키지는 사람들이 선택하여 사용할 수 있지만 아직 완전히 신뢰하기에는 준비되지 않았을 수 있는 새로운 API들을 제한하는 unstable 기능을 가지고 있습니다.
