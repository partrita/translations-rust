The Cargo Book

Rust 버전

rust-version 필드는 패키지에서 지원하는 Rust 툴체인 버전을 Cargo에 알려주는 선택적 키입니다.

[package]
# ...
rust-version = "1.56"

Rust 버전은 최소 하나 이상의 구성 요소가 포함된 순수 버전 번호여야 합니다. SemVer 연산자나 프리릴리스 식별자를 포함할 수 없습니다. Rust 버전을 확인할 때 -nightly 와 같은 컴파일러 프리릴리스 식별자는 무시됩니다.

MSRV: 1.56 버전부터 적용됨

사용 사례

진단:

When your package is compiled on an unsupported toolchain, Cargo will report that as an error to the user. This makes the support expectations clear and avoids reporting a less direct diagnostic like invalid syntax or missing functionality in the standard library. This affects all Cargo targets in the package, including binaries, examples, test suites, benchmarks, etc. A user can opt-in to an unsupported build of a package with the --ignore-rust-version flag.

개발 지원:

cargo add 는 의존성의 버전 요구 사항을 사용자의 rust-version 과 호환되는 최신 버전으로 자동으로 선택합니다. 그것이 최신 버전이 아닌 경우, cargo add 는 사용자에게 이를 알려 사용자가 해당 버전을 유지할지 아니면 rust-version 을 업데이트할지 선택할 수 있게 합니다.

리졸버 는 의존성을 선택할 때 Rust 버전을 고려할 수 있습니다.

Other tools may also take advantage of it, like cargo clippy’s incompatible_msrv lint.

참고: rust-version 은 --ignore-rust-version 옵션을 사용하여 무시될 수 있습니다.

지원 기대 사항

이는 일반적인 기대 사항입니다. 일부 패키지는 이를 따르지 않을 때 별도로 문서화할 수 있습니다.

완전성:

바이너리 및 API를 포함한 모든 기능은 모든 기능(feature) 하에서 지원되는 Rust 버전에서 사용 가능해야 합니다.

검증:

패키지의 기능은 자동화된 테스트를 포함하여 지원되는 Rust 버전에서 검증되어야 합니다. Rust 버전 CI 가이드 도 참조하세요.

패치 가능성:

라이선스가 허용하는 경우, 사용자는 패키지의 포크를 사용하여 로컬 의존성을 재정의 할 수 있습니다. 이 상황에서 Cargo는 패치된 의존성을 위해 전체 워크스페이스를 로드할 수 있으며, 워크스페이스의 다른 패키지들이 서로 다른 지원 Rust 버전을 가지고 있더라도 지원되는 Rust 버전에서 작동해야 합니다.

의존성 지원:

위 사항을 지원하기 위해, 각 의존성의 버전 요구 사항이 사용자의 rust-version 과 호환되는 버전을 적어도 하나 이상 지원할 것으로 기대됩니다. 하지만 의존성 명세에서 사용자의 rust-version 과 호환되지 않는 버전을 반드시 제외해야 하는 것은 아닙니다. 사실 두 가지 모두를 지원함으로써 이전 Rust 버전을 사용하는 사용자와 그렇지 않은 사용자 사이의 요구 사항을 균형 있게 맞출 수 있습니다.

Rust 버전 설정 및 업데이트

어떤 Rust 버전을 지원할지는 다음과 같은 사항들 사이의 절충안입니다.

  • Rust 툴체인이나 그 의존성의 최신 기능을 사용하지 못하는 유지 관리자의 비용
  • 툴체인의 새로운 기능을 사용하는 패키지를 통해 혜택을 볼 수 있는 사용자의 비용 (예: 폴필(polyfill) 대신 표준 라이브러리 기능을 사용하여 빌드 시간 단축)
  • 이전 Rust 버전을 사용하는 사용자들에 대한 패키지 가용성

참고: rust-version 변경 은 마이너 비호환성으로 간주됩니다.

권장 사항: 어떤 Rust 버전을 지원할지, 그리고 언제 변경할지에 대한 정책을 선택하세요. 이를 통해 사용자가 자신의 정책과 비교하고, 호환되지 않는 경우 일반적인 개선 사항의 손실이나 수정되지 않을 버그의 위험을 감수할지 결정할 수 있게 하세요.

지원하기 가장 간단한 정책은 항상 최신 Rust 버전을 사용하는 것입니다.

위험 프로필에 따라, 다음으로 간단한 방법은 이전 Rust 버전을 지원하는 패키지의 이전 주 버전이나 부 버전을 계속 지원하는 것입니다.

지원되는 Rust 버전 선택

패키지 사용자는 다음과 같은 기준으로 지원되는 Rust 버전을 추적할 가능성이 높습니다.

  • Rust 툴체인 벤더의 지원 정책 (예: Rust 프로젝트 또는 리눅스 배포판)
    • 참고: Rust 프로젝트는 최신 버전에 대해서만 버그 수정 및 보안 업데이트를 제공합니다.
  • 새로운 툴체인으로 패키지를 재검증하기 위한 고정된 일정 (예: 매년 첫 번째 릴리스, 매 5번째 릴리스 등)

또한 사용자가 새로운 Rust 버전을 즉시 사용하기보다는 이를 인지하고 재검증할 시간이 필요하거나, 일정이 정확히 일치하지 않을 수도 있습니다.

버전 정책 예시:

  • “N-2”: 최신 버전에서 2회 릴리스만큼의 업데이트 유예 기간을 둠을 의미함
  • 모든 짝수 릴리스마다 2회 릴리스의 업데이트 유예 기간을 둠
  • 이번 역년(calendar year)의 모든 버전에 대해 1년의 업데이트 유예 기간을 둠

참고: 현재 프로젝트와 호환되는 최소 rust-version 을 찾으려면 cargo-msrv 와 같은 타사 도구를 사용할 수 있습니다.

업데이트 타임라인

정책에 따라 더 이상 특정 Rust 버전을 지원할 필요가 없는 경우, 즉시 또는 필요할 때 rust-version 을 업데이트할 수 있습니다.

rust-version 이 정책에서 벗어나도록 허용함으로써 사용자에게 업그레이드를 위한 유예 기간을 더 많이 제공할 수 있습니다. 그러나 이는 사용자가 추적하는 Rust 버전과 맞추기 위해 신뢰하기에는 너무 예측 불가능합니다.

rust-version 이 지정된 정책에서 멀어질수록 사용자가 의도하지 않은 정책을 추측할 가능성이 높아지며, 이는 기대에 미치지 못하는 결과로 인한 불만으로 이어질 수 있습니다.

차이를 허용할 때, 지원되는 버전을 중단하기 위해 무엇이 “충분히 정당화“되는지에 대한 문제가 발생합니다. 사람마다 합리적으로 다른 정당성을 제시할 수 있으며, 그러한 논의 과정은 관련 당사자들에게 좌절감을 줄 수 있습니다. 이는 갈등을 피하고 싶어 하는 사람들의 의욕을 꺾을 수 있으며, 특히 질문을 제기할 입장이 아니라고 느끼거나 갈등이 자신의 변경 사항이 병합될 기회를 해칠 수 있다고 생각하는 신규 또는 비정기 기여자들에게 그렇습니다.

워크스페이스 내의 여러 정책

Cargo는 하나의 워크스페이스 내에서 여러 정책을 지원할 수 있도록 허용합니다.

특정 Rust 버전에서 특정 패키지를 검증하는 것은 복잡해질 수 있습니다. cargo-hack 과 같은 도구가 도움이 될 수 있습니다.

여러 정책에서 공유되는 의존성의 경우, Cargo가 SemVer 호환 버전을 통합 하므로 가장 낮은 공통 버전을 사용해야 하며, 이로 인해 더 높은 rust-version 을 가진 워크스페이스 구성원이 공유 의존성의 기능에 접근하는 것이 제한될 수 있습니다.

사용자가 워크스페이스 구성원 중 하나에 대한 의존성을 패치할 수 있도록 하려면 워크스페이스의 모든 패키지가 워크스페이스에서 지원하는 가장 오래된 Rust 버전에서 로드될 수 있어야 합니다.

incompatible-rust-versions = "fallback" 을 사용할 때, 한 패키지의 Rust 버전이 다른 Rust 버전을 가진 다른 패키지에 대해 선택된 의존성 버전에 영향을 줄 수 있습니다. 자세한 내용은 리졸버 장을 참조하세요.

하나 이상의 정책

이전 Rust 버전을 지원하는 것의 단점을 완화하는 한 가지 방법은 계속 지원하는 패키지의 이전 주 또는 부 버전에 정책을 적용하는 것입니다. 해당 주 또는 부 버전의 릴리스 브랜치와 비교하여 개발 브랜치가 어떤 Rust 버전을 지원하는지에 대한 정책이 여전히 필요할 것입니다.

“필요할 때“만 개발 브랜치를 업데이트하면 지원되는 릴리스 브랜치의 수를 줄이는 데 도움이 될 수 있습니다.

이러한 릴리스 브랜치로 무엇을 백포트(backport)할 수 있는지의 문제가 있습니다. 부 버전 사이에 새로운 기능을 백포트하면 다음 가용 버전에는 해당 기능이 누락되어 SemVer를 위반하는 파괴적 변경(breaking change)으로 간주될 수 있습니다. 변경 사항을 백포트하는 것에는 버그가 발생할 위험도 따릅니다.

이전 버전을 지원하는 데는 비용이 듭니다. 이 비용은 패키지 내 버그의 위험과 영향, 그리고 백포트에 허용되는 수준에 따라 달라집니다. 필요에 따라 릴리스 브랜치를 생성하고 백포트 부담을 커뮤니티에 맡기는 것이 이러한 비용의 균형을 맞추는 방법입니다.

의존성 관리 도구가 최신이 아닌 버전이 여전히 지원된다는 것을 보고할 방법이 아직 없으므로, 문서에서 이를 인지하는 책임은 사용자에게 전가됩니다.

예를 들어 Rust 버전 지원 정책은 다음과 같을 수 있습니다.

  • 개발 브랜치는 Rust 프로젝트의 최신 안정 릴리스를 추적하며, 필요할 때 업데이트됨
    • rust-version 을 변경할 때 부 버전이 상향됨
  • 프로젝트는 이번 연도의 모든 버전을 지원하며, 추가로 1년의 유예 기간을 둠
    • 지원되는 Rust 버전을 지원하는 마지막 부 버전은 커뮤니티에서 제공하는 버그 수정을 받음
    • 수정 사항은 개발 브랜치와 필요한 지원 Rust 버전 사이의 지원되는 모든 부 릴리스로 백포트되어야 함
