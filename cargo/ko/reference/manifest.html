The Cargo Book

매니페스트 형식

각 패키지의 Cargo.toml 파일은 매니페스트 라고 합니다. TOML 형식으로 작성됩니다. 패키지를 컴파일하는 데 필요한 메타데이터를 포함합니다. cargo가 매니페스트 파일을 찾는 방법에 대한 자세한 내용은 cargo locate-project 섹션을 확인하세요.

모든 매니페스트 파일은 다음 섹션으로 구성됩니다:

  • cargo-features — 불안정한, 나이틀리 전용 기능입니다.
  • [package] — 패키지를 정의합니다.
    • name — 패키지의 이름입니다.
    • version — 패키지의 버전입니다.
    • authors — 패키지의 저자입니다.
    • edition — Rust 에디션입니다.
    • rust-version — 최소 지원 Rust 버전입니다.
    • description — 패키지에 대한 설명입니다.
    • documentation — 패키지 문서의 URL입니다.
    • readme — 패키지의 README 파일 경로입니다.
    • homepage — 패키지 홈페이지의 URL입니다.
    • repository — 패키지 소스 저장소의 URL입니다.
    • license — 패키지 라이선스입니다.
    • license-file — 라이선스 텍스트 파일의 경로입니다.
    • keywords — 패키지의 키워드입니다.
    • categories — 패키지의 카테고리입니다.
    • workspace — 패키지의 워크스페이스 경로입니다.
    • build — 패키지 빌드 스크립트의 경로입니다.
    • links — 패키지가 연결되는 네이티브 라이브러리의 이름입니다.
    • exclude — 게시할 때 제외할 파일입니다.
    • include — 게시할 때 포함할 파일입니다.
    • publish — 패키지 게시를 방지하는 데 사용할 수 있습니다.
    • metadata — 외부 도구를 위한 추가 설정입니다.
    • default-run — cargo run 이 실행할 기본 바이너리입니다.
    • autolib — 라이브러리 자동 검색을 비활성화합니다.
    • autobins — 바이너리 자동 검색을 비활성화합니다.
    • autoexamples — 예제 자동 검색을 비활성화합니다.
    • autotests — 테스트 자동 검색을 비활성화합니다.
    • autobenches — 벤치 자동 검색을 비활성화합니다.
    • resolver — 사용할 의존성 해결사를 설정합니다.
  • 타겟 테이블: (설정은 구성 참조)
    • [lib] — 라이브러리 타겟 설정입니다.
    • [[bin]] — 바이너리 타겟 설정입니다.
    • [[example]] — 예제 타겟 설정입니다.
    • [[test]] — 테스트 타겟 설정입니다.
    • [[bench]] — 벤치마크 타겟 설정입니다.
  • 의존성 테이블:
    • [dependencies] — 패키지 라이브러리 의존성입니다.
    • [dev-dependencies] — 예제, 테스트, 벤치마크를 위한 의존성입니다.
    • [build-dependencies] — 빌드 스크립트를 위한 의존성입니다.
    • [target] — 플랫폼별 의존성입니다.
  • [badges] — 레지스트리에 표시할 배지입니다.
  • [features] — 조건부 컴파일 기능입니다.
  • [lints] — 이 패키지를 위한 린터를 구성합니다.
  • [hints] — Provide hints for compiling this package.
  • [patch] — 의존성을 재정의합니다.
  • [replace] — 의존성을 재정의합니다 (사용 중단됨).
  • [profile] — 컴파일러 설정 및 최적화입니다.
  • [workspace] — 워크스페이스 정의입니다.

[package] 섹션

Cargo.toml 의 첫 번째 섹션은 [package] 입니다.

[package]
name = "hello_world" # 패키지 이름
version = "0.1.0"    # 현재 버전 (semver 준수)

Cargo가 요구하는 유일한 필드는 name 입니다. 레지스트리에 게시하는 경우 레지스트리에서 추가 필드를 요구할 수 있습니다. 아래 참고 사항과 게시 장 에서 crates.io 게시 요건을 확인하세요.

name 필드

패키지 이름은 패키지를 참조하는 데 사용되는 식별자입니다. 다른 패키지에서 의존성으로 나열될 때 사용되며, 유추된 lib 및 bin 타겟의 기본 이름으로도 사용됩니다.

이름은 영문자 및 숫자 문자 또는 -, _ 만 사용해야 하며 비워 둘 수 없습니다.

cargo new 및 cargo init 은 패키지 이름에 대해 유효한 Rust 식별자여야 하고 키워드가 아니어야 하는 등의 추가 제한을 둡니다. crates.io 는 다음과 같은 더 많은 제한을 둡니다.

  • ASCII 문자만 허용됩니다.
  • 예약된 이름을 사용하지 마세요.
  • “nul“과 같은 특수한 Windows 이름을 사용하지 마세요.
  • 최대 길이는 64자입니다.

version 필드

version 필드는 SemVer 사양에 따라 형식이 지정됩니다.

버전은 주 버전(major), 부 버전(minor), 패치 버전(patch)의 세 가지 숫자 부분으로 구성되어야 합니다.

대시 뒤에 1.0.0-alpha 와 같은 프리릴리스(pre-release) 부분을 추가할 수 있습니다. 프리릴리스 부분은 개별 구성 요소를 구분하기 위해 마침표로 나눌 수 있습니다. 숫자 구성 요소는 숫자 비교를 사용하고 다른 모든 구성 요소는 사전식 비교를 사용합니다. 예를 들어, 1.0.0-alpha.11 은 1.0.0-alpha.4 보다 높습니다.

플러스 뒤에 1.0.0+21AF26D3 과 같은 메타데이터 부분을 추가할 수 있습니다. 이는 정보 제공 목적으로만 사용되며 일반적으로 Cargo에서는 무시됩니다.

Cargo는 시맨틱 버저닝(Semantic Versioning) 개념을 기반으로 하므로, 가장 왼쪽의 0이 아닌 주/부/패치 구성 요소가 같으면 버전이 호환 되는 것으로 간주됩니다. Cargo가 의존성 해결을 위해 버전을 사용하는 방법에 대한 자세한 내용은 리졸버(Resolver) 장을 참조하세요.

이 필드는 선택 사항이며 기본값은 0.0.0 입니다. 패키지를 게시하려면 이 필드가 필수입니다.

MSRV: 1.75 이전에는 이 필드가 필수였습니다.

authors 필드

경고: 이 필드는 사용 중단되었습니다.

선택 사항인 authors 필드에는 패키지의 “저자“로 간주되는 사람이나 조직을 배열로 나열합니다. 각 저자 항목 끝의 꺽쇠 괄호 안에 선택 사항으로 이메일 주소를 포함할 수 있습니다.

[package]
# ...
authors = ["Graydon Hoare", "Fnu Lnu <no-reply@rust-lang.org>"]

이 필드는 패키지 메타데이터와 하위 호환성을 위해 build.rs 내의 CARGO_PKG_AUTHORS 환경 변수에 나타납니다.

edition 필드

edition 키는 패키지가 어떤 Rust 에디션 으로 컴파일되는지에 영향을 미치는 선택 사항 키입니다. [package] 에서 edition 키를 설정하면 테스트 모음, 벤치마크, 바이너리, 예제 등을 포함하여 패키지의 모든 타겟/크레이트에 영향을 미칩니다.

[package]
# ...
edition = '2024'

대부분의 매니페스트는 cargo new 에 의해 최신 안정 에디션으로 edition 필드가 자동 채워집니다. 현재 cargo new 는 기본적으로 2024 에디션으로 매니페스트를 생성합니다.

Cargo.toml 에 edition 필드가 없으면 하위 호환성을 위해 2015 에디션으로 간주됩니다. cargo new 로 생성된 모든 매니페스트는 edition 이 더 최신 값으로 명시적으로 지정되므로 이 과거 폴백을 사용하지 않습니다.

rust-version 필드

rust-version 필드는 패키지에 대해 지원하는 Rust 툴체인 버전을 Cargo에 알려줍니다. 자세한 내용은 Rust 버전 장 을 참조하세요.

description 필드

설명은 패키지에 대한 짧은 홍보 문구입니다. crates.io 는 패키지와 함께 이를 표시합니다. 마크다운이 아닌 일반 텍스트여야 합니다.

[package]
# ...
description = "내 패키지에 대한 짧은 설명"

참고: crates.io 에서는 description 설정이 필수입니다.

documentation 필드

documentation 필드는 크레이트 문서를 호스팅하는 웹사이트의 URL을 지정합니다. 매니페스트 파일에 URL이 지정되지 않은 경우, 문서가 빌드되어 사용 가능해지면 crates.io 가 자동으로 크레이트를 해당 docs.rs 페이지에 연결합니다 (docs.rs 큐 참조).

[package]
# ...
documentation = "https://docs.rs/bitflags"

readme 필드

readme 필드는 패키지에 대한 일반적인 정보가 포함된 패키지 루트의 파일 경로(Cargo.toml 기준 상대 경로)여야 합니다. 이 파일은 게시할 때 레지스트리로 전송됩니다. crates.io 는 이를 마크다운으로 해석하여 크레이트 페이지에 렌더링합니다.

[package]
# ...
readme = "README.md"

이 필드에 값이 지정되지 않고 패키지 루트에 README.md, README.txt 또는 README 라는 파일이 있으면 해당 파일의 이름이 사용됩니다. 이 필드를 false 로 설정하여 이 동작을 방지할 수 있습니다. 필드가 true 로 설정되면 기본값 README.md 가 가정됩니다.

homepage 필드

homepage 필드는 패키지의 홈페이지인 사이트의 URL이어야 합니다.

[package]
# ...
homepage = "https://serde.rs"

homepage 값은 소스 저장소나 API 문서 외에 크레이트를 위한 전용 웹사이트가 있는 경우에만 설정해야 합니다. homepage 가 documentation 또는 repository 값과 중복되지 않도록 하세요.

repository 필드

repository 필드는 패키지의 소스 저장소 URL이어야 합니다.

[package]
# ...
repository = "https://github.com/rust-lang/cargo"

license 및 license-file 필드

license 필드에는 패키지가 출시되는 소프트웨어 라이선스 이름이 포함됩니다. license-file 필드에는 라이선스 텍스트가 포함된 파일의 경로(Cargo.toml 기준 상대 경로)가 포함됩니다.

crates.io 는 license 필드를 SPDX 2.3 라이선스 식 으로 해석합니다. 이름은 SPDX 라이선스 목록 3.20 에 있는 알려진 라이선스여야 합니다. 자세한 내용은 SPDX 사이트 를 참조하세요.

SPDX 라이선스 식은 여러 라이선스를 결합하기 위해 AND 및 OR 연산자를 지원합니다.1

[package]
# ...
license = "MIT OR Apache-2.0"

OR 를 사용하면 사용자가 두 라이선스 중 하나를 선택할 수 있음을 나타냅니다. AND 를 사용하면 사용자가 두 라이선스를 동시에 준수해야 함을 나타냅니다. WITH 연산자는 특수한 예외가 있는 라이선스를 나타냅니다. 몇 가지 예시:

  • MIT OR Apache-2.0
  • LGPL-2.1-only AND MIT AND BSD-2-Clause
  • GPL-2.0-or-later WITH Bison-exception-2.2

패키지가 비표준 라이선스를 사용하는 경우 license 필드 대신 license-file 필드를 지정할 수 있습니다.

[package]
# ...
license-file = "LICENSE.txt"

참고: crates.io 에서는 license 또는 license-file 중 하나를 설정해야 합니다.

keywords 필드

keywords 필드는 이 패키지를 설명하는 문자열 배열입니다. 레지스트리에서 패키지를 검색할 때 도움이 될 수 있으며, 누군가 이 크레이트를 찾는 데 도움이 될 단어를 선택할 수 있습니다.

[package]
# ...
keywords = ["gamedev", "graphics"]

참고: crates.io 는 최대 5개의 키워드를 허용합니다. 각 키워드는 ASCII 텍스트여야 하고, 최대 20자여야 하며, 영문자나 숫자로 시작해야 하고, 문자, 숫자, _, - 또는 + 만 포함해야 합니다.

categories 필드

categories 필드는 이 패키지가 속한 카테고리의 문자열 배열입니다.

categories = ["command-line-utilities", "development-tools::cargo-plugins"]

참고: crates.io 는 최대 5개의 카테고리를 가집니다. 각 카테고리는 https://crates.io/category_slugs에서 사용 가능한 문자열 중 하나와 정확히 일치해야 합니다.

workspace 필드

workspace 필드는 이 패키지가 구성원이 될 워크스페이스를 구성하는 데 사용할 수 있습니다. 지정하지 않으면 파일 시스템 상위 방향으로 [workspace] 가 있는 첫 번째 Cargo.toml로 유추됩니다. 구성원이 워크스페이스 루트의 하위 디렉토리에 있지 않은 경우 이를 설정하는 것이 유용합니다.

[package]
# ...
workspace = "워크스페이스/루트/경로"

매니페스트에 이미 [workspace] 테이블이 정의되어 있는 경우 이 필드를 지정할 수 없습니다. 즉, 크레이트가 워크스페이스의 루트 크레이트([workspace] 포함)이면서 동시에 다른 워크스페이스의 구성원 크레이트(package.workspace 포함)일 수는 없습니다.

자세한 내용은 워크스페이스 장 을 참조하세요.

build 필드

build 필드는 네이티브 코드 빌드를 위한 빌드 스크립트 인 패키지 루트의 파일을 지정합니다. 자세한 내용은 빌드 스크립트 가이드 에서 확인할 수 있습니다.

[package]
# ...
build = "build.rs"

기본값은 "build.rs" 이며, 패키지 루트에 있는 build.rs 라는 파일에서 스크립트를 로드합니다. 다른 파일에 대한 경로를 지정하려면 build = "custom_build_name.rs" 를 사용하고, 빌드 스크립트 자동 감지를 비활성화하려면 build = false 를 사용하세요.

links 필드

links 필드는 링크되는 네이티브 라이브러리의 이름을 지정합니다. 자세한 내용은 빌드 스크립트 가이드의 links 섹션에서 확인할 수 있습니다.

예를 들어, “git2”(예: 리눅스의 libgit2.a)라는 네이티브 라이브러리를 링크하는 크레이트는 다음과 같이 지정할 수 있습니다.

[package]
# ...
links = "git2"

exclude 및 include 필드

exclude 및 include 필드는 프로젝트를 게시 하기 위해 패키징할 때 어떤 파일이 포함될지 명시적으로 지정하는 데 사용되며, 특정 종류의 변경 사항 추적(아래 설명)에도 사용됩니다. exclude 필드에 지정된 패턴은 포함되지 않는 파일 집합을 식별하고, include 에 지정된 패턴은 명시적으로 포함되는 파일을 지정합니다. cargo package --list 를 실행하여 패키지에 어떤 파일이 포함될지 확인할 수 있습니다.

[package]
# ...
exclude = ["/ci", "images/", ".*"]

[package]
# ...
include = ["/src", "COPYRIGHT", "/examples", "!/examples/big_example"]

두 필드가 모두 지정되지 않은 경우의 기본값은 아래 나열된 제외 대상을 제외하고 패키지 루트의 모든 파일을 포함하는 것입니다.

include 가 지정되지 않은 경우 다음 파일들이 제외됩니다.

  • 패키지가 git 저장소에 있지 않은 경우, 점(.)으로 시작하는 모든 “숨김” 파일은 건너뜁니다.
  • 패키지가 git 저장소에 있는 경우, 저장소의 gitignore 규칙 및 전역 git 구성에 의해 무시되는 모든 파일은 건너뜁니다.

exclude 또는 include 가 지정되었는지 여부와 관계없이 다음 파일은 항상 제외됩니다.

  • 모든 하위 패키지는 건너뜁니다 (Cargo.toml 파일을 포함하는 모든 하위 디렉토리).
  • 패키지 루트에 있는 target 이라는 디렉토리는 건너뜁니다.

다음 파일은 항상 포함됩니다.

  • 패키지 자체의 Cargo.toml 파일은 항상 포함되며, include 에 나열할 필요가 없습니다.
  • A minimized Cargo.lock is automatically included. See cargo package for more information.
  • license-file 가 지정된 경우 항상 포함됩니다.

이 옵션들은 상호 배타적입니다. include 를 설정하면 exclude 가 무시됩니다. include 파일 집합에서 특정 파일을 제외해야 하는 경우 아래 설명된 ! 연산자를 사용하세요.

패턴은 gitignore 스타일 패턴이어야 합니다. 요약하자면 다음과 같습니다.

  • foo 는 패키지 내 어디에서나 이름이 foo 인 모든 파일 또는 디렉토리에 일치합니다. 이는 **/foo 패턴과 동일합니다.
  • /foo 는 패키지 루트에 있는 이름이 foo 인 파일 또는 디렉토리에만 일치합니다.
  • foo/ 는 패키지 내 어디에서나 이름이 foo 인 모든 디렉토리 에 일치합니다.
  • *, ?, [] 와 같은 일반적인 글로브(glob) 패턴이 지원됩니다.
    • * 는 / 를 제외한 0개 이상의 문자에 일치합니다. 예를 들어 *.html 은 패키지 내 어디에서나 확장자가 .html 인 모든 파일 또는 디렉토리에 일치합니다.
    • ? 는 / 를 제외한 모든 한 문자에 일치합니다. 예를 들어 foo? 는 food 에는 일치하지만 foo 에는 일치하지 않습니다.
    • [] 는 문자 범위를 일치시킬 수 있게 합니다. 예를 들어 [ab] 는 a 또는 b 중 하나에 일치합니다. [a-z] 는 a부터 z까지의 문자에 일치합니다.
  • **/ 접두사는 모든 디렉토리에서 일치합니다. 예를 들어 **/foo/bar 는 foo 디렉토리 바로 아래에 있는 bar 파일 또는 디렉토리에 일치합니다.
  • /** 접미사는 내부의 모든 항목에 일치합니다. 예를 들어 foo/** 는 foo 디렉토리 내부의 모든 파일과 foo 아래 하위 디렉토리의 모든 파일에 일치합니다.
  • /**/ 는 0개 이상의 디렉토리에 일치합니다. 예를 들어 a/**/b 는 a/b, a/x/b, a/x/y/b 등에 일치합니다.
  • ! 접두사는 패턴을 반전시킵니다. 예를 들어 src/*.rs 와 !foo.rs 패턴은 src 디렉토리 내에서 확장자가 .rs 인 모든 파일에 일치하지만, 이름이 foo.rs 인 파일은 제외합니다.

include/exclude 목록은 일부 상황에서 변경 사항 추적을 위해서도 사용됩니다. rustdoc 으로 빌드되는 타겟의 경우, 타겟을 다시 빌드해야 하는지 결정하기 위해 추적할 파일 목록을 결정하는 데 사용됩니다. 패키지에 rerun-if-* 지시문을 내보내지 않는 빌드 스크립트 가 있는 경우, 해당 파일 중 하나라도 변경되면 빌드 스크립트를 다시 실행해야 하는지 추적하는 데 include/exclude 목록이 사용됩니다.

publish 필드

publish 필드를 사용하여 패키지를 게시할 수 있는 레지스트리 이름을 제어할 수 있습니다.

[package]
# ...
publish = ["레지스트리-이름"]

예를 들어 회사의 패키지를 비공개로 유지하기 위해 실수로 레지스트리(예: crates.io)에 패키지가 게시되는 것을 방지하려면 version 필드를 생략할 수 있습니다. 더 명시적으로 하려면 게시를 비활성화할 수 있습니다.

[package]
# ...
publish = false

publish 배열에 단일 레지스트리가 포함된 경우, --registry 플래그가 지정되지 않았을 때 cargo publish 명령어가 해당 레지스트리를 사용합니다.

metadata 테이블

Cargo는 기본적으로 오타 등을 감지하는 데 도움을 주기 위해 Cargo.toml 에서 사용되지 않는 키에 대해 경고를 표시합니다. 하지만 package.metadata 테이블은 Cargo에 의해 완전히 무시되며 경고도 발생하지 않습니다. 이 섹션은 Cargo.toml 에 패키지 구성을 저장하려는 도구에 사용할 수 있습니다. 예를 들어 다음과 같습니다.

[package]
name = "..."
# ...

# 예를 들어 안드로이드 APK 생성 시 사용되는 메타데이터
[package.metadata.android]
package-name = "my-awesome-android-app"
assets = "static/경로"

이 필드를 사용하는 방법은 해당 도구의 문서를 참조해야 합니다. package.metadata 테이블을 사용하는 러스트 프로젝트의 예시는 다음과 같습니다.

  • docs.rs

워크스페이스 수준에도 workspace.metadata 라는 유사한 테이블이 있습니다. Cargo는 이러한 테이블의 내용에 대한 형식을 지정하지 않지만, 외부 도구에서 해당 도구에 적합하다면 package.metadata 에 데이터가 없는 경우 workspace.metadata 의 데이터를 참조하는 등 일관된 방식으로 사용하도록 제안됩니다.

default-run 필드

매니페스트의 [package] 섹션에 있는 default-run 필드는 cargo run 에서 선택할 기본 바이너리를 지정하는 데 사용할 수 있습니다. 예를 들어, src/bin/a.rs 와 src/bin/b.rs 가 모두 있는 경우 다음과 같이 설정할 수 있습니다.

[package]
default-run = "a"

[lints] 섹션

테이블에서 서로 다른 도구의 린트 기본 레벨을 새로운 레벨로 재정의할 수 있습니다. 예를 들어 다음과 같습니다.

[lints.rust]
unsafe_code = "forbid"

이것은 다음과 같은 표현의 축약형입니다.

[lints.rust]
unsafe_code = { level = "forbid", priority = 0 }

level 은 rustc 의 린트 레벨 에 대응합니다.

  • forbid
  • deny
  • warn
  • allow

priority 는 어떤 린트나 린트 그룹이 다른 린트 그룹을 재정의할지 제어하는 부호 있는 정수입니다.

  • 낮은 수(특히 음수)는 우선순위가 낮으며 높은 수에 의해 재정의되고, rustc 와 같은 도구의 명령줄에 먼저 나타납니다.

[lints] 아래의 어느 테이블에 특정 린트가 속하는지 알기 위해서는 린트 이름에서 :: 앞부분을 확인하면 됩니다. :: 가 없다면 해당 도구는 rust 입니다. 예를 들어 unsafe_code 에 대한 경고는 lints.rust.unsafe_code 가 되고, clippy::enum_glob_use 에 대한 린트는 lints.clippy.enum_glob_use 가 됩니다.

예를 들어 다음과 같습니다.

[lints.rust]
unsafe_code = "forbid"

[lints.clippy]
enum_glob_use = "deny"

일반적으로 이러한 설정은 현재 패키지의 로컬 개발에만 영향을 미칩니다. Cargo는 이를 현재 패키지에만 적용하고 의존성에는 적용하지 않습니다. 의존하는 패키지들의 경우, Cargo는 --cap-lints 와 같은 기능을 사용하여 경로 기반이 아닌 의존성에서 발생하는 린트를 억제합니다.

MSRV: 1.74 버전부터 적용됨

The [hints] section

The [hints] section allows specifying hints for compiling this package. Cargo will respect these hints by default when compiling this package, though the top-level package being built can override these values through the [profile] mechanism. Hints are, by design, always safe for Cargo to ignore; if Cargo encounters a hint it doesn’t understand, or a hint it understands but with a value it doesn’t understand, it will warn, but not error. As a result, specifying hints in a crate does not impact the MSRV of the crate.

Individual hints may have an associated unstable feature gate that you need to pass in order to apply the configuration they specify, but if you don’t specify that unstable feature gate, you will again get only a warning, not an error.

There are no stable hints at this time. See the hint-mostly-unused documentation for information on an unstable hint.

MSRV: Respected as of 1.90.

[badges] 섹션

[badges] 섹션은 패키지가 게시될 때 레지스트리 웹사이트에 표시될 수 있는 상태 배지(status badge)를 지정하기 위한 것입니다.

참고: crates.io 는 이전에 웹사이트의 크레이트 옆에 배지를 표시했지만, 해당 기능은 제거되었습니다. 패키지는 배지를 README 파일에 포함해야 하며, 해당 파일은 crates.io 에 표시됩니다 (readme 필드 참조).

[badges]
# `maintenance` 테이블은 크레이트의 유지 관리 상태를 나타냅니다.
# 이는 레지스트리에서 사용될 수 있지만, 현재 crates.io에서는 사용되지 않습니다.
# 자세한 내용은 https://github.com/rust-lang/crates.io/issues/2437 및
# https://github.com/rust-lang/crates.io/issues/2438 을 참조하세요.
#
# `status` 필드는 필수입니다. 사용 가능한 옵션은 다음과 같습니다.
# - `actively-developed`: 새로운 기능이 추가되고 버그가 수정되고 있음.
# - `passively-maintained`: 새로운 기능 계획은 없지만, 제기된 이슈에 대해
#   유지 관리자가 대응할 의사가 있음.
# - `as-is`: 크레이트의 기능이 완성되었으며, 유지 관리자가 작업을 계속하거나
#   지원을 제공할 계획은 없지만, 설계된 목적대로 작동함.
# - `experimental`: 저자가 커뮤니티와 공유하고 싶어 하지만, 특정 유스케이스를
#   충족시키려는 의도는 없음.
# - `looking-for-maintainer`: 현재 유지 관리자가 크레이트를 다른 사람에게
#   양도하고 싶어 함.
# - `deprecated`: 유지 관리자가 이 크레이트 사용을 권장하지 않음 (크레이트 설명에
#   그 이유를 기재할 수 있으며, 더 나은 대안이 있거나 저자가 수정하고 싶지 않은
#   문제가 있을 수 있음).
# - `none`: 유지 관리자가 의도를 명시하지 않았으므로 crates.io에 배지를 표시하지 않음.
#   잠재적 사용자는 직접 조사해야 함.
maintenance = { status = "..." }
