의존성 재정의하기

의존성을 재정의하려는 욕구는 여러 시나리오를 통해 발생할 수 있습니다. 하지만 대부분은 크레이트가 crates.io에 게시되기 전에 해당 크레이트로 작업할 수 있는 기능으로 요약됩니다. 예를 들면 다음과 같습니다.

  • 작업 중인 크레이트가 작업 중인 훨씬 더 큰 애플리케이션에서도 사용되고 있으며, 해당 큰 애플리케이션 내부에서 라이브러리의 버그 수정을 테스트하고 싶은 경우
  • 직접 작업하지 않는 상위(upstream) 크레이트의 git 저장소 master 브랜치에 새로운 기능이나 버그 수정이 있어 이를 테스트하고 싶은 경우
  • 크레이트의 새로운 주 버전을 게시하려는데, 새 주 버전이 잘 작동하는지 확인하기 위해 전체 패키지에 걸쳐 통합 테스트를 수행하고 싶은 경우
  • 발견한 버그에 대해 상위 크레이트에 수정을 제출했지만, 버그 수정이 병합될 때까지 기다리지 않고 즉시 애플리케이션이 크레이트의 수정된 버전에 의존하기 시작하게 하고 싶은 경우

이러한 시나리오는 [patch] 매니페스트 섹션으로 해결할 수 있습니다.

이 장에서는 몇 가지 다른 유스케이스를 살펴보고, 의존성을 재정의하는 다양한 방법에 대한 세부 정보를 포함합니다.

  • 유스케이스 예시
    • 버그 수정 테스트
    • 게시되지 않은 부 버전으로 작업하기
      • 저장소 URL 재정의하기
    • 파괴적 변경 사전 게시하기
    • 여러 버전에 [patch] 사용하기
  • 참조
    • [patch] 섹션
    • [replace] 섹션
    • paths 재정의

참고: 로컬 패키지의 단일 의존성 선언에 대한 소스를 재정의하는 데 사용할 수 있는 여러 위치(multiple locations)로 의존성 지정하기를 참조하세요.

버그 수정 테스트하기

uuid 크레이트를 사용하여 작업하던 중 버그를 발견했다고 가정해 봅시다. 당신은 매우 진취적인 사람이기에 직접 버그를 수정해 보기로 했습니다! 원래 매니페스트는 다음과 같을 것입니다.

[package]
name = "my-library"
version = "0.1.0"

[dependencies]
uuid = "1.0"

가장 먼저 할 일은 uuid 저장소를 로컬로 복제하는 것입니다.

$ git clone https://github.com/uuid-rs/uuid.git

다음으로 my-library의 매니페스트를 다음과 같이 수정합니다.

[patch.crates-io]
uuid = { path = "../uuid/경로" }

여기서 우리는 crates-io 소스를 새로운 의존성으로 _패칭(patching)_한다고 선언합니다. 이렇게 하면 로컬 패키지의 crates.io 레지스트리에 로컬로 체크아웃된 uuid 버전이 실질적으로 추가됩니다.

다음으로, 패키지가 crates.io의 복사본 대신 로컬로 체크아웃된 복사본을 사용하도록 uuid의 새 버전을 사용하도록 락 파일을 업데이트해야 합니다. [patch]가 작동하는 방식은 ../path/to/uuid에 있는 의존성을 로드한 다음, uuid 버전에 대해 crates.io를 쿼리할 때마다 로컬 버전도 함께 반환하는 것입니다.

이는 로컬 체크아웃의 버전 번호가 중요하며 패치 사용 여부에 영향을 미친다는 것을 의미합니다. 매니페스트에서 uuid = "1.0"을 선언했으므로 >= 1.0.0, < 2.0.0만 해결(resolve)하며, Cargo의 탐욕적(greedy) 해결 알고리즘은 해당 범위 내에서 최대 버전을 해결함을 의미합니다. 보통 git 저장소의 버전이 이미 crates.io에 게시된 최대 버전보다 크거나 같으므로 문제가 되지 않지만, 이를 염두에 두는 것이 중요합니다!

어떤 경우든, 보통 지금 필요한 작업은 다음과 같습니다.

$ cargo build
   Compiling uuid v1.0.0 (.../uuid)
   Compiling my-library v0.1.0 (.../my-library)
    Finished dev [unoptimized + debuginfo] target(s) in 0.32 secs

이것으로 끝입니다! 이제 uuid의 로컬 버전으로 빌드하게 됩니다 (빌드 출력의 괄호 안에 있는 경로를 확인하세요). 로컬 경로 버전이 빌드되는 것이 보이지 않는다면 cargo update uuid --precise $version을 실행해야 할 수도 있습니다. 여기서 $version은 로컬로 체크아웃된 uuid 사본의 버전입니다.

원래 발견했던 버그를 수정한 후 다음에 할 일은 아마도 이를 uuid 크레이트 자체에 풀 리퀘스트로 제출하는 것입니다. 이 작업을 마친 후에는 [patch] 섹션도 업데이트할 수 있습니다. [patch] 내부의 목록은 [dependencies] 섹션과 똑같으므로, 풀 리퀘스트가 병합되면 다음과 같이 path 의존성을 변경할 수 있습니다.

[patch.crates-io]
uuid = { git = 'https://github.com/uuid-rs/uuid.git' }

게시되지 않은 부 버전으로 작업하기

이제 버그 수정에서 기능 추가로 주제를 바꿔보겠습니다. my-library를 작업하던 중 uuid 크레이트에 완전히 새로운 기능이 필요하다는 것을 발견했습니다. 이 기능을 구현했고, 위에서 [patch]를 사용하여 로컬에서 테스트했으며, 풀 리퀘스트를 제출했습니다. 실제로 게시되기 전에 이를 어떻게 계속 사용하고 테스트하는지 살펴보겠습니다.

또한 crates.io의 현재 uuid 버전은 1.0.0이지만, 그 이후 git 저장소의 master 브랜치가 1.0.1로 업데이트되었다고 가정해 봅시다. 이 브랜치에는 이전에 제출한 새로운 기능이 포함되어 있습니다. 이 저장소를 사용하기 위해 Cargo.toml을 다음과 같이 수정합니다.

[package]
name = "my-library"
version = "0.1.0"

[dependencies]
uuid = "1.0.1"

[patch.crates-io]
uuid = { git = 'https://github.com/uuid-rs/uuid.git' }

크레이트가 게시된 후 실제로 필요하게 될 버전인 1.0.1로 uuid에 대한 로컬 의존성을 업데이트했습니다. 하지만 이 버전은 crates.io에 존재하지 않으므로 매니페스트의 [patch] 섹션으로 제공합니다.

이제 라이브러리가 빌드될 때 git 저장소에서 uuid를 가져오고, crates.io에서 버전을 다운로드하는 대신 저장소 내부의 1.0.1로 해결합니다. 1.0.1이 crates.io에 게시되면 [patch] 섹션을 삭제할 수 있습니다.

[patch]가 전이적으로(transitively) 적용된다는 점도 주목할 가치가 있습니다. 다음과 같이 더 큰 패키지에서 my-library를 사용한다고 가정해 봅시다.

[package]
name = "my-binary"
version = "0.1.0"

[dependencies]
my-library = { git = 'https://example.com/git/my-library' }
uuid = "1.0"

[patch.crates-io]
uuid = { git = 'https://github.com/uuid-rs/uuid.git' }

[patch]는 전이적으로 적용 가능하지만 _최상위 수준(top level)_에서만 정의될 수 있으므로, my-library의 소비자인 우리는 필요한 경우 [patch] 섹션을 반복해야 합니다. 하지만 여기서는 새로운 uuid 크레이트가 우리의 uuid 의존성과 my-library -> uuid 의존성 _둘 다_에 적용됩니다. uuid 크레이트는 이 전체 크레이트 그래프에 대해 하나의 버전인 1.0.1로 해결되며 git 저장소에서 가져오게 됩니다.

저장소 URL 재정의하기

재정의하려는 의존성이 crates.io에서 로드되지 않는 경우, [patch] 사용법을 약간 변경해야 합니다. 예를 들어 의존성이 git 의존성인 경우 다음과 같이 로컬 경로로 재정의할 수 있습니다.

[patch."https://github.com/your/repository"]
my-library = { path = "../my-library/경로" }

이게 전부입니다!

파괴적 변경 사전 게시하기

보통 파괴적 변경을 동반하는 크레이트의 새로운 주 버전으로 작업하는 경우를 살펴보겠습니다. 이전 크레이트의 예를 이어가자면, uuid 크레이트의 2.0.0 버전을 만들려고 합니다. 상위에 모든 변경 사항을 제출한 후, my-library의 매니페스트를 다음과 같이 업데이트할 수 있습니다.

[dependencies]
uuid = "2.0"

[patch.crates-io]
uuid = { git = "https://github.com/uuid-rs/uuid.git", branch = "2.0.0" }

이게 전부입니다! 이전 예제와 마찬가지로 2.0.0 버전은 실제로 crates.io에 존재하지 않지만, [patch] 섹션을 사용하여 git 의존성을 통해 추가할 수 있습니다. 사고 실험으로 위에서 본 my-binary 매니페스트를 다시 한번 살펴보겠습니다.

[package]
name = "my-binary"
version = "0.1.0"

[dependencies]
my-library = { git = 'https://example.com/git/my-library' }
uuid = "1.0"

[patch.crates-io]
uuid = { git = 'https://github.com/uuid-rs/uuid.git', branch = '2.0.0' }

이 경우 실제로 uuid 크레이트의 두 버전으로 해결됩니다. my-binary 크레이트는 계속해서 uuid 크레이트의 1.x.y 시리즈를 사용하지만, my-library 크레이트는 2.0.0 버전의 uuid를 사용하게 됩니다. 이를 통해 모든 것을 한꺼번에 업데이트하도록 강요받지 않고도 의존성 그래프를 통해 크레이트에 파괴적 변경을 점진적으로 적용할 수 있습니다.

여러 버전에 [patch] 사용하기

의존성 이름을 변경하는 데 사용되는 package 키를 사용하여 동일한 크레이트의 여러 버전을 패치할 수 있습니다. 예를 들어, serde 크레이트의 1.* 시리즈에 사용하고 싶은 버그 수정이 있지만, 동시에 우리 git 저장소에 있는 2.0.0 버전의 serde를 사용하여 프로토타입을 만들고 싶다고 가정해 봅시다. 이를 위해 다음과 같이 구성할 수 있습니다.

[patch.crates-io]
serde = { git = 'https://github.com/serde-rs/serde.git' }
serde2 = { git = 'https://github.com/example/serde.git', package = 'serde', branch = 'v2' }

첫 번째 serde = ... 지시어는 git 저장소에서 serde 1.*를 사용해야 함을 나타내고(필요한 버그 수정을 가져옴), 두 번째 serde2 = ... 지시어는 https://github.com/example/serde의 v2 브랜치에서도 serde 패키지를 가져와야 함을 나타냅니다. 여기서는 해당 브랜치의 Cargo.toml이 2.0.0 버전을 명시하고 있다고 가정합니다.

package 키를 사용할 때 여기서 serde2 식별자는 실제로 무시됩니다. 우리는 단지 다른 패치된 크레이트와 충돌하지 않는 고유한 이름이 필요할 뿐입니다.

[patch] 섹션

Cargo.toml의 [patch] 섹션은 의존성을 다른 사본으로 재정의하는 데 사용될 수 있습니다. 구문은 [dependencies] 섹션과 유사합니다.

[patch.crates-io]
foo = { git = 'https://github.com/example/foo.git' }
bar = { path = 'my/local/bar' }

[dependencies.baz]
git = 'https://github.com/example/baz.git'

[patch.'https://github.com/example/baz']
baz = { git = 'https://github.com/example/patched-baz.git', branch = 'my-branch' }

참고: [patch] 테이블은 .cargo/config.toml 파일이나 --config 'patch.crates-io.rand.path="rand"'와 같은 CLI 옵션을 통해 구성 옵션으로 지정할 수도 있습니다. 이는 커밋하고 싶지 않은 로컬 전용 변경 사항이나 임시로 패치를 테스트할 때 유용할 수 있습니다.

[patch] 테이블은 의존성과 유사한 하위 테이블들로 구성됩니다. [patch] 뒤의 각 키는 패치하려는 소스의 URL 또는 레지스트리 이름입니다. crates-io라는 이름을 사용하여 기본 레지스트리인 crates.io를 재정의할 수 있습니다. 위의 예에서 첫 번째 [patch]는 crates.io를 재정의하는 것을 보여주고, 두 번째 [patch]는 git 소스를 재정의하는 것을 보여줍니다.

이러한 테이블의 각 항목은 매니페스트의 [dependencies] 섹션에 있는 것과 동일한 일반적인 의존성 사양입니다. [patch] 섹션에 나열된 의존성들은 해결(resolve)되어 지정된 URL의 소스를 패치하는 데 사용됩니다. 위의 매니페스트 코드 조각은 foo 크레이트와 bar 크레이트로 crates-io 소스(예: crates.io 자체)를 패치합니다. 또한 https://github.com/example/baz 소스를 다른 곳에서 가져온 my-branch로 패치합니다.

소스는 존재하지 않는 버전의 크레이트로 패치될 수도 있고, 이미 존재하는 버전의 크레이트로 패치될 수도 있습니다. 소스에 이미 존재하는 크레이트 버전으로 패치하는 경우, 소스의 원래 크레이트가 대체됩니다.

Cargo는 워크스페이스 루트의 Cargo.toml 매니페스트에 있는 패치 설정만 확인합니다. 의존성에 정의된 패치 설정은 무시됩니다.

[replace] 섹션

참고: [replace]는 사용 중단되었습니다. 대신 [patch] 테이블을 사용해야 합니다.

Cargo.toml의 이 섹션은 의존성을 다른 사본으로 재정의하는 데 사용될 수 있습니다. 구문은 [dependencies] 섹션과 유사합니다.

[replace]
"foo:0.1.0" = { git = 'https://github.com/example/foo.git' }
"bar:1.0.2" = { path = 'my/local/bar' }

[replace] 테이블의 각 키는 패키지 ID 사양(package ID specification)이며, 의존성 그래프에서 재정의할 노드를 임의로 선택할 수 있습니다(3부분으로 된 버전 번호가 필수입니다). 각 키의 값은 기능을 지정할 수 없다는 점을 제외하고는 의존성을 지정하는 [dependencies] 구문과 동일합니다. 크레이트가 재정의될 때 대체되는 사본은 이름과 버전이 모두 동일해야 하지만, 다른 소스(예: git 또는 로컬 경로)에서 가져올 수 있습니다.

Cargo는 워크스페이스 루트의 Cargo.toml 매니페스트에 있는 replace 설정만 확인합니다. 의존성에 정의된 replace 설정은 무시됩니다.

paths 재정의

때로는 크레이트에서 일시적으로만 작업하고 있어 위의 [patch] 섹션처럼 Cargo.toml을 수정하고 싶지 않을 때가 있습니다. 이 유스케이스를 위해 Cargo는 **경로 재정의(path overrides)**라는 훨씬 더 제한적인 형태의 재정의를 제공합니다.

경로 재정의는 Cargo.toml 대신 .cargo/config.toml을 통해 지정됩니다. .cargo/config.toml 내부에 paths라는 키를 지정합니다.

paths = ["/uuid/경로"]

이 배열은 Cargo.toml을 포함하는 디렉토리들로 채워져야 합니다. 이 예에서는 uuid만 추가하고 있으므로 해당 크레이트만 재정의됩니다. 이 경로는 절대 경로일 수도 있고 .cargo 폴더를 포함하는 디렉토리를 기준으로 한 상대 경로일 수도 있습니다.

Path overrides are more restricted than the [patch] section, however, in that they cannot change the structure of the dependency graph. When a path replacement is used then the previous set of dependencies must all match exactly to the new Cargo.toml specification. For example this means that path overrides cannot be used to test out adding a dependency to a crate. Instead, [patch] must be used in that situation. As a result, usage of a path override is typically isolated to quick bug fixes rather than larger changes.

참고: 경로를 재정의하기 위해 로컬 구성을 사용하는 것은 crates.io에 게시된 크레이트에 대해서만 작동합니다. 이 기능을 사용하여 Cargo에 로컬의 게시되지 않은 크레이트를 찾는 방법을 알려줄 수는 없습니다.
