The Cargo Book

자주 묻는 질문

GitHub을 패키지 저장소로 사용할 계획입니까?

아니요. Cargo의 계획은 npm이 npmjs.com 을 사용하고 Rubygems가 rubygems.org 를 사용하는 것처럼 crates.io 를 사용하는 것입니다.

사람들이 레지스트리를 패키지의 기본 소스로 사용하더라도 초기 개발 및 임시 패치에 사용될 수 있기 때문에, 패키지 소스로서의 git 저장소는 영원히 지원할 계획입니다.

GitHub를 레지스트리로 사용하는 대신 왜 crates.io를 만드나요?

우리는 GitHub에서 다운로드하거나 패키지 자체에 패키지를 복사하는 것을 포함하여 패키지를 다운로드하는 여러 방법을 지원하는 것이 매우 중요하다고 생각합니다.

그렇긴 하지만, 우리는 crates.io 가 여러 가지 중요한 이점을 제공하며 사람들이 Cargo에서 패키지를 다운로드하는 주요 방법이 될 가능성이 높다고 생각합니다.

선례를 보면 Node.js의 npm 과 Ruby의 bundler 모두 중앙 레지스트리 모델과 Git 기반 모델을 모두 지원하며, 해당 생태계에서 대부분의 패키지는 레지스트리를 통해 다운로드되고 중요한 소수의 패키지가 Git 기반 패키지를 활용합니다.

중앙 레지스트리가 다른 언어에서 인기 있는 이유 중 몇 가지 장점은 다음과 같습니다:

  • 발견 용이성(Discoverability). 중앙 레지스트리는 기존 패키지를 찾기 쉬운 장소를 제공합니다. 태깅과 결합하여 레지스트리가 가장 인기 있는 패키지 목록이나 가장 많이 의존되는 패키지 목록과 같은 생태계 전반의 정보를 제공할 수도 있습니다.
  • 속도(Speed). 중앙 레지스트리를 사용하면 패키지의 메타데이터만 쉽고 빠르고 효율적으로 가져올 수 있으며, 저장소에 존재하는 다른 불필요한 데이터가 아닌 게시된 패키지만 효율적으로 다운로드할 수 있습니다. 이는 의존성 해결 및 가져오기 속도를 크게 향상시킵니다. 의존성 그래프가 커짐에 따라 모든 Git 저장소를 다운로드하는 것은 금방 느려집니다. 또한 모든 사람이 고속의 저지연 인터넷 연결을 가지고 있지는 않다는 점을 기억하세요.

Cargo가 C 코드(또는 다른 언어)와 연동되나요?

네!

Cargo는 Rust 코드 컴파일을 처리하지만, 많은 Rust 패키지가 C 코드와 연결된다는 것을 알고 있습니다. 또한 Rust 이외의 언어를 컴파일하는 것과 관련하여 수십 년 동안 구축된 도구들이 있다는 것도 알고 있습니다.

우리의 솔루션: Cargo는 패키지가 rustc 를 호출하기 전에 실행할 스크립트를 지정(Rust로 작성됨)할 수 있도록 합니다. Rust는 플랫폼별 구성을 구현하고 패키지 간의 공통 빌드 기능을 리팩토링하는 데 활용됩니다.

make(또는 ninja 등) 내부에서 Cargo를 사용할 수 있나요?

그렇습니다. Cargo가 최상위 수준에서 Rust 패키지를 컴파일하는 독립적인 방법으로 유용하게 사용되기를 의도하지만, 일부 사람들이 다른 빌드 도구에서 Cargo를 호출하기를 원할 것이라는 점도 알고 있습니다.

우리는 에러 코드와 기계 판독 가능한 출력 모드와 같은 것들에 주의를 기울여 Cargo가 그러한 문맥에서 잘 작동하도록 설계했습니다. 아직 해야 할 일이 좀 더 남아있지만, 전통적인 스크립트 문맥에서 Cargo를 사용하는 것은 우리가 처음부터 설계한 부분이며 앞으로도 계속 우선순위를 둘 것입니다.

Cargo는 멀티 플랫폼 패키지나 교차 컴파일(cross-compilation)을 처리하나요?

Rust 자체적으로 플랫폼에 따라 코드 섹션을 구성하는 기능을 제공합니다. Cargo 또한 플랫폼별 의존성 을 지원하며, 향후 Cargo.toml 에서 더 많은 플랫폼별 구성을 지원할 계획입니다.

장기적으로는 Cargo를 사용하여 패키지를 편리하게 교차 컴파일하는 방법을 모색하고 있습니다.

Cargo는 production 이나 test 와 같은 환경을 지원하나요?

우리는 다음을 지원하기 위해 프로파일 을 사용하여 환경을 지원합니다:

  • 환경별 플래그(예: 개발용 -g --opt-level=0, 운영용 --opt-level=3 등).
  • 환경별 의존성(예: 테스트 단언을 위한 hamcrest).
  • 환경별 #[cfg]
  • cargo test 명령어

Cargo는 Windows에서 작동하나요?

네!

Cargo에 대한 모든 커밋은 Windows에서 로컬 테스트 스위트를 통과해야 합니다. Windows에서 실행 중에 문제가 발생하면 이를 버그로 간주하므로, 이슈를 제기해 주세요.

왜 Cargo.lock 을 버전 관리에 두어야 하나요?

cargo new 는 기본적으로 버전 관리에서 Cargo.lock 을 추적하도록 설정되지만, 그렇게 할지 여부는 패키지의 필요에 따라 다릅니다.

Cargo.lock 잠금 파일의 목적은 성공적인 빌드 시점의 상태를 기술하는 것입니다. Cargo는 이 잠금 파일을 사용하여 Cargo.lock 파일이 처음 생성되었을 때와 정확히 동일한 의존성과 버전이 사용되도록 보장함으로써, 서로 다른 시점과 서로 다른 시스템에서 결정론적인 빌드를 제공합니다.

결정론적인 빌드는 다음에 도움이 됩니다:

  • 버그의 근본 원인을 찾기 위해 git bisect 실행하기
  • 외부 요인이 아닌 새로운 커밋으로 인해서만 CI가 실패하도록 보장하기
  • 기여자가 다른 기여자나 CI와 비교하여 다른 동작을 볼 때 발생하는 혼란 줄이기

의존성의 스냅샷을 갖는 것은 다음과 같이 프로젝트를 일관된 버전의 의존성에 대해 검증해야 할 때도 도움이 됩니다.

  • 의존성의 최신 버전이 지원하는 버전보다 낮은 최소 지원 Rust 버전(MSRV)을 검증할 때
  • 호환성이 보장되지 않는 사람이 읽을 수 있는 출력을 검증할 때(예: 에러 메시지가 “이해하기 쉬운지” 확인하기 위한 스냅샷 테스트. 이는 자동화하기에는 너무 모호한 지표임)

하지만 이러한 결정론은 잘못된 보안감을 줄 수 있습니다. Cargo.lock 은 패키지 사용자에게 영향을 미치지 않고 Cargo.toml 만 영향을 미치기 때문입니다. 예를 들어:

  • --locked 를 전달하지 않는 한 cargo install 은 최신 의존성을 선택합니다.
  • cargo add 로 추가된 것과 같은 새로운 의존성은 최신 버전으로 고정됩니다.

잠금 파일은 머지 충돌의 원인이 될 수도 있습니다.

CI를 통해 최신 버전의 의존성을 검증하는 전략에 대해서는 최신 의존성 확인 을 참조하세요.

라이브러리가 의존성 버전으로 * 를 사용할 수 있나요?

2016년 1월 22일부로, crates.io 는 와일드카드 의존성 제약 조건이 있는 모든 패키지(라이브러리뿐만 아니라)를 거부합니다.

엄밀히 말해 라이브러리가 사용할 수는 있지만, 사용해서는 안 됩니다. * 버전 요구 사항은 “이것은 지금까지 나온 모든 버전과 연동됩니다“라고 말하는 것과 같으며, 이는 결코 사실일 수 없습니다. 라이브러리는 “모든 1.x.y 버전“과 같이 일반적인 것이라도 항상 연동되는 범위를 지정해야 합니다.

왜 Cargo.toml 인가요?

Cargo를 사용하면서 가장 자주 접하는 질문 중 하나가 왜 설정 파일 이름이 Cargo.toml 인가 하는 것입니다. 맨 앞의 대문자 C 는 디렉토리 목록에서 매니페스트가 다른 유사한 설정 파일들과 함께 묶이도록 선택되었습니다. 파일을 정렬할 때 종종 대문자가 소문자보다 앞에 오기 때문에, Makefile 과 Cargo.toml 같은 파일들이 함께 놓이게 됩니다. 뒤의 .toml 은 이 파일이 TOML 설정 형식 임을 강조하기 위해 선택되었습니다.

Cargo는 저장소를 쉽게 식별할 수 있도록 cargo.toml 이나 Cargofile 과 같은 다른 이름을 허용하지 않습니다. 다양한 이름을 허용할 경우 과거에 한 가지 경우는 처리되었지만 다른 경우는 실수로 잊혀지는 등의 혼란이 발생했기 때문입니다.

Cargo를 어떻게 오프라인에서 사용할 수 있나요?

--offline 또는 --frozen 플래그는 Cargo가 네트워크에 접속하지 않도록 합니다. 네트워크에 접속해야 하는 경우 에러를 반환합니다. 오프라인으로 전환하기 전에 한 프로젝트에서 cargo fetch 를 사용하여 의존성을 다운로드한 다음, 다른 프로젝트에서 동일한 의존성을 사용할 수 있습니다. Cargo 설정을 통해 이를 설정하려면 설정 값 을 참조하세요.

벤더링(Vendoring)도 이와 관련이 있습니다. 자세한 내용은 소스 교체(source replacement) 문서를 참조하세요.

왜 Cargo가 제 코드를 다시 빌드하나요?

Cargo는 프로젝트의 크레이트를 점진적으로 컴파일할 책임이 있습니다. 이는 예를 들어 cargo build 를 두 번 입력했을 때 두 번째 빌드에서는 crates.io 의존성을 다시 빌드하지 않아야 함을 의미합니다. 그럼에도 불구하고 버그가 발생할 수 있으며, Cargo가 예상치 못한 시점에 코드를 다시 빌드할 수도 있습니다.

우리는 오랫동안 이에 대해 더 나은 진단 기능을 제공하기를 원해 왔지만, 불행히도 꽤 오랫동안 해당 이슈에서 진전을 보지 못했습니다. 하지만 그동안 CARGO_LOG 환경 변수를 설정하여 재빌드 문제를 어느 정도 디버깅할 수 있습니다.

$ CARGO_LOG=cargo::core::compiler::fingerprint=info cargo build

이렇게 하면 Cargo가 진단 및 재빌드에 대한 많은 정보를 출력합니다. 이 출력에는 프로젝트가 왜 다시 빌드되는지에 대한 단서가 포함되어 있을 수 있지만, 출력이 아직 읽기 쉽지 않기 때문에 직접 단서들을 연결해 보아야 할 수도 있습니다. CARGO_LOG 는 재빌드되지 않아야 한다고 생각되는 명령을 실행할 때 설정되어야 한다는 점에 유의하세요. 안타깝게도 현재 Cargo에는 사후에 “왜 그것이 다시 빌드되었는가?“를 디버깅할 수 있는 방법이 없습니다.

역사적으로 크레이트가 다시 빌드되게 만든 몇 가지 이슈들은 다음과 같습니다:

  • 빌드 스크립트가 존재하지 않고 아무것도 생성하지 않는 파일 foo 에 대해 cargo::rerun-if-changed=foo 를 출력하는 경우입니다. 이 경우 Cargo는 스크립트가 해당 파일을 생성할 것이라고 생각하여 빌드 스크립트를 계속 실행하지만, 실제로는 아무것도 생성되지 않습니다. 해결 방법은 이러한 시나리오에서 rerun-if-changed 를 출력하지 않는 것입니다.

  • 두 번의 연속된 Cargo 빌드에서 일부 의존성에 대해 활성화된 기능(feature) 집합이 다를 수 있습니다. 예를 들어, 첫 번째 빌드 명령이 전체 워크스페이스를 빌드하고 두 번째 명령이 하나의 크레이트만 빌드하는 경우, crates.io의 의존성에 대해 다른 기능 집합이 활성화되어 해당 의존성과 이를 의존하는 모든 것이 다시 빌드될 수 있습니다. 안타깝게도 이에 대한 완벽한 해결책은 없지만, 가능하다면 워크스페이스에서 무엇을 빌드하든 크레이트에 활성화된 기능 집합을 일정하게 유지하는 것이 최선입니다.

  • 일부 파일 시스템은 타임스탬프와 관련하여 특이한 동작을 보입니다. Cargo는 주로 파일의 타임스탬프를 사용하여 재빌드 여부를 결정하지만, 비표준 파일 시스템을 사용하는 경우 타임스탬프에 영향을 줄 수 있습니다(예: 타임스탬프 잘림, 드리프트 발생 등). 이러한 시나리오에서는 자유롭게 이슈를 제기해 주시면 파일 시스템을 지원할 수 있는 방법을 찾아보겠습니다.

  • 동시 빌드 프로세스가 결과물을 삭제하거나 파일을 수정하고 있는 경우입니다. 때때로 프로젝트를 빌드하거나 체크하려고 시도하는 백그라운드 프로세스가 있을 수 있습니다. 이러한 백그라운드 프로세스가 예상치 못하게 빌드 결과물을 삭제하거나 파일을 건드려서(또는 실수로) 재빌드가 불필요해 보이게 만들 수 있습니다! 여기서 최선의 해결책은 백그라운드 프로세스가 작업과 충돌하지 않도록 조정하는 것입니다.

하지만 문제를 디버깅해 본 후에도 여전히 문제가 발생한다면, 언제든지 이슈를 제기해 주세요!

“버전 충돌“이 무엇을 의미하며 어떻게 해결하나요?

x 에 대해 이 충돌을 해결할 수 있는 버전을 선택하는 데 실패했습니다

위와 같은 에러 메시지를 본 적이 있으신가요?

This is one of the most annoying error messages for Cargo users. There are several situations which may lead to a version conflict. Below we’ll walk through possible causes and provide diagnostic techniques to help you out there:

  • The project and its dependencies use links to repeatedly link the local library. Cargo forbids linking two packages with the same native library, so even with multiple layers of dependencies it is not allowed. In this case, the error message will prompt: Only one package in the dependency graph may specify the same links value, you may need to manually check and delete duplicate link values. The community also have conventions in place to alleviate this.

  • 프로젝트에서 서로 다른 크레이트에 의존할 때, 이 크레이트들이 동일한 라이브러리를 의존하지만 사용되는 버전이 제한되어 있어 올바른 버전을 결정할 수 없는 경우에도 충돌이 발생합니다. 에러 메시지는 all possible versions conflict with previously selected packages 라고 안내합니다. 버전 요구 사항을 일관되게 수정해야 할 수 있습니다.

  • 프로젝트에 여러 버전의 의존성이 있을 때, direct-minimal-versions 를 사용하면 최소 버전 요구 사항을 충족할 수 없어 충돌이 발생할 수 있습니다. 이에 맞춰 최소 SemVer 버전을 충족하도록 직접 의존성의 버전 요구 사항을 수정해야 할 수 있습니다.

  • 의존하는 크레이트에 선택한 기능(feature)이 없는 경우에도 충돌이 발생합니다. 이 경우 의존하는 버전과 해당 기능을 확인해야 합니다.

  • 브랜치나 PR을 머지할 때 충돌이 발생할 수 있습니다. 복잡한 충돌이 있는 경우 “자신의(yours)” 변경 사항을 모두 리셋하고 브랜치의 다른 모든 충돌을 해결한 다음, cargo tree 나 cargo check 와 같은 명령어를 실행하면 로컬 변경 사항으로 잠금 파일을 다시 업데이트합니다. 이전에 브랜치에서 cargo update 명령을 실행했다면 지금 다시 실행하면 됩니다. 커뮤니티에서는 커스텀 머지 도구 를 사용하여 Cargo.lock 및 Cargo.toml 의 머지 충돌을 해결하는 방법을 모색해 왔습니다.

Why does my build take up so much space?

Cargo trades off disk space for faster builds including:

  • Maintaining a cache of intermediate build artifacts to avoid rebuilding everything when making changes to one package
  • Maintaining distinct cache entries for different combinations of toolchain versions, package versions, features, etc to avoid rebuilding packages when switching back and forth between configurations
  • Enabling incremental compilation for local packages for faster rebuilds for the package that changed
  • Enabling debuginfo in the dev profile in case you use a debugger
