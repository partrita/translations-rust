Send와 Sync

하지만 모든 것이 상속된 가변성(inherited mutability)을 따르는 것은 아닙니다. 어떤 타입들은 메모리 위치를 수정하면서 동시에 여러 개의 별칭(alias)을 갖는 것을 허용합니다. 이러한 타입들이 동기화를 사용하여 접근을 관리하지 않는 한, 그것들은 절대로 스레드 세이프(thread-safe)하지 않습니다. 러스트는 Send 와 Sync 트레잇을 통해 이를 규정합니다.

  • 어떤 타입을 다른 스레드로 보내는 것이 안전하다면, 그 타입은 Send 입니다.
  • 스레드 간에 공유하는 것이 안전하다면 그 타입은 Sync 입니다(&T 가 Send 라면 T 는 Sync 이고, 그 역도 성립합니다).

Send 와 Sync 는 러스트 동시성 이야기의 핵심입니다. 따라서 이들을 올바르게 작동시키기 위한 상당한 양의 특별한 도구들이 존재합니다. 무엇보다도, 이들은 비안전 트레잇 입니다. 이는 이들을 구현하는 것이 안전하지 않으며, 다른 비안전 코드는 이들이 올바르게 구현되었다고 가정할 수 있다는 것을 의미합니다. 이들은 마커 트레잇(메서드와 같은 연관 항목이 없음)이기 때문에, 올바르게 구현되었다는 것은 단순히 구현체가 가져야 할 고유한 속성을 가지고 있음을 의미합니다. Send 나 Sync 를 잘못 구현하면 정의되지 않은 동작을 유발할 수 있습니다.

Send 와 Sync 는 또한 자동으로 파생되는 트레잇입니다. 이는 다른 모든 트레잇과 달리, 어떤 타입이 전적으로 Send 또는 Sync 인 타입들로만 구성되어 있다면, 그 타입도 Send 또는 Sync 가 된다는 것을 의미합니다. 거의 모든 기본 타입은 Send 이자 Sync 이며, 결과적으로 여러분이 다루게 될 거의 모든 타입은 Send 이자 Sync 입니다.

주요 예외는 다음과 같습니다:

  • 원시 포인터는 Send 도 아니고 Sync 도 아닙니다(안전 장치가 없기 때문입니다).
  • UnsafeCell 은 Sync 가 아닙니다(따라서 Cell 과 RefCell 도 아닙니다).
  • Rc 는 Send 나 Sync 가 아닙니다(참조 카운트가 공유되며 동기화되지 않기 때문입니다).

Rc 와 UnsafeCell 은 매우 근본적으로 스레드에 안전하지 않습니다. 이들은 동기화되지 않은 공유 가변 상태를 가능하게 합니다. 하지만 엄밀히 말하면 원시 포인터는 일종의 린트(lint) 로서 스레드에 안전하지 않다고 표시된 것입니다. 원시 포인터로 유용한 작업을 하려면 역참조가 필요한데, 이는 이미 안전하지 않습니다. 그런 의미에서, 원시 포인터를 스레드 세이프하다고 표시해도 “괜찮다“고 주장할 수도 있습니다.

하지만 원시 포인터를 포함하는 타입들이 자동으로 스레드 세이프하다고 표시되는 것을 방지하기 위해 이들이 스레드에 안전하지 않은 것이 중요합니다. 이러한 타입들은 사소하지 않은 추적되지 않는 소유권을 가지고 있으며, 작성자가 반드시 스레드 안전성에 대해 깊이 고민했을 것 같지는 않습니다. Rc 의 경우, *mut 를 포함하며 확실히 스레드 세이프하지 않은 타입의 좋은 예입니다.

자동으로 파생되지 않는 타입들도 원한다면 간단히 이들을 구현할 수 있습니다:

#![allow(unused)]
fn main() {
struct MyBox(*mut u8);

unsafe impl Send for MyBox {}
unsafe impl Sync for MyBox {}
}

어떤 타입이 부적절하게 Send 나 Sync 로 자동 파생되는 믿을 수 없을 정도로 드문 경우에는, Send 와 Sync 구현을 제거할 수도 있습니다:

#![allow(unused)]
#![feature(negative_impls)]

fn main() {
// 어떤 동기화 기본 요소를 위한 마법 같은 의미론이 있습니다!
struct SpecialThreadToken(u8);

impl !Send for SpecialThreadToken {}
impl !Sync for SpecialThreadToken {}
}

그 자체로는 Send 와 Sync 를 잘못 파생하는 것이 불가능하다는 점에 유의하세요. 다른 비안전 코드에 의해 특별한 의미가 부여된 타입들만이 잘못된 Send 또는 Sync 로 인해 문제를 일으킬 수 있습니다.

대부분의 원시 포인터 사용은 Send 와 Sync 가 파생될 수 있도록 충분한 추상화 뒤에 캡슐화되어야 합니다. 예를 들어, 러스트의 모든 표준 컬렉션은 할당과 복잡한 소유권을 관리하기 위해 원시 포인터를 광범위하게 사용함에도 불구하고 (Send 와 Sync 타입을 포함할 때) Send 이자 Sync 입니다. 마찬가지로, 이러한 컬렉션에 대한 대부분의 반복자(iterator)들도 컬렉션에 대한 & 나 &mut 처럼 동작하므로 Send 이자 Sync 입니다.

예제

Box 는 여러 이유로 인해 컴파일러에 의해 자체적인 특별한 내장 타입으로 구현되어 있지만, Send 와 Sync 를 구현하는 것이 언제 건전한지에 대한 예를 보기 위해 비슷한 동작을 하는 무언가를 직접 구현해 볼 수 있습니다. 이것을 Carton 이라고 부르기로 합시다.

스택에 할당된 값을 가져와 힙으로 전송하는 코드를 작성하는 것으로 시작합니다.

#![allow(unused)]
fn main() {
pub mod libc {
   pub use ::std::os::raw::{c_int, c_void};
   #[allow(non_camel_case_types)]
   pub type size_t = usize;
   unsafe extern "C" { pub fn posix_memalign(memptr: *mut *mut c_void, align: size_t, size: size_t) -> c_int; }
}
use std::{
    mem::{align_of, size_of},
    ptr,
    cmp::max,
};

struct Carton<T>(ptr::NonNull<T>);

impl<T> Carton<T> {
    pub fn new(value: T) -> Self {
        // T 하나를 저장할 수 있을 만큼의 힙 메모리를 할당합니다.
        assert_ne!(size_of::<T>(), 0, "크기가 0인 타입은 이 예제의 범위를 벗어납니다");
        let mut memptr: *mut T = ptr::null_mut();
        unsafe {
            let ret = libc::posix_memalign(
                (&mut memptr as *mut *mut T).cast(),
                max(align_of::<T>(), size_of::<usize>()),
                size_of::<T>()
            );
            assert_eq!(ret, 0, "할당 실패 또는 잘못된 정렬");
        };

        // NonNull은 포인터가 null이 아님을 강제하는 래퍼일 뿐입니다.
        let ptr = {
            // 안전성: memptr은 참조로부터 생성했고 배타적인 접근 권한을 가지고 있으므로
            // 역참조 가능합니다.
            ptr::NonNull::new(memptr)
                .expect("posix_memalign이 0을 반환하면 null이 아님을 보장함")
        };

        // 스택에서 힙에 할당한 위치로 값을 이동합니다.
        unsafe {
            // 안전성: null이 아니라면, posix_memalign은 쓰기에 유효하고
            // 적절히 정렬된 포인터를 제공합니다.
            ptr.as_ptr().write(value);
        }

        Self(ptr)
    }
}
}

이것은 별로 유용하지 않습니다. 사용자가 우리에게 값을 주면, 그들은 그것에 접근할 방법이 없기 때문입니다. Box는 Deref 와 DerefMut 를 구현하여 내부 값에 접근할 수 있게 합니다. 우리도 그렇게 해봅시다.

#![allow(unused)]
fn main() {
use std::ops::{Deref, DerefMut};

struct Carton<T>(std::ptr::NonNull<T>);

impl<T> Deref for Carton<T> {
    type Target = T;

    fn deref(&self) -> &Self::Target {
        unsafe {
            // 안전성: 포인터는 [`Self::new`] 의 로직에 의해 정렬되고, 초기화되었으며,
            //   역참조 가능합니다. 우리는 읽는이가 Carton을 대여하도록 요구하며,
            //   반환 값의 수명은 입력의 수명으로 생략됩니다.
            //   이는 반환된 참조가 드롭될 때까지 아무도 Carton의 내용을
            //   변경할 수 없도록 대여 검사기가 강제한다는 것을 의미합니다.
            self.0.as_ref()
        }
    }
}

impl<T> DerefMut for Carton<T> {
    fn deref_mut(&mut self) -> &mut Self::Target {
        unsafe {
            // 안전성: 포인터는 [`Self::new`] 의 로직에 의해 정렬되고, 초기화되었으며,
            //   역참조 가능합니다. 우리는 쓰는이가 Carton을 가변 대여하도록 요구하며,
            //   반환 값의 수명은 입력의 수명으로 생략됩니다.
            //   이는 반환된 가변 참조가 드롭될 때까지 다른 누구도 Carton의
            //   내용에 접근할 수 없도록 대여 검사기가 강제한다는 것을 의미합니다.
            self.0.as_mut()
        }
    }
}
}

마지막으로, 우리의 Carton 이 Send 이고 Sync 인지 생각해 봅시다. 배타적 접근을 강제하지 않으면서 다른 무언가와 가변 상태를 공유하지 않는 한, 어떤 것은 안전하게 Send 가 될 수 있습니다. 각 Carton 은 유일한 포인터를 가지고 있으므로 문제 없습니다.

#![allow(unused)]
fn main() {
struct Carton<T>(std::ptr::NonNull<T>);
// 안전성: 우리 외에는 아무도 원시 포인터를 가지고 있지 않으므로,
// T가 안전하게 전송될 수 있다면 Carton을 다른 스레드로 안전하게 전송할 수 있습니다.
unsafe impl<T> Send for Carton<T> where T: Send {}
}

Sync 는 어떨까요? Carton 이 Sync 가 되려면, 다른 &Carton 에서 읽거나 쓰고 있는 동안 &Carton 에 저장된 무언가에 쓸 수 없도록 강제해야 합니다. 포인터에 쓰려면 &mut Carton 이 필요하고, 대여 검사기가 가변 참조의 배타성을 강제하므로, Carton 을 Sync 로 만드는 데에도 건전성 문제는 없습니다.

#![allow(unused)]
fn main() {
struct Carton<T>(std::ptr::NonNull<T>);
// 안전성: 동기화되지 않은 방식(`Deref` 등)으로 `&Carton<T>` 에서 `&T` 로 가는
// 공개된 방법이 존재하므로, `T` 가 `Sync` 가 아니면 `Carton<T>` 도 `Sync` 일 수
// 없습니다.
// 반대로, `Carton` 자체는 내부 가변성을 전혀 사용하지 않습니다.
// 모든 변경은 배타적 참조(`&mut`)를 통해 수행됩니다. 이는 `Carton<T>` 가
// `Sync` 이기 위해 `T` 가 `Sync` 인 것으로 충분하다는 것을 의미합니다:
unsafe impl<T> Sync for Carton<T> where T: Sync  {}
}

우리의 타입이 Send 이고 Sync 라고 주장할 때, 우리는 보통 포함된 모든 타입이 Send 이고 Sync 임을 강제해야 합니다. 표준 라이브러리 타입처럼 동작하는 사용자 정의 타입을 작성할 때, 우리는 동일한 요구 사항을 갖는다고 주장할 수 있습니다. 예를 들어, 다음 코드는 동일한 종류의 Box 가 Send 라면 Carton 도 Send 라고 주장하는데, 이 경우 이는 T 가 Send 라는 말과 같습니다.

#![allow(unused)]
fn main() {
struct Carton<T>(std::ptr::NonNull<T>);
unsafe impl<T> Send for Carton<T> where Box<T>: Send {}
}

지금 Carton<T> 는 할당한 메모리를 절대 해제하지 않으므로 메모리 누수가 있습니다. 이를 수정하면 Send 가 되기 위해 충족해야 할 새로운 요구 사항이 생깁니다. 우리는 다른 스레드에서 수행된 할당으로 얻은 포인터에 대해 free 를 호출할 수 있는지 알아야 합니다. libc::free 문서에서 이것이 사실임을 확인할 수 있습니다.

#![allow(unused)]
fn main() {
struct Carton<T>(std::ptr::NonNull<T>);
mod libc {
    pub use ::std::os::raw::c_void;
    unsafe extern "C" { pub fn free(p: *mut c_void); }
}
impl<T> Drop for Carton<T> {
    fn drop(&mut self) {
        unsafe {
            libc::free(self.0.as_ptr().cast());
        }
    }
}
}

이것이 성립하지 않는 좋은 예는 MutexGuard 입니다. Send 가 아님에 주목하세요. MutexGuard 의 구현은 다른 스레드에서 획득한 잠금을 해제하려고 시도하지 않도록 보장해야 하는 라이브러리를 사용합니다. 만약 MutexGuard 를 다른 스레드로 보낼(Send) 수 있다면, 소멸자가 보낸 스레드에서 실행되어 요구 사항을 위반하게 될 것입니다. MutexGuard 는 여전히 Sync 일 수 있는데, 다른 스레드로 보낼 수 있는 것은 &MutexGuard 뿐이고 참조를 드롭하는 것은 아무 일도 하지 않기 때문입니다.
