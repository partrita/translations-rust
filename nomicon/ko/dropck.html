드롭 체크

우리는 수명이 매달린 참조(dangling references)를 읽지 않도록 보장하는 몇 가지 간단한 규칙을 제공하는 것을 보았습니다. 하지만 지금까지 우리는 outlives 관계를 포함적인(inclusive) 방식으로만 다루어 왔습니다. 즉, 'a: 'b 라고 말할 때, 'a 가 'b 와 정확히 같은 기간 동안 사는 것도 괜찮았습니다. 언뜻 보기에 이는 무의미한 구분처럼 보입니다. 어떤 것도 다른 것과 동시에 드롭(dropped)되지 않으니까요, 그렇죠? 이것이 우리가 let 문을 다음과 같이 설탕 제거(desugaring)하여 사용한 이유입니다.

let x;
let y;

다음과 같이 설탕 제거됨:

{
    let x;
    {
        let y;
    }
}

스코프를 사용하여 설탕 제거를 할 수 없는 더 복잡한 상황들이 있지만, 순서는 여전히 정의되어 있습니다. 변수는 정의된 역순으로 드롭되고, 구조체와 튜플의 필드는 정의된 순서대로 드롭됩니다. 드롭 순서에 대한 더 자세한 내용은 RFC 1857에서 확인할 수 있습니다.

이렇게 해봅시다:

let tuple = (vec![], vec![]);

왼쪽 벡터가 먼저 드롭됩니다. 하지만 이것이 대여 검사기(borrow checker)의 관점에서 오른쪽 벡터가 왼쪽 벡터보다 엄격하게 더 오래 산다는 것을 의미할까요? 대답은 아니오 입니다. 대여 검사기는 튜플의 필드를 개별적으로 추적할 수도 있겠지만, 대여 검사기가 이해하지 못하는 순수 라이브러리 코드를 통해 수동으로 드롭되는 벡터 요소의 경우, 무엇이 무엇보다 더 오래 사는지 결정할 수 없을 것입니다.

그렇다면 왜 우리가 이것에 신경을 써야 할까요? 타입 시스템이 주의를 기울이지 않으면 실수로 매달린 포인터를 만들 수 있기 때문입니다. 다음의 간단한 프로그램을 고려해 보세요.

struct Inspector<'a>(&'a u8);

struct World<'a> {
    inspector: Option<Inspector<'a>>,
    days: Box<u8>,
}

fn main() {
    let mut world = World {
        inspector: None,
        days: Box::new(1),
    };
    world.inspector = Some(Inspector(&world.days));
}

이 프로그램은 완전히 건전하며 현재 컴파일됩니다. days 가 inspector 보다 엄격하게 더 오래 살지 않는다는 사실은 중요하지 않습니다. inspector 가 살아있는 동안은 days 도 살아있기 때문입니다.

하지만 소멸자를 추가하면, 프로그램은 더 이상 컴파일되지 않습니다!

struct Inspector<'a>(&'a u8);

impl<'a> Drop for Inspector<'a> {
    fn drop(&mut self) {
        println!("은퇴까지 고작 {}일 남았었는데!", self.0);
    }
}

struct World<'a> {
    inspector: Option<Inspector<'a>>,
    days: Box<u8>,
}

fn main() {
    let mut world = World {
        inspector: None,
        days: Box::new(1),
    };
    world.inspector = Some(Inspector(&world.days));
    // 만약 `days` 가 우연히 먼저 드롭된다고 가정해 봅시다.
    // 그러면 Inspector가 드롭될 때, 이미 해제된 메모리를 읽으려고 시도할 것입니다!
}

필드의 순서를 바꾸거나 구조체 대신 튜플을 사용해 볼 수도 있지만, 여전히 컴파일되지 않을 것입니다.

Drop을 구현하면 Inspector가 소멸하는 동안 임의의 코드를 실행할 수 있게 됩니다. 이는 자신만큼 오래 살아야 하는 타입들이 실제로는 먼저 파괴되었다는 것을 잠재적으로 관찰할 수 있음을 의미합니다.

흥미롭게도, 제네릭 타입만이 이를 걱정해야 합니다. 제네릭이 아니라면, 가질 수 있는 유일한 수명은 진정으로 영원히 사는 'static 뿐이기 때문입니다. 이것이 바로 이 문제를 건전한 제네릭 드롭(sound generic drop) 이라고 부르는 이유입니다. 건전한 제네릭 드롭은 드롭 검사기(drop checker) 에 의해 강제됩니다. 이 글을 쓰는 시점에서 드롭 검사기(dropck라고도 함)가 타입을 검증하는 방식에 대한 세부적인 사항 중 일부는 아직 정해지지 않았습니다. 하지만 ’커다란 규칙(The Big Rule)’은 우리가 이 섹션 전체에서 집중해 온 미묘한 차이입니다.

제네릭 타입이 건전하게 드롭을 구현하려면, 그 제네릭 인자들이 해당 타입보다 엄격하게 더 오래 살아야 합니다.

이 규칙을 준수하는 것은 (대개) 대여 검사기를 만족시키기 위해 필요합니다. 이 규칙을 준수하는 것은 건전함을 보장하기에 충분하지만 필수적인 것은 아닙니다. 즉, 여러분의 타입이 이 규칙을 따른다면 드롭하는 것이 확실히 건전합니다.

위의 규칙을 항상 만족해야 할 필요가 없는 이유는, 일부 Drop 구현이 타입상으로는 접근할 수 있음에도 불구하고 빌린 데이터에 접근하지 않거나, 우리가 특정 드롭 순서를 알고 있어서 대여 검사기가 모르더라도 빌린 데이터가 여전히 유효하기 때문입니다.

예를 들어, 위 Inspector 예제의 이 변형은 빌려온 데이터에 절대 접근하지 않습니다.

struct Inspector<'a>(&'a u8, &'static str);

impl<'a> Drop for Inspector<'a> {
    fn drop(&mut self) {
        println!("Inspector(_, {})는 언제 검사하지 *않아야* 할지 알고 있습니다.", self.1);
    }
}

struct World<'a> {
    inspector: Option<Inspector<'a>>,
    days: Box<u8>,
}

fn main() {
    let mut world = World {
        inspector: None,
        days: Box::new(1),
    };
    world.inspector = Some(Inspector(&world.days, "gadget"));
    // 만약 `days` 가 우연히 먼저 드롭된다고 가정해 봅시다.
    // Inspector가 드롭되더라도, 소멸자는 빌려온 `days` 에 접근하지 않을 것입니다.
}

마찬가지로, 이 변형 또한 빌려온 데이터에 절대 접근하지 않습니다.

struct Inspector<T>(T, &'static str);

impl<T> Drop for Inspector<T> {
    fn drop(&mut self) {
        println!("Inspector(_, {})는 언제 검사하지 *않아야* 할지 알고 있습니다.", self.1);
    }
}

struct World<T> {
    inspector: Option<Inspector<T>>,
    days: Box<u8>,
}

fn main() {
    let mut world = World {
        inspector: None,
        days: Box::new(1),
    };
    world.inspector = Some(Inspector(&world.days, "gadget"));
    // 만약 `days` 가 우연히 먼저 드롭된다고 가정해 봅시다.
    // Inspector가 드롭되더라도, 소멸자는 빌려온 `days` 에 접근하지 않을 것입니다.
}

하지만 위 두 변형 모두 fn main 분석 과정에서 days 가 충분히 오래 살지 않는다는 이유로 대여 검사기에 의해 거부됩니다.

그 이유는 main 의 빌림 검사 분석이 각 Inspector 의 Drop 구현 내부를 모르기 때문입니다. 빌림 검사기가 main 을 분석하는 동안 아는 한, 검사기(inspector)의 소멸자 본문은 빌려온 데이터에 접근할 수도 있습니다.

따라서 드롭 검사기는 값 안에 있는 모든 빌려온 데이터가 해당 값보다 엄격하게 더 오래 살도록 강제합니다.

탈출구 (An Escape Hatch)

드롭 검사를 규정하는 정확한 규칙은 미래에 덜 제한적으로 변할 수도 있습니다.

The current analysis is deliberately conservative; it forces all borrowed data in a value to outlive that value, which is certainly sound.

미래 버전의 언어에서는 분석이 더 정밀해져서, 건전한 코드가 안전하지 않다는 이유로 거부되는 사례를 줄일 수 있을 것입니다. 이는 위에서 살펴본 파괴되는 동안 검사하지 않는 두 Inspector 와 같은 사례를 해결하는 데 도움이 될 것입니다.

그동안은 제네릭 타입의 소멸자가 만료된 데이터에 접근하지 않는다는 것이 (비안전하게) 보장됨을 단언(assert)하기 위해 사용할 수 있는 불안정(unstable) 속성이 있습니다. 타입 자체는 접근 능력을 제공하더라도 말이죠.

그 속성은 may_dangle 이라고 불리며 RFC 1327 에서 도입되었습니다. 위의 Inspector 에 적용하려면 다음과 같이 작성합니다:

#![feature(dropck_eyepatch)]

struct Inspector<'a>(&'a u8, &'static str);

unsafe impl<#[may_dangle] 'a> Drop for Inspector<'a> {
    fn drop(&mut self) {
        println!("Inspector(_, {})는 언제 검사하지 *않아야* 할지 알고 있습니다.", self.1);
    }
}

struct World<'a> {
    days: Box<u8>,
    inspector: Option<Inspector<'a>>,
}

fn main() {
    let mut world = World {
        inspector: None,
        days: Box::new(1),
    };
    world.inspector = Some(Inspector(&world.days, "gadget"));
}

이 속성을 사용하려면 Drop 구현을 unsafe 로 표시해야 합니다. 컴파일러는 잠재적으로 만료된 데이터(예: 위의 self.0)에 접근하지 않는다는 암묵적인 단언을 검사하지 않기 때문입니다.

이 속성은 임의의 개수의 수명 및 타입 매개변수에 적용될 수 있습니다. 다음 예제에서, 우리는 수명 'b 를 가진 참조 뒤의 데이터에 접근하지 않으며 T 는 이동되거나 드롭되는 용도로만 사용된다고 단언합니다. 하지만 'a 와 U 에서는 이 속성을 생략했는데, 해당 수명과 타입을 가진 데이터에는 실제로 접근하기 때문입니다:

#![allow(unused)]
#![feature(dropck_eyepatch)]
fn main() {
use std::fmt::Display;

struct Inspector<'a, 'b, T, U: Display>(&'a u8, &'b u8, T, U);

unsafe impl<'a, #[may_dangle] 'b, #[may_dangle] T, U: Display> Drop for Inspector<'a, 'b, T, U> {
    fn drop(&mut self) {
        println!("Inspector({}, _, _, {})", self.0, self.3);
    }
}
}

위의 경우처럼 그런 접근이 발생할 수 없음이 명백할 때도 있습니다. 하지만 제네릭 타입 매개변수를 다룰 때는 그런 접근이 간접적으로 발생할 수 있습니다. 간접적인 접근의 예는 다음과 같습니다:

  • 콜백 호출,
  • 트레잇 메서드 호출을 경유.

(구현 전문화(impl specialization)와 같이 미래에 언어에 추가될 변경 사항들이 이런 간접 접근의 또 다른 통로를 열 수도 있습니다.)

콜백 호출의 예시는 다음과 같습니다:

#![allow(unused)]
fn main() {
struct Inspector<T>(T, &'static str, Box<for <'r> fn(&'r T) -> String>);

impl<T> Drop for Inspector<T> {
    fn drop(&mut self) {
        // `self.2` 호출은 가령 `T` 가 `&'a _` 라면 빌림에 접근할 수 있습니다.
        println!("Inspector({}, {})는 무의식중에 만료된 데이터를 검사합니다.",
                 (self.2)(&self.0), self.1);
    }
}
}

트레잇 메서드 호출의 예시는 다음과 같습니다:

#![allow(unused)]
fn main() {
use std::fmt;

struct Inspector<T: fmt::Display>(T, &'static str);

impl<T: fmt::Display> Drop for Inspector<T> {
    fn drop(&mut self) {
        // 아래에는 숨겨진 `<T as Display>::fmt` 호출이 있으며,
        // 가령 `T` 가 `&'a _` 라면 빌림에 접근할 수 있습니다.
        println!("Inspector({}, {})는 무의식중에 만료된 데이터를 검사합니다.",
                 self.0, self.1);
    }
}
}

물론, 이러한 모든 접근은 소멸자 내에 직접 작성되는 대신 소멸자가 호출하는 다른 메서드 내에 더 깊이 숨겨져 있을 수도 있습니다.

소멸자에서 &'a u8 에 접근하는 위의 모든 사례에서 #[may_dangle] 속성을 추가하면, 빌림 검사기가 잡아내지 못하는 오용에 타입이 취약해져 대혼란을 초래하게 됩니다. 속성을 추가하지 않는 것이 좋습니다.

드롭 순서에 관한 관련 사이드 노트

구조체 내부 필드의 드롭 순서는 정의되어 있지만, 여기에 의존하는 것은 깨지기 쉽고 미묘합니다. 순서가 중요하다면 ManuallyDrop 래퍼를 사용하는 것이 더 좋습니다.

드롭 검사기에 대한 내용은 이게 전부인가요?

알고 보니 unsafe 코드를 작성할 때, 우리는 일반적으로 드롭 검사기를 위해 올바른 일을 하는 것에 대해 전혀 걱정할 필요가 없습니다. 하지만 걱정해야 할 한 가지 특별한 경우가 있는데, 그것은 다음 섹션에서 살펴보겠습니다.
