서브타이핑과 공변성

러스트는 대여와 소유권 간의 관계를 추적하기 위해 수명을 사용합니다. 하지만 수명을 단순히 구현하면 너무 제한적이거나 정의되지 않은 동작을 허용하게 될 수 있습니다.

수명의 유연한 사용을 허용하면서도 오용을 방지하기 위해, 러스트는 서브타이핑(subtyping) 과 공변성(variance) 을 사용합니다.

예제를 통해 시작해 보겠습니다.

// 참고: debug는 *동일한* 수명을 가진 두 개의 매개변수를 기대합니다
fn debug<'a>(a: &'a str, b: &'a str) {
    println!("a = {a:?} b = {b:?}");
}

fn main() {
    let hello: &'static str = "hello";
    {
        let world = String::from("world");
        let world = &world; // 'world는 'static보다 수명이 짧습니다
        debug(hello, world);
    }
}

보수적인 수명 구현에서는 hello 와 world 가 서로 다른 수명을 가지므로 다음과 같은 에러를 보게 될 것입니다.

error[E0308]: mismatched types
 --> src/main.rs:10:16
   |
10 |         debug(hello, world);
   |                      ^
   |                      |
   |                      expected `&'static str`, found struct `&'world str`

이것은 꽤 안타까운 일입니다. 이 경우 우리가 원하는 것은 적어도 'world 만큼 오래 사는 모든 타입을 수용하는 것입니다. 수명에 서브타이핑을 적용해 봅시다.

서브타이핑(Subtyping)

서브타이핑은 하나의 타입을 다른 타입 대신 사용할 수 있다는 개념입니다.

Sub 가 Super 의 서브타입이라고 정의합시다(이 장 전체에서 Sub <: Super 라는 표기법을 사용할 것입니다).

이것이 우리에게 시사하는 바는 Super 가 정의하는 요구 사항 의 집합을 Sub 가 완전히 만족시킨다는 것입니다. Sub 는 그보다 더 많은 요구 사항을 가질 수도 있습니다.

이제 수명과 함께 서브타이핑을 사용하기 위해, 수명의 요구 사항을 정의해야 합니다.

'a 는 코드의 영역을 정의합니다.

이제 수명에 대한 요구 사항 집합이 정의되었으므로, 이들이 서로 어떻게 관련되는지 정의할 수 있습니다.

'long 이 'short 를 완전히 포함하는 코드 영역을 정의할 때만 'long <: 'short 가 성립합니다.

'long 은 'short 보다 더 큰 영역을 정의할 수 있지만, 이는 여전히 우리의 정의에 부합합니다.

이 장의 나머지 부분에서 보게 되겠지만, 서브타이핑은 이보다 훨씬 복잡하고 미묘합니다. 하지만 이 간단한 규칙은 99%의 경우에 매우 좋은 직관을 제공합니다. 그리고 안전하지 않은 코드를 작성하지 않는 한, 컴파일러가 여러분을 위해 모든 코너 케이스를 자동으로 처리해 줄 것입니다.

하지만 여기는 러스트노미콘입니다. 우리는 안전하지 않은 코드를 작성하고 있으므로, 이 모든 것이 실제로 어떻게 작동하는지, 그리고 우리가 어떻게 망칠 수 있는지 이해해야 합니다.

위의 예제로 돌아가서, 우리는 'static <: 'world 라고 말할 수 있습니다. 지금은 수명의 서브타입이 참조를 통해 전달될 수 있다는 개념도 받아들입시다(공변성 에서 자세히 다룰 예정입니다). 즉, &'static str 은 &'world str 의 서브타입이므로, &'static str 을 &'world str 로 “다운그레이드“할 수 있습니다. 그러면 위의 예제는 컴파일될 것입니다.

fn debug<'a>(a: &'a str, b: &'a str) {
    println!("a = {a:?} b = {b:?}");
}

fn main() {
    let hello: &'static str = "hello";
    {
        let world = String::from("world");
        let world = &world; // 'world는 'static보다 수명이 짧습니다
        debug(hello, world); // hello는 조용히 `&'static str` 에서 `&'world str` 로 다운그레이드됩니다
    }
}

공변성(Variance)

위에서 우리는 'static <: 'b 가 &'static T <: &'b T 를 의미한다는 사실을 대충 넘어갔습니다. 이는 공변성(variance) 이라고 알려진 속성을 사용합니다. 하지만 항상 이 예제처럼 간단한 것은 아닙니다. 이를 이해하기 위해 예제를 조금 확장해 보겠습니다.

fn assign<T>(input: &mut T, val: T) {
    *input = val;
}

fn main() {
    let mut hello: &'static str = "hello";
    {
        let world = String::from("world");
        assign(&mut hello, &world);
    }
    println!("{hello}"); // 해제 후 사용(use after free) 😿
}

assign 에서 우리는 hello 참조가 world 를 가리키도록 설정하고 있습니다. 하지만 그 후 world 가 스코프를 벗어나고, 나중에 println! 에서 hello 가 사용됩니다!

이것은 전형적인 해제 후 사용 버그입니다!

우리의 첫 번째 본능은 assign 구현을 탓하는 것이겠지만, 여기에는 잘못된 것이 전혀 없습니다. T 를 T 에 할당하고 싶어 하는 것은 놀라운 일이 아닙니다.

문제는 &mut &'static str 과 &mut &'b str 가 호환된다고 가정할 수 없다는 점입니다. 이는 'static 이 'b 의 서브타입이라 하더라도, &mut &'static str 는 &mut &'b str 의 서브타입 이 될 수 없음 을 의미합니다.

공변성은 러스트가 제네릭 매개변수를 통해 서브타입 관계를 정의하기 위해 차용한 개념입니다.

참고: 편의를 위해 T 에 대해 쉽게 이야기할 수 있도록 제네릭 타입 F<T> 를 정의하겠습니다. 문맥상 명확하기를 바랍니다.

타입 F 의 공변성 은 입력의 서브타이핑이 출력의 서브타이핑에 어떤 영향을 미치는지에 대한 것입니다. 러스트에는 세 가지 종류의 공변성이 있습니다. Sub 가 Super 의 서브타입인 두 타입 Sub 와 Super 가 주어졌을 때:

  • F<Sub> 가 F<Super> 의 서브타입이면 F 는 공변적(covariant) 입니다 (서브타입 속성이 전달됨)
  • F<Super> 가 F<Sub> 의 서브타입이면 F 는 반공변적(contravariant) 입니다 (서브타입 속성이 “뒤집힘”)
  • 그렇지 않으면 F 는 불변적(invariant) 입니다 (서브타이핑 관계가 존재하지 않음)

위의 예제를 기억한다면, 'a <: 'b 일 때 &'a T 를 &'b T 의 서브타입으로 취급하는 것이 괜찮았습니다. 따라서 우리는 &'a T 가 'a 에 대해 공변적 이라고 말할 수 있습니다.

Also, we saw that it was not ok for us to treat &mut &'a T as a subtype of &mut &'b T, therefore we can say that &mut T is invariant over T

다음은 몇 가지 다른 제네릭 타입과 그 공변성에 대한 표입니다:

’aTU
&'a T 공변(covariant)공변(covariant)
&'a mut T공변(covariant)불변(invariant)
Box<T>공변(covariant)
Vec<T>공변(covariant)
UnsafeCell<T>불변(invariant)
Cell<T>불변(invariant)
fn(T) -> U반공변(contravariant)공변(covariant)
*const T공변(covariant)
*mut T불변(invariant)

이들 중 일부는 다른 것들과의 관계를 통해 간단히 설명될 수 있습니다.

  • Vec<T> 와 다른 모든 소유 포인터 및 컬렉션은 Box<T> 와 동일한 논리를 따릅니다
  • Cell<T> 와 다른 모든 내부 가변성 타입은 UnsafeCell<T> 와 동일한 논리를 따릅니다
  • UnsafeCell<T> 는 내부 가변성을 가지므로 &mut T 와 동일한 공변성 속성을 가집니다
  • *const T 는 &T 의 논리를 따릅니다
  • *mut T 는 &mut T(또는 UnsafeCell<T>)의 논리를 따릅니다

더 많은 타입에 대해서는 참조서의 “공변성” 섹션 을 확인하세요.

참고: 언어에서 반공변성의 유일한 원천은 함수의 인자입니다. 이것이 실제로 실전에서 반공변성이 많이 등장하지 않는 이유입니다. 반공변성을 호출하려면 특정 수명을 가진 참조를 받는 함수 포인터를 사용하는 고차 프로그래밍이 필요합니다(일반적인 “아무 수명이나” 받는 것과는 대조적인데, 후자는 서브타이핑과 독립적으로 작동하는 고차 수명으로 들어갑니다).

이제 공변성에 대해 좀 더 형식적으로 이해했으니, 몇 가지 예제를 더 자세히 살펴보겠습니다.

fn assign<T>(input: &mut T, val: T) {
    *input = val;
}

fn main() {
    let mut hello: &'static str = "hello";
    {
        let world = String::from("world");
        assign(&mut hello, &world);
    }
    println!("{hello}");
}

그리고 이것을 실행하면 무엇을 얻게 될까요?

error[E0597]: `world` 가 충분히 오래 살지 않습니다
  --> src/main.rs:9:28
   |
6  |     let mut hello: &'static str = "hello";
   |                    ------------ 타입 주석은 `world` 가 `'static` 동안 대여될 것을 요구합니다
...
9  |         assign(&mut hello, &world);
   |                            ^^^^^^ 대여된 값이 충분히 오래 살지 않습니다
10 |     }
   |     - `world` 가 여기서 드롭되지만 여전히 대여 중입니다

좋습니다. 컴파일되지 않는군요! 여기서 무슨 일이 일어나고 있는지 자세히 분석해 봅시다.

먼저 assign 함수를 살펴봅시다.

#![allow(unused)]
fn main() {
fn assign<T>(input: &mut T, val: T) {
    *input = val;
}
}

이 함수가 하는 일은 가변 참조와 값을 받아서 지칭 대상을 덮어쓰는 것뿐입니다. 이 함수에서 중요한 점은 타입 동일성 제약을 생성한다는 것입니다. 시그니처에서 지칭 대상과 값이 정확히 같은 타입이어야 한다고 명확하게 명시하고 있습니다.

한편, 호출자에서는 &mut &'static str 과 &'world str 을 전달합니다.

&mut T 는 T 에 대해 불변적이므로, 컴파일러는 첫 번째 인자에 어떤 서브타이핑도 적용할 수 없다고 결론 내립니다. 따라서 T 는 정확히 &'static str 이어야 합니다.

이는 &T 의 경우와는 반대됩니다.

#![allow(unused)]
fn main() {
fn debug<T: std::fmt::Debug>(a: T, b: T) {
    println!("a = {a:?} b = {b:?}");
}
}

여기서도 마찬가지로 a 와 b 는 같은 타입 T 를 가져야 합니다. 하지만 &'a T 는 'a 에 대해 공변적 이므로 서브타이핑을 수행할 수 있습니다. 따라서 컴파일러는 &'static str 이 &'b str 의 서브타입인 경우에만 &'b str 이 될 수 있다고 결정합니다. 이는 'static <: 'b 일 때 성립합니다. 이것은 참이므로 컴파일러는 기꺼이 이 코드를 계속 컴파일합니다.

알고 보니, Box(그리고 Vec, HashMap 등)가 공변적이어도 괜찮은 이유는 수명이 공변적이어도 괜찮은 이유와 꽤 비슷합니다. 가변 참조와 같은 것에 넣으려고 하는 순간 불변성을 상속받게 되어 나쁜 짓을 하는 것이 방지됩니다.

하지만 Box는 우리가 부분적으로 대충 넘어갔던 참조의 값에 의한(by-value) 측면에 집중하기 쉽게 해줍니다.

값이 항상 자유롭게 별칭되는 것을 허용하는 많은 언어들과 달리, 러스트는 매우 엄격한 규칙을 가지고 있습니다. 값을 변경하거나 이동할 수 있다면, 당신만이 그 값에 접근할 수 있는 유일한 존재임이 보장됩니다.

다음 코드를 고려해 보세요:

let hello: Box<&'static str> = Box::new("hello");

let mut world: Box<&'b str>;
world = hello;

hello 가 'static 동안 살아있었다는 사실을 잊어버린 것은 전혀 문제가 되지 않습니다. hello 를 'b 동안만 살아있다고 아는 변수로 이동시키는 순간, 우리는 그것이 더 오래 산다는 것을 기억하는 우주의 유일한 존재를 파괴했기 때문입니다!

설명할 것이 하나 남았습니다: 함수 포인터.

fn(T) -> U 가 U 에 대해 공변적이어야 하는 이유를 알아보기 위해 다음 시그니처를 고려해 보세요:

fn get_str() -> &'a str;

이 함수는 어떤 수명 'a 에 의해 제한된 str 을 생성한다고 주장합니다. 따라서 대신 다음 시그니처를 가진 함수를 제공하는 것은 완전히 유효합니다.

fn get_static() -> &'static str;

따라서 함수가 호출될 때 기대하는 것은 적어도 'a 수명만큼 사는 &str 일 뿐이며, 값이 실제로 더 오래 사는지는 중요하지 않습니다.

하지만 동일한 논리가 인자 에는 적용되지 않습니다. 다음을 만족시키려 한다고 가정해 보세요:

fn store_ref(&'a str);

다음을 사용해서:

fn store_static(&'static str);

첫 번째 함수는 적어도 'a 만큼 사는 한 어떤 문자열 참조든 받을 수 있지만, 두 번째 함수는 'static 보다 짧게 사는 문자열 참조를 받을 수 없으므로 충돌이 발생합니다. 여기서는 공변성이 작동하지 않습니다. 하지만 뒤집어 보면 실제로 작동합니다! &'static str 을 처리할 수 있는 함수가 필요하다면, 어떤 참조 수명이라도 처리할 수 있는 함수는 확실히 잘 작동할 것입니다.

실제로 확인해 봅시다

use std::cell::RefCell;
thread_local! {
    pub static StaticVecs: RefCell<Vec<&'static str>> = RefCell::new(Vec::new());
}

/// 주어진 입력을 스레드 로컬 `Vec<&'static str>` 에 저장합니다
fn store(input: &'static str) {
    StaticVecs.with_borrow_mut(|v| v.push(input));
}

/// 입력을 가지고 함수를 호출합니다 (반드시 같은 수명이어야 합니다!)
fn demo<'a>(input: &'a str, f: fn(&'a str)) {
    f(input);
}

fn main() {
    demo("hello", store); // "hello"는 'static입니다. `store` 를 문제없이 호출할 수 있습니다

    {
        let smuggle = String::from("smuggle");

        // `&smuggle` 은 static이 아닙니다. 만약 `store` 를 `&smuggle` 로 호출한다면,
        // 유효하지 않은 수명을 `StaticVecs` 에 넣게 됩니다.
        // 따라서 `fn(&'static str)` 은 `fn(&'a str)` 의 서브타입이 될 수 없습니다
        demo(&smuggle, store);
    }

    // 해제 후 사용(use after free) 😿
    StaticVecs.with_borrow(|v| println!("{v:?}"));
}

이것이 바로 함수 타입이 언어의 다른 것들과 달리 그 인자에 대해 반공변성(contravariant) 을 갖는 이유입니다.

표준 라이브러리가 제공하는 타입들의 경우는 이 정도로 충분하지만, 여러분이 직접 정의하는 타입의 공변성은 어떻게 결정될까요? 비공식적으로 말하자면, 구조체는 필드의 공변성을 상속합니다. 만약 구조체 MyType 이 필드 a 에서 사용되는 제네릭 인자 A 를 가지고 있다면, MyType 의 A 에 대한 공변성은 정확히 a 의 A 에 대한 공변성과 같습니다.

하지만 A 가 여러 필드에서 사용된다면 다음과 같습니다.

  • A 를 사용하는 모든 곳이 공변적이라면, MyType 은 A 에 대해 공변합니다.
  • A 를 사용하는 모든 곳이 반공변적이라면, MyType 은 A 에 대해 반공변합니다.
  • 그렇지 않으면 MyType 은 A 에 대해 불변입니다.
#![allow(unused)]
fn main() {
use std::cell::Cell;

struct MyType<'a, 'b, A: 'a, B: 'b, C, D, E, F, G, H, In, Out, Mixed> {
    a: &'a A,     // 'a와 A에 대해 공변
    b: &'b mut B, // 'b에 대해 공변이고 B에 대해 불변

    c: *const C,  // C에 대해 공변
    d: *mut D,    // D에 대해 불변

    e: E,         // E에 대해 공변
    f: Vec<F>,    // F에 대해 공변
    g: Cell<G>,   // G에 대해 불변

    h1: H,        // ...가 아니었다면 H에 대해서도 공변이었을 것입니다
    h2: Cell<H>,  // H에 대해 불변입니다. 모든 충돌에서 불변성이 우선하기 때문입니다.

    i: fn(In) -> Out,       // In에 대해 반공변, Out에 대해 공변

    k1: fn(Mixed) -> usize, // ...가 아니었다면 Mixed에 대해 반공변이었을 것입니다
    k2: Mixed,              // Mixed에 대해 불변입니다. 모든 충돌에서 불변성이 우선하기 때문입니다.
}
}
