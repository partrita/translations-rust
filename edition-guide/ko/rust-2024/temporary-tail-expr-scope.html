The Rust Edition Guide

꼬리 표현식 임시 스코프

요약

  • 함수 또는 클로저 본문, 또는 블록의 꼬리 표현식을 평가할 때 생성된 임시 값은 이제 지역 변수보다 먼저 삭제될 수 있으며, 때로는 다음으로 큰 임시 스코프로 확장되지 않을 수 있습니다.

세부 정보

2024 에디션은 꼬리 표현식의 임시 값 삭제 순서를 변경합니다. 2024 에디션 이전에는 꼬리 표현식의 임시 값이 블록 자체보다 오래 살 수 있고 지역 변수 바인딩보다 나중에 삭제된다는 사실이 종종 놀라움으로 다가왔습니다. 다음 예를 참조하세요:

#![allow(unused)]
fn main() {
// 2024년 이전
use std::cell::RefCell;
fn f() -> usize {
    let c = RefCell::new("..");
    c.borrow().len() // error[E0597]: `c`가 충분히 오래 지속되지 않습니다
}
}

2021 에디션에서는 다음과 같은 오류가 발생합니다:

error[E0597]: `c` does not live long enough
 --> src/lib.rs:4:5
  |
3 |     let c = RefCell::new("..");
  |         - binding `c` declared here
4 |     c.borrow().len() // error[E0597]: `c` does not live long enough
  |     ^---------
  |     |
  |     borrowed value does not live long enough
  |     a temporary with access to the borrow is created here ...
5 | }
  | -
  | |
  | `c` dropped here while still borrowed
  | ... and the borrow might be used here, when that temporary is dropped and runs the destructor for type `Ref<'_, &str>`

          consider forcing this temporary to be dropped sooner, before the block's local variables are dropped
help: for example, you could save the expression's value in a new local variable `x` and then make `x` be the expression at the end of the block
  |
4 |     let x = c.borrow().len(); x // error[E0597]: `c` does not live long enough
  |     +++++++                 +++

For more information about this error, try `rustc --explain E0597`.

2021 에디션에서는 지역 변수 c가 c.borrow()에 의해 생성된 임시 값보다 먼저 드롭됩니다. 2024 에디션에서는 이를 변경하여 임시 값 c.borrow()가 먼저 드롭되고 이어서 지역 변수 c가 드롭되도록 하여 코드가 예상대로 컴파일되도록 합니다.

임시 스코프가 축소될 수 있습니다

When a temporary is created in order to evaluate an expression, the temporary is dropped based on the temporary scope rules. Those rules define how long the temporary will be kept alive. Before 2024, temporaries from tail expressions of a block would be extended outside the block to the next temporary scope boundary. In many cases this would be the end of a statement or function body. In 2024, the temporaries of the tail expression may now be dropped immediately at the end of the block (before any local variables in the block).

이러한 임시 스코프의 축소로 인해 2024년에는 프로그램 컴파일이 실패할 수 있습니다. 예를 들어:

// 이 예제는 2021에서는 작동하지만 2024에서는 컴파일에 실패합니다.
fn main() {
    let x = { &String::from("1234") }.len();
}

이 예제에서 2021 에디션의 경우 임시 String은 블록 외부로 확장되어 len() 호출을 지나 문장 끝에서 드롭됩니다. 2024 에디션에서는 블록 끝에서 즉시 드롭되므로, 빌려온(borrowed) 상태에서 임시 값이 드롭된다는 컴파일 오류가 발생합니다.

이러한 상황에 대한 해결책은 임시 값이 충분히 오래 유지되도록 블록 표현식을 지역 변수로 끌어내는 것입니다.

fn main() {
    let s = { &String::from("1234") };
    let x = s.len();
}

이 특정 예제는 임시 라이프타임 연장(temporary lifetime extension)을 활용합니다. 임시 라이프타임 연장은 임시 값이 평소보다 더 오래 유지되도록 허용하는 특정 규칙 세트입니다. String 임시 값이 참조 뒤에 있기 때문에, String 임시 값은 다음 문장에서 len()을 호출할 수 있을 만큼 충분히 연장됩니다.

if let 표현식의 임시 스코프에 적용된 유사한 변경 사항은 if let 임시 스코프 장을 참조하세요.

마이그레이션

안타깝게도, 테일 표현식의 임시 값 라이프타임을 단축하기 위한 의미 보존형 다시 작성(semantics-preserving rewrites) 방법은 없습니다1. tail_expr_drop_order 린트는 테일 표현식에서 사용자 정의, 사소하지 않은 Drop 소멸자가 있는 임시 값이 생성되는지 감지합니다. 이 린트의 경고는 cargo fix --edition을 실행할 때 나타나지만, 그 외에는 자동으로 변경 사항을 만들지 않습니다. 경고를 수동으로 검사하고 조정이 필요한지 여부를 결정하는 것이 권장됩니다.

에디션 마이그레이션을 수행하지 않고 이러한 경고를 수동으로 검사하려면 다음과 같이 린트를 활성화할 수 있습니다.

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(tail_expr_drop_order)]
}
1

자세한 내용은 RFC 3606에 기록되어 있습니다.
