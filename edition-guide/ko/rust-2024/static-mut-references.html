static mut 에 대한 참조 금지

요약

  • static_mut_refs 린트 레벨이 이제 기본적으로 deny 입니다. 이는 static mut 에 대한 공유 또는 가변 참조를 가져오는 것을 검사합니다.

세부 정보

static_mut_refs 린트는 static mut 에 대한 참조를 가져오는 것을 감지합니다. 2024 에디션에서는 이러한 참조를 피해야 함을 강조하기 위해 이 린트가 기본적으로 deny 로 설정됩니다.

#![allow(unused)]
fn main() {
static mut X: i32 = 23;
static mut Y: i32 = 24;

unsafe {
    let y = &X;             // ERROR: 가변 static에 대한 공유 참조
    let ref x = X;          // ERROR: 가변 static에 대한 공유 참조
    let (x, y) = (&X, &Y);  // ERROR: 가변 static에 대한 공유 참조
}
}

Rust의 가변성 XOR 앨리어싱 요구 사항을 위반하여 이러한 참조를 가져오는 것은 해당 참조를 읽거나 쓰지 않더라도 항상 즉각적인 정의되지 않은 동작 이었습니다. 게다가 static mut 에 대한 가변성 XOR 앨리어싱을 준수하려면 코드를 전역적으로 추론 해야 하는데, 이는 재진입성이나 멀티스레딩 상황에서 특히 어려울 수 있습니다.

보이는 & 연산자 없이 암시적 참조가 자동으로 생성되는 경우가 있다는 점에 유의하세요. 예를 들어, 다음과 같은 상황에서도 린트가 발생합니다:

#![allow(unused)]
fn main() {
static mut NUMS: &[u8; 3] = &[0, 1, 2];

unsafe {
    println!("{NUMS:?}");   // ERROR: 가변 static에 대한 공유 참조
    let n = NUMS.len();     // ERROR: 가변 static에 대한 공유 참조
}
}

대안

가능하다면, Rust의 가변성 XOR 앨리어싱 요구 사항 준수를 보장하는 복잡성을 크게 줄여주는 지역적으로 추론된 추상화 뒤에서 내부 가변성 을 제공하는 타입의 불변 static 을 대신 사용하는 것이 강력히 권장됩니다.

지역적으로 추론된 추상화가 불가능하여 어쩔 수 없이 static 변수에 대한 접근을 전역적으로 추론해야 하는 상황이라면, 이제 &raw const 또는 &raw mut 연산자 를 통해 얻을 수 있는 원시 포인터를 사용해야 합니다. 참조를 직접 가져오는 대신 먼저 원시 포인터를 얻음으로써, 해당 포인터를 통한 접근(의 안전 요구 사항)은 unsafe 개발자에게 더 친숙할 것이며 더 작은 코드 영역으로 제한하거나 미룰 수 있습니다.

다음 예제들은 단지 설명일 뿐이며 완전한 구현을 의도한 것이 아닙니다. 있는 그대로 복사하지 마세요. 여러분의 특정 상황에 맞춰 변경해야 할 세부 사항이 있을 수 있습니다. 이 예제들은 문제에 접근하는 다양한 방법을 보여주기 위한 것입니다.

표준 라이브러리의 특정 타입에 대한 문서, 정의되지 않은 동작 에 대한 참조, Rustonomicon 을 읽어보는 것을 권장하며, 질문이 있다면 Users Forum 과 같은 Rust 포럼에 문의하세요.

전역 변수 사용하지 않기

이미 알고 계시겠지만, 가능하다면 가변 전역 상태는 피하는 것이 가장 좋습니다. 물론 때로는 조금 더 어색하거나 어려울 수 있으며, 특히 많은 함수 사이에서 가변 참조를 전달해야 할 때 그렇습니다.

아토믹 (Atomics)

아토믹 타입 은 static 에서 (mut 없이) 사용할 수 있는 정수, 포인터, 불리언을 제공합니다.

use std::sync::atomic::Ordering;
use std::sync::atomic::AtomicU64;

// 이렇게 변경:
//   static mut COUNTER: u64 = 0;
// 이것으로:
static COUNTER: AtomicU64 = AtomicU64::new(0);

fn main() {
    // 사용 사례를 분석하여 올바른 Ordering을 사용하도록 하세요.
    COUNTER.fetch_add(1, Ordering::Relaxed);
}

Mutex 또는 RwLock

타입이 아토믹보다 더 복잡하다면, Mutex 나 RwLock 을 사용하여 전역 값에 대한 올바른 접근을 보장하는 것을 고려하세요.

use std::sync::Mutex;
use std::collections::VecDeque;

// 이렇게 변경:
//     static mut QUEUE: VecDeque<String> = VecDeque::new();
// 이것으로:
static QUEUE: Mutex<VecDeque<String>> = Mutex::new(VecDeque::new());

fn main() {
    QUEUE.lock().unwrap().push_back(String::from("abc"));
    let first = QUEUE.lock().unwrap().pop_front();
}

OnceLock 또는 LazyLock

const 로 할 수 없는 일회성 초기화를 위해 static mut 을 사용하고 있다면, 대신 OnceLock 이나 LazyLock 을 사용할 수 있습니다.

use std::sync::LazyLock;

struct GlobalState;

impl GlobalState {
    fn new() -> GlobalState {
        GlobalState
    }
    fn example(&self) {}
}

// 다음과 같은 임시 또는 초기화되지 않은 타입 대신:
//     static mut STATE: Option<GlobalState> = None;
// 대신 이것을 사용하세요:
static STATE: LazyLock<GlobalState> = LazyLock::new(|| {
    GlobalState::new()
});

fn main() {
    STATE.example();
}

OnceLock 은 LazyLock 과 비슷하지만, 생성자에 정보를 전달해야 하는 경우 사용할 수 있습니다. 이는 (main 과 같은) 단일 초기화 지점과 잘 작동하거나, 전역 변수에 접근하는 곳 어디에서나 입력을 사용할 수 있는 경우에 유용합니다.

use std::sync::OnceLock;

struct GlobalState;

impl GlobalState {
    fn new(verbose: bool) -> GlobalState {
        GlobalState
    }
    fn example(&self) {}
}

struct Args {
    verbose: bool
}
fn parse_arguments() -> Args {
    Args { verbose: true }
}

static STATE: OnceLock<GlobalState> = OnceLock::new();

fn main() {
    let args = parse_arguments();
    let state = GlobalState::new(args.verbose);
    let _ = STATE.set(state);
    // ...
    STATE.get().unwrap().example();
}

no_std 일회성 초기화

이 예제는 전역 변수의 일회성 초기화를 제공한다는 점에서 OnceLock 과 유사하지만, std 를 필요로 하지 않아 no_std 환경에서 유용합니다. 타겟이 아토믹을 지원한다고 가정하면, 아토믹을 사용하여 전역 변수의 초기화 여부를 확인할 수 있습니다. 패턴은 다음과 비슷할 것입니다:

use core::sync::atomic::AtomicUsize;
use core::sync::atomic::Ordering;

struct Args {
    verbose: bool,
}
fn parse_arguments() -> Args {
    Args { verbose: true }
}

struct GlobalState {
    verbose: bool,
}

impl GlobalState {
    const fn default() -> GlobalState {
        GlobalState { verbose: false }
    }
    fn new(verbose: bool) -> GlobalState {
        GlobalState { verbose }
    }
    fn example(&self) {}
}

const UNINITIALIZED: usize = 0;
const INITIALIZING: usize = 1;
const INITIALIZED: usize = 2;

static STATE_INITIALIZED: AtomicUsize = AtomicUsize::new(UNINITIALIZED);
static mut STATE: GlobalState = GlobalState::default();

fn set_global_state(state: GlobalState) {
    if STATE_INITIALIZED
        .compare_exchange(
            UNINITIALIZED,
            INITIALIZING,
            Ordering::SeqCst,
            Ordering::SeqCst,
        )
        .is_ok()
    {
        // SAFETY: STATE에 대한 읽기 및 쓰기는 INITIALIZED 가드로 보호됩니다.
        unsafe {
            STATE = state;
        }
        STATE_INITIALIZED.store(INITIALIZED, Ordering::SeqCst);
    } else {
        panic!("이미 초기화되었거나 동시에 초기화 중");
    }
}

fn get_state() -> &'static GlobalState {
    if STATE_INITIALIZED.load(Ordering::Acquire) != INITIALIZED {
        panic!("초기화되지 않음");
    } else {
        // SAFETY: 상태가 초기화된 후에는 가변 접근이 불가능합니다.
        unsafe { &*&raw const STATE }
    }
}

fn main() {
    let args = parse_arguments();
    let state = GlobalState::new(args.verbose);
    set_global_state(state);
    // ...
    let state = get_state();
    state.example();
}

이 예제는 초기화 전에 static에 기본값을 넣을 수 있다고 가정합니다(이 예제에서는 const default 생성자). 불가능한 경우, MaybeUninit 을 사용하거나, 동적 트레이트 디스패치(트레이트를 구현하는 더미 타입 사용)를 사용하거나, 기본 자리 표시자를 갖는 다른 접근 방식을 고려해보세요.

비슷한 일회성 초기화를 제공하는 커뮤니티 제공 크레이트들이 있습니다. 예를 들어 static-cell 크레이트(portable-atomic 을 사용하여 아토믹이 없는 타겟도 지원함)가 있습니다.

원시 포인터

일부 경우에는 static mut 을 계속 사용할 수 있지만, 참조 생성은 피해야 합니다. 예를 들어, 원시 포인터 를 C 라이브러리에 전달하기만 하면 되는 경우 중간 참조를 생성하지 마세요. 대신 다음 예제처럼 원시 대여 연산자 를 사용할 수 있습니다:

#[repr(C)]
struct GlobalState {
    value: i32
}

impl GlobalState {
    const fn new() -> GlobalState {
        GlobalState { value: 0 }
    }
}

static mut STATE: GlobalState = GlobalState::new();

unsafe extern "C" {
    fn example_ffi(state: *mut GlobalState);
}

fn main() {
    unsafe {
        // 이렇게 변경:
        //     example_ffi(&mut STATE as *mut GlobalState);
        // 이것으로:
        example_ffi(&raw mut STATE);
    }
}

단, 가변 포인터에 대한 앨리어싱 제약 조건은 여전히 준수해야 합니다. 이를 위해서는 내부 또는 외부 동기화가 필요하거나, 스레드, 인터럽트 핸들러, 재진입성에 걸쳐 어떻게 사용되는지에 대한 증명이 필요할 수 있습니다.

Sync 와 함께 UnsafeCell 사용

UnsafeCell 은 Sync 를 구현하지 않으므로 static 에서 사용할 수 없습니다. UnsafeCell 을 감싸는 래퍼를 만들어 Sync 구현을 추가하면 static 에서 사용하여 내부 가변성을 구현할 수 있습니다. 이 접근 방식은 외부 락이나 가변 포인터에 필요한 안전 불변성을 유지하는 다른 보장이 있는 경우 유용할 수 있습니다.

이것은 원시 포인터 예제와 거의 동일하다는 점에 유의하세요. 래퍼는 타입을 어떻게 사용하는지 강조하고, 어떤 안전 요구 사항을 주의해야 하는지에 집중하는 데 도움을 줍니다. 하지만 그 외에는 거의 동일합니다.

#![allow(unused)]
fn main() {
use std::cell::UnsafeCell;

fn with_interrupts_disabled<T: Fn()>(f: T) {
    // 실제 예제에서는 인터럽트를 비활성화할 것입니다.
    f();
}

#[repr(C)]
struct GlobalState {
    value: i32,
}

impl GlobalState {
    const fn new() -> GlobalState {
        GlobalState { value: 0 }
    }
}

#[repr(transparent)]
pub struct SyncUnsafeCell<T>(UnsafeCell<T>);

unsafe impl<T: Sync> Sync for SyncUnsafeCell<T> {}

static STATE: SyncUnsafeCell<GlobalState> = SyncUnsafeCell(UnsafeCell::new(GlobalState::new()));

fn set_value(value: i32) {
    with_interrupts_disabled(|| {
        let state = STATE.0.get();
        unsafe {
            // SAFETY: 이 값은 인터럽트 핸들러에서만 읽히며,
            // 인터럽트는 비활성화되어 있고, 하나의 스레드에서만 사용합니다.
            (*state).value = value;
        }
    });
}
}

표준 라이브러리에는 UnsafeCell 의 나이틀리 전용(불안정) 변형인 SyncUnsafeCell 이 있습니다. 위 예제는 표준 라이브러리 타입의 매우 단순화된 버전을 보여주지만, 대략 같은 방식으로 사용될 것입니다. 더 나은 격리를 제공할 수 있으므로 자세한 내용은 해당 구현을 확인해 보세요.

이 예제에는 임베디드 환경에서 볼 수 있는 가상의 with_interrupts_disabled 함수가 포함되어 있습니다. 예를 들어, critical-section 크레이트는 임베디드 환경에서 사용할 수 있는 유사한 기능을 제공합니다.

안전한 참조

어떤 경우에는 static mut 의 참조를 생성하는 것이 안전할 수 있습니다. static_mut_refs 린트의 핵심은 이것을 올바르게 수행하기가 매우 어렵다는 것입니다! 하지만 불가능 하다는 말은 아닙니다. static이 좁은 범위(작은 모듈이나 함수 내에서만 사용됨)로 지정되어 있거나, 내부 또는 외부 동기화가 있거나, 인터럽트 핸들러와 재진입성, 패닉 안전성, 드롭 핸들러 등을 고려하는 등 앨리어싱 요구 사항이 준수됨을 보장할 수 있는 상황이라면 참조를 가져오는 것도 괜찮을 수 있습니다.

이를 위해 두 가지 접근 방식을 취할 수 있습니다. static_mut_refs 린트를 허용하거나(가급적 좁은 범위에서), &mut *&raw mut MY_STATIC 과 같이 원시 포인터를 참조로 변환할 수 있습니다.

수명이 짧은 참조

static mut 에 대한 참조를 생성해야 한다면, 그 참조가 존재하는 범위를 최소화하는 것이 좋습니다. 참조를 어딘가에 숨겨두거나 코드의 큰 섹션 동안 계속 살아있게 하지 마세요. 수명을 짧게 유지하면 감사가 쉬워지고 해당 기간 동안 배타적 접근이 유지되는지 확인하는 데 도움이 됩니다. 포인터를 기본 단위로 사용하고, 절대적으로 필요한 경우에만 요청 시 포인터를 참조로 변환하세요.

마이그레이션

static mut 에 대한 이러한 참조를 수정하는 자동 마이그레이션은 없습니다. 정의되지 않은 동작을 피하려면 대안 섹션에서 권장하는 다른 접근 방식을 사용하도록 코드를 다시 작성해야 합니다.
