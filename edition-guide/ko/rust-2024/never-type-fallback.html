Never 타입 폴백 변경

요약

  • Never 타입(!)에서 임의의 타입으로의("never-to-any") 강제 변환은 이제 유닛 타입(())이 아닌 never 타입(!)으로 폴백됩니다.
  • The never_type_fallback_flowing_into_unsafe lint is now deny by default.

세부 정보

컴파일러가 강제 변환 위치 에서 ! (never) 타입의 값을 보면, 타입 검사기가 어떤 타입이든 추론할 수 있도록 암시적으로 강제 변환을 삽입합니다:

#![allow(unused)]
fn main() {
#![feature(never_type)]
// 이것은:
let x: u8 = panic!();

// ...컴파일러에 의해 (본질적으로) 다음과 같이 변환됩니다:
let x: u8 = absurd(panic!());

// ...여기서 `absurd` 는 다음 함수입니다
// (`!` 는 항상 도달할 수 없는 코드를 표시하므로 안전합니다):
fn absurd<T>(x: !) -> T { x }
}

타입을 추론할 수 없는 경우 컴파일 오류가 발생할 수 있습니다:

#![allow(unused)]
fn main() {
#![feature(never_type)]
fn absurd<T>(x: !) -> T { x }
// 이것은:
{ panic!() };

// ...다음과 같이 변환됩니다:
{ absurd(panic!()) }; //~ ERROR `absurd` 의 타입을 추론할 수 없음
}

이러한 오류를 방지하기 위해 컴파일러는 absurd 호출을 삽입한 위치를 기억하고, 타입을 추론할 수 없는 경우 대신 폴백 타입을 사용합니다:

#![allow(unused)]
fn main() {
#![feature(never_type)]
fn absurd<T>(x: !) -> T { x }
type Fallback = /* 임의로 선택된 타입! */ !;
{ absurd::<Fallback>(panic!()) }
}

이것이 바로 "never 타입 폴백"이라고 알려진 것입니다.

역사적으로 폴백 타입은 () (유닛)이었습니다. 이로 인해 컴파일러가 폴백 없이는 () 를 추론하지 않을 때조차 ! 가 () 로 저절로 강제 변환되는 일이 발생했습니다. 이는 혼란스러웠으며 ! 타입의 안정화를 막아왔습니다.

2024 에디션에서는 폴백 타입이 이제 ! 입니다. (추후 모든 에디션에 걸쳐 이 변경을 적용할 계획입니다.) 이렇게 하면 더 직관적으로 작동합니다. 이제 ! 를 전달할 때 다른 것으로 강제 변환할 이유가 없다면 ! 로 유지됩니다.

어떤 경우에는 코드가 폴백 타입이 () 인 것에 의존할 수 있으므로, 이로 인해 컴파일 오류나 동작 변경이 발생할 수 있습니다.

never_type_fallback_flowing_into_unsafe

The default level of the never_type_fallback_flowing_into_unsafe lint has been raised from warn to deny in the 2024 Edition. This lint helps detect a particular interaction with the fallback to ! and unsafe code which may lead to undefined behavior. See the link for a complete description.

마이그레이션

자동 수정 기능은 없지만, 에디션 변경으로 인해 깨질 코드를 자동으로 감지하는 기능은 있습니다. 이전 에디션을 사용하는 동안 코드가 깨질 경우 경고가 표시됩니다.

해결 방법은 폴백 타입이 사용되지 않도록 타입을 명시적으로 지정하는 것입니다. 불행히도 어떤 타입을 지정해야 하는지 파악하는 것이 쉽지 않을 수 있습니다.

이 변경으로 인해 깨지는 가장 일반적인 패턴 중 하나는 f()?; 를 사용하는 경우인데, 여기서 f 는 반환 타입의 Ok 부분에 대해 제네릭입니다:

#![allow(unused)]
fn main() {
#![allow(dependency_on_unit_never_type_fallback)]
fn outer<T>(x: T) -> Result<T, ()> {
fn f<T: Default>() -> Result<T, ()> {
    Ok(T::default())
}

f()?;
Ok(x)
}
}

이 예제에서 타입 T 를 추론할 수 없다고 생각할 수 있습니다. 하지만 현재 ? 연산자의 탈탕(desugaring) 방식 때문에 이전에는 () 로 추론되었으나, 이제는 ! 로 추론됩니다.

이 문제를 해결하려면 T 타입을 명시적으로 지정해야 합니다:

#![allow(unused)]
fn main() {
#![deny(dependency_on_unit_never_type_fallback)]
fn outer<T>(x: T) -> Result<T, ()> {
fn f<T: Default>() -> Result<T, ()> {
    Ok(T::default())
}
f::<()>()?;
// ...또는:
() = f()?;
Ok(x)
}
}

비교적 흔한 또 다른 케이스는 클로저 내에서의 패닉입니다:

#![allow(unused)]
fn main() {
#![allow(dependency_on_unit_never_type_fallback)]
trait Unit {}
impl Unit for () {}

fn run<R: Unit>(f: impl FnOnce() -> R) {
    f();
}

run(|| panic!());
}

이전에는 panic! 의 ! 가 Unit 을 구현하는 () 로 강제 변환되었습니다. 하지만 이제는 ! 가 ! 로 유지되므로 ! 가 Unit 을 구현하지 않아 이 코드는 실패합니다. 이 문제를 해결하려면 클로저의 반환 타입을 지정할 수 있습니다:

#![allow(unused)]
fn main() {
#![deny(dependency_on_unit_never_type_fallback)]
trait Unit {}
impl Unit for () {}

fn run<R: Unit>(f: impl FnOnce() -> R) {
    f();
}
run(|| -> () { panic!() });
}

f()? 와 유사한 사례로, 한 분기에서는 ! 타입 표현식을 사용하고 다른 분기에서는 반환 타입이 제한되지 않은 함수를 사용할 때 볼 수 있습니다:

#![allow(unused)]
fn main() {
#![allow(dependency_on_unit_never_type_fallback)]
if true {
    Default::default()
} else {
    return
};
}

이전에는 return 의 ! 가 () 로 잘못 강제 변환되었기 때문에 Default::default() 의 반환 타입으로 () 가 추론되었습니다. 이제는 대신 ! 가 추론되어 ! 가 Default 를 구현하지 않으므로 이 코드는 컴파일되지 않습니다.

마찬가지로, 타입을 명시적으로 지정하여 해결할 수 있습니다:

#![allow(unused)]
fn main() {
#![deny(dependency_on_unit_never_type_fallback)]
() = if true {
    Default::default()
} else {
    return
};

// ...또는:

if true {
    <() as Default>::default()
} else {
    return
};
}
