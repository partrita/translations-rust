에디션이란 무엇인가요?

2015년 5월, Rust 1.0 릴리스 는 "정체 없는 안정성"을 핵심 Rust 원칙으로 확립했습니다. 그 이후로 Rust는 중요한 규칙을 약속했습니다. 기능이 안정 버전을 통해 릴리스되면 기여자는 향후 모든 릴리스에서 해당 기능을 계속 지원할 것입니다.

그러나 언어에 하위 호환되지 않는 변경을 하는 것이 유용할 때가 있습니다. 일반적인 예는 새 키워드를 도입하는 것입니다. 예를 들어, 초기 버전의 Rust에는 async 및 await 키워드가 없었습니다.

만약 Rust가 갑자기 이러한 새 키워드를 도입했다면 일부 코드가 깨졌을 것입니다. let async = 1; 은 더 이상 작동하지 않을 것입니다.

Rust는 이 문제를 해결하기 위해 에디션 을 사용합니다. 하위 호환되지 않는 변경 사항이 있을 때, 다음 에디션으로 푸시됩니다. 에디션은 옵트인이므로, 기존 크레이트는 명시적으로 새 에디션으로 마이그레이션하지 않는 한 변경 사항을 사용하지 않습니다. 예를 들어, 최신 버전의 Rust는 2018 에디션 이상을 선택하지 않는 한 async 를 키워드로 취급하지 않습니다.

각 크레이트는 Cargo.toml 파일 내에서 에디션을 선택합니다. Cargo로 새 크레이트를 만들 때, 자동으로 최신 안정 에디션을 선택합니다.

에디션은 생태계를 분할하지 않습니다

에디션을 만들 때 가장 중요한 규칙이 하나 있습니다. 한 에디션의 크레이트는 다른 에디션으로 컴파일된 크레이트와 반드시 원활하게 상호 운용되어야 합니다.

즉, 각 크레이트는 독립적으로 새 에디션으로 마이그레이션할 시기를 결정할 수 있습니다. 이 결정은 '비공개'이며 생태계의 다른 크레이트에 영향을 미치지 않습니다.

Rust의 경우, 이러한 필수 호환성은 에디션에 포함될 수 있는 변경 종류에 몇 가지 제한을 의미합니다. 결과적으로, 새로운 Rust 에디션에서 발견되는 변경 사항은 '피상적인' 경향이 있습니다. 에디션에 관계없이 모든 Rust 코드는 궁극적으로 컴파일러 내에서 동일한 내부 표현으로 컴파일됩니다.

에디션 마이그레이션은 쉽고 대부분 자동화됩니다

Rust는 새 에디션으로의 업그레이드를 쉬운 프로세스로 만드는 것을 목표로 합니다. 새 에디션이 출시되면 크레이트 작성자는 cargo 내의 자동 마이그레이션 도구 를 사용하여 마이그레이션할 수 있습니다. 그러면 Cargo는 새 버전과 호환되도록 코드에 사소한 변경을 합니다.

예를 들어, Rust 2018로 마이그레이션할 때 async 라는 이름의 모든 것은 이제 동등한 원시 식별자 구문 인 r#async 를 사용합니다.

Cargo의 자동 마이그레이션은 완벽하지 않습니다. 수동 변경이 필요한 예외적인 경우가 여전히 있을 수 있습니다. 코드의 정확성이나 성능에 영향을 줄 수 있는 의미 체계 변경을 피하는 것을 목표로 합니다.

기존 프로젝트를 새 에디션으로 전환하기

Rust에는 프로젝트를 한 에디션에서 다음 에디션으로 자동으로 전환하는 도구가 포함되어 있습니다. 소스 코드를 다음 에디션과 호환되도록 업데이트합니다. 간단히 말해, 다음 에디션으로 업데이트하는 단계는 다음과 같습니다.

  1. cargo update 를 실행하여 종속성을 최신 버전으로 업데이트하십시오.
  2. cargo fix --edition 실행
  3. Cargo.toml 을 편집하고 edition 필드를 다음 에디션으로 설정합니다(예: edition = "2024").
  4. cargo build 또는 cargo test 를 실행하여 수정 사항이 작동하는지 확인합니다.
  5. cargo fmt 를 실행하여 프로젝트를 다시 포맷합니다.

다음 섹션에서는 이러한 단계의 세부 정보와 그 과정에서 발생할 수 있는 몇 가지 문제에 대해 자세히 설명합니다.

새 에디션으로의 마이그레이션이 가능한 한 순조로운 경험이 되도록 하는 것이 저희의 의도입니다. 최신 에디션으로 업그레이드하기 어려운 경우 버그로 간주합니다. 이 프로세스에 문제가 발생하면 버그 보고서를 제출 해 주십시오. 감사합니다!

마이그레이션 시작

예를 들어, 2015 에디션에서 2018 에디션으로 전환하는 것을 살펴보겠습니다. 2021과 같은 다른 에디션으로 전환할 때도 단계는 기본적으로 동일합니다.

src/lib.rs 에 이 코드가 있는 크레이트가 있다고 상상해 보십시오.

#![allow(unused)]
fn main() {
trait Foo {
    fn foo(&self, i32);
}
}

이 코드는 익명 매개변수인 i32 를 사용합니다. 이것은 Rust 2018에서 지원되지 않으므로 컴파일에 실패합니다. 이 코드를 최신 상태로 만들어 봅시다!

종속성 업데이트

시작하기 전에 종속성을 업데이트하는 것이 좋습니다. 일부 종속성, 특히 일부 proc-매크로 또는 빌드 타임 코드 생성을 수행하는 종속성은 최신 에디션과 호환성 문제가 있을 수 있습니다. 마지막으로 업데이트한 이후에 이러한 문제를 해결할 수 있는 새 릴리스가 만들어졌을 수 있습니다. 다음을 실행하십시오.

cargo update

업데이트 후 테스트를 실행하여 모든 것이 작동하는지 확인하는 것이 좋습니다. git 과 같은 소스 제어 도구를 사용하는 경우 커밋의 논리적 분리를 유지하기 위해 이러한 변경 사항을 별도로 커밋할 수 있습니다.

새 에디션과 호환되도록 코드 업데이트

코드가 새 에디션과 호환되지 않는 기능을 사용할 수도 있고 사용하지 않을 수도 있습니다. 다음 에디션으로의 전환을 돕기 위해 Cargo에는 소스 코드를 자동으로 업데이트하는 cargo fix 하위 명령이 포함되어 있습니다. 시작하려면 실행해 봅시다.

cargo fix --edition

이렇게 하면 코드를 확인하고 가능한 모든 문제를 자동으로 수정합니다. src/lib.rs 를 다시 살펴보겠습니다.

#![allow(unused)]
fn main() {
trait Foo {
    fn foo(&self, _: i32);
}
}

코드를 다시 작성하여 i32 값에 대한 매개변수 이름을 도입했습니다. 이 경우 이름이 없었기 때문에 cargo fix 는 사용되지 않는 변수에 대한 관례인 _ 로 바꿉니다.

cargo fix 가 항상 코드를 자동으로 수정할 수 있는 것은 아닙니다. cargo fix 가 무언가를 수정할 수 없는 경우 수정할 수 없다는 경고를 콘솔에 인쇄합니다. 이러한 경고 중 하나가 표시되면 코드를 수동으로 업데이트해야 합니다. 마이그레이션 프로세스 작업에 대한 자세한 내용은 고급 마이그레이션 전략 장을 참조하고 필요한 변경 사항을 설명하는 이 가이드의 장을 읽으십시오. 문제가 있는 경우 사용자 포럼 에서 도움을 요청하십시오.

새 에디션에서 코드 테스트

다음 단계는 새 에디션에서 프로젝트를 테스트하는 것입니다. cargo test 를 실행하는 것과 같이 프로젝트 테스트를 실행하여 모든 것이 여전히 작동하는지 확인하십시오. 새 경고가 발행되면 컴파일러에서 제공하는 제안을 적용하기 위해 cargo fix 를 다시 실행( --edition 플래그 없이)하는 것을 고려할 수 있습니다.

이 시점에서 여전히 일부 수동 변경이 필요할 수 있습니다. 예를 들어, 자동 마이그레이션은 doctest를 업데이트하지 않으며 빌드 타임 코드 생성 또는 매크로는 수동 업데이트가 필요할 수 있습니다. 자세한 내용은 고급 마이그레이션 장 을 참조하십시오.

rustfmt로 다시 포맷하기

프로젝트 내에서 포맷을 자동으로 유지하기 위해 rustfmt 를 사용하는 경우 새 에디션의 새 포맷 규칙을 사용하여 다시 포맷하는 것을 고려해야 합니다.

다시 포맷하기 전에 git 과 같은 소스 제어 도구를 사용하는 경우 이 단계를 수행하기 전에 지금까지 변경한 모든 내용을 커밋하는 것이 좋습니다. 포맷 변경 사항을 별도의 커밋에 넣는 것이 유용할 수 있습니다. 그러면 어떤 변경 사항이 단지 포맷인지 다른 코드 변경 사항인지 확인할 수 있고 git blame 에서 포맷 변경 사항을 무시할 수도 있습니다.

cargo fmt

자세한 내용은 스타일 에디션 장 을 참조하십시오.

불안정한 에디션으로 마이그레이션

에디션이 출시된 후 다음 에디션까지는 대략 3년의 기간이 있습니다. 그 기간 동안 다음 에디션에 새로운 기능이 추가될 수 있으며, 이는 nightly 채널 에서만 사용할 수 있습니다. 안정화되기 전에 이러한 새로운 기능을 테스트하는 데 도움을 주고 싶다면 nightly 채널을 사용하여 사용해 볼 수 있습니다.

단계는 안정 채널과 거의 유사합니다.

  1. 최신 nightly 설치: rustup update nightly.
  2. cargo +nightly fix --edition 실행.
  3. Edit Cargo.toml and place cargo-features = ["edition20xx"] at the top (above [package]), and change the edition field to say edition = "20xx" where 20xx is the edition you are upgrading to.
  4. cargo +nightly check 를 실행하여 새 에디션에서 작동하는지 확인합니다.

⚠ 주의: 다음 에디션에 구현된 기능에는 cargo fix 로 구현된 자동 마이그레이션이 없을 수 있으며 기능 자체가 완료되지 않았을 수 있습니다. 가능하면 이 가이드에는 상태에 대한 자세한 정보와 함께 nightly에 구현된 기능에 대한 정보가 포함되어야 합니다. 에디션이 안정화되기 몇 달 전에 모든 새로운 기능이 완전히 구현되어야 하며 Rust 블로그 에서 테스트 요청을 발표할 것입니다.

마이그레이션 작동 방식

cargo fix --edition 은 다음 에디션에서 컴파일되지 않을 수 있는 코드를 감지하는 특수 lints 가 활성화된 상태에서 프로젝트에서 cargo check 와 동등한 것을 실행하여 작동합니다. 이러한 린트에는 현재 및 다음 에디션 모두에서 호환되도록 코드를 수정하는 방법에 대한 지침이 포함되어 있습니다. cargo fix 는 이러한 변경 사항을 소스 코드에 적용한 다음 cargo check 를 다시 실행하여 수정 사항이 작동하는지 확인합니다. 수정 사항이 실패하면 변경 사항을 되돌리고 경고를 표시합니다.

코드를 현재 및 다음 에디션 모두와 동시에 호환되도록 변경하면 코드를 점진적으로 마이그레이션하기가 더 쉬워집니다. 자동화된 마이그레이션이 완전히 성공하지 못하거나 수동 도움이 필요한 경우 Cargo.toml 을 다음 에디션을 사용하도록 변경하기 전에 원래 에디션에 머무르면서 반복할 수 있습니다.

cargo fix --edition 이 적용하는 린트는 린트 그룹 의 일부입니다. 예를 들어, 2018에서 2021로 마이그레이션할 때 Cargo는 rust-2021-compatibility 린트 그룹을 사용하여 코드를 수정합니다. 마이그레이션에 도움이 되는 개별 린트 사용에 대한 팁은 아래 부분 마이그레이션 섹션을 확인하십시오.

cargo fix 는 cargo check 를 여러 번 실행할 수 있습니다. 예를 들어, 한 세트의 수정 사항을 적용한 후 추가 수정이 필요한 새 경고가 트리거될 수 있습니다. Cargo는 새 경고가 생성되지 않을 때까지 이 작업을 반복합니다.

여러 구성 마이그레이션

cargo fix 는 한 번에 하나의 구성으로만 작동할 수 있습니다. Cargo 기능 또는 조건부 컴파일 을 사용하는 경우 다른 플래그로 cargo fix 를 여러 번 실행해야 할 수 있습니다.

예를 들어, #[cfg] 속성을 사용하여 다른 플랫폼에 대해 다른 코드를 포함하는 코드가 있는 경우 다른 대상에 대해 수정하기 위해 --target 옵션으로 cargo fix 를 실행해야 할 수 있습니다. 교차 컴파일을 사용할 수 없는 경우 시스템 간에 코드를 이동해야 할 수 있습니다.

Similarly, if you have conditions on Cargo features, like #[cfg(feature = "my-optional-thing")], it is recommended to use the --all-features flag to allow cargo fix to migrate all the code behind those feature gates. If you want to migrate feature code individually, you can use the --features flag to migrate one at a time.

대규모 프로젝트 또는 작업 공간 마이그레이션

문제가 발생하면 대규모 프로젝트를 점진적으로 마이그레이션하여 프로세스를 더 쉽게 만들 수 있습니다.

Cargo 작업 공간 에서 각 패키지는 자체 에디션을 정의하므로 프로세스는 자연스럽게 한 번에 하나의 패키지를 마이그레이션하는 것을 포함합니다.

Cargo 패키지 내에서 전체 패키지를 한 번에 마이그레이션하거나 개별 Cargo 대상 을 한 번에 하나씩 마이그레이션할 수 있습니다. 예를 들어, 여러 바이너리, 테스트 및 예제가 있는 경우 cargo fix --edition 과 함께 특정 대상 선택 플래그를 사용하여 해당 대상 하나만 마이그레이션할 수 있습니다. 기본적으로 cargo fix 는 --all-targets 를 사용합니다.

더 고급 사례의 경우 Cargo.toml 에서 각 개별 대상에 대한 에디션을 다음과 같이 지정할 수 있습니다.

이것은 일반적으로 필요하지 않지만 대상이 많고 모두 함께 마이그레이션하는 데 어려움이 있는 경우 옵션입니다.

손상된 코드로 부분 마이그레이션

때때로 컴파일러에서 제안한 수정 사항이 작동하지 않을 수 있습니다. 이런 일이 발생하면 Cargo는 무슨 일이 일어났는지와 오류가 무엇인지 나타내는 경고를 보고합니다. 그러나 기본적으로 변경 사항을 자동으로 되돌립니다. 코드를 손상된 상태로 유지하고 문제를 수동으로 해결하는 것이 도움이 될 수 있습니다. 일부 수정 사항은 정확했을 수 있으며 손상된 수정 사항은 대부분 정확하지만 약간의 조정만 필요할 수 있습니다.

이 상황에서는 cargo fix 와 함께 --broken-code 옵션을 사용하여 Cargo에 변경 사항을 되돌리지 않도록 지시하십시오. 그런 다음 수동으로 오류를 검사하고 수정하는 데 필요한 사항을 조사할 수 있습니다.

프로젝트를 점진적으로 마이그레이션하는 또 다른 옵션은 개별 수정 사항을 한 번에 하나씩 별도로 적용하는 것입니다. 개별 린트를 경고로 추가한 다음 cargo fix(--edition 플래그 없이)를 실행하거나 편집기 또는 IDE가 "빠른 수정"을 지원하는 경우 해당 제안을 적용하여 이 작업을 수행할 수 있습니다.

예를 들어, 2018 에디션은 keyword-idents 린트를 사용하여 충돌하는 키워드를 수정합니다. 각 크레이트의 맨 위에(예: src/lib.rs 또는 src/main.rs 의 맨 위) #![warn(keyword_idents)] 를 추가할 수 있습니다. 그런 다음 cargo fix 를 실행하면 해당 린트에 대한 제안만 적용됩니다.

린트 그룹 페이지에서 각 에디션에 대해 활성화된 린트 목록을 보거나 rustc -Whelp 명령을 실행할 수 있습니다.

매크로 마이그레이션

일부 매크로는 다음 에디션을 위해 수정하기 위해 수동 작업이 필요할 수 있습니다. 예를 들어, cargo fix --edition 은 다음 에디션에서 작동하지 않는 구문을 생성하는 매크로를 자동으로 수정하지 못할 수 있습니다.

이것은 proc 매크로 와 macro_rules 스타일 매크로 모두에 문제가 될 수 있습니다. macro_rules 매크로는 동일한 크레이트 내에서 매크로가 사용되는 경우 때때로 자동으로 업데이트될 수 있지만, 그렇지 않은 경우도 여러 있습니다. 일반적으로 Proc 매크로는 자동으로 수정할 수 없습니다.

예를 들어, 2015년에서 2018년으로 이 (억지로 만든) 매크로 foo 를 포함하는 크레이트를 마이그레이션하면 foo 는 자동으로 수정되지 않습니다.

#![allow(unused)]
fn main() {
#[macro_export]
macro_rules! foo {
    () => {
        let dyn = 1;
        println!("it is {}", dyn);
    };
}
}

이 매크로가 2015 크레이트에 정의된 경우 매크로 위생(아래에서 설명)으로 인해 다른 에디션의 크레이트에서 사용할 수 있습니다. 2015년에는 dyn 이 일반 식별자이며 제한 없이 사용할 수 있습니다.

그러나 2018년에는 dyn 이 더 이상 유효한 식별자가 아닙니다. cargo fix --edition 을 사용하여 2018로 마이그레이션할 때 Cargo는 경고나 오류를 전혀 표시하지 않습니다. 그러나 foo 는 어떤 크레이트에서 호출해도 작동하지 않습니다.

매크로가 있는 경우 매크로의 구문을 완전히 다루는 테스트가 있는지 확인하는 것이 좋습니다. 또한 여러 에디션의 크레이트에서 매크로를 가져와 사용하여 모든 곳에서 올바르게 작동하는지 확인하는 것이 좋습니다. 문제가 발생하면 이 가이드의 장을 읽고 모든 에디션에서 작동하도록 코드를 변경하는 방법을 이해해야 합니다.

매크로 위생

매크로는 매크로 내의 토큰이 어떤 에디션에서 왔는지 표시되는 "에디션 위생"이라는 시스템을 사용합니다. 이를 통해 외부 매크로를 호출하는 에디션에 대해 걱정할 필요 없이 다양한 에디션의 크레이트에서 호출할 수 있습니다.

dyn 을 식별자로 사용하는 macro_rules 매크로를 정의하는 위의 예를 자세히 살펴보겠습니다. 해당 매크로가 2015 에디션을 사용하는 크레이트에 정의된 경우 dyn 이 키워드이고 일반적으로 구문 오류가 발생하는 2018 크레이트에서 호출되더라도 해당 매크로는 잘 작동합니다. let dyn = 1; 토큰은 2015년산으로 표시되며 컴파일러는 해당 코드가 확장되는 모든 곳에서 이를 기억합니다. 파서는 토큰의 에디션을 보고 해석 방법을 알 수 있습니다.

문제는 정의된 크레이트에서 에디션을 2018로 변경할 때 발생합니다. 이제 해당 토큰은 2018 에디션으로 태그가 지정되며 구문 분석에 실패합니다. 그러나 크레이트에서 매크로를 호출한 적이 없기 때문에 cargo fix --edition 은 매크로를 검사하고 수정할 기회가 없었습니다.

문서 테스트

현재 cargo fix 는 문서 테스트 를 업데이트할 수 없습니다. Cargo.toml 에서 에디션을 업데이트한 후 cargo test 를 실행하여 모든 것이 여전히 통과하는지 확인해야 합니다. 문서 테스트가 새 에디션에서 지원되지 않는 구문을 사용하는 경우 수동으로 업데이트해야 합니다.

드문 경우지만 각 테스트에 대해 에디션을 수동으로 설정할 수 있습니다. 예를 들어, 삼중 백틱에 edition2018 주석 을 사용하여 rustdoc 에 사용할 에디션을 알릴 수 있습니다.

생성된 코드

자동 수정이 적용될 수 없는 또 다른 영역은 컴파일 타임에 Rust 코드를 생성하는 빌드 스크립트가 있는 경우입니다(예는 코드 생성 참조). 이 상황에서 다음 에디션에서 작동하지 않는 코드가 생기면 호환되는 코드를 생성하도록 빌드 스크립트를 수동으로 변경해야 합니다.

비-Cargo 프로젝트 마이그레이션

프로젝트가 빌드 시스템으로 Cargo를 사용하지 않는 경우에도 자동화된 린트를 사용하여 다음 에디션으로 마이그레이션하는 데 도움을 받을 수 있습니다. 적절한 린트 그룹 을 활성화하여 위에서 설명한 대로 마이그레이션 린트를 활성화할 수 있습니다. 예를 들어, #![warn(rust_2021_compatibility)] 속성 또는 -Wrust-2021-compatibility 또는 --force-warns=rust-2021-compatibility CLI 플래그 를 사용할 수 있습니다.

다음 단계는 해당 린트를 코드에 적용하는 것입니다. 여기에는 몇 가지 옵션이 있습니다.

  • 경고를 수동으로 읽고 컴파일러에서 권장하는 제안을 적용합니다.
  • 제안을 자동으로 적용하는 것을 지원하는 편집기나 IDE를 사용하십시오. 예를 들어, Rust Analyzer 확장 이 포함된 Visual Studio Code 는 "빠른 수정" 링크를 사용하여 제안을 자동으로 적용하는 기능이 있습니다. 다른 많은 편집기와 IDE에도 유사한 기능이 있습니다.
  • rustfix 라이브러리를 사용하여 마이그레이션 도구를 작성하십시오. 이것은 Cargo가 내부적으로 컴파일러에서 JSON 메시지 를 가져와 소스 코드를 수정하는 데 사용하는 라이브러리입니다. 라이브러리 사용 방법에 대한 예는 examples 디렉토리 를 확인하십시오.

새 에디션에서 관용적인 코드 작성

에디션은 새로운 기능을 추가하고 오래된 기능을 제거하는 것만이 아닙니다. 모든 프로그래밍 언어에서 관용구는 시간이 지남에 따라 변하며 Rust도 예외는 아닙니다. 이전 코드는 계속 컴파일되지만 오늘날에는 다른 관용구로 작성될 수 있습니다.

예를 들어, Rust 2015에서는 외부 크레이트를 다음과 같이 extern crate 로 나열해야 합니다.

// src/lib.rs
extern crate rand;

Rust 2018에서는 더 이상 포함할 필요가 없습니다.

cargo fix 에는 이러한 관용구 중 일부를 새 구문으로 자동 전환하는 --edition-idioms 옵션이 있습니다.

경고: 현재 _"관용구 린트"_에는 몇 가지 문제가 있는 것으로 알려져 있습니다. 컴파일에 실패할 수 있는 잘못된 제안을 할 수 있습니다. 현재 린트는 다음과 같습니다.

  • 2018 에디션:
    • unused-extern-crates
    • explicit-outlives-requirements
  • 2021 에디션에는 관용구 린트가 없습니다.

다음 지침은 몇 가지 컴파일러/Cargo 버그를 해결하려는 용감한 사람들에게만 권장됩니다! 문제가 발생하면 위에서 설명한 --broken-code 옵션을 사용하여 가능한 한 많은 진전을 이룬 다음 나머지 문제를 수동으로 해결할 수 있습니다.

이제 Cargo에 코드 스니펫을 수정하도록 지시할 수 있습니다.

cargo fix --edition-idioms

그런 다음 src/lib.rs 의 extern crate rand; 줄이 제거됩니다.

이제 더 관용적이 되었고 코드를 수동으로 수정할 필요가 없었습니다!

Rust 2015

Rust 2015의 주제는 "안정성"입니다. 1.0 릴리스와 함께 시작되었으며 "기본 에디션"입니다. 에디션 시스템은 2017년 후반에 고안되었지만 Rust 1.0은 2015년 5월에 출시되었습니다. 따라서 2015는 하위 호환성을 위해 특정 에디션을 지정하지 않을 때 얻는 에디션입니다.

"안정성"은 Rust 2015의 주제입니다. 왜냐하면 1.0은 Rust 개발에 큰 변화를 가져왔기 때문입니다. Rust 1.0 이전에는 Rust가 매일 바뀌었습니다. 이로 인해 Rust로 대규모 소프트웨어를 작성하기가 매우 어려웠고 배우기도 어려웠습니다. Rust 1.0 및 Rust 2015 릴리스를 통해 우리는 하위 호환성을 약속하여 사람들이 프로젝트를 구축할 수 있는 견고한 기반을 보장했습니다.

기본 에디션이므로 코드를 Rust 2015로 포팅할 방법이 없습니다. 그냥 그렇습니다. 2015에서 벗어나 전환하게 되지만 실제로 2015로 전환 하지는 않습니다. 따라서 그것에 대해 더 이상 할 말이 없습니다!

에디션 시스템은 Rust 2018 릴리스를 위해 만들어졌습니다. Rust 2018 에디션 릴리스는 생산성 이라는 주제를 중심으로 조정된 여러 다른 기능과 동시에 이루어졌습니다. 이러한 기능의 대부분은 하위 호환이 가능하며 이제 모든 에디션에서 사용할 수 있습니다. 그러나 이러한 변경 사항 중 일부는 에디션 메커니즘이 필요했습니다(가장 주목할 만한 것은 모듈 시스템 변경 입니다).

경로 및 모듈 시스템 변경사항

요약

  • use 선언의 경로는 이제 다른 경로와 동일하게 작동합니다.
  • :: 로 시작하는 경로는 이제 외부 크레이트가 뒤따라야 합니다.
  • pub(in path) 가시성 한정자의 경로는 이제 crate, self 또는 super 로 시작해야 합니다.

동기

모듈 시스템은 종종 Rust를 처음 접하는 사람들에게 가장 어려운 것 중 하나입니다. 물론 모든 사람이 마스터하는 데 시간이 걸리는 자신만의 것이 있지만, 많은 사람들에게 혼란스러운 근본적인 원인이 있습니다. 모듈 시스템을 정의하는 간단하고 일관된 규칙이 있지만 그 결과는 일관성이 없고 직관에 반하며 신비롭게 느껴질 수 있습니다.

따라서 2018 에디션의 Rust는 몇 가지 새로운 모듈 시스템 기능을 도입하지만, 결국 모듈 시스템을 단순화 하여 무슨 일이 일어나고 있는지 더 명확하게 만듭니다.

간단한 요약은 다음과 같습니다.

  • extern crate 는 99%의 경우에 더 이상 필요하지 않습니다.
  • crate 키워드는 현재 크레이트를 나타냅니다.
  • 경로는 하위 모듈 내에서도 크레이트 이름으로 시작할 수 있습니다.
  • :: 로 시작하는 경로는 외부 크레이트를 참조해야 합니다.
  • foo.rs 와 foo/ 하위 디렉토리가 공존할 수 있습니다. 하위 모듈을 하위 디렉토리에 배치할 때 mod.rs 는 더 이상 필요하지 않습니다.
  • use 선언의 경로는 다른 경로와 동일하게 작동합니다.

이렇게 말하면 임의의 새로운 규칙처럼 보일 수 있지만, 이제 정신 모델이 전반적으로 크게 단순화되었습니다. 자세한 내용은 계속 읽어보십시오!

자세한 내용

각 새로운 기능에 대해 차례로 이야기해 봅시다.

extern crate 더 이상 없음

이것은 매우 간단합니다. 더 이상 프로젝트에 크레이트를 가져오기 위해 extern crate 를 작성할 필요가 없습니다. 이전:

// Rust 2015

extern crate futures;

mod submodule {
    use futures::Future;
}

이후:

// Rust 2018

mod submodule {
    use futures::Future;
}

이제 프로젝트에 새 크레이트를 추가하려면 Cargo.toml 에 추가하면 두 번째 단계가 없습니다. Cargo를 사용하지 않는 경우 이미 --extern 플래그를 전달하여 rustc 에 외부 크레이트의 위치를 제공해야 했으므로 거기서 하던 일을 계속하면 됩니다.

예외

이 규칙에는 한 가지 예외가 있는데, 바로 "sysroot" 크레이트입니다. 이것들은 Rust 자체와 함께 배포되는 크레이트입니다.

일반적으로 이것들은 매우 특수한 상황에서만 필요합니다. 1.41부터 rustc 는 extern crate 와 유사한 방식으로 주어진 크레이트 이름을 자동으로 추가하는 --extern=CRATE_NAME 플래그를 허용합니다. 빌드 도구는 이를 사용하여 sysroot 크레이트를 크레이트의 프렐루드에 주입할 수 있습니다. Cargo는 proc_macro 크레이트에 사용하지만 이를 표현하는 일반적인 방법이 없습니다.

sysroot 크레이트를 명시적으로 가져와야 하는 몇 가지 예는 다음과 같습니다.

  • std: 일반적으로 이것은 필요하지 않습니다. 왜냐하면 std 는 크레이트가 #![no_std]로 표시되지 않는 한 자동으로 가져오기 때문입니다.
  • core: Usually this is not necessary, because core is automatically imported, unless the crate is marked with #![no_core]. For example, some of the internal crates used by the standard library itself need this.
  • proc_macro: This is automatically imported by Cargo if it is a proc-macro crate starting in 1.42. extern crate proc_macro; would be needed if you want to support older releases, or if using another build tool that does not pass the appropriate --extern flags to rustc.
  • alloc: alloc 크레이트의 항목은 일반적으로 std 크레이트의 재내보내기를 통해 액세스됩니다. 할당을 지원하는 no_std 크레이트로 작업하는 경우 alloc 을 명시적으로 가져와야 할 수 있습니다.
  • test: 이것은 nightly 채널 에서만 사용할 수 있으며 일반적으로 불안정한 벤치마크 지원에만 사용됩니다.

매크로

extern crate 의 또 다른 용도는 매크로를 가져오는 것이었습니다. 더 이상 필요하지 않습니다. 매크로는 다른 항목과 마찬가지로 use 로 가져올 수 있습니다. 예를 들어, 다음과 같은 extern crate 사용:

#[macro_use]
extern crate bar;

fn main() {
    baz!();
}

다음과 같이 변경할 수 있습니다.

use bar::baz;

fn main() {
    baz!();
}

크레이트 이름 바꾸기

다음과 같이 as 를 사용하여 크레이트 이름을 바꾸고 있었다면:

extern crate futures as f;

use f::Future;

extern crate 줄을 자체적으로 제거하는 것은 작동하지 않습니다. 다음과 같이 해야 합니다.

use futures as f;

use self::f::Future;

이 변경은 f 를 사용하는 모든 모듈에서 발생해야 합니다.

crate 키워드는 현재 크레이트를 나타냅니다.

use 선언 및 기타 코드에서 crate:: 접두사를 사용하여 현재 크레이트의 루트를 참조할 수 있습니다. 예를 들어, crate::foo::bar 는 동일한 크레이트의 다른 곳에서 항상 foo 모듈 내의 bar 이름을 참조합니다.

이전에는 접두사 :: 가 크레이트 루트 또는 외부 크레이트를 참조했지만 이제는 명확하게 외부 크레이트를 참조합니다. 예를 들어, ::foo::bar 는 항상 외부 크레이트 foo 내의 bar 이름을 참조합니다.

외부 크레이트 경로

이전에는 use 가져오기 없이 모듈에서 외부 크레이트를 사용하려면 경로에 선행 :: 가 필요했습니다.

// Rust 2015

extern crate chrono;

fn foo() {
    // 이것은 크레이트 루트에서 작동합니다
    let x = chrono::Utc::now();
}

mod submodule {
    fn function() {
        // 그러나 하위 모듈에서는 `use` 로 가져오지 않으면 선행 ::가 필요합니다
        let x = ::chrono::Utc::now();
    }
}

이제 외부 크레이트 이름은 하위 모듈을 포함하여 전체 크레이트의 범위에 있습니다.

// Rust 2018

fn foo() {
    // 이것은 크레이트 루트에서 작동합니다
    let x = chrono::Utc::now();
}

mod submodule {
    fn function() {
        // 크레이트는 하위 모듈에서도 직접 참조할 수 있습니다
        let x = chrono::Utc::now();
    }
}

외부 크레이트와 이름이 같은 로컬 모듈이나 항목이 있는 경우 해당 이름으로 시작하는 경로는 로컬 모듈이나 항목을 참조하는 것으로 간주됩니다. 외부 크레이트를 명시적으로 참조하려면 ::name 형식을 사용하십시오.

mod.rs 더 이상 없음

Rust 2015에서 하위 모듈이 있는 경우:

// 이 `mod` 선언은 `foo` 모듈을
// `foo.rs` 또는 `foo/mod.rs`에서 찾습니다.
mod foo;

foo.rs 또는 foo/mod.rs 에 있을 수 있습니다. 자체 하위 모듈이 있는 경우 반드시 foo/mod.rs 여야 합니다. 따라서 foo 의 bar 하위 모듈은 foo/bar.rs 에 있습니다.

Rust 2018에서는 하위 모듈이 있는 모듈의 이름이 mod.rs 여야 한다는 제한이 해제되었습니다. foo.rs 는 그냥 foo.rs 일 수 있으며 하위 모듈은 여전히 foo/bar.rs 입니다. 이렇게 하면 특수 이름이 제거되고 편집기에서 여러 파일을 열어두면 mod.rs 라는 이름의 여러 탭 대신 이름을 명확하게 볼 수 있습니다.

Rust 2015 Rust 2018 
.
├── lib.rs
└── foo/
    ├── mod.rs
    └── bar.rs

.
├── lib.rs
├── foo.rs
└── foo/
    └── bar.rs

use 경로

Rust 2018은 Rust 2015에 비해 경로 처리를 단순화하고 통합합니다. Rust 2015에서는 use 선언의 경로가 다른 곳과 다르게 작동합니다. 특히 use 선언의 경로는 항상 크레이트 루트에서 시작하는 반면 다른 코드의 경로는 암시적으로 현재 범위에서 시작했습니다. 이러한 차이점은 최상위 모듈에 영향을 미치지 않았으므로 하위 모듈이 있는 충분히 큰 프로젝트에서 작업할 때까지 모든 것이 간단해 보였습니다.

Rust 2018에서는 use 선언 및 기타 코드의 경로가 최상위 모듈과 모든 하위 모듈에서 동일하게 작동합니다. 현재 범위의 상대 경로, 외부 크레이트 이름으로 시작하는 경로 또는 ::, crate, super 또는 self 로 시작하는 경로를 사용할 수 있습니다.

다음과 같이 보이는 코드:

// Rust 2015

extern crate futures;

use futures::Future;

mod foo {
    pub struct Bar;
}

use foo::Bar;

fn my_poll() -> futures::Poll { ... }

enum SomeEnum {
    V1(usize),
    V2(String),
}

fn func() {
    let five = std::sync::Arc::new(5);
    use SomeEnum::*;
    match ... {
        V1(i) => { ... }
        V2(s) => { ... }
    }
}

extern crate 줄을 삭제할 수 있다는 점을 제외하고 Rust 2018에서도 똑같이 보일 것입니다.

// Rust 2018

use futures::Future;

mod foo {
    pub struct Bar;
}

use foo::Bar;

fn my_poll() -> futures::Poll { ... }

enum SomeEnum {
    V1(usize),
    V2(String),
}

fn func() {
    let five = std::sync::Arc::new(5);
    use SomeEnum::*;
    match ... {
        V1(i) => { ... }
        V2(s) => { ... }
    }
}

동일한 코드는 하위 모듈에서도 완전히 수정되지 않은 상태로 작동합니다.

// Rust 2018

mod submodule {
    use futures::Future;

    mod foo {
        pub struct Bar;
    }

    use foo::Bar;

    fn my_poll() -> futures::Poll { ... }

    enum SomeEnum {
        V1(usize),
        V2(String),
    }

    fn func() {
        let five = std::sync::Arc::new(5);
        use SomeEnum::*;
        match ... {
            V1(i) => { ... }
            V2(s) => { ... }
        }
    }
}

이렇게 하면 프로젝트에서 코드를 쉽게 이동할 수 있고 다중 모듈 프로젝트에 추가적인 복잡성을 도입하는 것을 방지할 수 있습니다.

익명 트레이트 함수 매개변수 사용 중단

요약

  • 트레이트 함수 매개변수 는 함수에 본문이 있을 때 모든 반박할 수 없는 패턴을 사용할 수 있습니다.

세부 정보

RFC #1685 에 따라 트레이트 메서드 선언의 매개변수는 더 이상 익명으로 허용되지 않습니다.

예를 들어, 2015 에디션에서는 이것이 허용되었습니다.

#![allow(unused)]
fn main() {
trait Foo {
    fn foo(&self, u8);
}
}

2018 에디션에서는 모든 매개변수에 인수 이름이 지정되어야 합니다(단지 _ 인 경우에도).

#![allow(unused)]
fn main() {
trait Foo {
    fn foo(&self, baz: u8);
}
}

새로운 키워드

요약

  • dyn 은 엄격한 키워드 이며, 2015년에는 약한 키워드 입니다.
  • async 및 await 는 엄격한 키워드 입니다.
  • try 는 예약된 키워드 입니다.

동기

트레이트 객체를 위한 dyn Trait

dyn Trait 기능은 트레이트 객체를 사용하기 위한 새로운 구문입니다. 간단히 말해서:

  • Box<Trait> 는 Box<dyn Trait> 가 됩니다.
  • &Trait 및 &mut Trait 는 &dyn Trait 및 &mut dyn Trait 가 됩니다.

등등. 코드에서:

#![allow(unused)]
fn main() {
trait Trait {}

impl Trait for i32 {}

// 이전
fn function1() -> Box<Trait> {
unimplemented!()
}

// 새로운
fn function2() -> Box<dyn Trait> {
unimplemented!()
}
}

그게 다입니다!

왜?

트레이트 객체에 트레이트 이름만 사용하는 것은 잘못된 결정으로 판명되었습니다. 현재 구문은 베테랑에게도 종종 모호하고 혼란스러우며, 대안보다 더 자주 사용되지 않고, 때로는 더 느리며, 대안을 사용할 수 있을 때 전혀 사용할 수 없는 기능을 선호합니다.

또한 impl Trait 가 등장하면서 "impl Trait 대 dyn Trait"는 "impl Trait 대 Trait"보다 훨씬 대칭적이므로 조금 더 좋습니다. impl Trait는 여기에서 설명합니다.

따라서 새 에디션에서는 트레이트 객체가 필요한 경우 Trait 대신 dyn Trait 를 선호해야 합니다.

async 및 await

이러한 키워드는 Rust의 async-await 기능을 구현하기 위해 예약되었으며, 궁극적으로 1.39.0에서 안정 버전으로 출시되었습니다.

try 키워드

try 키워드는 try 블록에서 사용하기 위해 예약되었으며, (이 글을 쓰는 시점에는) 안정화되지 않았습니다(추적 문제).

Cargo 변경사항

요약

  • Cargo.toml 매니페스트에 대상 정의가 있는 경우 더 이상 다른 대상의 자동 검색을 자동으로 비활성화하지 않습니다.
  • path 필드가 설정되지 않은 대상에 대해 src/{target_name}.rs 형식의 대상 경로는 더 이상 유추되지 않습니다.
  • 현재 디렉토리에 대한 cargo install 은 더 이상 허용되지 않으며, 현재 패키지를 설치하려면 cargo install --path . 를 지정해야 합니다.

Rust 2021 에디션에는 언어에 새로운 기능과 일관성을 더하고 향후 확장을 위한 공간을 여는 몇 가지 변경 사항이 포함되어 있습니다. 다음 장에서는 각 변경 사항의 세부 정보를 자세히 살펴보고 기존 코드를 마이그레이션하는 방법에 대한 지침을 포함합니다.

프렐루드에 추가된 항목

요약

  • TryInto, TryFrom 및 FromIterator 트레이트는 이제 프렐루드의 일부입니다.
  • 이로 인해 트레이트 메서드 호출이 모호해져 일부 코드가 컴파일되지 않을 수 있습니다.

세부 정보

표준 라이브러리의 프렐루드 는 모든 모듈에서 자동으로 가져오는 모든 것을 포함하는 모듈입니다. Option, Vec, drop 및 Clone 과 같이 일반적으로 사용되는 항목을 포함합니다.

Rust 컴파일러는 프렐루드에 추가된 항목이 기존 코드를 깨뜨리지 않도록 하기 위해 프렐루드의 항목보다 수동으로 가져온 항목을 우선시합니다. 예를 들어, pub struct Option; 을 포함하는 example 이라는 크레이트나 모듈이 있는 경우 use example::*; 은 Option 이 표준 라이브러리의 것이 아니라 example 의 것을 명확하게 참조하도록 합니다.

그러나 프렐루드에 _트레이트 _를 추가하면 미묘한 방식으로 기존 코드가 깨질 수 있습니다. 예를 들어, MyTryInto 트레이트에서 오는 x.try_into() 호출은 std 의 TryInto 도 가져온 경우 컴파일에 실패할 수 있습니다. try_into 호출이 이제 모호하고 어느 트레이트에서든 올 수 있기 때문입니다. 이것이 우리가 아직 TryInto 를 프렐루드에 추가하지 않은 이유입니다. 이런 식으로 깨질 코드가 많기 때문입니다.

해결책으로 Rust 2021은 새로운 프렐루드를 사용합니다. 세 가지 새로운 추가 사항을 제외하고는 현재 것과 동일합니다.

  • std::convert::TryInto
  • std::convert::TryFrom
  • std::iter::FromIterator

추적 문제는 여기 에서 찾을 수 있습니다.

마이그레이션

2021 에디션의 일부로 Rust 2018 코드베이스를 Rust 2021로 자동 마이그레이션하는 데 도움이 되도록 마이그레이션 린트인 rust_2021_prelude_collisions 가 추가되었습니다.

코드를 Rust 2021 에디션과 호환되도록 마이그레이션하려면 다음을 실행하십시오.

cargo fix --edition

린트는 새 프렐루드 트레이트 중 하나에 정의된 메서드와 이름이 같은 함수나 메서드가 호출되는 경우를 감지합니다. 경우에 따라 이전에 호출했던 것과 동일한 함수를 계속 호출하도록 호출을 다양한 방식으로 다시 작성할 수 있습니다.

코드를 수동으로 마이그레이션하거나 cargo fix 가 수행하는 작업을 더 잘 이해하고 싶다면 아래에 마이그레이션이 필요한 상황과 필요하지 않은 경우의 반대 예를 간략하게 설명했습니다.

마이그레이션 필요

충돌하는 트레이트 메서드

범위에 있는 두 트레이트의 메서드 이름이 같으면 어떤 트레이트 메서드를 사용해야 할지 모호합니다. 예를 들어:

trait MyTrait<A> {
  // 이 이름은 `std` 의 `FromIterator` 트레이트에 있는 `from_iter` 메서드와 같습니다. 
  fn from_iter(x: Option<A>);
}

impl<T> MyTrait<()> for Vec<T> {
  fn from_iter(_: Option<()>) {}
}

fn main() {
  // Vec<T>는 `std::iter::FromIterator` 와 `MyTrait` 를 모두 구현합니다
  // 두 트레이트가 모두 범위에 있는 경우(Rust 2021의 경우와 같음),
  // 그러면 어떤 `from_iter` 메서드를 호출해야 할지 모호해집니다
  <Vec<i32>>::from_iter(None);
}

완전 정규화된 구문을 사용하여 이 문제를 해결할 수 있습니다.

fn main() {
  // 이제 어떤 트레이트 메서드를 참조하는지 명확합니다
  <Vec<i32> as MyTrait<()>>::from_iter(None);
}

일부 사용자는 dyn Trait 값에 대해 메서드를 호출하는데, 이 메서드 이름이 새로운 prelude 특성과 겹칩니다.

#![allow(unused)]
fn main() {
mod submodule {
  pub trait MyTrait {
    // 이것은 `TryInto::try_into` 와 같은 이름을 갖습니다.
    fn try_into(&self) -> Result<u32, ()>;
  }
}

// `MyTrait` 는 여기 범위에 포함되지 않으며 경로 `submodule::MyTrait`
"을 통해서만 참조될 수 있습니다fn bar(f: Box<dyn submodule::MyTrait>) {

  f.try_into();

정적 디스패치 메서드와 달리, 트레이트 객체에서 트레이트 메서드를 호출할 때 해당 트레이트가 범위 내에 있어야 할 필요는 없습니다. 위 코드는 범위 내에 메서드 이름이 충돌하는 트레이트가 없는 한 정상적으로 작동합니다. TryInto 트레이트가 범위 내에 있는 경우(Rust 2021의 경우), 이는 모호성을 발생시킵니다. MyTrait::try_into 를 호출해야 할까요, 아니면 std::convert::TryInto::try_into 를 호출해야 할까요?

이런 경우에는 추가적인 역참조를 추가하거나, 메서드 수신자의 타입을 명확히 하여 이 문제를 해결할 수 있습니다. 이렇게 하면 prelude 특성의 메서드 대신 dyn Trait 메서드가 선택됩니다. 예를 들어, 위의 "f.try_into()를 "(&*f).try_into()로 변환하면 dyn MyTrait에서 MyTrait::try_into 메서드만 참조할 수 있는 try_into를 호출하게 됩니다.

마이그레이션이 필요 없습니다

고유한 방법

많은 타입이 특성과 동일한 이름을 가진 자체 내장 메서드를 정의합니다.메서드입니다. 예를 들어, 아래 구조체 MyStruct 는 from_iter 를 구현합니다.이 메서드는 표준 라이브러리에 있는 특성 FromIterator 의 메서드와 동일한 이름을 공유합니다.

#![allow(unused)]
fn main() {
use std::iter::IntoIterator;

struct MyStruct {
  data: Vec<u32>
}

impl MyStruct {
  // 이것은 `std::iter::FromIterator::from_iter` 와 이름이 같습니다
  fn from_iter(iter: impl IntoIterator<Item = u32>) -> Self {
    Self {
      data: iter.into_iter().collect()
    }
  }
}

impl std::iter::FromIterator<u32> for MyStruct {
    fn from_iter<I: IntoIterator<Item = u32>>(iter: I) -> Self {
      Self {
        data: iter.into_iter().collect()
      }
    }
}
}

고유 메서드는 항상 트레이트 메서드보다 우선하므로 마이그레이션이 필요하지 않습니다.

구현 참조

린트는 2021 에디션을 코드베이스에 도입하는 것이 이름 확인 충돌을 일으킬지(따라서 에디션 변경 후 코드가 깨지는지) 여부를 결정할 때 몇 가지 요소를 고려해야 합니다. 이러한 요소는 다음과 같습니다.

  • 호출이 완전 정규화된 호출 입니까, 아니면 점-호출 메서드 구문 을 사용합니까?
    • 이것은 메서드 호출 구문에서 자동 참조 및 자동 역참조로 인해 이름이 확인되는 방식에 영향을 줍니다. 수동으로 역참조/참조하면 점-호출 메서드 구문의 경우 우선순위를 지정할 수 있지만, 완전 정규화된 호출은 메서드 경로에서 유형과 트레이트 이름을 지정해야 합니다(예: <Type as Trait>::method).
  • 이것은 고유 메서드 입니까, 아니면 트레이트 메서드 입니까?
    • self 를 사용하는 고유 메서드는 TryInto::try_into 보다 우선합니다. 고유 메서드가 트레이트 메서드보다 우선하기 때문입니다. 그러나 &self 또는 &mut self 를 사용하는 고유 메서드는 자동 참조가 필요하기 때문에 우선하지 않습니다(TryInto::try_into 는 self 를 사용하므로 그렇지 않습니다).
  • 이 메서드의 출처는 core/std 입니까? (트레이트는 자신과 충돌할 수 없으므로)
  • 주어진 유형이 충돌할 수 있는 트레이트를 구현합니까?
  • 메서드가 동적 디스패치를 통해 호출됩니까? (즉, self 유형이 dyn Trait 입니까)
    • 그렇다면 트레이트 가져오기는 확인에 영향을 미치지 않으며 마이그레이션 린트가 발생할 필요가 없습니다.

기본 Cargo 기능 해결자

요약

  • edition = "2021" 은 Cargo.toml 에서 resolver = "2" 를 의미합니다.

세부 정보

Rust 1.51.0부터 Cargo는 Cargo.toml 에서 resolver = "2" 로 활성화할 수 있는 새로운 기능 해석기 에 대한 옵트인 지원을 제공합니다.

Rust 2021부터 이것이 기본값이 됩니다. 즉, Cargo.toml 에 edition = "2021" 을 작성하면 resolver = "2" 를 의미합니다.

해석기는 작업 공간 에 대한 전역 설정이며 종속성에서는 설정이 무시됩니다. 이 설정은 작업 공간의 최상위 패키지에만 적용됩니다. 가상 작업 공간 을 사용하는 경우 새 해석기를 옵트인하려면 [workspace] 정의에서 resolver 필드 를 명시적으로 설정해야 합니다.

새로운 기능 해석기는 더 이상 여러 방식으로 종속된 크레이트에 대해 요청된 모든 기능을 병합하지 않습니다. 자세한 내용은 Rust 1.51 발표 를 참조하십시오.

마이그레이션

새 해석기로 업데이트하기 위한 자동화된 마이그레이션 도구는 없습니다. 대부분의 프로젝트에서는 업데이트로 인해 변경 사항이 거의 또는 전혀 없습니다.

cargo fix --edition 으로 업데이트할 때 새 해석기가 다른 기능으로 종속성을 빌드하는 경우 Cargo는 보고서를 표시합니다. 다음과 같이 보일 수 있습니다.

참고: 2021 에디션으로 전환하면 Cargo에서 버전 2 기능 해석기를 사용할 수 있습니다. 이로 인해 일부 종속성이 이전보다 더 적은 기능으로 빌드될 수 있습니다. 해석기 변경에 대한 자세한 내용은 https://doc.rust-lang.org/nightly/edition-guide/rust-2021/default-cargo-resolver.html에서 확인할 수 있습니다.
다음 종속성을 빌드할 때 주어진 기능은 더 이상 사용되지 않습니다.

  bstr v0.2.16: default, lazy_static, regex-automata, unicode
  libz-sys v1.1.3 (as host dependency): libc

이를 통해 특정 종속성이 더 이상 주어진 기능으로 빌드되지 않음을 알 수 있습니다.

빌드 실패

변경 후 프로젝트가 올바르게 빌드되지 않는 경우가 있을 수 있습니다. 한 패키지의 종속성 선언이 다른 패키지에서 특정 기능이 활성화되어 있다고 가정하고 해당 기능이 이제 비활성화된 경우 컴파일에 실패할 수 있습니다.

예를 들어, 다음과 같은 종속성이 있다고 가정해 보겠습니다.

# Cargo.toml

[dependencies]
bstr = { version = "0.2.16", default-features = false }
# ...

그리고 종속성 트리의 어딘가에 다른 패키지에 다음이 있습니다.

# Another package's Cargo.toml

[build-dependencies]
bstr = "0.2.16"

우리 패키지에서는 bstr 의 words_with_breaks 메서드를 사용하고 있는데, 이를 위해서는 bstr 의 "unicode" 기능을 활성화해야 합니다. 이는 과거에는 Cargo가 두 패키지 간에 bstr 의 기능을 통합했기 때문에 작동했습니다. 하지만 Rust 2021로 업데이트한 후에는, 새로운 리졸버가 bstr 을 두 번 빌드하는데, 한 번은 기본 기능으로(빌드 의존성으로서), 다른 한 번은 기능 없이(일반 의존성으로서) 빌드합니다. bstr 이 이제 "unicode" 기능 없이 빌드되기 때문에 words_with_breaks 메서드가 존재하지 않으며, 메서드가 없다는 오류와 함께 빌드가 실패합니다.

해결책은 의존성이 실제로 사용하는 기능과 함께 선언되도록 하는 것입니다. 예를 들면:

[dependencies]
bstr = { version = "0.2.16", default-features = false, features = ["unicode"] }

어떤 경우에는, 직접 제어할 수 없는 서드파티 의존성에서 문제가 발생할 수 있습니다. 해당 프로젝트에 패치를 제출하여 문제가 되는 의존성에 대해 올바른 기능 세트를 선언하도록 시도해 볼 수 있습니다. 또는, 자신의 Cargo.toml 파일 내에서 어떤 의존성에든 기능을 추가할 수 있습니다. 예를 들어, 위에 제시된 bstr 예제가 어떤 서드파티 의존성에 선언되어 있다면, 올바른 의존성 선언을 자신의 프로젝트에 복사하기만 하면 됩니다. 새로운 리졸버의 통합 규칙과 일치하는 한 기능들은 통합될 것입니다. 그 규칙들은 다음과 같습니다:

  • 현재 빌드되지 않는 타겟에 대한 플랫폼별 의존성에서 활성화된 기능은 무시됩니다.
  • 빌드 의존성과 절차적 매크로(proc-macros)는 일반 의존성과 기능을 공유하지 않습니다.
  • 개발 의존성(Dev-dependencies)은 이를 필요로 하는 타겟(테스트나 예제 등)을 빌드하지 않는 한 기능을 활성화하지 않습니다.

실제 예로 diesel 과 diesel_migrations 를 사용하는 경우가 있습니다. 이 패키지들은 데이터베이스 지원을 제공하며, 다음과 같이 기능을 사용하여 데이터베이스를 선택합니다:

[dependencies]
diesel = { version = "1.4.7", features = ["postgres"] }
diesel_migrations = "1.4.0"

문제는 diesel_migrations 가 내부적으로 diesel 에 의존하는 절차적 매크로를 가지고 있는데, 이 절차적 매크로는 자신의 diesel 사본이 나머지 의존성 그래프와 동일한 기능을 활성화하고 있다고 가정한다는 점입니다. 새로운 리졸버로 업데이트한 후에는 diesel 의 사본이 두 개가 되고, 절차적 매크로를 위해 빌드된 사본에는 "postgres" 기능이 빠져 있기 때문에 빌드에 실패하게 됩니다.

해결책은 diesel 을 필요한 기능과 함께 빌드 의존성으로 추가하는 것입니다. 예를 들면:

[build-dependencies]
diesel = { version = "1.4.7", features = ["postgres"] }

이렇게 하면 Cargo가 "postgres"를 호스트 의존성(절차적 매크로 및 빌드 의존성)을 위한 기능으로 추가하게 됩니다. 이제 diesel_migrations 절차적 매크로는 "postgres" 기능이 활성화된 상태로 올바르게 빌드될 것입니다.

diesel 2.0 릴리스(현재 개발 중)는 이 종속성 요구 사항이 없도록 구조가 변경되었으므로 이 문제가 발생하지 않습니다.

기능 탐색

cargo tree 명령은 새로운 리졸버로의 이전을 돕기 위해 상당히 개선되었습니다. cargo tree 를 사용하여 의존성 그래프를 탐색하고, 어떤 기능이 활성화되었는지, 그리고 중요하게는 왜 활성화되었는지 확인할 수 있습니다.

한 가지 방법은 --duplicates 플래그(줄여서 -d)를 사용하는 것입니다. 이 플래그는 패키지가 여러 번 빌드될 때 이를 알려줍니다. 앞서 본 bstr 예제를 보면 다음과 같습니다:

> cargo tree -d
bstr v0.2.16
└── foo v0.1.0 (/MyProjects/foo)

bstr v0.2.16
[build-dependencies]
└── bar v0.1.0
    └── foo v0.1.0 (/MyProjects/foo)

이 출력은 bstr 이 두 번 빌드되었음을 알려주며, 두 경우 모두 포함되게 된 의존성 체인을 보여줍니다.

-f 플래그를 사용하여 각 패키지가 어떤 기능을 사용하고 있는지 다음과 같이 출력할 수 있습니다:

cargo tree -f '{p} {f}'

이는 Cargo에게 출력의 "형식"을 변경하도록 지시하여, 패키지와 활성화된 기능을 모두 출력하게 합니다.

또한 -e 플래그를 사용하여 표시할 "엣지(edges)"를 지정할 수 있습니다. 예를 들어, cargo tree -e features 는 각 의존성 사이에서 어떤 기능이 각 의존성에 의해 추가되고 있는지 보여줍니다. 이 옵션은 트리를 "반전"시키는 데 사용할 수 있는 -i 플래그와 함께 사용할 때 더 유용합니다. 이를 통해 기능이 특정 의존성으로 어떻게 흘러 들어가는지 볼 수 있습니다. 예를 들어, 의존성 그래프가 크고 누가 bstr 에 의존하고 있는지 확실하지 않은 경우, 다음 명령이 이를 보여줍니다:

> cargo tree -e features -i bstr
bstr v0.2.16
├── bstr feature "default"
│   [build-dependencies]
│   └── bar v0.1.0
│       └── bar feature "default"
│           └── foo v0.1.0 (/MyProjects/foo)
├── bstr feature "lazy_static"
│   └── bstr feature "unicode"
│       └── bstr feature "default" (*)
├── bstr feature "regex-automata"
│   └── bstr feature "unicode" (*)
├── bstr feature "std"
│   └── bstr feature "default" (*)
└── bstr feature "unicode" (*)

이 출력 스니펫은 foo 프로젝트가 "default" 기능을 가진 bar 에 의존하고 있음을 보여줍니다. 그리고 bar 는 "default" 기능을 가진 bstr 에 빌드 의존성으로 의존합니다. 더 나아가 bstr 의 "default" 기능이 (다른 기능들 중에서) "unicode"를 활성화한다는 것을 알 수 있습니다.

배열을 위한 IntoIterator

요약

  • 배열은 모든 에디션에서 IntoIterator 를 구현합니다.
  • IntoIterator::into_iter 호출은 메서드 호출 구문(예: array.into_iter())을 사용할 때 Rust 2015 및 Rust 2018에서 숨겨져 있습니다. 따라서 array.into_iter() 는 이전과 같이 여전히 (&array).into_iter() 로 확인됩니다.
  • array.into_iter() 는 Rust 2021에서 IntoIterator::into_iter 호출을 의미하도록 변경됩니다.

세부 정보

Rust 1.53까지는 배열에 대한 참조 만 IntoIterator 를 구현합니다. 즉, &[1, 2, 3] 및 &mut [1, 2, 3] 에 대해 반복할 수 있지만 [1, 2, 3] 에 대해서는 직접 반복할 수 없습니다.

for &e in &[1, 2, 3] {} // 정상 :)

for e in [1, 2, 3] {} // 오류 :(

이것은 오래된 문제 였지만 해결책은 보이는 것처럼 간단하지 않습니다. 단순히 트레이트 구현을 추가 하면 기존 코드가 깨질 것입니다. array.into_iter() 는 메서드 호출 구문이 작동하는 방식 때문에 암시적으로 (&array).into_iter() 를 호출하므로 오늘날 이미 컴파일됩니다. 트레이트 구현을 추가하면 의미가 변경됩니다.

일반적으로 이러한 유형의 손상(트레이트 구현 추가)은 '사소한' 것으로 분류되어 허용됩니다. 그러나 이 경우에는 너무 많은 코드가 깨질 것입니다.

It has been suggested many times to "only implement IntoIterator for arrays in Rust 2021". However, this is simply not possible. You can't have a trait implementation exist in one edition and not in another, since editions can be mixed.

대신, 트레이트 구현은 모든 에디션(Rust 1.53.0부터 시작)에 추가되었지만 Rust 2021까지 손상을 피하기 위한 작은 핵이 있습니다. Rust 2015 및 2018 코드에서 컴파일러는 이전과 같이 트레이트 구현이 존재하지 않는 것처럼 array.into_iter() 를 (&array).into_iter() 로 계속 확인합니다. 이것은 .into_iter() 메서드 호출 구문에만 _적용 _됩니다. for e in [1, 2, 3], iter.zip([1, 2, 3]) 또는 IntoIterator::into_iter([1, 2, 3]) 와 같은 다른 구문에는 영향을 미치지 않습니다. 이것들은 모든 에디션에서 작동하기 시작할 것입니다.

이것이 파손을 피하기 위해 작은 해킹이 필요했다는 것은 안타까운 일이지만, 이 솔루션은 에디션 간의 차이를 절대적인 최소값으로 유지합니다.

마이그레이션

array_into_iter 린트는 Rust 2021에서 의미가 변경될 into_iter() 에 대한 일부 호출이 있을 때마다 트리거됩니다. array_into_iter 린트는 1.41 릴리스 이후 모든 에디션에서 기본적으로 경고였습니다(1.55에서 몇 가지 개선 사항이 적용됨). 코드가 이미 경고가 없다면 Rust 2021을 사용할 준비가 된 것입니다!

코드를 Rust 2021 에디션과 호환되도록 자동으로 마이그레이션하거나 다음을 실행하여 이미 호환되는지 확인할 수 있습니다.

cargo fix --edition

에디션 간의 차이가 작기 때문에 Rust 2021로의 마이그레이션은 매우 간단합니다.

배열에서 into_iter 의 메서드 호출의 경우 구현되는 요소가 참조에서 소유 값으로 변경됩니다.

예를 들어:

fn main() {
  let array = [1u8, 2, 3];
  for x in array.into_iter() {
    // x is a `&u8` in Rust 2015 and Rust 2018
    // x is a `u8` in Rust 2021
  }
}

Rust 2021에서 마이그레이션하는 가장 간단한 방법은 이전 에디션의 정확한 동작을 유지하기 위해 iter() 를 호출하는 것입니다. 이 함수는 참조로 소유된 배열을 반복합니다.

fn main() {
  let array = [1u8, 2, 3];
  for x in array.iter() { // <- This line changed
    // x is a `&u8` in all editions
  }
}

선택적 마이그레이션

이전 에디션에서 완전 정규화된 메서드 구문(예: IntoIterator::into_iter(array))을 사용하는 경우 메서드 호출 구문(예: array.into_iter())으로 업그레이드할 수 있습니다.

클로저에서의 분리된 캡처

요약

  • || a.x + 1 은 이제 a 대신 a.x 만 캡처합니다.
  • 이로 인해 다른 시간에 항목이 삭제되거나 클로저가 Send 또는 Clone 과 같은 트레이트를 구현하는지 여부에 영향을 줄 수 있습니다.
    • 가능한 변경 사항이 감지되면 cargo fix 는 클로저가 전체 변수를 캡처하도록 강제하기 위해 let _ = &a 와 같은 문을 삽입합니다.

세부 정보

클로저 는 본문 내에서 참조하는 모든 것을 자동으로 캡처합니다. 예를 들어 || a + 1 은 주변 컨텍스트에서 a 에 대한 참조를 자동으로 캡처합니다.

Rust 2018 및 이전 버전에서 클로저는 클로저가 하나의 필드만 사용하더라도 전체 변수를 캡처합니다. 예를 들어 || a.x + 1 은 a.x 뿐만 아니라 a 에 대한 참조를 캡처합니다. a 전체를 캡처하면 a 의 다른 필드에서 변경 또는 이동이 방지되므로 다음과 같은 코드는 컴파일되지 않습니다.

let a = SomeStruct::new();
drop(a.x); // Move out of one field of the struct
println!("{}", a.y); // Ok: 구조체의 다른 필드를 계속 사용
let c = || println!("{}", a.y); // Error: Tries to capture all of `a`
c();

Rust 2021부터 클로저 캡처는 더 정확해집니다. 일반적으로 사용하는 필드만 캡처합니다(경우에 따라 사용하는 것 이상을 캡처할 수 있으며 자세한 내용은 Rust 참조를 참조하십시오). 따라서 위의 예는 Rust 2021에서 잘 컴파일됩니다.

분리된 캡처는 RFC 2229 의 일부로 제안되었으며 RFC에는 동기에 대한 자세한 내용이 포함되어 있습니다.

마이그레이션

2021 에디션의 일부로 Rust 2018 코드베이스를 Rust 2021로 자동 마이그레이션하는 데 도움이 되도록 마이그레이션 린트인 rust_2021_incompatible_closure_captures 가 추가되었습니다.

코드를 Rust 2021 에디션과 호환되도록 마이그레이션하려면 다음을 실행하십시오.

cargo fix --edition

아래에서는 자동 마이그레이션이 실패하거나 마이그레이션 작동 방식을 더 잘 이해하고 싶은 경우 Rust 2021과 호환되는 클로저 캡처를 사용하도록 코드를 수동으로 마이그레이션하는 방법을 살펴봅니다.

클로저에 의해 캡처된 변수를 변경하면 두 가지 경우에 프로그램 동작이 변경되거나 컴파일이 중지될 수 있습니다.

  • 삭제 순서 변경 또는 소멸자가 실행될 때(세부 정보);
  • 클로저가 구현하는 트레이트 변경(세부 정보).

아래 시나리오 중 하나라도 감지되면, cargo fix 는 클로저에 "dummy let"을 삽입하여 전체 변수를 캡처하도록 강제합니다:

#![allow(unused)]
fn main() {
let x = (vec![22], vec![23]);
let c = move || {
    // "Dummy let" that forces `x` to be captured in its entirety
    let _ = &x;

    // 그렇지 않으면 여기에서 `x.0` 만 캡처됩니다
    println!("{:?}", x.0);
};
}

이것은 보수적인 분석입니다. 많은 경우 이러한 더미 let은 안전하게 제거할 수 있으며 프로그램은 정상적으로 작동합니다.

와일드카드 패턴

이제 클로저는 읽어야 할 데이터만 캡처하므로 다음 클로저는 x 를 캡처하지 않습니다.

#![allow(unused)]
fn main() {
let x = 10;
let c = || {
    let _ = x; // 아무 작업도 안 함
};

let c = || match x {
    _ => println!("Hello World!")
};
}

여기서 let _ = x 문은 _ 패턴이 오른쪽을 완전히 무시하고 x 가 메모리의 한 위치(이 경우 변수)에 대한 참조이므로 아무 작업도 수행하지 않습니다.

이 변경 자체(더 적은 값 캡처)는 제안을 트리거하지 않지만 아래의 "삭제 순서" 변경과 함께 트리거할 수 있습니다.

미묘함: 우리가 삽입하는 "더미 let" let _ = &x 와 같이 no-op이 아닌 다른 유사한 표현식이 있습니다. 이것은 오른쪽(&x)이 메모리의 한 위치에 대한 참조가 아니라 먼저 평가되어야 하는 표현식이기 때문입니다(그리고 그 결과는 버려집니다).

삭제 순서

클로저가 변수 t 에서 값의 소유권을 가져오면 해당 값은 변수 t 가 범위를 벗어날 때가 아니라 클로저가 삭제될 때 삭제됩니다.

#![allow(unused)]
fn main() {
fn move_value<T>(_: T){}
{
    let t = (vec![0], vec![0]);

    {
        let c = || move_value(t); // t가 여기로 이동됩니다
    } // c is dropped, which drops the tuple `t` as well
} // t goes out of scope here
}

위의 코드는 Rust 2018과 Rust 2021 모두에서 동일하게 실행됩니다. 그러나 클로저가 변수의 일부 에 대한 소유권만 갖는 경우에는 차이가 있을 수 있습니다.

#![allow(unused)]
fn main() {
fn move_value<T>(_: T){}
{
    let t = (vec![0], vec![0]);

    {
        let c = || {
            // Rust 2018에서는 `t` 전체를 캡처합니다.
            // Rust 2021에서는 `t.0` 만 캡처합니다.
            move_value(t.0);
        };

        // Rust 2018에서는 이 블록을 나갈 때 `c` (그리고 `t`)가 모두 드롭됩니다.
        //
        //
        // Rust 2021에서는 이 블록을 나갈 때 `c` 와 `t.0` 가 모두 드롭됩니다.
        //
    }

// Rust 2018에서는 `t` 의 값이 이동되어 삭제되지 않습니다.
//
// Rust 2021에서는 `t.0` 의 값이 이동되었지만 `t.1` 은 남아 있으므로 여기서 삭제됩니다.
}
}

대부분의 경우 다른 시간에 값을 삭제하는 것은 메모리가 해제되는 시점에만 영향을 미치며 중요하지 않습니다. 그러나 일부 Drop 구현(일명 소멸자)에는 부작용이 있으며 이러한 경우 삭제 순서를 변경하면 프로그램의 의미 체계가 변경될 수 있습니다. 이러한 경우 컴파일러는 전체 변수를 캡처하도록 강제하기 위해 더미 let 을 삽입하도록 제안합니다.

트레이트 구현

클로저는 캡처하는 값에 따라 다음 트레이트를 자동으로 구현합니다.

  • Clone: 모든 캡처된 값이 Clone 인 경우.
  • 자동 트레이트 와 같은 Send, Sync 및 UnwindSafe: 모든 캡처된 값이 주어진 트레이트를 구현하는 경우.

Rust 2021에서는 다른 값이 캡처되므로 클로저가 구현할 트레이트에 영향을 줄 수 있습니다. 마이그레이션 린트는 각 클로저를 테스트하여 이전에 주어진 트레이트를 구현했는지 여부와 지금도 구현하는지 여부를 확인합니다. 이전에 구현되었지만 더 이상 구현되지 않는 트레이트를 발견하면 "더미 let"이 삽입됩니다.

예를 들어, 스레드 간에 원시 포인터를 전달할 수 있도록 허용하는 일반적인 방법은 이를 구조체로 래핑한 다음 래퍼에 대해 Send/Sync 자동 트레이트를 구현하는 것입니다. thread::spawn 에 전달되는 클로저는 래퍼 내의 특정 필드를 사용하지만 전체 래퍼는 관계없이 캡처됩니다. 래퍼가 Send/Sync 이므로 코드는 안전한 것으로 간주되어 성공적으로 컴파일됩니다.

분리된 캡처를 사용하면 클로저에 언급된 특정 필드만 캡처되므로 원래 Send/Sync 가 아니었던 래퍼의 목적을 무효화합니다.

#![allow(unused)]
fn main() {
use std::thread;

struct Ptr(*mut i32);
unsafe impl Send for Ptr {}


let mut x = 5;
let px = Ptr(&mut x as *mut i32);

let c = thread::spawn(move || {
    unsafe {
        *(px.0) += 10;
    }
}); // 클로저가 Send가 아닌 px.0을 캡처했습니다
}

Panic 매크로 일관성

요약

  • panic!(..) 은 이제 println!() 처럼 항상 format_args!(..) 를 사용합니다.
  • panic!("{") 은 { 를 {{ 로 이스케이프하지 않으면 더 이상 허용되지 않습니다.
  • x 가 문자열 리터럴이 아니면 panic!(x) 는 더 이상 허용되지 않습니다.
    • 문자열이 아닌 페이로드로 패닉하려면 std::panic::panic_any(x) 를 사용하십시오.
    • 또는 panic!("{}", x) 를 사용하여 x 의 Display 구현을 사용하십시오.
  • assert!(expr, ..) 에도 동일하게 적용됩니다.

세부 정보

panic!() 매크로는 Rust에서 가장 잘 알려진 매크로 중 하나입니다. 그러나 하위 호환성으로 인해 변경할 수 없는 몇 가지 미묘한 놀라움 이 있습니다.

// Rust 2018
panic!("{}", 1); // Ok, 메시지 "1"로 패닉
panic!("{}"); // Ok, panics with the message "{}"

panic!() 매크로는 두 개 이상의 인수로 호출될 때만 문자열 서식을 사용합니다. 단일 인수로 호출될 때는 해당 인수를 보지도 않습니다.

// Rust 2018
let a = "{";
println!(a); // 오류: 첫 번째 인수는 형식 문자열 리터럴이어야 합니다
panic!(a); // Ok: The panic macro doesn't care

It even accepts non-strings such as panic!(123), which is uncommon and rarely useful since it produces a surprisingly unhelpful message: panicked at 'Box<Any>'.

This will especially be a problem once implicit format arguments are stabilized. That feature will make println!("hello {name}") a short-hand for println!("hello {}", name). However, panic!("hello {name}") would not work as expected, since panic!() doesn't process a single argument as format string.

이러한 혼란스러운 상황을 피하기 위해 Rust 2021에는 더 일관된 panic!() 매크로가 있습니다. 새 panic!() 매크로는 더 이상 임의의 표현식을 유일한 인수로 허용하지 않습니다. println!() 과 마찬가지로 항상 첫 번째 인수를 형식 문자열로 처리합니다. panic!() 이 더 이상 임의의 페이로드를 허용하지 않으므로 panic_any() 가 서식이 지정된 문자열 이외의 것으로 패닉하는 유일한 방법이 될 것입니다.

// Rust 2021
panic!("{}", 1); // Ok, 메시지 "1"로 패닉
panic!("{}"); // 오류, 인수 누락
panic!(a); // Error, must be a string literal

또한 core::panic!() 과 std::panic!() 은 Rust 2021에서 동일합니다. 현재 이 둘 사이에는 몇 가지 역사적 차이점이 있으며, 이는 #![no_std] 를 켜거나 끌 때 눈에 띌 수 있습니다.

마이그레이션

non_fmt_panics 린트는 Rust 2021에서 오류가 발생할 일부 사용되지 않는 동작을 사용하는 panic 에 대한 일부 호출이 있을 때마다 트리거됩니다. non_fmt_panics 린트는 1.50 릴리스 이후 모든 에디션에서 기본적으로 경고였습니다(이후 릴리스에서 몇 가지 개선 사항이 적용됨). 코드가 이미 경고가 없다면 Rust 2021을 사용할 준비가 된 것입니다!

코드를 Rust 2021 에디션과 호환되도록 자동으로 마이그레이션하거나 다음을 실행하여 이미 호환되는지 확인할 수 있습니다.

cargo fix --edition

수동으로 마이그레이션하도록 선택하거나 필요하다면 모든 패닉 호출을 println 과 동일한 서식을 사용하거나 std::panic::panic_any 를 사용하여 문자열이 아닌 데이터로 패닉하도록 업데이트해야 합니다.

예를 들어, panic!(MyStruct) 의 경우 std::panic::panic_any 를 사용하도록 변환해야 합니다(이것은 매크로가 아닌 함수임에 유의하십시오). std::panic::panic_any(MyStruct).

In the case of panic messages that include curly braces but the wrong number of arguments (e.g., panic!("Some curlies: {}")), you can panic with the string literal by either using the same syntax as println! (i.e., panic!("{}", "Some curlies: {}")) or by escaping the curly braces (i.e., panic!("Some curlies: {{}}")).

예약된 구문

요약

  • any_identifier#, any_identifier"...", any_identifier'...', and 'any_identifier#는 이제 예약된 구문이며 더 이상 토큰화되지 않습니다.
  • 이것은 주로 매크로와 관련이 있습니다. 예를 들어 quote!{ #a#b } 는 더 이상 허용되지 않습니다.
  • 키워드를 특별하게 취급하지 않으므로 예를 들어 match"..." {} 는 더 이상 허용되지 않습니다.
  • 오류를 방지하려면 식별자와 그 뒤에 오는 #, ", 또는 ' 사이에 공백을 삽입하세요.
  • 에디션 마이그레이션은 이러한 경우에 공백을 삽입하는 데 도움이 됩니다.

세부 정보

To make space for new syntax in the future, we've decided to reserve syntax for prefixed identifiers, literals, and lifetimes: prefix#identifier, prefix"string", prefix'c', prefix#123, and 'prefix#, where prefix can be any identifier. (Except those prefixes that already have a meaning, such as b'...' (byte chars) and r"..." (raw strings).)

이것은 에디션 경계 없이 향후 확장할 수 있는 구문을 제공합니다. 다음 에디션까지 임시 구문으로 사용하거나 적절한 경우 영구 구문으로 사용할 수 있습니다.

Without an edition, this would be a breaking change, since macros can currently accept syntax such as hello"world", which they will see as two separate tokens: hello and "world". The (automatic) fix is simple though: just insert a space: hello "world". Likewise, prefix#ident should become prefix #ident. Edition migrations will help with this fix.

이것들을 토큰화 오류로 바꾸는 것 외에, RFC 는 아직 어떤 접두사에도 의미를 부여하지 않습니다. 특정 접두사에 의미를 할당하는 것은 향후 제안에 맡겨지며, 이제 이러한 접두사를 예약한 덕분에 주요 변경 사항이 되지 않을 것입니다.

향후에 볼 수 있는 몇 가지 새로운 접두사(아직 어떤 것도 약속하지 않았지만):

  • k#keyword 는 현재 에디션에 아직 존재하지 않는 키워드를 작성할 수 있도록 허용합니다. 예를 들어, async 가 2015 에디션에서 키워드가 아니었지만 이 접두사를 사용하면 2018 에디션에서 async 를 키워드로 예약할 때까지 기다릴 필요 없이 2015 에디션에서 k#async 를 허용할 수 있었습니다.

  • f"" as a short-hand for a format string. For example, f"hello {name}" as a short-hand for the equivalent format!() invocation.

  • String 리터럴에 대한 s"".

다음과 같이 정의된 매크로가 있다고 가정해 보겠습니다.

#![allow(unused)]
fn main() {
macro_rules! my_macro {
    ($a:tt $b:tt) => {};

Rust 2015 및 2018에서는 첫 번째 토큰 트리와 두 번째 토큰 트리 사이에 공백 없이 이 매크로를 호출하는 것이 합법적입니다.

이 z 접두사는 Rust 2021에서 더 이상 허용되지 않으므로 이 매크로를 호출하려면 접두사 뒤에 공백을 추가해야 합니다.

my_macro!(z "hey");

요약

세부 정보

원시 라이프타임(Raw lifetimes)은 새로운 키워드를 도입하는 최신 에디션으로 마이그레이션할 수 있는 기능을 지원하기 위해 2021 에디션에 도입되었습니다. 이는 식별자에 대해 동일한 기능을 제공하는 원시 식별자(raw identifiers) 와 유사합니다. 예를 들어, 2024 에디션에는 gen 키워드가 도입되었습니다. 라이프타임은 키워드가 될 수 없으므로, 'gen 라이프타임을 사용하는 코드는 컴파일에 실패하게 됩니다. 원시 라이프타임을 사용하면 마이그레이션 린트가 이러한 라이프타임을 키워드로 허용되는 'r#gen 으로 수정할 수 있습니다.

2021 이전 에디션에서는 원시 라이프타임이 별도의 토큰으로 구문 분석됩니다. 예를 들어 'r#foo 는 'r, # 및 foo 의 세 가지 토큰으로 구문 분석됩니다.

my_macro!('r# foo);

오류로 승격된 경고

요약

  • bare_trait_objects 및 ellipsis_inclusive_range_patterns 린트를 트리거한 코드는 Rust 2021에서 오류가 발생합니다.

세부 정보

두 개의 기존 린트는 Rust 2021에서 하드 오류가 되지만 이러한 린트는 이전 에디션에서는 경고로 유지됩니다.

bare_trait_objects:

트레이트 객체 를 식별하기 위해 dyn 키워드를 사용하는 것은 Rust 2021에서 필수가 됩니다.

예를 들어, &MyTrait 에 dyn 키워드를 포함하지 않는 다음 코드는 Rust 2021에서 단순한 린트 대신 오류를 생성합니다.

#![allow(unused)]
fn main() {
pub trait MyTrait {}

pub fn my_function(_trait_object: &MyTrait) { // `&dyn MyTrait` 여야 합니다
  unimplemented!()
}
}

ellipsis_inclusive_range_patterns:

포괄적 범위 패턴(즉, 끝 값이 범위에 포함 되는 범위)에 대한 사용되지 않는 ... 구문 은 Rust 2021에서 더 이상 허용되지 않습니다. 표현식과 일치하는 ..= 로 대체되었습니다.

예를 들어, 패턴에서 ... 를 사용하는 다음 코드는 Rust 2021에서 단순한 린트 대신 오류를 생성합니다.

#![allow(unused)]
fn main() {
pub fn less_or_eq_to_100(n: u8) -> bool {
  matches!(n, 0...100) // `0..=100` 이어야 합니다
}
}

마이그레이션

Rust 2015 또는 2018 코드가 bare_trait_objects 또는 ellipsis_inclusive_range_patterns 에 대한 경고를 생성하지 않고 #![allow()] 또는 다른 메커니즘을 사용하여 이러한 린트를 허용하지 않은 경우 마이그레이션할 필요가 없습니다.

패턴에서 ... 를 사용하거나 트레이트 객체와 함께 dyn 을 사용하지 않는 모든 크레이트를 자동으로 마이그레이션하려면 cargo fix --edition 을 실행할 수 있습니다.

매크로 규칙에서의 OR 패턴

요약

  • macro_rules 매크로에서 패턴이 작동하는 방식이 약간 변경됩니다.
    • macro_rules 의 $_:pat 는 이제 | 의 사용법과도 일치합니다(예: A | B).
    • 새로운 $_:pat_param 은 이전의 $_:pat 처럼 동작합니다. (최상위 수준) | 와 일치하지 않습니다.
    • $_:pat_param 은 모든 에디션에서 사용할 수 있습니다.

세부 정보

Rust 1.53.0부터 패턴 은 패턴의 어느 곳에나 중첩된 | 를 지원하도록 확장되었습니다. 이를 통해 Some(1) | Some(2) 대신 Some(1 | 2) 를 작성할 수 있습니다. 이전에는 이것이 허용되지 않았기 때문에 이것은 주요 변경 사항이 아닙니다.

그러나 이 변경 사항은 macro_rules 매크로 에도 영향을 줍니다. 이러한 매크로는 :pat 프래그먼트 지정자를 사용하여 패턴을 허용할 수 있습니다. 현재 :pat 는 최상위 수준 | 와 일치하지 않습니다. Rust 1.53 이전에는 모든 패턴(모든 중첩 수준에서)에 | 가 포함될 수 없었기 때문입니다. A | B 와 같은 패턴을 허용하는 매크로(예: matches!())는 $($_:pat)|+ 와 같은 것을 사용합니다.

이것이 기존 매크로를 잠재적으로 깨뜨릴 수 있기 때문에 :pat 의 의미는 Rust 1.53.0에서 | 를 포함하도록 변경되지 않았습니다. 대신 해당 변경은 Rust 2021에서 발생합니다. 새 에디션에서 :pat 프래그먼트 지정자는 A | B 와 일치 합니다.

Rust 2021의 $_:pat 프래그먼트 뒤에는 명시적인 | 가 올 수 없습니다. 여전히 | 가 뒤따르는 패턴 프래그먼트와 일치시키고 싶을 때가 있으므로 이전 동작을 유지하기 위해 :pat_param 으로 지정된 프래그먼트가 추가되었습니다.

에디션은 크레이트별 이므로 매크로가 정의된 크레이트의 에디션만 관련이 있다는 것을 기억하는 것이 중요합니다. 매크로가 사용되는 크레이트의 에디션은 매크로 작동 방식을 변경하지 않습니다.

마이그레이션

rust_2021_incompatible_or_patterns 린트는 Rust 2021에서 의미가 변경될 $_:pat 사용이 있을 때마다 트리거됩니다.

코드를 Rust 2021 에디션과 호환되도록 자동으로 마이그레이션하거나 다음을 실행하여 이미 호환되는지 확인할 수 있습니다.

cargo fix --edition

패턴에서 최상위 수준의 | 사용과 일치하지 않는 $_:pat 에 의존하는 매크로가 있는 경우 각 $_:pat 발생을 $_:pat_param 으로 변경해야 합니다.

예를 들어:

#![allow(unused)]
fn main() {
macro_rules! my_macro { 
	($x:pat | $y:pat) => {
		// TODO: 구현
	} 
}

// 이 매크로는 `$x:pat` 가 `|` 와 일치하지 않기 때문에 Rust 2018에서 작동합니다:
my_macro!(1 | 2);

// In Rust 2021 however, the `$_:pat` fragment matches `|` and is not allowed
// to be followed by a `|`. To make sure this macro still works in Rust 2021
// change the macro to the following:
macro_rules! my_macro { 
	($x:pat_param | $y:pat) => { // <- 이 줄은 다릅니다
		// TODO: 구현
	} 
}
}

C 문자열 리터럴

요약

  • Literals of the form c"foo" or cr"foo" represent a string of type &core::ffi::CStr.

세부 정보

Rust 1.77부터 C-문자열은 c 또는 cr 접두사와 함께 C-문자열 리터럴 구문을 사용하여 작성할 수 있습니다.

이전에는 NUL 바이트로 끝나는 C API와 상호 운용할 수 있는 유효한 문자열 리터럴을 올바르게 생성하기가 어려웠습니다. cstr 크레이트가 널리 사용되는 해결책이었지만, 이는 매우 비싼 proc-매크로를 컴파일해야 했습니다. 이제 C-문자열은 리터럴 구문 표기법을 사용하여 직접 작성할 수 있으며, 이는 자동으로 NUL 바이트로 끝나는 &core::ffi::CStr 유형의 값을 생성합니다.

#![allow(unused)]
fn main() {
use core::ffi::CStr;

assert_eq!(c"hello", CStr::from_bytes_with_nul(b\"hello\0\").unwrap());
assert_eq!(
    c"byte escapes \xff work",
    CStr::from_bytes_with_nul(b"byte escapes \xff work\0").unwrap()
);
assert_eq!(
    c"unicode escapes \u{00E6} work",
    CStr::from_bytes_with_nul(b"unicode escapes \xc3\xa6 work\0").unwrap()
);
assert_eq!(
    c"unicode characters αβγ encoded as UTF-8",
    CStr::from_bytes_with_nul(
        b"unicode characters \xce\xb1\xce\xb2\xce\xb3 encoded as UTF-8\0"
    )
    .unwrap()
);
assert_eq!(
    c"strings can continue \
        on multiple lines",
    CStr::from_bytes_with_nul(b"strings can continue on multiple lines\0").unwrap()
);
}

C-문자열은 내부 NUL 바이트(\0 이스케이프와 같은)를 허용하지 않습니다.

일반 문자열과 마찬가지로 C-문자열도 cr 접두사가 붙은 "원시(raw)" 구문을 지원합니다. 이러한 원시 C-문자열은 백슬래시 이스케이프를 처리하지 않으므로 백슬래시가 포함된 문자열을 작성하기가 더 쉽습니다. 따옴표를 # 문자로 감싸서 큰따옴표를 포함할 수 있습니다. 내부의 "# 시퀀스와의 모호성을 피하기 위해 여러 개의 # 문자를 사용할 수 있습니다.

#![allow(unused)]
fn main() {
assert_eq!(cr"foo", c"foo");
// 숫자 기호(#)를 사용하여 내부에 큰따옴표를 포함할 수 있습니다.
assert_eq!(cr#""foo""#, c"\"foo\"");
// 이것은 두 개의 #이 필요합니다.
assert_eq!(cr##""foo"#"##, c"\"foo\"#");
// 이스케이프는 처리되지 않습니다.
assert_eq!(cr"C:\foo", c"C:\\foo");
}

자세한 내용은 참조 문서를 확인하세요.

마이그레이션

마이그레이션은 c"…" 또는 cr"…" 와 유사하게 보이는 토큰 시퀀스를 가정했을 수 있는 매크로에 대해서만 필요합니다. 이러한 시퀀스는 2021 에디션 이전에는 두 개의 별개 토큰으로 토큰화되었지만, 2021 에디션에서는 단일 토큰으로 나타납니다.

2021 에디션을 위한 구문 예약 의 일환으로, 이 문제가 발생할 수 있는 모든 매크로 입력은 rust_2021_prefixes_incompatible_syntax 마이그레이션 린트에서 경고를 발생시켜야 합니다. 자세한 내용은 해당 챕터를 참조하세요.

RPIT 라이프타임 캡처 규칙

이 장에서는 RFC 3498 에서 도입된 라이프타임 캡처 규칙 2024(Lifetime Capture Rules 2024) 와 관련된 변경 사항을 설명하며, 여기에는 코드를 마이그레이션하기 위해 불투명 타입(opaque type)의 정밀 캡처(precise capturing)(RFC 3617 에서 도입됨)를 사용하는 방법이 포함됩니다.

요약

  • Rust 2024에서는 use<..> 바운드가 없을 때 라이프타임 매개변수를 포함한 범위 내의 모든 제네릭 매개변수가 암시적으로 캡처됩니다.
  • Captures 트릭(Captures<..> 바운드) 및 outlives 트릭(예: '_ 바운드)의 사용은 (모든 에디션에서) use<..> 바운드로 대체되거나 (Rust 2024에서) 완전히 제거될 수 있습니다.

세부 정보

캡처링

RPIT(반환 위치 impl Trait) 불투명 타입에서 제네릭 매개변수를 캡처 하면 해당 매개변수를 해당하는 숨겨진 타입에서 사용할 수 있습니다. Rust 1.82에서는 캡처할 제네릭 매개변수를 명시적으로 지정할 수 있는 use<..> 바운드를 추가했습니다. 이는 코드를 Rust 2024로 마이그레이션하는 데 도움이 되며, 이 챕터에서 에디션별 암시적 캡처 규칙이 어떻게 작동하는지 설명하는 데에도 유용합니다. 이러한 use<..> 바운드는 다음과 같습니다:

#![allow(unused)]
fn main() {
fn capture<'a, T>(x: &'a (), y: T) -> impl Sized + use<'a, T> {
    //                                ~~~~~~~~~~~~~~~~~~~~~~~
    //                             이것은 RPIT 불투명 타입입니다.
    //
    //                                이것은 `'a` 와 `T` 를 캡처합니다.
    (x, y)
  //~~~~~~
  // 숨겨진 타입은: `(&'a (), T)` 입니다.
  //
  // 이 타입은 `'a` 와 `T` 가 캡처되었기 때문에 이를 사용할 수 있습니다.
}
}

캡처된 제네릭 매개변수는 불투명 타입을 사용하는 방식에 영향을 미칩니다. 예를 들어, 숨겨진 타입이 라이프타임을 사용하지 않음에도 불구하고 라이프타임이 캡처되기 때문에 다음은 오류입니다:

#![allow(unused)]
fn main() {
fn capture<'a>(_: &'a ()) -> impl Sized + use<'a> {}

fn test<'a>(x: &'a ()) -> impl Sized + 'static {
    capture(x)
    //~^ 오류: 라이프타임이 충분히 오래 지속되지 않을 수 있습니다
}
}

반대로, 다음은 괜찮습니다:

#![allow(unused)]
fn main() {
fn capture<'a>(_: &'a ()) -> impl Sized + use<> {}

fn test<'a>(x: &'a ()) -> impl Sized + 'static {
    capture(x) //~ OK
}
}

use<..> 바운드가 없을 때의 에디션별 규칙

use<..> 바운드가 없는 경우, 컴파일러는 에디션별 규칙을 사용하여 범위 내의 어떤 제네릭 매개변수를 암시적으로 캡처할지 결정합니다.

모든 에디션에서 use<..> 바운드가 없을 때 범위 내의 모든 타입 및 상수 제네릭 매개변수는 암시적으로 캡처됩니다. 예를 들면:

#![allow(unused)]
fn main() {
fn f_implicit<T, const C: usize>() -> impl Sized {}
//                                    ~~~~~~~~~~
//                         여기에 `use<..>` 바운드가 없습니다.
//
// 모든 에디션에서 위 코드는 다음과 동일합니다:
fn f_explicit<T, const C: usize>() -> impl Sized + use<T, C> {}
}

In Rust 2021 and earlier editions, when the use<..> bound is not present, generic lifetime parameters are only captured when they appear syntactically within a bound in RPIT opaque types in the signature of bare functions and associated functions and methods within inherent impls. However, starting in Rust 2024, these in-scope generic lifetime parameters are unconditionally captured. E.g.:

#![allow(unused)]
fn main() {
fn f_implicit(_: &()) -> impl Sized {}
// Rust 2021 및 이전 버전에서 위 코드는 다음과 같습니다:
fn f_2021(_: &()) -> impl Sized + use<> {}
// Rust 2024 및 이후 버전에서 이는 다음과 같습니다:
fn f_2024(_: &()) -> impl Sized + use<'_> {}
}

이로써 트레이트 impl 내의 연관 함수 및 메서드 시그니처에 있는 RPIT 불투명 타입, 트레이트 정의 내에서의 RPIT 사용(RPITIT), async fn 에 의해 생성된 불투명 Future 타입과 동작이 일관되게 됩니다. 이들은 모두 use<..> 바운드가 없을 때 모든 에디션에서 범위 내의 모든 제네릭 라이프타임 매개변수를 암시적으로 캡처합니다.

외부 제네릭 매개변수

암시적으로 캡처되는 것을 결정할 때 외부 impl의 제네릭 매개변수는 범위 내에 있는 것으로 간주됩니다. 예:

#![allow(unused)]
fn main() {
struct S<T, const C: usize>((T, [(); C]));
impl<T, const C: usize> S<T, C> {
//   ~~~~~~~~~~~~~~~~~
// 이 제네릭 매개변수들은 범위 내에 있습니다.
    fn f_implicit<U>() -> impl Sized {}
    //            ~       ~~~~~~~~~~
    //            ^ 이 제네릭도 범위 내에 있습니다.
    //                    ^
    //                    |
    //     여기에 `use<..>` 바운드가 없습니다.
    //
    // 모든 에디션에서 이는 다음과 같습니다:
    fn f_explicit<U>() -> impl Sized + use<T, U, C> {}
}
}

고차(higher-ranked) 바인더의 라이프타임

마찬가지로, 고차 for<..> 바인더에 의해 범위에 도입된 제네릭 라이프타임 매개변수도 범위 내에 있는 것으로 간주됩니다. 예:

#![allow(unused)]
fn main() {
trait Tr<'a> { type Ty; }
impl Tr<'_> for () { type Ty = (); }

fn f_implicit() -> impl for<'a> Tr<'a, Ty = impl Copy> {}
// Rust 2021 및 이전 버전에서 위 코드는 다음과 같습니다:
fn f_2021() -> impl for<'a> Tr<'a, Ty = impl Copy + use<>> {}
// Rust 2024 및 이후 버전에서 이는 다음과 같습니다:
//fn f_2024() -> impl for<'a> Tr<'a, Ty = impl Copy + use<'a>> {}
//                                        ~~~~~~~~~~~~~~~~~~~~
// 그러나 고차 라이프타임의 캡처는
// 중첩된 불투명 타입에서는 아직 지원되지 않습니다.
}

인수 위치 impl Trait (APIT)

APIT(인수 위치 impl Trait) 사용으로 생성된 익명(즉, 이름 없는) 제네릭 매개변수도 범위 내에 있는 것으로 간주됩니다. 예:

#![allow(unused)]
fn main() {
fn f_implicit(_: impl Sized) -> impl Sized {}
//               ~~~~~~~~~~
//           이것을 APIT라고 부릅니다.
//
// 위 코드는 *대략* 다음과 같습니다:
fn f_explicit<_0: Sized>(_: _0) -> impl Sized + use<_0> {}
}

전자가 후자와 정확히 동일하지는 않다는 점에 유의하세요. 제네릭 매개변수에 이름을 지정하면 터보피시(turbofish) 구문을 사용하여 인수를 제공할 수 있기 때문입니다. 익명 제네릭 매개변수를 명명된 제네릭 매개변수로 변환하지 않고 use<..> 바운드에 명시적으로 포함할 수 있는 방법은 없습니다.

마이그레이션

과도한 캡처(overcapturing)를 피하면서 마이그레이션하기

impl_trait_overcaptures 린트는 Rust 2024에서 추가 라이프타임을 캡처하게 될 RPIT 불투명 타입을 플래그합니다. 이 린트는 cargo fix --edition 을 실행할 때 자동으로 적용되는 rust-2024-compatibility 린트 그룹의 일부입니다. 대부분의 경우, 이 린트는 Rust 2024에서 추가 라이프타임이 캡처되지 않도록 필요한 곳에 use<..> 바운드를 자동으로 삽입할 수 있습니다.

코드를 Rust 2024와 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

예를 들어, 이것은 다음과 같이 변경됩니다:

#![allow(unused)]
fn main() {
fn f<'a>(x: &'a ()) -> impl Sized { *x }
}

...을 다음으로:

#![allow(unused)]
fn main() {
fn f<'a>(x: &'a ()) -> impl Sized + use<> { *x }
}

Rust 2024에서 이 use<> 바운드가 없으면 불투명 타입은 'a 라이프타임 매개변수를 캡처하게 됩니다. 이 바운드를 추가함으로써 마이그레이션 린트는 기존 의미를 보존합니다.

APIT와 관련된 케이스 마이그레이션

경우에 따라 린트가 변경을 자동으로 수행할 수 없는 경우가 있는데, 이는 제네릭 매개변수가 use<..> 바운드 내에 나타날 수 있도록 이름을 지정해야 하기 때문입니다. 이러한 경우 린트는 수동으로 변경해야 할 수도 있음을 알려줍니다. 예를 들어, 다음과 같은 경우:

#![allow(unused)]
fn main() {
fn f<'a>(x: &'a (), y: impl Sized) -> impl Sized { (*x, y) }
//   ^^                ~~~~~~~~~~
//               이것은 APIT의 사용입니다.
//
//~^ 경고 `impl Sized` 는 2024 에디션에서 의도한 것보다 더 많은 라이프타임을 캡처할 것입니다
//~| 참고 구체적으로, 이 라이프타임은 범위 내에 있지만 타입의 바운드에서 언급되지 않았습니다

fn test<'a>(x: &'a (), y: ()) -> impl Sized + 'static {
    f(x, y)
}
}

APIT의 사용과 제네릭 타입 매개변수가 use<..> 바운드에서 명명되어야 한다는 사실 때문에 코드를 자동으로 변환할 수 없습니다. 라이프타임을 캡처하지 않고 이 코드를 Rust 2024로 변환하려면 해당 타입 매개변수의 이름을 지정해야 합니다. 예:

#![allow(unused)]
fn main() {
#![deny(impl_trait_overcaptures)]
fn f<'a, T: Sized>(x: &'a (), y: T) -> impl Sized + use<T> { (*x, y) }
//       ~~~~~~~~
// 여기에 타입 매개변수의 이름이 지정되었습니다.

fn test<'a>(x: &'a (), y: ()) -> impl Sized + use<> {
    f(x, y)
}
}

Note that this changes the API of the function slightly as a type argument can now be explicitly provided for this parameter using turbofish syntax. If this is undesired, you might consider instead whether you can simply continue to omit the use<..> bound and allow the lifetime to be captured. This might be particularly desirable if you might in the future want to use that lifetime in the hidden type and would like to save space for that.

Captures 트릭에서 마이그레이션하기

Rust 1.82에서 정확한 캡처 use<..> 바운드가 도입되기 전에는 RPIT 불투명 타입에서 라이프타임을 올바르게 캡처하려면 종종 Captures 트릭을 사용해야 했습니다. 예:

#![allow(unused)]
fn main() {
#[doc(hidden)]
pub trait Captures<T: ?Sized> {}
impl<T: ?Sized, U: ?Sized> Captures<T> for U {}

fn f<'a, T>(x: &'a (), y: T) -> impl Sized + Captures<(&'a (), T)> {
//                                           ~~~~~~~~~~~~~~~~~~~~~
//                            이것을 `Captures` 트릭이라고 부릅니다.
    (x, y)
}

fn test<'t, 'x>(t: &'t (), x: &'x ()) {
    f(t, x);
}
}

use<..> 바운드 구문을 사용하면 Captures 트릭은 더 이상 필요하지 않으며 모든 에디션에서 다음과 같이 대체될 수 있습니다.

#![allow(unused)]
fn main() {
fn f<'a, T>(x: &'a (), y: T) -> impl Sized + use<'a, T> {
    (x, y)
}

fn test<'t, 'x>(t: &'t (), x: &'x ()) {
    f(t, x);
}
}

Rust 2024에서는 use<..> 바운드를 완전히 생략할 수 있는 경우가 많으며, 위 코드는 다음과 같이 간단하게 작성할 수 있습니다.

#![allow(unused)]
fn main() {
fn f<'a, T>(x: &'a (), y: T) -> impl Sized {
    (x, y)
}

fn test<'t, 'x>(t: &'t (), x: &'x ()) {
    f(t, x);
}
}

이에 대한 자동 마이그레이션은 없으며 Captures 트릭은 Rust 2024에서도 여전히 작동하지만, 이 오래된 트릭을 사용하는 코드에서 수동으로 마이그레이션하는 것을 고려할 수 있습니다.

Outlives 트릭에서 마이그레이션하기

Rust 1.82에서 정확한 캡처 use<..> 바운드가 도입되기 전에는 일부 불투명 타입의 숨겨진 타입에서 라이프타임을 사용해야 할 때 "outlives 트릭"을 사용하는 것이 일반적이었습니다. 예:

#![allow(unused)]
fn main() {
fn f<'a, T: 'a>(x: &'a (), y: T) -> impl Sized + 'a {
    //    ~~~~                                 ~~~~
    //    ^                     This is the outlives trick.
    //    |
    // This bound is needed only for the trick.
    (x, y)
//  ~~~~~~
// 숨겨진 타입은 `(&'a (), T)` 입니다.
}
}

This trick was less baroque than the Captures trick, but also less correct. As we can see in the example above, even though any lifetime components within T are independent of the lifetime 'a, we're required to add a T: 'a bound in order to make the trick work. This created undue and surprising restrictions on callers.

정밀 캡처를 사용하면 모든 에디션에서 위 코드를 다음과 같이 대신 작성할 수 있습니다.

#![allow(unused)]
fn main() {
fn f<T>(x: &(), y: T) -> impl Sized + use<'_, T> {
    (x, y)
}

fn test<'t, 'x>(t: &'t (), x: &'x ()) {
   f(t, x);
}
}

Rust 2024에서는 use<..> 바운드를 완전히 생략할 수 있는 경우가 많으며, 위 코드는 다음과 같이 간단하게 작성할 수 있습니다.

#![allow(unused)]
fn main() {
fn f<T>(x: &(), y: T) -> impl Sized {
    (x, y)
}

fn test<'t, 'x>(t: &'t (), x: &'x ()) {
   f(t, x);
}
}

이에 대한 자동 마이그레이션은 없으며 outlives 트릭은 Rust 2024에서도 여전히 작동하지만, 이 오래된 트릭을 사용하는 코드에서 수동으로 마이그레이션하는 것을 고려할 수 있습니다.

if let 임시 스코프

요약

  • if let $pat = $expr { .. } else { .. } 표현식에서, $expr 을 평가하여 생성된 임시 값은 프로그램이 else 분기로 들어가기 전(종료 후가 아님)에 드롭됩니다.

세부 정보

2024 에디션은 if let 표현식의 검사 대상(scrutinee)1 에 있는 임시 값 의 드롭 범위를 변경합니다. 이는 임시 값이 너무 오래 유지됨으로써 발생할 수 있는 잠재적인 예기치 않은 동작을 줄이기 위한 것입니다.

2024년 이전에는 임시 변수가 if let 표현식 자체를 넘어 확장될 수 있었습니다. 예를 들어:

#![allow(unused)]
fn main() {
// 2024년 이전
use std::sync::RwLock;

fn f(value: &RwLock<Option<bool>>) {
    if let Some(x) = *value.read().unwrap() {
        println!("값은 {x}");
    } else {
        let mut v = value.write().unwrap();
        if v.is_none() {
            *v = Some(true);
        }
    }
    // <--- 2021에서는 여기에서 읽기 잠금이 해제됩니다
}
}

이 예제에서 value.read() 호출로 생성된 임시 읽기 잠금은 if let 표현식 이후(즉, else 블록 이후)까지 드롭되지 않습니다. else 블록이 실행되는 경우, 쓰기 잠금을 획득하려고 시도할 때 교착 상태(deadlock)가 발생합니다.

2024 에디션에서는 임시 변수의 라이프타임을 then 블록이 완전히 평가되거나 프로그램 제어가 else 블록으로 들어가는 시점으로 단축합니다.

#![allow(unused)]
fn main() {
// 2024부터
use std::sync::RwLock;

fn f(value: &RwLock<Option<bool>>) {
    if let Some(x) = *value.read().unwrap() {
        println!("값은 {x}");
    }
    // <--- 2024에서는 여기에서 읽기 잠금이 해제됩니다
    else {
        let mut v = value.write().unwrap();
        if v.is_none() {
            *v = Some(true);
        }
    }
}
}

See the temporary scope rules for more information about how temporary scopes are extended. See the tail expression temporary scope chapter for a similar change made to tail expressions.

마이그레이션

if let 을 match 로 다시 작성하는 것은 항상 안전합니다. match 검사 대상의 임시 값은 match 표현식의 끝(보통 문장의 끝)까지 연장되는데, 이는 if let 의 2021 에디션 동작과 동일합니다.

if_let_rescope 린트는 이 변경으로 인해 라이프타임 문제가 발생하거나 린트가 if let 의 검사 대상(scrutinee)에서 사용자 정의, 사소하지 않은 Drop 소멸자가 있는 임시 값이 생성됨을 감지할 때 수정을 제안합니다. 예를 들어, cargo fix 의 제안이 수락되면 이전 예제는 다음과 같이 다시 작성될 수 있습니다:

#![allow(unused)]
fn main() {
use std::sync::RwLock;
fn f(value: &RwLock<Option<bool>>) {
    match *value.read().unwrap() {
        Some(x) => {
            println!("값은 {x}");
        }
        _ => {
            let mut s = value.write().unwrap();
            if s.is_none() {
                *s = Some(true);
            }
        }
    }
    // <--- 2021 및 2024 모두에서 여기에서 읽기 잠금이 해제됩니다
}
}

이 특정 예제에서는 앞서 언급한 교착 상태 때문에 이것이 원하는 바가 아닐 수 있습니다! 그러나 일부 시나리오에서는 임시 변수가 else 절을 지나서 유지된다고 가정할 수 있으며, 이 경우 이전 동작을 유지하고 싶을 수 있습니다.

if_let_rescope 린트는 자동 에디션 마이그레이션에 포함된 rust-2024-compatibility 린트 그룹의 일부입니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요.

cargo fix --edition

마이그레이션 후에는 if let 에서 match 로의 모든 변경 사항을 검토하고, 임시 값이 드롭되는 시점과 관련하여 필요한 동작이 무엇인지 결정하는 것이 권장됩니다. 변경이 불필요하다고 판단되면 변경 사항을 다시 if let 으로 되돌릴 수 있습니다.

에디션 마이그레이션을 수행하지 않고 이러한 경고를 수동으로 검사하려면 다음과 같이 린트를 활성화할 수 있습니다.

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(if_let_rescope)]
}

let chains in if and while

요약

  • Allow chaining of let expressions in the condition operand of if and while.

세부 정보

Starting with the 2024 Edition, it is now allowed to have chaining of let expressions inside if and while condition operands, where chaining refers to && chains. The let expressions still have to appear at the top level, so if (let Some(hi) = foo || let Some(hi) = bar) is not allowed.

Before 2024, the let had to appear directly after the if or while, forming a if let or while let special variant. Now, if and while allow chains of one or more let expressions, possibly mixed with expressions that are bool typed.

#![allow(unused)]
fn main() {
fn sum_first_two(nums: &[u8]) -> Option<u8> {
    let mut iter = nums.iter();
    if let Some(first) = iter.next()
        && let Some(second) = iter.next()
    {
        first.checked_add(*second)
    } else {
        None
    }
}
}

The feature is edition gated due to requiring if let rescoping, which is a Edition 2024 change.

마이그레이션

The switch to Edition 2024 doesn't neccessitate any migrations due to this feature, as it creates a true extension of the set of allowed Rust programs.

꼬리 표현식 임시 스코프

요약

  • 함수 또는 클로저 본문, 또는 블록 의 꼬리 표현식을 평가할 때 생성된 임시 값은 이제 지역 변수보다 먼저 삭제될 수 있으며, 때로는 다음으로 큰 임시 스코프로 확장되지 않을 수 있습니다.

세부 정보

2024 에디션은 꼬리 표현식의 임시 값 삭제 순서를 변경합니다. 2024 에디션 이전에는 꼬리 표현식의 임시 값이 블록 자체보다 오래 살 수 있고 지역 변수 바인딩보다 나중에 삭제된다는 사실이 종종 놀라움으로 다가왔습니다. 다음 예를 참조하세요:

#![allow(unused)]
fn main() {
// 2024년 이전
use std::cell::RefCell;
fn f() -> usize {
    let c = RefCell::new("..");
    c.borrow().len() // error[E0597]: `c` 가 충분히 오래 지속되지 않습니다
}
}

2021 에디션에서는 지역 변수 c 가 c.borrow() 에 의해 생성된 임시 값보다 먼저 드롭됩니다. 2024 에디션에서는 이를 변경하여 임시 값 c.borrow() 가 먼저 드롭되고 이어서 지역 변수 c 가 드롭되도록 하여 코드가 예상대로 컴파일되도록 합니다.

임시 스코프가 축소될 수 있습니다

When a temporary is created in order to evaluate an expression, the temporary is dropped based on the temporary scope rules. Those rules define how long the temporary will be kept alive. Before 2024, temporaries from tail expressions of a block would be extended outside the block to the next temporary scope boundary. In many cases this would be the end of a statement or function body. In 2024, the temporaries of the tail expression may now be dropped immediately at the end of the block (before any local variables in the block).

이러한 임시 스코프의 축소로 인해 2024년에는 프로그램 컴파일이 실패할 수 있습니다. 예를 들어:

// 이 예제는 2021에서는 작동하지만 2024에서는 컴파일에 실패합니다.
fn main() {
    let x = { &String::from("1234") }.len();
}

이 예제에서 2021 에디션의 경우 임시 String 은 블록 외부로 확장되어 len() 호출을 지나 문장 끝에서 드롭됩니다. 2024 에디션에서는 블록 끝에서 즉시 드롭되므로, 빌려온(borrowed) 상태에서 임시 값이 드롭된다는 컴파일 오류가 발생합니다.

이러한 상황에 대한 해결책은 임시 값이 충분히 오래 유지되도록 블록 표현식을 지역 변수로 끌어내는 것입니다.

fn main() {
    let s = { &String::from("1234") };
    let x = s.len();
}

이 특정 예제는 임시 라이프타임 연장(temporary lifetime extension) 을 활용합니다. 임시 라이프타임 연장은 임시 값이 평소보다 더 오래 유지되도록 허용하는 특정 규칙 세트입니다. String 임시 값이 참조 뒤에 있기 때문에, String 임시 값은 다음 문장에서 len() 을 호출할 수 있을 만큼 충분히 연장됩니다.

if let 표현식의 임시 스코프에 적용된 유사한 변경 사항은 if let 임시 스코프 장을 참조하세요.

마이그레이션

안타깝게도, 테일 표현식의 임시 값 라이프타임을 단축하기 위한 의미 보존형 다시 작성(semantics-preserving rewrites) 방법은 없습니다 1. tail_expr_drop_order 린트는 테일 표현식에서 사용자 정의, 사소하지 않은 Drop 소멸자가 있는 임시 값이 생성되는지 감지합니다. 이 린트의 경고는 cargo fix --edition 을 실행할 때 나타나지만, 그 외에는 자동으로 변경 사항을 만들지 않습니다. 경고를 수동으로 검사하고 조정이 필요한지 여부를 결정하는 것이 권장됩니다.

에디션 마이그레이션을 수행하지 않고 이러한 경고를 수동으로 검사하려면 다음과 같이 린트를 활성화할 수 있습니다.

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(tail_expr_drop_order)]
}

매치 인체공학 예약

요약

  • 바인딩에 mut, ref 또는 ref mut 을 작성하는 것은 해당 바인딩으로 이어지는 패턴이 완전히 명시적인 경우(즉, 매치 인간공학을 사용하지 않는 경우)에만 패턴 내에서 허용됩니다.
    • 달리 표현하면, 기본 바인딩 모드가 move 가 아닐 때 바인딩에 mut, ref 또는 ref mut 을 작성하는 것은 오류입니다.
  • 참조 패턴(& 또는 &mut)은 패턴의 완전히 명시적인 접두사 내에서만 허용됩니다.
    • 달리 표현하면, 참조 패턴은 기본 바인딩 모드가 move 일 때만 검사 대상(scrutinee)의 참조와 매칭될 수 있습니다.

세부 정보

배경

match, let 및 기타 구문 내에서 우리는 검사 대상(scrutinee) 에 대해 패턴 을 매칭합니다. 예:

#![allow(unused)]
fn main() {
let &[&mut [ref x]] = &[&mut [()]]; // x: &()
//  ~~~~~~~~~~~~~~~   ~~~~~~~~~~~~
//        패턴           검사 대상
}

이러한 패턴은 검사 대상 내의 어떤 참조도 생략(즉, "건너뛰기" 또는 "통과")하지 않기 때문에 완전히 명시적(fully explicit)이라고 불립니다. 대조적으로, 기능적으로 동일한 다음 패턴은 완전히 명시적이지 않습니다.

#![allow(unused)]
fn main() {
let [[x]] = &[&mut [()]]; // x: &()
}

이와 같은 패턴은 원래 RFC 2005 에서 도입된 매치 인간공학을 사용한다고 합니다.

매치 인간공학하에서, 패턴을 검사 대상에 대해 점진적으로 매칭함에 따라 기본 바인딩 모드를 추적합니다. 이 모드는 move, ref mut 또는 ref 중 하나일 수 있으며, move 로 시작합니다. 바인딩에 도달했을 때 명시적인 바인딩 모드가 제공되지 않으면, 기본 바인딩 모드를 사용하여 바인딩의 타입을 결정합니다.

예를 들어, 여기서 명시적인 바인딩 모드를 제공하여 x 가 참조에 의해 바인딩되도록 합니다.

#![allow(unused)]
fn main() {
let ref x = (); // &()
}

대조적으로:

#![allow(unused)]
fn main() {
let [x] = &[()]; // &()
}

여기서 패턴에서 우리는 검사 대상의 바깥쪽 공유 참조를 통과시킵니다. 이로 인해 기본 바인딩 모드가 move 에서 ref 로 전환됩니다. 명시적인 바인딩 모드가 지정되지 않았으므로, x 를 바인딩할 때 ref 바인딩 모드가 사용됩니다.

mut 제약

Rust 2021 및 이전 에디션에서는 다음과 같은 특이한 허용이 있었습니다:

#![allow(unused)]
fn main() {
let [x, mut y] = &[(), ()]; // x: &(), mut y: ()
}

여기서 패턴에 공유 참조를 전달하기 때문에 기본 바인딩 모드가 ref 로 전환됩니다. 하지만 이 에디션들에서는 바인딩에 mut 을 쓰면 기본 바인딩 모드가 move 로 재설정됩니다.

가변성이 타입에 영향을 미친다는 것은 직관적이지 않기 때문에 놀라울 수 있습니다.

이 문제를 해결할 여지를 남겨두기 위해, Rust 2024에서는 기본 바인딩 모드가 move 가 아닐 때 바인딩에 mut 을 쓰는 것이 오류가 됩니다. 즉, mut 은 패턴(해당 바인딩까지 이어지는)이 완전히 명시적일 때만 바인딩에 쓸 수 있습니다.

Rust 2024에서는 위 예제를 다음과 같이 쓸 수 있습니다:

#![allow(unused)]
fn main() {
let &[ref x, mut y] = &[(), ()]; // x: &(), mut y: ()
}

ref / ref mut 제약

Rust 2021 및 이전 에디션에서는 다음을 허용합니다:

#![allow(unused)]
fn main() {
let [ref x] = &[()]; // x: &()
}

여기서 ref 명시적 바인딩 모드는 불필요합니다. 공유 참조를 전달함으로써(즉, 패턴에서 언급하지 않음으로써) 바인딩 모드가 ref 로 전환되기 때문입니다.

다른 언어 가능성을 위한 공간을 남겨두기 위해, Rust 2024에서는 중복되는 명시적 바인딩 모드를 허용하지 않습니다. 위 예제는 다음과 같이 간단하게 다시 쓸 수 있습니다:

#![allow(unused)]
fn main() {
let [x] = &[()]; // x: &()
}

참조 패턴 제약

Rust 2021 및 이전 에디션에서는 다음과 같은 특이한 허용이 있었습니다:

#![allow(unused)]
fn main() {
let [&x, y] = &[&(), &()]; // x: (), y: &&()
}

여기서 패턴의 & 는 &() 의 참조와 매칭되면서 동시에 기본 바인딩 모드를 move 로 재설정합니다. 패턴의 단일 & 가 참조의 두 레이어를 모두 제거함으로써 예상보다 큰 타입 변화를 일으키기 때문에 놀라울 수 있습니다.

이 문제를 해결할 여지를 남겨두기 위해, Rust 2024에서는 기본 바인딩 모드가 move 가 아닐 때 패턴에 & 나 &mut 을 쓰는 것이 오류가 됩니다. 즉, & 나 &mut 은 패턴(해당 지점까지 이어지는)이 완전히 명시적일 때만 쓸 수 있습니다.

Rust 2024에서는 위 예제를 다음과 같이 쓸 수 있습니다:

#![allow(unused)]
fn main() {
let &[&x, ref y] = &[&(), &()];
}

마이그레이션

rust_2024_incompatible_pat 린트는 Rust 2024에서 허용되지 않는 패턴을 표시합니다. 이 린트는 rust-2024-compatibility 린트 그룹의 일부이며 cargo fix --edition 을 실행할 때 자동으로 적용됩니다. 이 린트는 영향을 받는 패턴을 Rust 2024 및 모든 이전 에디션에서 올바르게 작동하는 완전히 명시적인 패턴으로 자동 변환합니다.

코드를 Rust 2024와 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

예를 들어, 이것은 다음을...

#![allow(unused)]
fn main() {
let [x, mut y] = &[(), ()];
let [ref x] = &[()];
let [&x, y] = &[&(), &()];
}

...다음과 같이 변환합니다:

#![allow(unused)]
fn main() {
let &[ref x, mut y] = &[(), ()];
let &[ref x] = &[()];
let &[&x, ref y] = &[&(), &()];
}

대안으로, 마이그레이션이 필요한 패턴을 찾기 위해 수동으로 린트를 활성화할 수 있습니다:

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(rust_2024_incompatible_pat)]
}

안전하지 않은 extern 블록

요약

  • extern 블록 은 이제 unsafe 키워드로 표시되어야 합니다.

세부 정보

Rust 1.82는 모든 에디션에서 extern 블록 을 unsafe 키워드로 표시할 수 있는 기능을 추가했습니다.1 unsafe 키워드를 추가하는 것은 시그니처가 올바른지 확인하는 것이 extern 블록 작성자의 책임임을 강조하는 데 도움이 됩니다. 시그니처가 올바르지 않으면 정의되지 않은 동작이 발생할 수 있습니다.

unsafe extern 블록의 문법은 다음과 같습니다:

#![allow(unused)]
fn main() {
unsafe extern "C" {
    // sqrt (libm에서 가져옴)는 어떤 `f64` 로도 호출될 수 있음
    pub safe fn sqrt(x: f64) -> f64;

    // strlen (libc에서 가져옴)은 유효한 포인터를 요구하므로,
    // unsafe fn으로 표시함
    pub unsafe fn strlen(p: *const std::ffi::c_char) -> usize;

    // 이 함수는 safe나 unsafe를 명시하지 않았으므로, 기본적으로 unsafe임
    pub fn free(p: *mut core::ffi::c_void);

    pub safe static IMPORTANT_BYTES: [u8; 256];
}
}

extern 블록을 unsafe 로 표시하는 것 외에도, extern 블록 내의 개별 항목이 safe 인지 unsafe 인지 지정할 수도 있습니다. safe 로 표시된 항목은 unsafe 블록 없이 사용할 수 있습니다.

2024 에디션부터는 extern 블록에 unsafe 키워드를 포함하는 것이 필수입니다. 이는 extern 정의가 준수해야 할 안전 요구 사항이 있음을 매우 명확하게 하기 위함입니다.

1

원래 제안은 RFC 3484 를 참조하세요.

마이그레이션

missing_unsafe_on_extern 린트는 extern 블록을 업데이트하여 unsafe 키워드를 추가할 수 있습니다. 이 린트는 자동 에디션 마이그레이션에 포함된 rust-2024-compatibility 린트 그룹의 일부입니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

단, 이 자동 마이그레이션은 extern 블록 내의 시그니처가 올바른지 확인할 수 없으므로 주의하세요. 정의를 수동으로 검토하는 것은 여전히 여러분의 책임입니다.

대안으로, 업데이트가 필요한 unsafe 블록이 있는 곳을 찾기 위해 수동으로 린트를 활성화할 수 있습니다.

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(missing_unsafe_on_extern)]
}

안전하지 않은 속성

요약

  • 다음 속성들은 이제 unsafe 로 표시되어야 합니다:
    • export_name
    • link_section
    • no_mangle

세부 정보

Rust 1.82는 모든 에디션에서 특정 속성을 unsafe 로 표시하여 준수해야 할 건전성 요구 사항이 있음을 나타내는 기능을 추가했습니다.1 unsafe 속성의 문법은 다음과 같습니다:

#![allow(unused)]
fn main() {
// SAFETY: 이 이름을 가진 다른 전역 함수는 없음
#[unsafe(no_mangle)]
pub fn example() {}
}

속성을 unsafe 로 표시하면 컴파일러가 자체적으로 확인할 수 없는 안전 요구 사항을 준수해야 함을 강조하게 됩니다.

2024 에디션부터는 이러한 속성을 unsafe 로 표시해야 합니다. 다음 섹션에서는 이러한 속성에 대한 안전 요구 사항을 설명합니다.

1

원래 제안은 RFC 3325 를 참조하세요.

안전 요구 사항

no_mangle, export_name, link_section 속성은 항목의 심볼 이름과 링킹 동작에 영향을 줍니다. 이 속성들을 올바르게 사용하도록 주의해야 합니다.

연결된 모든 라이브러리의 심볼 집합은 전역 네임스페이스이므로 라이브러리 간에 심볼 이름 충돌이 발생하면 문제가 생길 수 있습니다. 일반적으로 심볼 맹글링 이 심볼 이름의 고유성을 보장하는 데 도움이 되기 때문에 일반적인 함수 정의에서는 문제가 되지 않습니다. 하지만 export_name 과 같은 속성은 그러한 고유성 가정을 깨뜨릴 수 있습니다.

예를 들어, 이전 에디션에서는 안전한 코드만 포함되어 있음에도 불구하고 다음 코드가 대부분의 유닉스 계열 플랫폼에서 충돌을 일으킵니다:

fn main() {
    println!("Hello, world!");
}

#[export_name = "malloc"]
fn foo() -> usize { 1 }

2024 에디션에서는 심볼이 올바르게 정의되었는지 확인해야 함을 강조하기 위해 이러한 속성을 unsafe로 표시하는 것이 필수입니다:

#![allow(unused)]
fn main() {
// SAFETY: loop 심볼의 정의는 하나만 있어야 함.
#[unsafe(export_name="loop")]
fn arduino_loop() {
    // ...
}
}

마이그레이션

unsafe_attr_outside_unsafe 린트는 이러한 속성이 unsafe(...) 형식을 사용하도록 업데이트할 수 있습니다. 이 린트는 자동 에디션 마이그레이션에 포함된 rust-2024-compatibility 린트 그룹의 일부입니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

단, 이 자동 마이그레이션은 이 속성들이 올바르게 사용되고 있는지 확인할 수 없으므로 주의하세요. 사용법을 수동으로 검토하는 것은 여전히 여러분의 책임입니다.

대안으로, 이 속성들의 업데이트가 필요한 곳을 찾기 위해 수동으로 린트를 활성화할 수 있습니다.

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(unsafe_attr_outside_unsafe)]
}

unsafe_op_in_unsafe_fn 경고

요약

  • unsafe_op_in_unsafe_fn 린트가 이제 기본적으로 경고를 발생시킵니다. 이 경고는 unsafe 함수 내에서 명시적인 unsafe 블록 없이 수행되는 unsafe 연산 호출을 감지합니다.

세부 정보

unsafe_op_in_unsafe_fn 린트는 unsafe 함수 내에 명시적인 unsafe {} 블록 없이 unsafe 연산 이 있을 경우 발생합니다.

#![allow(unused)]
fn main() {
#![warn(unsafe_op_in_unsafe_fn)]
unsafe fn get_unchecked<T>(x: &[T], i: usize) -> &T {
  x.get_unchecked(i) // WARNING: unsafe 블록이 필요함
}
}

해결 방법은 모든 unsafe 연산을 unsafe 블록으로 감싸는 것입니다:

#![allow(unused)]
fn main() {
#![deny(unsafe_op_in_unsafe_fn)]
unsafe fn get_unchecked<T>(x: &[T], i: usize) -> &T {
  unsafe { x.get_unchecked(i) }
}
}

이 변경 사항은 unsafe 함수 내에서 우발적으로 unsafe 연산을 사용하는 것을 방지하기 위한 것입니다. unsafe 함수 키워드는 두 가지 역할을 수행하고 있었습니다. 하나는 함수 호출 에 unsafe가 필요하며 호출자가 추가적인 안전 요구 사항을 준수할 책임이 있음을 선언하는 것이었습니다. 다른 하나는 함수 내부에서 unsafe 연산 사용을 허용하는 것이었습니다. 이 두 번째 역할은 명시적인 unsafe 블록 없이는 너무 위험하다고 판단되었습니다.

더 자세한 정보와 동기는 RFC #2585 에서 확인할 수 있습니다.

마이그레이션

unsafe_op_in_unsafe_fn 린트는 rust-2024-compatibility 린트 그룹의 일부입니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

대안으로, unsafe 블록 추가가 필요한 곳을 찾기 위해 수동으로 린트를 활성화하거나, 린트를 완전히 끄기 위해 allow 로 전환할 수 있습니다.

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(unsafe_op_in_unsafe_fn)]
}

static mut 에 대한 참조 금지

요약

  • static_mut_refs 린트 레벨이 이제 기본적으로 deny 입니다. 이는 static mut 에 대한 공유 또는 가변 참조를 가져오는 것을 검사합니다.

세부 정보

static_mut_refs 린트는 static mut 에 대한 참조를 가져오는 것을 감지합니다. 2024 에디션에서는 이러한 참조를 피해야 함을 강조하기 위해 이 린트가 기본적으로 deny 로 설정됩니다.

#![allow(unused)]
fn main() {
static mut X: i32 = 23;
static mut Y: i32 = 24;

unsafe {
    let y = &X;             // ERROR: 가변 static에 대한 공유 참조
    let ref x = X;          // ERROR: 가변 static에 대한 공유 참조
    let (x, y) = (&X, &Y);  // ERROR: 가변 static에 대한 공유 참조
}
}

Rust의 가변성 XOR 앨리어싱 요구 사항을 위반하여 이러한 참조를 가져오는 것은 해당 참조를 읽거나 쓰지 않더라도 항상 즉각적인 정의되지 않은 동작 이었습니다. 게다가 static mut 에 대한 가변성 XOR 앨리어싱을 준수하려면 코드를 전역적으로 추론 해야 하는데, 이는 재진입성이나 멀티스레딩 상황에서 특히 어려울 수 있습니다.

보이는 & 연산자 없이 암시적 참조가 자동으로 생성되는 경우가 있다는 점에 유의하세요. 예를 들어, 다음과 같은 상황에서도 린트가 발생합니다:

#![allow(unused)]
fn main() {
static mut NUMS: &[u8; 3] = &[0, 1, 2];

unsafe {
    println!("{NUMS:?}");   // ERROR: 가변 static에 대한 공유 참조
    let n = NUMS.len();     // ERROR: 가변 static에 대한 공유 참조
}
}

대안

가능하다면, Rust의 가변성 XOR 앨리어싱 요구 사항 준수를 보장하는 복잡성을 크게 줄여주는 지역적으로 추론된 추상화 뒤에서 내부 가변성 을 제공하는 타입의 불변 static 을 대신 사용하는 것이 강력히 권장됩니다.

지역적으로 추론된 추상화가 불가능하여 어쩔 수 없이 static 변수에 대한 접근을 전역적으로 추론해야 하는 상황이라면, 이제 &raw const 또는 &raw mut 연산자 를 통해 얻을 수 있는 원시 포인터를 사용해야 합니다. 참조를 직접 가져오는 대신 먼저 원시 포인터를 얻음으로써, 해당 포인터를 통한 접근(의 안전 요구 사항)은 unsafe 개발자에게 더 친숙할 것이며 더 작은 코드 영역으로 제한하거나 미룰 수 있습니다.

다음 예제들은 단지 설명일 뿐이며 완전한 구현을 의도한 것이 아닙니다. 있는 그대로 복사하지 마세요. 여러분의 특정 상황에 맞춰 변경해야 할 세부 사항이 있을 수 있습니다. 이 예제들은 문제에 접근하는 다양한 방법을 보여주기 위한 것입니다.

표준 라이브러리의 특정 타입에 대한 문서, 정의되지 않은 동작 에 대한 참조, Rustonomicon 을 읽어보는 것을 권장하며, 질문이 있다면 Users Forum 과 같은 Rust 포럼에 문의하세요.

전역 변수 사용하지 않기

이미 알고 계시겠지만, 가능하다면 가변 전역 상태는 피하는 것이 가장 좋습니다. 물론 때로는 조금 더 어색하거나 어려울 수 있으며, 특히 많은 함수 사이에서 가변 참조를 전달해야 할 때 그렇습니다.

아토믹 (Atomics)

아토믹 타입 은 static 에서 (mut 없이) 사용할 수 있는 정수, 포인터, 불리언을 제공합니다.

use std::sync::atomic::Ordering;
use std::sync::atomic::AtomicU64;

// 이렇게 변경:
//   static mut COUNTER: u64 = 0;
// 이것으로:
static COUNTER: AtomicU64 = AtomicU64::new(0);

fn main() {
    // 사용 사례를 분석하여 올바른 Ordering을 사용하도록 하세요.
    COUNTER.fetch_add(1, Ordering::Relaxed);
}

Mutex 또는 RwLock

타입이 아토믹보다 더 복잡하다면, Mutex 나 RwLock 을 사용하여 전역 값에 대한 올바른 접근을 보장하는 것을 고려하세요.

use std::sync::Mutex;
use std::collections::VecDeque;

// 이렇게 변경:
//     static mut QUEUE: VecDeque<String> = VecDeque::new();
// 이것으로:
static QUEUE: Mutex<VecDeque<String>> = Mutex::new(VecDeque::new());

fn main() {
    QUEUE.lock().unwrap().push_back(String::from("abc"));
    let first = QUEUE.lock().unwrap().pop_front();
}

OnceLock 또는 LazyLock

const 로 할 수 없는 일회성 초기화를 위해 static mut 을 사용하고 있다면, 대신 OnceLock 이나 LazyLock 을 사용할 수 있습니다.

use std::sync::LazyLock;

struct GlobalState;

impl GlobalState {
    fn new() -> GlobalState {
        GlobalState
    }
    fn example(&self) {}
}

// 다음과 같은 임시 또는 초기화되지 않은 타입 대신:
//     static mut STATE: Option<GlobalState> = None;
// 대신 이것을 사용하세요:
static STATE: LazyLock<GlobalState> = LazyLock::new(|| {
    GlobalState::new()
});

fn main() {
    STATE.example();
}

OnceLock 은 LazyLock 과 비슷하지만, 생성자에 정보를 전달해야 하는 경우 사용할 수 있습니다. 이는 (main 과 같은) 단일 초기화 지점과 잘 작동하거나, 전역 변수에 접근하는 곳 어디에서나 입력을 사용할 수 있는 경우에 유용합니다.

use std::sync::OnceLock;

struct GlobalState;

impl GlobalState {
    fn new(verbose: bool) -> GlobalState {
        GlobalState
    }
    fn example(&self) {}
}

struct Args {
    verbose: bool
}
fn parse_arguments() -> Args {
    Args { verbose: true }
}

static STATE: OnceLock<GlobalState> = OnceLock::new();

fn main() {
    let args = parse_arguments();
    let state = GlobalState::new(args.verbose);
    let _ = STATE.set(state);
    // ...
    STATE.get().unwrap().example();
}

no_std 일회성 초기화

이 예제는 전역 변수의 일회성 초기화를 제공한다는 점에서 OnceLock 과 유사하지만, std 를 필요로 하지 않아 no_std 환경에서 유용합니다. 타겟이 아토믹을 지원한다고 가정하면, 아토믹을 사용하여 전역 변수의 초기화 여부를 확인할 수 있습니다. 패턴은 다음과 비슷할 것입니다:

use core::sync::atomic::AtomicUsize;
use core::sync::atomic::Ordering;

struct Args {
    verbose: bool,
}
fn parse_arguments() -> Args {
    Args { verbose: true }
}

struct GlobalState {
    verbose: bool,
}

impl GlobalState {
    const fn default() -> GlobalState {
        GlobalState { verbose: false }
    }
    fn new(verbose: bool) -> GlobalState {
        GlobalState { verbose }
    }
    fn example(&self) {}
}

const UNINITIALIZED: usize = 0;
const INITIALIZING: usize = 1;
const INITIALIZED: usize = 2;

static STATE_INITIALIZED: AtomicUsize = AtomicUsize::new(UNINITIALIZED);
static mut STATE: GlobalState = GlobalState::default();

fn set_global_state(state: GlobalState) {
    if STATE_INITIALIZED
        .compare_exchange(
            UNINITIALIZED,
            INITIALIZING,
            Ordering::SeqCst,
            Ordering::SeqCst,
        )
        .is_ok()
    {
        // SAFETY: STATE에 대한 읽기 및 쓰기는 INITIALIZED 가드로 보호됩니다.
        unsafe {
            STATE = state;
        }
        STATE_INITIALIZED.store(INITIALIZED, Ordering::SeqCst);
    } else {
        panic!("이미 초기화되었거나 동시에 초기화 중");
    }
}

fn get_state() -> &'static GlobalState {
    if STATE_INITIALIZED.load(Ordering::Acquire) != INITIALIZED {
        panic!("초기화되지 않음");
    } else {
        // SAFETY: 상태가 초기화된 후에는 가변 접근이 불가능합니다.
        unsafe { &*&raw const STATE }
    }
}

fn main() {
    let args = parse_arguments();
    let state = GlobalState::new(args.verbose);
    set_global_state(state);
    // ...
    let state = get_state();
    state.example();
}

이 예제는 초기화 전에 static에 기본값을 넣을 수 있다고 가정합니다(이 예제에서는 const default 생성자). 불가능한 경우, MaybeUninit 을 사용하거나, 동적 트레이트 디스패치(트레이트를 구현하는 더미 타입 사용)를 사용하거나, 기본 자리 표시자를 갖는 다른 접근 방식을 고려해보세요.

비슷한 일회성 초기화를 제공하는 커뮤니티 제공 크레이트들이 있습니다. 예를 들어 static-cell 크레이트(portable-atomic 을 사용하여 아토믹이 없는 타겟도 지원함)가 있습니다.

원시 포인터

일부 경우에는 static mut 을 계속 사용할 수 있지만, 참조 생성은 피해야 합니다. 예를 들어, 원시 포인터 를 C 라이브러리에 전달하기만 하면 되는 경우 중간 참조를 생성하지 마세요. 대신 다음 예제처럼 원시 대여 연산자 를 사용할 수 있습니다:

#[repr(C)]
struct GlobalState {
    value: i32
}

impl GlobalState {
    const fn new() -> GlobalState {
        GlobalState { value: 0 }
    }
}

static mut STATE: GlobalState = GlobalState::new();

unsafe extern "C" {
    fn example_ffi(state: *mut GlobalState);
}

fn main() {
    unsafe {
        // 이렇게 변경:
        //     example_ffi(&mut STATE as *mut GlobalState);
        // 이것으로:
        example_ffi(&raw mut STATE);
    }
}

단, 가변 포인터에 대한 앨리어싱 제약 조건은 여전히 준수해야 합니다. 이를 위해서는 내부 또는 외부 동기화가 필요하거나, 스레드, 인터럽트 핸들러, 재진입성에 걸쳐 어떻게 사용되는지에 대한 증명이 필요할 수 있습니다.

Sync 와 함께 UnsafeCell 사용

UnsafeCell 은 Sync 를 구현하지 않으므로 static 에서 사용할 수 없습니다. UnsafeCell 을 감싸는 래퍼를 만들어 Sync 구현을 추가하면 static 에서 사용하여 내부 가변성을 구현할 수 있습니다. 이 접근 방식은 외부 락이나 가변 포인터에 필요한 안전 불변성을 유지하는 다른 보장이 있는 경우 유용할 수 있습니다.

이것은 원시 포인터 예제와 거의 동일하다는 점에 유의하세요. 래퍼는 타입을 어떻게 사용하는지 강조하고, 어떤 안전 요구 사항을 주의해야 하는지에 집중하는 데 도움을 줍니다. 하지만 그 외에는 거의 동일합니다.

#![allow(unused)]
fn main() {
use std::cell::UnsafeCell;

fn with_interrupts_disabled<T: Fn()>(f: T) {
    // 실제 예제에서는 인터럽트를 비활성화할 것입니다.
    f();
}

#[repr(C)]
struct GlobalState {
    value: i32,
}

impl GlobalState {
    const fn new() -> GlobalState {
        GlobalState { value: 0 }
    }
}

#[repr(transparent)]
pub struct SyncUnsafeCell<T>(UnsafeCell<T>);

unsafe impl<T: Sync> Sync for SyncUnsafeCell<T> {}

static STATE: SyncUnsafeCell<GlobalState> = SyncUnsafeCell(UnsafeCell::new(GlobalState::new()));

fn set_value(value: i32) {
    with_interrupts_disabled(|| {
        let state = STATE.0.get();
        unsafe {
            // SAFETY: 이 값은 인터럽트 핸들러에서만 읽히며,
            // 인터럽트는 비활성화되어 있고, 하나의 스레드에서만 사용합니다.
            (*state).value = value;
        }
    });
}
}

표준 라이브러리에는 UnsafeCell 의 나이틀리 전용(불안정) 변형인 SyncUnsafeCell 이 있습니다. 위 예제는 표준 라이브러리 타입의 매우 단순화된 버전을 보여주지만, 대략 같은 방식으로 사용될 것입니다. 더 나은 격리를 제공할 수 있으므로 자세한 내용은 해당 구현을 확인해 보세요.

이 예제에는 임베디드 환경에서 볼 수 있는 가상의 with_interrupts_disabled 함수가 포함되어 있습니다. 예를 들어, critical-section 크레이트는 임베디드 환경에서 사용할 수 있는 유사한 기능을 제공합니다.

안전한 참조

어떤 경우에는 static mut 의 참조를 생성하는 것이 안전할 수 있습니다. static_mut_refs 린트의 핵심은 이것을 올바르게 수행하기가 매우 어렵다는 것입니다! 하지만 불가능 하다는 말은 아닙니다. static이 좁은 범위(작은 모듈이나 함수 내에서만 사용됨)로 지정되어 있거나, 내부 또는 외부 동기화가 있거나, 인터럽트 핸들러와 재진입성, 패닉 안전성, 드롭 핸들러 등을 고려하는 등 앨리어싱 요구 사항이 준수됨을 보장할 수 있는 상황이라면 참조를 가져오는 것도 괜찮을 수 있습니다.

이를 위해 두 가지 접근 방식을 취할 수 있습니다. static_mut_refs 린트를 허용하거나(가급적 좁은 범위에서), &mut *&raw mut MY_STATIC 과 같이 원시 포인터를 참조로 변환할 수 있습니다.

수명이 짧은 참조

static mut 에 대한 참조를 생성해야 한다면, 그 참조가 존재하는 범위를 최소화하는 것이 좋습니다. 참조를 어딘가에 숨겨두거나 코드의 큰 섹션 동안 계속 살아있게 하지 마세요. 수명을 짧게 유지하면 감사가 쉬워지고 해당 기간 동안 배타적 접근이 유지되는지 확인하는 데 도움이 됩니다. 포인터를 기본 단위로 사용하고, 절대적으로 필요한 경우에만 요청 시 포인터를 참조로 변환하세요.

마이그레이션

static mut 에 대한 이러한 참조를 수정하는 자동 마이그레이션은 없습니다. 정의되지 않은 동작을 피하려면 대안 섹션에서 권장하는 다른 접근 방식을 사용하도록 코드를 다시 작성해야 합니다.

Never 타입 폴백 변경

요약

  • Never 타입(!)에서 임의의 타입으로의("never-to-any") 강제 변환은 이제 유닛 타입(())이 아닌 never 타입(!)으로 폴백됩니다.
  • The never_type_fallback_flowing_into_unsafe lint is now deny by default.

세부 정보

컴파일러가 강제 변환 위치 에서 ! (never) 타입의 값을 보면, 타입 검사기가 어떤 타입이든 추론할 수 있도록 암시적으로 강제 변환을 삽입합니다:

#![allow(unused)]
fn main() {
#![feature(never_type)]
// 이것은:
let x: u8 = panic!();

// ...컴파일러에 의해 (본질적으로) 다음과 같이 변환됩니다:
let x: u8 = absurd(panic!());

// ...여기서 `absurd` 는 다음 함수입니다
// (`!` 는 항상 도달할 수 없는 코드를 표시하므로 안전합니다):
fn absurd<T>(x: !) -> T { x }
}

타입을 추론할 수 없는 경우 컴파일 오류가 발생할 수 있습니다:

#![allow(unused)]
fn main() {
#![feature(never_type)]
fn absurd<T>(x: !) -> T { x }
// 이것은:
{ panic!() };

// ...다음과 같이 변환됩니다:
{ absurd(panic!()) }; //~ ERROR `absurd` 의 타입을 추론할 수 없음
}

이러한 오류를 방지하기 위해 컴파일러는 absurd 호출을 삽입한 위치를 기억하고, 타입을 추론할 수 없는 경우 대신 폴백 타입을 사용합니다:

#![allow(unused)]
fn main() {
#![feature(never_type)]
fn absurd<T>(x: !) -> T { x }
type Fallback = /* 임의로 선택된 타입! */ !;
{ absurd::<Fallback>(panic!()) }
}

이것이 바로 "never 타입 폴백"이라고 알려진 것입니다.

역사적으로 폴백 타입은 () (유닛)이었습니다. 이로 인해 컴파일러가 폴백 없이는 () 를 추론하지 않을 때조차 ! 가 () 로 저절로 강제 변환되는 일이 발생했습니다. 이는 혼란스러웠으며 ! 타입의 안정화를 막아왔습니다.

2024 에디션에서는 폴백 타입이 이제 ! 입니다. (추후 모든 에디션에 걸쳐 이 변경을 적용할 계획입니다.) 이렇게 하면 더 직관적으로 작동합니다. 이제 ! 를 전달할 때 다른 것으로 강제 변환할 이유가 없다면 ! 로 유지됩니다.

어떤 경우에는 코드가 폴백 타입이 () 인 것에 의존할 수 있으므로, 이로 인해 컴파일 오류나 동작 변경이 발생할 수 있습니다.

never_type_fallback_flowing_into_unsafe

The default level of the never_type_fallback_flowing_into_unsafe lint has been raised from warn to deny in the 2024 Edition. This lint helps detect a particular interaction with the fallback to ! and unsafe code which may lead to undefined behavior. See the link for a complete description.

마이그레이션

자동 수정 기능은 없지만, 에디션 변경으로 인해 깨질 코드를 자동으로 감지하는 기능은 있습니다. 이전 에디션을 사용하는 동안 코드가 깨질 경우 경고가 표시됩니다.

해결 방법은 폴백 타입이 사용되지 않도록 타입을 명시적으로 지정하는 것입니다. 불행히도 어떤 타입을 지정해야 하는지 파악하는 것이 쉽지 않을 수 있습니다.

이 변경으로 인해 깨지는 가장 일반적인 패턴 중 하나는 f()?; 를 사용하는 경우인데, 여기서 f 는 반환 타입의 Ok 부분에 대해 제네릭입니다:

#![allow(unused)]
fn main() {
#![allow(dependency_on_unit_never_type_fallback)]
fn outer<T>(x: T) -> Result<T, ()> {
fn f<T: Default>() -> Result<T, ()> {
    Ok(T::default())
}

f()?;
Ok(x)
}
}

이 예제에서 타입 T 를 추론할 수 없다고 생각할 수 있습니다. 하지만 현재 ? 연산자의 탈탕(desugaring) 방식 때문에 이전에는 () 로 추론되었으나, 이제는 ! 로 추론됩니다.

이 문제를 해결하려면 T 타입을 명시적으로 지정해야 합니다:

#![allow(unused)]
fn main() {
#![deny(dependency_on_unit_never_type_fallback)]
fn outer<T>(x: T) -> Result<T, ()> {
fn f<T: Default>() -> Result<T, ()> {
    Ok(T::default())
}
f::<()>()?;
// ...또는:
() = f()?;
Ok(x)
}
}

비교적 흔한 또 다른 케이스는 클로저 내에서의 패닉입니다:

#![allow(unused)]
fn main() {
#![allow(dependency_on_unit_never_type_fallback)]
trait Unit {}
impl Unit for () {}

fn run<R: Unit>(f: impl FnOnce() -> R) {
    f();
}

run(|| panic!());
}

이전에는 panic! 의 ! 가 Unit 을 구현하는 () 로 강제 변환되었습니다. 하지만 이제는 ! 가 ! 로 유지되므로 ! 가 Unit 을 구현하지 않아 이 코드는 실패합니다. 이 문제를 해결하려면 클로저의 반환 타입을 지정할 수 있습니다:

#![allow(unused)]
fn main() {
#![deny(dependency_on_unit_never_type_fallback)]
trait Unit {}
impl Unit for () {}

fn run<R: Unit>(f: impl FnOnce() -> R) {
    f();
}
run(|| -> () { panic!() });
}

f()? 와 유사한 사례로, 한 분기에서는 ! 타입 표현식을 사용하고 다른 분기에서는 반환 타입이 제한되지 않은 함수를 사용할 때 볼 수 있습니다:

#![allow(unused)]
fn main() {
#![allow(dependency_on_unit_never_type_fallback)]
if true {
    Default::default()
} else {
    return
};
}

이전에는 return 의 ! 가 () 로 잘못 강제 변환되었기 때문에 Default::default() 의 반환 타입으로 () 가 추론되었습니다. 이제는 대신 ! 가 추론되어 ! 가 Default 를 구현하지 않으므로 이 코드는 컴파일되지 않습니다.

마찬가지로, 타입을 명시적으로 지정하여 해결할 수 있습니다:

#![allow(unused)]
fn main() {
#![deny(dependency_on_unit_never_type_fallback)]
() = if true {
    Default::default()
} else {
    return
};

// ...또는:

if true {
    <() as Default>::default()
} else {
    return
};
}

매크로 조각 지정자

요약

  • expr 조각 지정자 는 이제 const 및 _ 표현식도 지원합니다.
  • 하위 호환성을 위해 expr_2021 조각 지정자가 추가되었습니다.

세부 정보

Rust에 새로운 문법이 추가됨에 따라, 기존 macro_rules 조각 지정자는 하위 호환성을 유지하기 위해 때로는 새 문법과 매칭되는 것이 허용되지 않습니다. 이전 조각 지정자에서 새 문법을 지원하는 것은 다음 에디션까지 미뤄지기도 하며, 이때가 이를 업데이트할 기회를 제공합니다.

실제로 1.79에 추가된 const 표현식 과 1.59에 추가된 _ 표현식 에서 이런 일이 발생했습니다. 2021 에디션 및 그 이전 버전에서 expr 조각 지정자는 이러한 표현식과 매칭되지 않습니다. 이는 다음과 같은 시나리오가 있을 수 있기 때문입니다:

macro_rules! example {
    ($e:expr) => { println!("첫 번째 규칙"); };
    (const $e:expr) => { println!("두 번째 규칙"); };
}

fn main() {
    example!(const { 1 + 1 });
}

여기서, 2021 에디션에서는 매크로가 두 번째 규칙과 매칭됩니다. 만약 이전 에디션에서 expr 이 새로 도입된 const 표현식과 매칭되도록 변경되었다면, 첫 번째 규칙과 매칭되었을 것이며, 이는 파괴적인 변경이 되었을 것입니다.

2024 에디션에서는 expr 지정자가 이제 const 및 _ 표현식과도 매칭됩니다. 이전 동작을 지원하기 위해, 새로운 표현식과 매칭되지 않는 expr_2021 조각 지정자가 추가되었습니다.

마이그레이션

edition_2024_expr_fragment_specifier 린트는 기존 매크로의 동작이 변경되지 않도록 expr 지정자의 모든 사용을 expr_2021 로 변경합니다. 이 린트는 자동 에디션 마이그레이션에 포함된 rust-2024-compatibility 린트 그룹의 일부입니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

대부분의 경우, 새로운 표현식을 지원하기 위해 대신 expr 지정자를 유지하고 싶을 것입니다. 매크로를 검토하여 const 또는 _ 와 매칭될 수 있는 다른 규칙이 있는지 확인하고 충돌이 있는지 판단해야 합니다. 새로운 동작을 원한다면 린트에 의해 변경된 내용을 되돌리면 됩니다.

대안으로, expr 지정자를 업데이트해야 할 수도 있는 매크로를 찾기 위해 수동으로 린트를 활성화할 수 있습니다.

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(edition_2024_expr_fragment_specifier)]
}

누락된 매크로 프래그먼트 지정자

NOTE: This was originally made a hard error only for the 2024 Edition. In Rust 1.89, released after Rust 2024, the lint was made into a hard error in all editions.

요약

  • The missing_fragment_specifier lint is now a hard error.

세부 정보

The missing_fragment_specifier lint detected a situation when an unused pattern in a macro_rules! macro definition had a meta-variable (e.g. $e) that was not followed by a fragment specifier (e.g. :expr). This was made into a hard error in the 2024 Edition.

macro_rules! foo {
   () => {};
   ($name) => { }; // ERROR: 조각 지정자가 누락됨
}

fn main() {
   foo!();
}

Calling the macro with arguments that would match a rule with a missing specifier (e.g., foo!($name)) was a hard error in all editions. However, simply defining a macro with missing fragment specifiers was not, though we did add a lint in Rust 1.17.

마이그레이션

To migrate your code to the 2024 Edition, remove the unused matcher rule from the macro.

There is no automatic migration for this change. We expect that this style of macro is extremely rare. The lint was a future-incompatibility lint since Rust 1.17, a deny-by-default lint since Rust 1.20, since Rust 1.82 it warned about dependencies using this pattern, and in Rust 1.89 it became a hard error.

gen 키워드

요약

  • gen 은 예약어 입니다.

세부 정보

gen 키워드는 Rust의 향후 릴리스에서 "gen 블록"을 도입하기 위해 RFC #3513 의 일부로 예약되었습니다. gen 블록은 특정 종류의 반복자를 더 쉽게 작성할 수 있는 방법을 제공할 것입니다. 지금 키워드를 예약하면 다음 에디션 전에 gen 블록을 안정화하기가 더 쉬워질 것입니다.

마이그레이션

gen 키워드를 도입하면 이미 gen 이라는 이름을 가진 식별자에 문제가 발생할 수 있습니다. 예를 들어, gen 이라는 이름의 변수나 함수는 새로운 키워드와 충돌합니다. 이를 극복하기 위해 Rust는 식별자가 키워드와 중복되는 것을 허용하는 원시 식별자(raw identifier) 를 위한 r# 접두사를 지원합니다.

keyword_idents_2024 린트는 코드가 두 에디션 모두에서 계속 작동하도록 gen 이라는 이름의 식별자를 r#gen 으로 자동 수정합니다. 이 린트는 rust-2024-compatibility 린트 그룹의 일부이며 cargo fix --edition 을 실행할 때 자동으로 적용됩니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

예를 들어, 이것은 다음과 같이 변경됩니다:

fn gen() {
    println!("generating!");
}

fn main() {
    gen();
}

변경 후:

fn r#gen() {
    println!("generating!");
}

fn main() {
    r#gen();
}

대안으로, gen 식별자를 r#gen 으로 수정해야 하는 곳을 찾기 위해 수동으로 린트를 활성화할 수 있습니다:

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(keyword_idents_2024)]
}

예약된 구문

요약

  • #"foo"# 형식의 접두사 없는 보호된 문자열(guarded strings)은 미래의 사용을 위해 예약되었습니다.
  • 두 개 이상의 # 문자는 미래의 사용을 위해 예약되었습니다.

세부 정보

RFC 3593 은 향후 가능한 언어 변경을 위한 공간을 확보하기 위해 2024 에디션에서 접두사가 없는 보호된 문자열 리터럴에 대한 구문을 예약했습니다. 2021 에디션은 ident##"foo"## 와 같이 접두사가 있는 보호된 문자열에 대한 구문을 예약 했습니다. 2024 에디션은 이를 확장하여 ident 접두사가 없는 문자열도 예약합니다.

두 가지 예약된 구문이 있습니다:

  • 하나 이상의 # 문자가 문자열 리터럴 바로 뒤에 오는 경우.
  • 두 개 이상의 # 문자가 (공백 없이) 연속으로 오는 경우.

이 예약은 토큰화 및 매크로와의 상호작용으로 인해 에디션 경계를 넘어 수행됩니다. 예를 들어, 다음 매크로를 고려해보세요:

#![allow(unused)]
fn main() {
macro_rules! demo {
    ( $a:tt ) => { println!("하나의 토큰") };
    ( $a:tt $b:tt $c:tt ) => { println!("세 개의 토큰") };
}

demo!("foo");
demo!(r#"foo"#);
demo!(#"foo"#);
demo!(###)
}

2024 에디션 이전에는, 이것은 다음을 생성합니다:

one token
one token
three tokens
three tokens

2024 에디션부터는, #"foo"# 라인과 ### 라인은 해당 형식이 이제 예약되었기 때문에 컴파일 오류를 발생시킵니다.

마이그레이션

rust_2024_guarded_string_incompatible_syntax 린트(lint)는 예약된 구문과 일치하는 모든 토큰을 식별하고, 토큰이 분리되어 파싱되도록 필요한 곳에 공백을 삽입하는 수정 제안을 할 것입니다.

이 린트는 자동 에디션 마이그레이션에 포함된 rust-2024-compatibility 린트 그룹의 일부입니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

또는, 토큰을 업데이트해야 할 수도 있는 매크로 호출을 찾기 위해 수동으로 린트를 활성화할 수 있습니다:

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(rust_2024_guarded_string_incompatible_syntax)]
}

표준 라이브러리

다음 장들에서는 2024 에디션에서의 표준 라이브러리 변경 사항을 자세히 설명합니다.

프렐루드 변경사항

요약

  • Future 와 IntoFuture 트레이트가 이제 프렐류드(prelude)의 일부가 되었습니다.
  • 이로 인해 트레이트 메서드 호출이 모호해져 일부 코드가 컴파일되지 않을 수 있습니다.

세부 정보

표준 라이브러리의 프렐류드 는 모든 모듈에 자동으로 임포트되는 모든 것을 포함하는 모듈입니다. 여기에는 Option, Vec, drop, Clone 과 같이 자주 사용되는 항목들이 포함되어 있습니다.

Rust 컴파일러는 프렐루드에 추가된 항목이 기존 코드를 깨뜨리지 않도록 하기 위해 프렐루드의 항목보다 수동으로 가져온 항목을 우선시합니다. 예를 들어, pub struct Option; 을 포함하는 example 이라는 크레이트나 모듈이 있는 경우 use example::*; 은 Option 이 표준 라이브러리의 것이 아니라 example 의 것을 명확하게 참조하도록 합니다.

하지만 프렐류드에 트레이트 를 추가하는 것은 기존 코드를 미묘한 방식으로 깨뜨릴 수 있습니다. 예를 들어, MyPoller 트레이트에서 온 x.poll() 호출은 std 의 Future 또한 임포트된 경우 컴파일에 실패할 수 있는데, 이는 poll 호출이 이제 모호해져서 어느 트레이트에서 온 것인지 알 수 없게 되기 때문입니다.

해결책으로, Rust 2024는 새로운 프렐류드를 사용할 것입니다. 다음 변경 사항을 제외하고는 현재의 것과 동일합니다:

  • 추가됨:
    • std::future::Future
    • std::future::IntoFuture

마이그레이션

충돌하는 트레이트 메서드

범위에 있는 두 트레이트의 메서드 이름이 같으면 어떤 트레이트 메서드를 사용해야 할지 모호합니다. 예를 들어:

trait MyPoller {
    // 이 이름은 `std` 의 `Future` 트레이트에 있는 `poll` 메서드와 동일합니다.
    fn poll(&self) {
        println!("폴링");
    }
}

impl<T> MyPoller for T {}

fn main() {
    // Pin<&mut async {}>는 `std::future::Future` 와 `MyPoller` 를 모두 구현합니다.
    // 두 트레이트 모두 스코프에 있다면 (Rust 2024의 경우가 그렇듯이),
    // 어떤 `poll` 메서드를 호출할지 모호해집니다
    core::pin::pin!(async {}).poll();
}

완전한 경로 문법(fully qualified syntax)을 사용하여 모든 에디션에서 작동하도록 수정할 수 있습니다:

fn main() {
    // 이제 어떤 트레이트 메서드를 참조하는지 명확합니다
    <_ as MyPoller>::poll(&core::pin::pin!(async {}));
}

rust_2024_prelude_collisions 린트는 모호한 메서드 호출을 완전한 경로 문법을 사용하도록 자동으로 수정합니다. 이 린트는 rust-2024-compatibility 린트 그룹의 일부이며, cargo fix --edition 을 실행할 때 자동으로 적용됩니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

또는, 이러한 한정자(qualifications)를 추가해야 하는 곳을 찾기 위해 수동으로 린트를 활성화할 수 있습니다:

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(rust_2024_prelude_collisions)]
}

Box<[T]> 에 대한 IntoIterator 추가

요약

  • 박싱된 슬라이스(Boxed slices)는 모든 에디션에서 IntoIterator를 구현합니다.
  • 메서드 호출 문법(예: boxed_slice.into_iter())을 사용할 때, 2024년 이전 에디션에서는 IntoIterator::into_iter 호출이 _가려집니다 _. 따라서 boxed_slice.into_iter() 는 이전과 같이 여전히 (&(*boxed_slice)).into_iter() 로 해석됩니다.
  • Rust 2024에서 boxed_slice.into_iter() 는 IntoIterator::into_iter를 호출하는 것으로 의미가 변경됩니다.

세부 정보

Rust 1.80 이전까지는, 박싱된 슬라이스에 대해 IntoIterator 가 구현되지 않았습니다. 이전 버전에서는 박싱된 슬라이스에 대해 .into_iter() 를 호출하면, 메서드 호출이 자동으로 Box<[T]> 에서 &[T] 로 역참조하고, &T 참조를 생성하는 반복자를 반환했습니다. 예를 들어, 다음 코드는 이전 버전에서 작동했습니다:

#![allow(unused)]
fn main() {
// 이전 에디션에서의 동작 예시입니다.
let my_boxed_slice: Box<[u32]> = vec![1, 2, 3].into_boxed_slice();
// 참고: .into_iter()는 1.80 이전 버전에서 필수였습니다
for x in my_boxed_slice.into_iter() {
    // 2024 이전 에디션에서 x의 타입은 &u32입니다
}
}

Rust 1.80에서는 박싱된 슬라이스에 대한 IntoIterator 구현이 추가되었습니다. 이를 통해 참조 대신 값으로 슬라이스의 요소를 반복할 수 있습니다:

#![allow(unused)]
fn main() {
// 1.80부터 새로 추가됨, 모든 에디션 적용
let my_boxed_slice: Box<[u32]> = vec![1, 2, 3].into_boxed_slice();
for x in my_boxed_slice { // .into_iter()를 호출할 필요가 없음을 주목하세요
    // x의 타입은 u32입니다
}
}

이 예제는 모든 에디션에서 허용되는데, 이전에는 for 루프가 .into_iter() 메서드 호출처럼 자동으로 역참조하지 않았기 때문에 오류였던 코드입니다.

하지만 박싱된 슬라이스에 대해 수동으로 .into_iter() 를 호출하던 기존 코드가 참조에 대한 반복자에서 값에 대한 반복자로 변경되므로, 이는 일반적으로 호환성을 깨는 변경(breaking change)이 될 것입니다. 이 문제를 해결하기 위해, 박싱된 슬라이스에서의 .into_iter() 메서드 호출은 에디션에 따라 다르게 동작합니다. 2024 이전 에디션에서는 계속해서 참조에 대한 반복자를 반환하고, 2024 에디션부터는 값에 대한 반복자를 반환합니다.

#![allow(unused)]
fn main() {
// 2024 에디션에서 변경된 동작 예시
let my_boxed_slice: Box<[u32]> = vec![1, 2, 3].into_boxed_slice();
// 여전히 수동으로 .into_iter()를 호출하는 오래된 코드 예시
for x in my_boxed_slice.into_iter() {
    // 2024 에디션에서 x는 이제 u32 타입입니다
}
}

마이그레이션

boxed_slice_into_iter 린트는 참조를 반환하는 기존 동작을 유지하기 위해 박싱된 슬라이스에 대한 .into_iter() 호출을 .iter() 호출로 자동으로 수정합니다. 이 린트는 rust-2024-compatibility 린트 그룹의 일부이며, cargo fix --edition 을 실행할 때 자동으로 적용됩니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

예를 들어, 이것은 다음과 같이 변경됩니다:

fn main() {
    let my_boxed_slice: Box<[u32]> = vec![1, 2, 3].into_boxed_slice();
    for x in my_boxed_slice.into_iter() {
        // x의 타입은 &u32입니다
    }
}

변경 후:

fn main() {
    let my_boxed_slice: Box<[u32]> = vec![1, 2, 3].into_boxed_slice();
    for x in my_boxed_slice.iter() {
        // x의 타입은 &u32입니다
    }
}

boxed_slice_into_iter 린트는 모든 에디션에서 경고하도록 기본 설정되어 있으므로, 수동으로 린트를 끄지 않았다면 마이그레이션 전에 이미 확인했을 것입니다.

Unsafe 함수

요약

  • 다음 함수들은 이제 unsafe로 표시됩니다:
    • std::env::set_var
    • std::env::remove_var
    • std::os::unix::process::CommandExt::before_exec

세부 정보

시간이 지남에 따라 표준 라이브러리의 특정 함수들은 unsafe 로 표시되었어야 함이 분명해졌습니다. 하지만 함수에 unsafe 를 추가하는 것은 기존 코드를 unsafe 블록 안에 넣어야 하기 때문에 호환성을 깨는 변경이 될 수 있습니다. 이러한 호환성 문제를 피하기 위해, 이 함수들은 이전 에디션에서는 unsafe 를 요구하지 않지만, 2024 에디션부터는 unsafe 로 표시됩니다.

std::env::{set_var, remove_var}

It can be unsound to call std::env::set_var or std::env::remove_var in a multithreaded program due to safety limitations of the way the process environment is handled on some platforms. The standard library originally defined these as safe functions, but it was later determined that was not correct.

다른 스레드가 실행 중일 때는 이 함수들이 호출되지 않도록 하는 것이 중요합니다. 더 자세한 내용은 해당 함수 문서의 Safety 섹션을 참조하세요.

std::os::unix::process::CommandExt::before_exec

std::os::unix::process::CommandExt::before_exec 함수는 exec 호출 전에 클로저를 실행하는 방법을 제공하는 유닉스 전용 함수입니다. 이 함수는 1.37 릴리스에서 사용 중단(deprecated)되었으며, 동일한 작업을 수행하지만 unsafe 로 표시된 pre_exec로 대체되었습니다.

before_exec 이 사용 중단되었음에도 불구하고, 2024 에디션부터는 unsafe 로 올바르게 표시됩니다. 이는 아직 pre_exec 으로 마이그레이션하지 않은 레거시 코드가 unsafe 블록을 필요로 하도록 보장하는 데 도움이 될 것입니다.

before_exec 클로저가 충족해야 하는 매우 엄격한 안전 요구 사항이 있습니다. 자세한 내용은 Safety 섹션을 참조하세요.

마이그레이션

코드가 2021 에디션과 2024 에디션 모두에서 컴파일되게 하려면, 이 함수들이 unsafe 블록 내에서만 호출되도록 해야 합니다.

⚠ 주의: 이 함수들에 대한 호출을 수동으로 검사하고, 해당 함수의 전제 조건을 충족하도록 코드를 다시 작성하는 것이 중요합니다. 특히, 여러 스레드가 실행 중일 수 있는 경우 set_var 와 remove_var 를 호출해서는 안 됩니다. 사용 사례를 관리하기 위해 환경 변수 이외의 다른 메커니즘을 사용하도록 선택해야 할 수도 있습니다.

deprecated_safe_2024 린트는 두 에디션 모두에서 컴파일될 수 있도록, 이러한 함수 사용을 unsafe 블록으로 자동으로 감싸서 수정합니다. 이 린트는 rust-2024-compatibility 린트 그룹의 일부이며, cargo fix --edition 을 실행할 때 자동으로 적용됩니다. 코드를 Rust 2024 에디션과 호환되도록 마이그레이션하려면 다음을 실행하세요:

cargo fix --edition

예를 들어, 이것은 다음과 같이 변경됩니다:

fn main() {
    std::env::set_var("FOO", "123");
}

변경 후:

fn main() {
    // TODO: 환경 변수 접근이 단일 스레드 코드에서만 발생하는지 감사(Audit)해야 합니다.
    unsafe { std::env::set_var("FOO", "123") };
}

단, 이 자동 마이그레이션은 이 함수들이 올바르게 사용되고 있는지 검증할 수 없다는 점에 유의하세요. 사용법을 수동으로 검토하는 것은 여전히 여러분의 책임입니다.

또는, 이 함수들이 호출되는 곳을 찾기 위해 수동으로 린트를 활성화할 수 있습니다:

#![allow(unused)]
fn main() {
// 수동 마이그레이션을 수행하려면 크레이트 루트에 이것을 추가하세요.
#![warn(deprecated_safe_2024)]
}

Cargo

다음 장들에서는 2024 에디션에서의 Cargo 변경 사항을 자세히 설명합니다.

Cargo: Rust 버전 인식 해결자

요약

  • Cargo.toml 의 edition = "2024" 는 resolver = "3" 를 암시하며, 이는 Rust 버전을 인식하는 의존성 리졸버(resolver)를 활성화합니다.

세부 정보

Rust 1.84.0부터 Cargo는 의존성 버전을 선택할 때 package.rust-version과의 호환성을 고려하기 위한 선택적 지원(opt-in support)을 가지고 있으며, 이는 .cargo/config.toml 에서 resolver.incompatible-rust-version = "fallback"으로 설정하여 활성화할 수 있습니다.

Rust 2024부터는 이것이 기본값이 됩니다. 즉, Cargo.toml 에 edition = "2024"를 작성하면 resolver = "3" 가 암시되고, 이는 다시 resolver.incompatible-rust-version = "fallback"을 암시하게 됩니다.

The resolver is a global setting for a workspace, and the setting is ignored in dependencies. The setting is only honored for the top-level package of the workspace. If you are using a virtual workspace, you will still need to explicitly set the resolver field in the [workspace] definition if you want to opt in to the new resolver.

Rust 버전을 인식하는 의존성 해결(resolution) 방식에 대한 자세한 내용은 Cargo 책을 참조하세요.

마이그레이션

새로운 리졸버(resolver)로 업데이트하기 위한 자동 마이그레이션 도구는 없습니다.

의존성 버그를 가능한 한 빨리 발견하기 위해, 프로젝트에서 CI에서 최신 의존성 검증을 수행할 것을 권장합니다.

Cargo: 테이블 및 키 이름 일관성

요약

  • 이전에 같은 것을 지정하는 두 가지 방법이 있었던 Cargo.toml 의 여러 테이블 및 키 이름이 제거되었습니다.
    • [project] 제거됨; 대신 [package] 를 사용하세요.
    • default_features 제거됨; 대신 default-features 를 사용하세요.
    • crate_type 제거됨; 대신 crate-type 을 사용하세요.
    • proc_macro 제거됨; 대신 proc-macro 를 사용하세요.
    • dev_dependencies 제거됨; 대신 dev-dependencies 를 사용하세요.
    • build_dependencies 제거됨; 대신 build-dependencies 를 사용하세요.

세부 정보

2024 에디션에서는 여러 테이블 및 키 이름이 더 이상 허용되지 않습니다. 이전에는 이러한 테이블이나 키를 지정하는 두 가지 방법이 있었지만, 이제는 이를 지정하는 방법이 단 하나만 있도록 보장합니다.

일부는 시간이 지남에 따라 결정이 변경되었기 때문이고, 일부는 의도치 않은 구현 결과였습니다. 혼란을 피하고 이러한 테이블과 키를 지정하는 단일 스타일을 강제하기 위해, 이제는 하나의 변형(variant)만 허용됩니다.

예를 들어:

[dev_dependencies]
rand = { version = "0.8.5", default_features = false }

다음과 같이 변경되어야 합니다:

[dev-dependencies]
rand = { version = "0.8.5", default-features = false }

dev_dependencies 와 default_features 에서 밑줄(underscores)이 대시(dashes)로 변경된 것에 주목하세요.

마이그레이션

cargo fix --edition 을 사용할 때, Cargo는 Cargo.toml 파일을 자동으로 업데이트하여 선호되는 테이블 및 키 이름을 사용하도록 합니다.

If you prefer to update your Cargo.toml manually, be sure to go through the list above and make sure only the new forms are used.

Cargo: 사용되지 않는 상속된 기본 기능 거부

요약

  • 워크스페이스 의존성이 default-features = true 를 지정하거나 (default-features를 지정하지 않은 경우) 상속된 워크스페이스 의존성에서 default-features = false를 사용하는 것은 더 이상 허용되지 않습니다.

세부 정보

워크스페이스 상속(Workspace inheritance)을 사용하면 의존성을 한 곳(워크스페이스)에 지정한 다음, 패키지 내에서 해당 워크스페이스 의존성을 참조할 수 있습니다. default-features 지정 방식에 의도치 않은 상호작용이 있었으며, 이는 2024 에디션에서 더 이상 허용되지 않습니다.

워크스페이스가 default-features = false 를 지정하지 않는 한, 상속된 패키지 의존성에서 default-features = false 를 지정하는 것은 더 이상 허용되지 않습니다. 예를 들어, 다음과 같이 지정된 워크스페이스가 있다고 가정해 보겠습니다:

[workspace.dependencies]
regex = "1.10.4"

다음은 이제 오류입니다:

[package]
name = "foo"
version = "1.0.0"
edition = "2024"

[dependencies]
regex = { workspace = true, default-features = false }  # ERROR

이 변경의 이유는 기본 기능(default features)이 이미 활성화된 상태에서 default-features = false를 지정할 때 발생하는 혼란을 피하기 위해서입니다. 왜냐하면 이는 아무런 효과가 없기 때문입니다.

의존성이 해당 의존성의 기본 기능을 활성화할지 여부를 결정하는 유연성을 원한다면, 워크스페이스 정의에서 default-features = false 로 설정해야 합니다. 단, 여러 워크스페이스 멤버를 동시에 빌드하는 경우 기능(feature)들이 통합되므로, 한 멤버가 default-features = true 로 설정하면(명시적으로 설정하지 않은 경우 이것이 기본값임) 해당 의존성을 사용하는 모든 멤버에 대해 기본 기능이 활성화된다는 점에 유의하세요.

마이그레이션

cargo fix --edition 을 사용할 때, Cargo는 Cargo.toml 파일에서 이러한 상황의 default-features = false 를 자동으로 제거합니다.

If you prefer to update your Cargo.toml manually, check for any warnings when running a build and remove the corresponding entries. Previous editions should display something like:

warning: /home/project/Cargo.toml: `default-features` is ignored for regex,
since `default-features` was not specified for `workspace.dependencies.regex`,
this could become a hard error in the future

Rustdoc

다음 장들에서는 2024 에디션에서의 Rustdoc 변경 사항을 자세히 설명합니다.

Rustdoc 통합 테스트

요약

  • 문서 테스트(Doctests)가 이제 단일 바이너리로 통합되어 상당한 성능 향상을 가져올 것입니다.

세부 정보

Prior to the 2024 Edition, rustdoc's "test" mode would compile each code block in your documentation as a separate executable. Although this was relatively simple to implement, it resulted in a significant performance burden when there were a large number of documentation tests. Starting with the 2024 Edition, rustdoc will attempt to combine documentation tests into a single binary, significantly reducing the overhead for compiling doctests.

#![allow(unused)]
fn main() {
/// 두 숫자를 더합니다
///
/// ```
/// assert_eq!(add(1, 1), 2);
/// ```
pub fn add(left: u64, right: u64) -> u64 {
    left + right
}

/// 두 숫자를 뺍니다
///
/// ```
/// assert_eq!(subtract(2, 1), 1);
/// ```
pub fn subtract(left: u64, right: u64) -> u64 {
    left - right
}
}

이 예제에서는 두 개의 doctest가 이제 단일 실행 파일로 컴파일됩니다. Rustdoc은 본질적으로 각 예제를 단일 바이너리 내의 별도 함수에 배치합니다. 테스트는 이전과 마찬가지로 독립적인 프로세스에서 계속 실행되므로 전역 상태(전역 정적과 같은)는 여전히 올바르게 작동해야 합니다.1

이 변경 사항은 결합된 실행 파일에서 작동하지 않을 수 있는 기존 doctest와의 잠재적인 비호환성을 피하기 위해 2024 에디션에서만 사용할 수 있습니다. 그러나 이러한 비호환성은 극히 드물 것으로 예상됩니다.

1

이 작업이 어떻게 수행되는지에 대한 자세한 내용은 "Doctests - How were they improved?"를 참조하세요.

standalone_crate 태그

어떤 상황에서는 rustdoc이 예제를 단일 실행 파일로 결합할 수 없습니다. rustdoc은 이것이 불가능한지 자동으로 감지하려고 시도합니다. 예를 들어, 테스트는 다음과 같은 경우 다른 테스트와 결합되지 않습니다.

  • compile_fail 태그를 사용합니다. 이 태그는 예제가 컴파일에 실패해야 함을 나타냅니다.
  • 예제의 에디션을 나타내는 edition 태그를 사용합니다.2
  • 다른 테스트와 잠재적으로 충돌할 수 있는 global_allocator 속성과 같은 전역 속성을 사용합니다.
  • 크레이트 전체 속성(예: #![feature(...)])을 정의합니다.
  • $crate 경로가 올바르게 작동하지 않으므로 $crate 를 사용하는 매크로를 정의합니다.

그러나 rustdoc은 예제를 다른 예제와 결합할 수 없는 모든 상황을 자동으로 결정할 수 없습니다. 이러한 상황에서는 standalone_crate 언어 태그를 추가하여 예제가 별도의 실행 파일로 빌드되어야 함을 나타낼 수 있습니다. 예를 들어:

#![allow(unused)]
fn main() {
//! ```
//! let location = std::panic::Location::caller();
//! assert_eq!(location.line(), 5);
//! ```
}

이것은 예제가 컴파일되는 코드 구조에 민감하며, doctest가 결합되는 방식에 따라 줄 번호가 변경되므로 "결합된" 접근 방식으로는 작동하지 않습니다. 이러한 상황에서는 standalone_crate 태그를 추가하여 이전 에디션과 마찬가지로 예제가 별도로 빌드되도록 강제할 수 있습니다. 예:

#![allow(unused)]
fn main() {
//! ```standalone_crate
//! let location = std::panic::Location::caller();
//! assert_eq!(location.line(), 5);
//! ```
}
2

rustdoc은 전체 크레이트가 2024 에디션 이상인 경우에만 테스트를 결합합니다. 이전 에디션에서 edition2024 태그를 사용해도 해당 테스트가 결합되지 않습니다.

마이그레이션

standalone_crate 태그로 주석을 달아야 하는 doctest를 결정하는 자동 마이그레이션은 없습니다. 마이그레이션할 때 주어진 doctest가 올바르게 작동하지 않을 가능성은 매우 낮습니다. 크레이트를 2024 에디션으로 업데이트한 다음 문서 테스트를 실행하여 실패하는 것이 있는지 확인하는 것이 좋습니다. 실패하는 것이 있다면 결합된 접근 방식과 호환되도록 다시 작성할 수 있는지 분석하거나, 또는 이전 동작을 유지하기 위해 standalone_crate 태그를 추가해야 합니다.

주의하고 피해야 할 몇 가지 사항은 다음과 같습니다.

  • std::panic::Location 의 값 또는 Location 을 사용하는 것들을 확인합니다. 여러 테스트가 이제 동일한 테스트 크레이트에 위치하므로 코드의 위치가 달라졌습니다.
  • 이제 다른 모듈 경로를 갖는 std::any::type_name 의 값을 확인합니다.

Rustdoc 중첩 include! 변경

요약

include_str! 로 doctest가 포함될 때, 해당 doctest 자체도 include!, include_str!, 또는 include_bytes! 를 사용하는 경우 경로는 Rust 소스 파일이 아닌 Markdown 파일에 상대적으로 해결됩니다.

세부 정보

2024 에디션 이전에는 #[doc=include_str!("path/file.md")] 로 문서를 추가할 때 해당 파일의 doctest로 스팬(span) 정보가 전달되지 않았습니다. 결과적으로 Markdown 파일이 소스와 다른 디렉터리에 있는 경우, 포함된 경로는 소스 파일을 기준으로 지정해야 했습니다.

예를 들어, 다음과 같은 파일이 있는 라이브러리 크레이트를 고려해 보십시오.

  • Cargo.toml
  • README.md
  • src/
    • lib.rs
  • examples/
    • data.bin

lib.rs 에 다음이 포함되어 있다고 가정해 보겠습니다.

#![doc=include_str!("../README.md")]

그리고 이 README.md 파일을 가정해 보겠습니다.

```
let _ = include_bytes!("../examples/data.bin");
//                      ^^^ notice this
```

2024 에디션 이전에는 README.md 의 경로가 lib.rs 파일에 상대적이어야 했습니다. 2024년 이후에는 README.md 자체에 상대적이므로 README.md 를 다음과 같이 업데이트합니다.

```
let _ = include_bytes!("examples/data.bin");
```

마이그레이션

영향을 받는 doctest의 경로를 변환하는 자동 마이그레이션은 없습니다. doctest 중 하나가 영향을 받는 경우 새 에디션으로 마이그레이션한 후 테스트를 빌드할 때 다음과 같은 오류가 표시됩니다.

error: couldn't read `../examples/data.bin`: No such file or directory (os error 2)
 --> src/../README.md:2:24
  |
2 | let _ = include_bytes!("../examples/data.bin");
  |         ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  = note: this error originates in the macro `include_bytes` (in Nightly builds, run with -Z macro-backtrace for more info)
help: there is a file with the same name in a different directory
  |
2 | let _ = include_bytes!("examples/data.bin");
  |                        ~~~~~~~~~~~~~~~~~~~

doctest를 Rust 2024로 마이그레이션하려면 영향을 받는 모든 경로를 doctest를 포함하는 파일에 상대적으로 업데이트하십시오.

Rustfmt: 스타일 에디션

요약

이제 사용자는 rustfmt 와 함께 사용할 스타일 에디션을 제어할 수 있습니다.

세부 정보

Rustfmt가 생성하는 기본 형식은 Rust 스타일 가이드 의 규칙에 따라 결정됩니다.

또한 Rustfmt는 Rust 툴체인을 업데이트할 때 사용자에게 시끄러운 형식 변경을 유발하지 않도록 하는 형식 안정성 보장을 제공합니다. 이 안정성 보장은 본질적으로 최신 버전의 Rustfmt가 이전 버전의 Rustfmt가 생성한 성공적으로 형식화된 출력을 수정할 수 없음을 의미합니다.

이 두 가지 제약 조건의 조합은 역사적으로 스타일 가이드와 Rustfmt의 기본 형식 지정 동작을 모두 고정시켰습니다. 이러한 교착 상태는 스타일 개선을 반복할 수 없게 하고, Rustfmt가 오래 전에 폐기된 레거시 형식 지정 특성(예: 중첩 튜플 액세스)을 유지하도록 요구하는 등 다양한 문제를 야기했습니다.

RFC 3338 은 Rust 스타일 가이드가 에디션 전반에 걸쳐 발전할 수 있도록 Rust의 에디션 모델에 맞춰 정렬되는 메커니즘을 구축하여 이 교착 상태를 해결했으며, rustfmt 는 사용자가 원하는 스타일 가이드 에디션(스타일 에디션이라고 함)을 지정할 수 있도록 했습니다.

2024 에디션에서 rustfmt 는 이제 사용자가 형식 지정에 사용되는 스타일 에디션을 제어할 수 있는 기능을 지원합니다. 2024 에디션의 스타일 가이드에는 이 에디션 가이드의 다른 곳에 자세히 설명된 스타일 가이드 개선 사항도 포함되어 있습니다.

기본적으로 rustfmt 는 구문 분석에 사용되는 표준 Rust 에디션과 동일한 스타일 에디션을 사용하지만, 스타일 에디션은 재정의하고 별도로 구성할 수도 있습니다.

2024 스타일 에디션으로 rustfmt 를 실행하는 방법은 여러 가지가 있습니다.

edition 이 2024 로 설정된 Cargo.toml 파일로 다음을 실행하십시오.

cargo fmt

또는 rustfmt 를 에디션 2024 로 직접 실행하여 구문 분석과 스타일 가이드의 2024 에디션 모두에 2024 에디션을 사용하십시오.

rustfmt lib.rs --edition 2024

The style edition can also be set in a rustfmt.toml or .rustfmt.toml configuration file:

style_edition = "2024"

그런 다음 rustfmt 를 직접 실행할 때 사용됩니다.

rustfmt lib.rs

또는 rustfmt 옵션에서 직접 스타일 에디션을 지정할 수 있습니다.

rustfmt lib.rs --style-edition 2024

마이그레이션

2024 에디션 또는 스타일 에디션으로 cargo fmt 또는 rustfmt 를 실행하면 형식이 2024 스타일 에디션 형식으로 자동 마이그레이션됩니다.

Projects who have contributors that may utilize their editor's format-on-save features are also strongly encouraged to add a rustfmt.toml file to their project that includes the corresponding style_edition utilized within their project, or to encourage their users to ensure their local editor format-on-save feature is configured to use that same style_edition.

이는 편집기의 저장 시 형식 지정 출력이 개발자가 수동으로 cargo fmt 를 실행하거나 프로젝트의 CI 프로세스(많은 편집기는 기본적으로 2015 에디션을 사용하는 rustfmt 를 직접 실행하는 반면, cargo fmt 는 Cargo.toml 파일에 지정된 에디션을 사용함)의 출력과 일치하도록 보장하기 위함입니다.

2024 에디션에서 rustfmt 는 이제 다음을 생성합니다:

use std::io::{self, Read, Write, stdout};
use std::num::{NonZeroU8, NonZeroU16, NonZeroU32, NonZeroU64};
