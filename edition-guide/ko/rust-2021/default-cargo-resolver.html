The Rust Edition Guide

기본 Cargo 기능 해결자

요약

  • edition = "2021" 은 Cargo.toml 에서 resolver = "2" 를 의미합니다.

세부 정보

Rust 1.51.0부터 Cargo는 Cargo.toml 에서 resolver = "2" 로 활성화할 수 있는 새로운 기능 해석기 에 대한 옵트인 지원을 제공합니다.

Rust 2021부터 이것이 기본값이 됩니다. 즉, Cargo.toml 에 edition = "2021" 을 작성하면 resolver = "2" 를 의미합니다.

해석기는 작업 공간 에 대한 전역 설정이며 종속성에서는 설정이 무시됩니다. 이 설정은 작업 공간의 최상위 패키지에만 적용됩니다. 가상 작업 공간 을 사용하는 경우 새 해석기를 옵트인하려면 [workspace] 정의에서 resolver 필드 를 명시적으로 설정해야 합니다.

새로운 기능 해석기는 더 이상 여러 방식으로 종속된 크레이트에 대해 요청된 모든 기능을 병합하지 않습니다. 자세한 내용은 Rust 1.51 발표 를 참조하십시오.

마이그레이션

새 해석기로 업데이트하기 위한 자동화된 마이그레이션 도구는 없습니다. 대부분의 프로젝트에서는 업데이트로 인해 변경 사항이 거의 또는 전혀 없습니다.

cargo fix --edition 으로 업데이트할 때 새 해석기가 다른 기능으로 종속성을 빌드하는 경우 Cargo는 보고서를 표시합니다. 다음과 같이 보일 수 있습니다.

참고: 2021 에디션으로 전환하면 Cargo에서 버전 2 기능 해석기를 사용할 수 있습니다. 이로 인해 일부 종속성이 이전보다 더 적은 기능으로 빌드될 수 있습니다. 해석기 변경에 대한 자세한 내용은 https://doc.rust-lang.org/nightly/edition-guide/rust-2021/default-cargo-resolver.html에서 확인할 수 있습니다.
다음 종속성을 빌드할 때 주어진 기능은 더 이상 사용되지 않습니다.

  bstr v0.2.16: default, lazy_static, regex-automata, unicode
  libz-sys v1.1.3 (as host dependency): libc

이를 통해 특정 종속성이 더 이상 주어진 기능으로 빌드되지 않음을 알 수 있습니다.

빌드 실패

변경 후 프로젝트가 올바르게 빌드되지 않는 경우가 있을 수 있습니다. 한 패키지의 종속성 선언이 다른 패키지에서 특정 기능이 활성화되어 있다고 가정하고 해당 기능이 이제 비활성화된 경우 컴파일에 실패할 수 있습니다.

예를 들어, 다음과 같은 종속성이 있다고 가정해 보겠습니다.

# Cargo.toml

[dependencies]
bstr = { version = "0.2.16", default-features = false }
# ...

그리고 종속성 트리의 어딘가에 다른 패키지에 다음이 있습니다.

# Another package's Cargo.toml

[build-dependencies]
bstr = "0.2.16"

우리 패키지에서는 bstr 의 words_with_breaks 메서드를 사용하고 있는데, 이를 위해서는 bstr 의 "unicode" 기능을 활성화해야 합니다. 이는 과거에는 Cargo가 두 패키지 간에 bstr 의 기능을 통합했기 때문에 작동했습니다. 하지만 Rust 2021로 업데이트한 후에는, 새로운 리졸버가 bstr 을 두 번 빌드하는데, 한 번은 기본 기능으로(빌드 의존성으로서), 다른 한 번은 기능 없이(일반 의존성으로서) 빌드합니다. bstr 이 이제 "unicode" 기능 없이 빌드되기 때문에 words_with_breaks 메서드가 존재하지 않으며, 메서드가 없다는 오류와 함께 빌드가 실패합니다.

해결책은 의존성이 실제로 사용하는 기능과 함께 선언되도록 하는 것입니다. 예를 들면:

[dependencies]
bstr = { version = "0.2.16", default-features = false, features = ["unicode"] }

어떤 경우에는, 직접 제어할 수 없는 서드파티 의존성에서 문제가 발생할 수 있습니다. 해당 프로젝트에 패치를 제출하여 문제가 되는 의존성에 대해 올바른 기능 세트를 선언하도록 시도해 볼 수 있습니다. 또는, 자신의 Cargo.toml 파일 내에서 어떤 의존성에든 기능을 추가할 수 있습니다. 예를 들어, 위에 제시된 bstr 예제가 어떤 서드파티 의존성에 선언되어 있다면, 올바른 의존성 선언을 자신의 프로젝트에 복사하기만 하면 됩니다. 새로운 리졸버의 통합 규칙과 일치하는 한 기능들은 통합될 것입니다. 그 규칙들은 다음과 같습니다:

  • 현재 빌드되지 않는 타겟에 대한 플랫폼별 의존성에서 활성화된 기능은 무시됩니다.
  • 빌드 의존성과 절차적 매크로(proc-macros)는 일반 의존성과 기능을 공유하지 않습니다.
  • 개발 의존성(Dev-dependencies)은 이를 필요로 하는 타겟(테스트나 예제 등)을 빌드하지 않는 한 기능을 활성화하지 않습니다.

실제 예로 diesel 과 diesel_migrations 를 사용하는 경우가 있습니다. 이 패키지들은 데이터베이스 지원을 제공하며, 다음과 같이 기능을 사용하여 데이터베이스를 선택합니다:

[dependencies]
diesel = { version = "1.4.7", features = ["postgres"] }
diesel_migrations = "1.4.0"

문제는 diesel_migrations 가 내부적으로 diesel 에 의존하는 절차적 매크로를 가지고 있는데, 이 절차적 매크로는 자신의 diesel 사본이 나머지 의존성 그래프와 동일한 기능을 활성화하고 있다고 가정한다는 점입니다. 새로운 리졸버로 업데이트한 후에는 diesel 의 사본이 두 개가 되고, 절차적 매크로를 위해 빌드된 사본에는 "postgres" 기능이 빠져 있기 때문에 빌드에 실패하게 됩니다.

해결책은 diesel 을 필요한 기능과 함께 빌드 의존성으로 추가하는 것입니다. 예를 들면:

[build-dependencies]
diesel = { version = "1.4.7", features = ["postgres"] }

이렇게 하면 Cargo가 "postgres"를 호스트 의존성(절차적 매크로 및 빌드 의존성)을 위한 기능으로 추가하게 됩니다. 이제 diesel_migrations 절차적 매크로는 "postgres" 기능이 활성화된 상태로 올바르게 빌드될 것입니다.

diesel 2.0 릴리스(현재 개발 중)는 이 종속성 요구 사항이 없도록 구조가 변경되었으므로 이 문제가 발생하지 않습니다.

기능 탐색

cargo tree 명령은 새로운 리졸버로의 이전을 돕기 위해 상당히 개선되었습니다. cargo tree 를 사용하여 의존성 그래프를 탐색하고, 어떤 기능이 활성화되었는지, 그리고 중요하게는 왜 활성화되었는지 확인할 수 있습니다.

한 가지 방법은 --duplicates 플래그(줄여서 -d)를 사용하는 것입니다. 이 플래그는 패키지가 여러 번 빌드될 때 이를 알려줍니다. 앞서 본 bstr 예제를 보면 다음과 같습니다:

> cargo tree -d
bstr v0.2.16
└── foo v0.1.0 (/MyProjects/foo)

bstr v0.2.16
[build-dependencies]
└── bar v0.1.0
    └── foo v0.1.0 (/MyProjects/foo)

이 출력은 bstr 이 두 번 빌드되었음을 알려주며, 두 경우 모두 포함되게 된 의존성 체인을 보여줍니다.

-f 플래그를 사용하여 각 패키지가 어떤 기능을 사용하고 있는지 다음과 같이 출력할 수 있습니다:

cargo tree -f '{p} {f}'

이는 Cargo에게 출력의 "형식"을 변경하도록 지시하여, 패키지와 활성화된 기능을 모두 출력하게 합니다.

또한 -e 플래그를 사용하여 표시할 "엣지(edges)"를 지정할 수 있습니다. 예를 들어, cargo tree -e features 는 각 의존성 사이에서 어떤 기능이 각 의존성에 의해 추가되고 있는지 보여줍니다. 이 옵션은 트리를 "반전"시키는 데 사용할 수 있는 -i 플래그와 함께 사용할 때 더 유용합니다. 이를 통해 기능이 특정 의존성으로 어떻게 흘러 들어가는지 볼 수 있습니다. 예를 들어, 의존성 그래프가 크고 누가 bstr 에 의존하고 있는지 확실하지 않은 경우, 다음 명령이 이를 보여줍니다:

> cargo tree -e features -i bstr
bstr v0.2.16
├── bstr feature "default"
│   [build-dependencies]
│   └── bar v0.1.0
│       └── bar feature "default"
│           └── foo v0.1.0 (/MyProjects/foo)
├── bstr feature "lazy_static"
│   └── bstr feature "unicode"
│       └── bstr feature "default" (*)
├── bstr feature "regex-automata"
│   └── bstr feature "unicode" (*)
├── bstr feature "std"
│   └── bstr feature "default" (*)
└── bstr feature "unicode" (*)

이 출력 스니펫은 foo 프로젝트가 "default" 기능을 가진 bar 에 의존하고 있음을 보여줍니다. 그리고 bar 는 "default" 기능을 가진 bstr 에 빌드 의존성으로 의존합니다. 더 나아가 bstr 의 "default" 기능이 (다른 기능들 중에서) "unicode"를 활성화한다는 것을 알 수 있습니다.
